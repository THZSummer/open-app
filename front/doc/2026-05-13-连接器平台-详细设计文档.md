# 连接器平台详细设计文档

## 一、文档概述

### 1.1 文档目的

本文档基于现有开放平台的能力,详细规划连接器平台的功能架构、页面设计、交互逻辑和实施优先级。文档旨在明确连接器平台的核心价值、功能定位和落地路径,为后续的开发实施提供指导。

### 1.2 设计背景

现有开放平台具备以下核心能力:

- **卡片能力**:群置顶卡片、群通知、点对点通知、助手广场卡片、链接增强
- **订阅能力**:API订阅、事件订阅、回调管理

这些能力为连接器平台提供了**通知触达**和**事件触发**的基础设施。

### 1.3 平台定位

连接器平台定位为\*\*"能力共享枢纽"\*\*,核心目标:

- 将**API调用**、**事件订阅**、**Webhook回调**作为输入
- 将**卡片能力**作为输出
- 通过**流程编排引擎**实现跨系统的自动化联动

### 1.4 技术方案选择

采用\*\*"可扩展的多步骤串联流程引擎"方案\*\*:

- 支持多个动作串联执行(A → B → C)
- 支持简单条件分支(if-else)
- 支持数据映射与转换
- 预留向完整流程引擎的扩展路径
- 中等开发周期(11-16周)

**架构详细说明:**

**1. 整体架构设计理念**

该架构采用分层设计思想,从上到下共分为六个层级,每层职责明确:

- **外部系统层**:业务系统作为事件和数据来源,包括HR系统(提供员工入职事件)、OA审批系统(提供审批完成事件)、CRM系统(提供客户数据变更事件)、监控系统(提供告警事件)等。这些系统通过事件订阅或Webhook回调与连接器平台交互。

- **现有开放平台**:提供连接器平台依赖的基础能力,包括:
  - **卡片能力平台**:封装了发送群置顶卡片、群通知卡片、点对点通知卡片、助手广场卡片、链接增强等能力
  - **API订阅管理**:管理平台上架的各类API,支持API的注册、审批、下架等生命周期管理
  - **事件订阅管理**:提供事件的发布和订阅能力,支持业务系统发布事件并被流程订阅
  - **回调管理**:管理Webhook回调地址的生成和接收

- **连接器平台核心**:是整个系统的核心处理层,包含五大子层:
  - **接入层**:负责接收外部请求,包括API网关(统一处理HTTP请求)、事件总线(统一事件分发和处理)、触发器管理(管理和配置各类触发器)
  - **核心引擎层**:负责流程的编排和执行,包括流程执行引擎(流程的调度和执行控制)、状态机(管理流程实例的生命周期和状态转换)、执行器(具体执行每个流程节点)
  - **连接器层**:封装各种能力的调用接口,包括卡片连接器(封装所有卡片发送能力)、API连接器(封装内部和第三方API调用)、逻辑连接器(封装条件判断、延迟等待、数据提取等逻辑)、事件连接器(封装事件的发布和订阅)
  - **数据层**:提供数据存储和消息传递能力,MySQL存储流程定义、执行记录等结构化数据,Redis缓存热点数据和会话信息,MQS消息队列保证异步处理和系统解耦
  - **监控运维层**:提供平台的运维支撑能力,包括监控大盘(实时展示平台运行状态和关键指标)、日志系统(记录所有操作和执行日志)、告警系统(异常情况及时告警通知)

**2. 数据流向说明**

**正向处理流程**:外部系统产生事件 → 事件总线接收事件 → 触发器管理器判断触发条件 → 流程执行引擎创建执行实例 → 执行器按顺序执行每个节点 → 连接器调用相应能力 → 开放平台执行实际业务操作。

**反向回调流程**:外部系统发送Webhook回调 → 回调管理系统接收 → 触发器管理器识别回调类型 → 流程执行引擎创建执行实例 → 执行器处理回调数据 → 执行后续流程节点。

**API触发流程**:外部系统调用API → API网关接收请求 → 验证权限和参数 → 流程执行引擎创建执行实例 → 执行器执行流程 → 返回执行结果。

**3. 关键设计要点**

**模块化设计**:各层职责清晰分离,便于独立开发、测试和维护,降低系统复杂度。

**连接器抽象**:通过连接器层封装各种能力,新增能力只需开发新的连接器,无需修改核心引擎。

**事件驱动架构**:采用事件总线设计,实现系统间松耦合,提高系统可扩展性和灵活性。

**异步处理机制**:使用消息队列处理耗时操作,提高系统吞吐量和响应速度。

**可观测性**:配备完整的监控、日志、告警体系,便于问题定位和性能优化。

**扩展性设计**:预留向完整流程引擎的升级路径,支持并行网关、循环执行、子流程等高级特性。

**触发器能力详细说明:**

**1. 四种触发器类型详解**

触发器是流程执行的起始点,连接器平台支持四种触发器类型:

**定时触发**:通过Cron表达式配置定时任务,支持秒、分、时、日、月、周多种配置方式。典型应用场景包括:每日定时汇总数据并发送报表、每周一定时发送周报、每月1号定时发送月报等。定时触发器需要配置执行时间(Cron表达式)和时区,支持配置重试策略和超时时间。

**API触发**:通过暴露REST API接口,允许外部系统通过HTTP请求触发流程执行。API触发支持GET和POST两种请求方法,支持路径参数和查询参数配置。调用方需要提供API Key进行身份验证,平台会记录每次API调用的日志和执行结果。典型应用场景包括:第三方系统回调触发、脚本触发、人工手动触发等。

**事件触发**:订阅特定业务系统发布的事件,当事件发生时自动触发流程执行。事件触发需要预先在事件源配置中订阅感兴趣的事件,并配置事件的过滤条件。平台支持事件的JSON Schema验证,确保接收的事件数据格式正确。典型应用场景包括:HR系统发布入职事件触发入职流程、OA系统发布审批完成事件触发后续处理、CRM系统发布客户变更事件触发数据同步等。

**Webhook触发**:平台为每个流程生成唯一的Webhook URL,外部系统通过POST请求向该URL发送数据即可触发流程。Webhook触发支持签名验证,确保请求来源的可靠性。支持配置请求体的JSON Path映射,提取关键字段作为流程输入参数。

**2. 配置方式对比**

可视化配置适合业务人员使用,通过界面操作即可完成触发器配置,无需编写代码。主要用于定时触发和事件触发的配置。

API接口配置适合开发人员使用,通过调用平台提供的API可以编程方式配置触发器,支持自动化配置和批量操作。主要用于API触发和Webhook触发的配置。

**3. 触发器选择建议**

选择触发器类型需要考虑以下因素:

- 如果流程需要定时执行,选择定时触发
- 如果需要外部系统主动触发,选择API触发或Webhook触发
- 如果需要响应业务系统的事件,选择事件触发
- 如果不确定触发时机,优先选择事件触发,因为最灵活

**连接器能力详细说明:**

**1. 四大连接器类型详解**

连接器是流程执行过程中调用的能力单元,封装了对外部系统的调用逻辑。

**卡片连接器**:封装了开放平台的卡片发送能力,是连接器平台的核心输出能力。包括五种卡片类型:

- **群置顶卡片**:发送到指定群聊并置顶显示,适用于紧急通知、重大公告等需要强提醒的场景。卡片会在群里始终显示在顶部,直到手动取消置顶。典型应用:系统故障紧急告警、公司重要通知发布。
- **群通知卡片**:发送到指定群聊的普通卡片消息,不会置顶。适用于流程执行结果通知、每日数据汇总报告等常规信息推送。典型应用:每日销售数据汇报、审批结果通知。
- **点对点通知**:发送消息给指定用户,是一对一的消息通知。适用于任务分配提醒、个人待办通知等需要个人关注的信息。典型应用:待办任务提醒、审批结果通知。
- **助手广场卡片**:发送交互式卡片,支持按钮点击、选项选择等交互操作。用户可以直接在卡片上完成审批、确认等操作。典型应用:权限申请待审批、投票调查收集。
- **链接增强**:对卡片中的链接进行增强处理,可以追踪链接的点击次数、记录点击用户信息、生成一次性访问令牌等。适用于需要追踪用户行为的场景。

**API连接器**:封装了对HTTP接口的调用能力,支持调用内部和第三方API。包括两种主要能力:

- **调用HTTP接口**:支持GET、POST、PUT、DELETE等HTTP方法,支持配置请求头、请求体、认证信息等。支持响应数据的提取和映射。典型应用:调用HR系统API获取员工信息、调用财务系统API创建报销单。
- **数据转换**:对输入数据进行格式转换、字段映射、计算处理等操作。包括JSON到JSON的转换、XML到JSON的转换、日期格式转换、数值计算等。典型应用:提取事件数据中的关键字段、计算金额的百分比。

**逻辑连接器**:提供流程控制的逻辑能力,不涉及外部系统调用。包括三种组件:

- **条件判断**:支持if-else分支逻辑,根据输入数据的值判断走不同的执行路径。支持比较运算(等于、大于、小于等)、逻辑运算(与、或、非)、正则匹配等条件类型。典型应用:根据审批结果判断是否发送通过或拒绝通知。
- **延迟等待**:暂停流程执行指定时间,支持固定时长延迟和定时延迟。固定时长延迟如等待10分钟后重试,定时延迟如等到某个时间点再继续执行。典型应用:审批超时提醒、失败重试。
- **数据提取**:从输入数据中提取指定字段或计算新的字段,作为后续节点的输入。支持JSON Path提取、正则提取、默认值设置等。典型应用:从事件数据提取客户ID作为后续API调用的参数。

**事件连接器**:提供事件的发布和订阅能力,用于流程之间的联动。包括两种能力:

- **事件输入**:将流程作为事件的消费者,接收其他流程发布的事件数据。用于流程间的解耦,一个流程可以触发多个下游流程。典型应用:主流程完成后发布事件,多个子流程分别订阅执行。
- **事件输出**:将流程作为事件的生产者,向事件总线发布事件数据。用于触发其他系统或流程。典型应用:流程执行完成后发布执行结果事件。

**2. 连接器使用场景推荐**

**卡片连接器使用建议**:优先使用群通知卡片作为常规通知方式,只有在紧急情况下才使用群置顶卡片以避免通知疲劳。点对点通知适用于需要个人确认的信息。助手广场卡片适用于需要用户交互确认的场景,但要注意用户可能不在线导致超时。

**API连接器使用建议**:调用外部API时要设置合理的超时时间和重试策略。对于调用失败的情况,建议记录日志并发送告警,便于问题追踪。数据转换尽量在前端完成,减少后端逻辑复杂度。

**逻辑连接器使用建议**:条件判断的条件表达式要清晰易懂,便于后期维护。延迟等待要根据业务场景设置合理的等待时间,避免等待时间过长影响用户体验。数据提取的字段名要有意义,便于后续节点引用。

**1. 四大功能区域概述**

流程编辑器是连接器平台的核心工作区,采用四区布局设计:

**顶部工具栏**:提供流程编辑的操作入口,位于页面最上方,方便快速操作。包含保存、发布、测试等高频操作按钮。

**节点面板区**:位于页面左侧,展示流程编辑所需的各类节点组件。包含触发器列表、连接器列表、逻辑组件列表。这个区域作为节点类型的参考库,用户在画布连线上添加节点时可以选择这些类型。

**画布编辑区**:占据页面中央大部分区域,是流程设计的主要工作区。用户在这里点击连线上的添加按钮来创建新节点、连接节点、调整节点位置和布局、可视化查看流程结构。与传统拖拽方式不同,节点通过点击连线的添加按钮来创建。

**属性配置区**:位于页面右侧或下方,当用户选中节点或连线时,显示对应的配置选项。用户在此配置节点的参数、数据映射等。

**2. 顶部工具栏功能详解**

**保存草稿**:将当前编辑的流程保存为草稿状态,不会影响线上运行的流程。草稿可以多次保存,方便分阶段编辑。

**发布流程**:将经过测试验证的流程正式发布到生产环境。发布前会进行配置校验,确保流程定义完整正确。

**测试运行**:在正式发布前验证流程的正确性。测试运行使用模拟数据,不会产生实际业务影响。

**撤销/重做**:提供编辑历史回溯能力,支持撤销最近50步编辑操作,避免误操作导致的工作丢失。

**缩放控制**:调整画布的显示比例,范围通常为50%-200%,便于查看整体结构或局部细节。

**3. 节点面板区功能详解**

**触发器列表**:展示平台支持的所有触发器类型,包括定时触发、API触发、事件触发、Webhook触发。每个流程必须包含一个且仅一个触发器节点,作为流程的起点。

**连接器列表**:展示平台提供的所有连接器能力,分为卡片连接器(发送各类卡片)、API连接器(调用HTTP接口)、事件连接器(发布/订阅事件)等类别。用户可以根据业务需求选择合适的连接器。

**逻辑组件列表**:提供流程控制的逻辑组件,包括条件判断(if-else分支)、延迟等待(暂停执行)、数据提取(字段映射)等。这些组件用于实现流程的逻辑控制和数据处理。

**4. 画布编辑区功能详解**

**节点创建操作**:在两个已有节点的连线上,会出现一个"+"添加按钮。点击该按钮后,会弹出节点类型选择菜单,用户可以选择要添加的节点类型(触发器/连接器/逻辑组件),选中后节点会被插入到连线中。这是与传统拖拽方式的核心区别。

**节点操作**:用户可以选中画布上的节点,调整节点位置,双击节点编辑名称,右键节点进行复制、删除等操作。

**连线操作**:节点之间会自动生成连线。当需要添加新节点时,点击连线上的添加按钮。当需要创建分支连线时(例如条件判断的真/假分支),在条件节点的分支连线上点击添加按钮,选择目标节点类型,即可创建条件分支连接。支持创建条件分支连线,区分条件为真和为假时的不同流向。

**画布操作**:支持画布的缩放、平移、多选、批量删除、对齐等操作,提高编辑效率。

**5. 属性配置区功能详解**

**节点属性**:配置选中节点的名称、描述、参数值等。每个节点类型有不同的配置项,例如API连接器需要配置调用的API和请求参数。

**数据映射**:定义节点间的数据传递关系。例如触发器节点的输出数据如何传递给后续节点,后续节点的输出又如何传递给再下一个节点。

**高级配置**:包括超时时间设置、重试策略配置、错误处理策略等,用于处理执行过程中的异常情况。

**6. 编辑器使用流程(钉钉实现方式)**

**创建流程**:在画布上创建一个触发器节点(作为流程起点) → 在触发器与结束位置之间的连线上点击添加按钮 → 选择需要的连接器类型 → 节点被插入到连线中 → 继续在后续连线上添加更多节点 → 配置每个节点的参数 → 配置节点间的数据映射 → 保存草稿 → 测试运行 → 发布流程。

**修改流程**:打开已有流程 → 修改需要调整的节点 → 调整节点间的连接关系 → 更新数据映射 → 保存并测试 → 重新发布。

执行日志详情页用于追溯单次流程执行的完整过程,是问题诊断的核心页面。采用四区布局设计:

**状态标识**:用颜色标识执行状态,绿色表示成功、红色表示失败、黄色表示执行中、灰色表示已终止。状态标识醒目展示,一眼判断执行结果。

**触发参数**:展示触发本次执行时的输入参数,包括触发类型(事件/API/定时等)、触发来源、触发数据等。这些信息对于分析执行行为和排查问题非常重要。

**节点状态标识**:每个节点下方显示该节点的执行状态图标,包括成功(✓)、失败(✗)、执行中(⟳)、跳过(⊘)等。失败的节点会用红色高亮,便于快速定位问题。

**时间戳展示**:每个节点显示精确到毫秒的开始时间和结束时间,便于分析执行耗时和时间分布。

**4. 步骤详情区功能详解**

**步骤基本信息**:展示选中节点的概要信息,包括节点名称、节点类型(触发器/连接器/逻辑组件)、执行时间、状态、耗时。

**输入参数JSON**:以JSON格式展示该节点的输入参数,包括来自前置节点的输出数据、系统注入的上下文数据、节点配置的参数值。JSON支持折叠和展开,便于查看复杂数据结构。

**输出结果JSON**:以JSON格式展示该节点的执行结果,包括API调用的响应数据、卡片发送的结果、系统处理的结果等。通过对比输入输出,可以验证节点是否按预期工作。

**执行日志**:以时间顺序展示节点执行过程中的详细日志,包括开始执行、调用连接器、处理响应、完成执行等关键事件。日志对于追踪执行细节和定位问题非常关键。

**错误信息(仅失败时显示)**:当节点执行失败时,显示详细的错误信息,包括错误码、错误消息、错误类型、堆栈信息等。错误信息帮助快速理解失败原因,有的错误信息还会提供解决方案建议。

**5. 操作区功能详解**

**重新执行**:基于本次执行的触发参数重新执行流程。用于测试修复后的流程或重新执行被中断的流程。重新执行会创建新的执行记录。

**编辑流程**:跳转到流程编辑器打开该流程。用于发现流程问题后快速进入编辑状态进行修改。编辑器会自动定位到出问题的节点。

**复制参数**:将输入参数JSON复制到剪贴板。用于获取测试数据或在其他地方使用这些参数。

**6. 问题诊断流程**

**定位问题节点**:通过时间线快速找到失败的节点(红色高亮)。

**查看错误信息**:点击失败节点,查看详细的错误码和错误消息。

**分析输入参数**:检查输入参数是否符合预期,排除参数错误。

**检查连接器配置**:如果参数正确,检查连接器配置是否正确。

**联系外部支持**:如果怀疑是外部系统问题,联系外部系统负责人。

**修改并测试**:在流程编辑器中修改流程,测试验证修复效果。


**流程执行详细说明:**

**1. 完整执行流程解析**

该时序图展示了从事件触发到流程执行完成的完整过程,涉及七个参与者的协作:

**事件源**:业务系统或监控系统的外部系统,当发生特定业务事件(如入职、审批完成、告警等)时,向事件总线发布事件消息。事件消息包含事件类型、事件数据、事件时间戳等信息。

**事件总线**:平台的事件统一入口,负责接收来自各个事件源的事件。事件总线根据事件类型将事件路由到对应的触发器管理器,同时对事件进行格式校验和初步处理。

**流程引擎**:流程执行的核心调度器,负责管理整个流程的生命周期。主要职责包括:根据触发条件匹配对应的流程定义、创建流程执行实例、管理执行状态、控制执行顺序、处理异常情况。流程引擎内部维护流程实例状态机,确保流程按正确顺序执行。

**执行器**:流程引擎的执行单元,负责具体执行每个流程节点。执行器从连接器层获取节点对应的连接器实现,调用连接器执行业务操作,并处理连接器返回的结果。

**连接器**:封装了各种业务能力的调用接口。连接器根据节点配置构造请求参数,调用底层服务(如卡片平台、API服务等),并对返回结果进行标准化处理。

**卡片平台**:开放平台提供的卡片发送服务,接收来自连接器的卡片发送请求,将卡片消息发送到目标群聊或用户。

**监控系统**:记录流程执行的完整日志,包括执行开始、步骤执行、执行完成、执行失败等关键事件。监控系统还负责向外部系统发送执行状态通知。

**2. 关键执行机制**

**步骤循环执行机制**:流程引擎会按照流程定义的节点顺序,依次调用执行器执行每个步骤。每个步骤执行完成后,执行器返回执行结果给流程引擎,引擎根据结果判断是继续下一步还是进入异常处理。

**成功分支处理**:当步骤执行成功时,流程引擎记录步骤结果到监控日志,然后继续执行下一个步骤。整个流程按顺序执行,直到所有步骤完成。

**失败分支处理**:当步骤执行失败时,流程引擎会立即记录失败日志到监控系统,并进入错误处理流程。错误处理包括:记录错误详情、触发告警通知(如果配置了告警)、根据配置决定是重试还是终止流程。

**状态记录机制**:监控系统会记录整个执行过程的关键状态:执行开始时间、每个步骤的开始时间和结束时间、执行结果、执行耗时等。这些状态信息用于后续的问题诊断和性能分析。

**3. 执行监控和告警**

执行过程中的监控信息包括:每个步骤的输入参数、输出结果、执行耗时、执行状态。监控日志采用结构化格式,便于日志检索和统计分析。

告警通知会在以下情况触发:步骤执行失败、流程执行超时、连续执行失败超过阈值等。告警通知通过配置的渠道发送(如邮件、短信、即时通讯等)。

**4. 性能优化设计**

**异步执行**:耗时的步骤采用异步执行方式,提高系统吞吐量。异步执行的步骤会立即返回执行ID,调用方通过轮询或回调获取执行结果。

**并发控制**:对资源占用较大的操作(如数据库写入、外部API调用)进行并发控制,避免系统过载。可以配置每个连接器的最大并发数。

**超时管理**:每个步骤都配置有超时时间,超时未完成的步骤会被强制终止并标记为失败。超时时间根据业务场景合理设置,避免过短导致正常操作被中断,或过长导致问题发现不及时。

### 6.3 问题诊断流程

```mermaid
flowchart LR
    A[执行失败告警] --> B[访问执行监控]
    B --> C[查看执行日志列表]
    C --> D[点击失败记录]
    D --> E[查看执行时间线]
    E --> F{定位问题节点}
    F -->|查看输入参数| G[检查参数是否正确]
    G -->|参数错误| H[复制参数]
    H --> I[跳转到流程编辑器]
    I --> J[修改节点配置]
    J --> K[重新执行测试]
    G -->|参数正确| L[检查连接器配置]
    L --> M[可能需要检查外部系统]
    F -->|查看错误日志| N[分析错误日志]
    N --> O[确定解决方案]
    
    style A fill:#ffcdd2
    style K fill:#c8e6c9
```

**问题诊断流程详细说明:**

**1. 诊断流程概览**

该流程图展示了从发现执行失败到解决问题的完整诊断过程。当流程执行出现异常时,运维人员需要按照此流程进行问题排查和修复。

**2. 各环节详解**

**执行失败告警**:系统检测到流程执行失败后,自动发送告警通知给相关人员。告警信息包括:流程名称、执行ID、失败时间、错误概要等。告警渠道可以根据重要性配置为邮件、短信、即时通讯等。

**访问执行监控**:运维人员收到告警后,登录连接器平台的执行监控页面。执行监控页面展示实时执行状态和历史执行记录,是问题诊断的起点。

**查看执行日志列表**:运维人员通过筛选条件(时间范围、流程名称、执行状态等)定位到问题流程的执行记录。执行日志列表展示每次执行的概览信息,包括执行时间、执行时长、执行状态等。

**点击失败记录**:运维人员点击失败记录进入执行日志详情页。详情页展示该次执行的完整时间线和每个步骤的详细信息。

**查看执行时间线**:执行时间线以可视化方式展示流程的执行过程,包括每个步骤的开始时间、结束时间、执行状态。失败的步骤会高亮显示,便于快速定位问题节点。

**定位问题节点**:根据时间线和执行日志,运维人员需要确定是哪个步骤出现了问题。问题可能出在输入参数、连接器配置、外部系统等环节。

**3. 两种排查路径**

**路径一:检查输入参数**

如果怀疑是输入参数错误导致的失败,运维人员需要:
- 查看该步骤的输入参数JSON
- 对比参数值是否符合预期
- 检查参数的格式、类型、范围是否正确
- 如果确认参数错误,复制参数后跳转到流程编辑器修改节点配置
- 修改完成后进行测试运行验证修复效果

**路径二:检查连接器配置**

如果参数正确但仍然失败,可能原因包括:
- 连接器的配置项(如目标群ID、API密钥等)配置错误
- 连接器对应的外部服务不可用
- 连接器的调用权限被撤销
- 连接器的调用配额超限

如果怀疑是外部系统问题,可能需要:
- 检查外部系统的服务状态
- 查看外部系统的访问日志
- 联系外部系统负责人确认
- 等待外部系统恢复后重新执行

**4. 查看错误日志分析**

错误日志是问题诊断的关键依据,包含以下信息:
- 错误代码:标准的错误分类代码,便于快速定位错误类型
- 错误消息:对错误的详细描述
- 堆栈信息:如果涉及代码异常,包含完整的调用堆栈
- 上下文信息:错误发生时的环境变量、请求参数等

根据错误日志分析,可以确定解决方案:
- 参数错误:修改流程节点配置
- 配置错误:更新连接器配置或联系管理员
- 外部系统错误:等待外部系统恢复或联系外部系统支持
- 代码bug:反馈给开发团队修复

**5. 修复验证流程**

问题修复后,需要验证修复效果:
- 在流程编辑器中进行测试运行
- 确认测试通过后再发布正式版本
- 监控修复后的执行情况,确保问题不再复现

**6. 问题预防建议**

为减少问题发生,建议:
- 在正式发布前进行充分的测试
- 配置合理的超时时间和重试策略
- 建立完善的监控告警机制
- 定期审查流程配置和连接器状态
- 记录常见问题的解决方案,形成知识库
