## 1 需求价值和概述

连接器配置当前是单份可编辑配置，无版本概念；HTTP 方法包含 PATCH；认证方式仅支持 SOA、APIG；入参和出参未按参数位置拆分；object、array 参数缺少保存前校验和层级限制。本次整改引入版本体系、空版本创建草稿和只读编辑模式，扩展认证方式，并提升参数配置质量。

价值：避免误改已发布配置；支持连接器版本化管理；增强 Cookie 和数字签名接入能力；提升参数配置可读性和可靠性。

## 6 功能设计

### 6.1 业界方案实现

接口契约管理通常采用草稿、已发布、已失效版本模型：草稿版本用于编辑和保存，已发布版本用于运行时引用，已失效版本用于保留历史但禁止继续使用。页面侧按版本状态控制可编辑性和危险操作，避免用户直接修改已发布契约。

参数配置方面，业界常见做法是将请求头、请求体、URL 参数、响应头、响应体按参数位置分区展示，降低参数混杂导致的理解成本；但数据提交时仍保持统一契约结构，由转换层负责聚合和拆分。

### 6.2 功能实现整体设计方案

页面由页面头部、版本选择卡、接口基础配置、认证方式配置、入参配置、出参配置组成。页面默认只读，只有草稿版本在点击编辑后进入编辑态。已发布和已失效版本始终只读，仅允许执行状态相关操作。

本次整改的核心原则是：**视图层改造，数据层和转换层尽量不变**。

| 层级 | 设计说明 | 关键约束 |
|---|---|---|
| 页面容器 | 持有完整接口配置数据，控制版本、编辑态、保存和发布流程 | 不直接实现 carrier 筛选细节 |
| 版本选择卡 | 负责版本下拉、状态展示、状态操作按钮和空版本创建草稿入口 | 只抛出事件，不直接调用保存逻辑；版本列表为空时仅展示创建草稿 |
| CarrierSchemaTabs | 按 carrier 拆分 Tab，合并当前 Tab 修改后的完整数组 | 只替换当前 carrier，保留其他 carrier |
| SchemaEditor | 作为纯参数编辑引擎，负责参数行、嵌套参数、增删改 | 不感知 Tab，不耦合 form |
| 转换层 | 将数组 + carrier 转换为后端需要的 `{ header, body, query }` 结构 | 保持现有逻辑不变 |

参数内部继续使用数组结构加 `carrier` 标识位置，展示层按 Tab 筛选，保存时仍通过现有转换层按 carrier 聚合。这样既满足新交互，又避免大范围改动 `transformToSchemaFormat`、`transformFromSchemaFormat`、`flowUtils` 等转换逻辑。

### 6.3 功能实现

| 模块 | 设计 |
|---|---|
| HTTP 方法 | 仅保留 GET、POST、PUT、DELETE，移除 PATCH |
| 版本状态 | 草稿、已发布、已失效 |
| 空版本操作 | 创建草稿 |
| 草稿操作 | 编辑、保存、发布 |
| 已发布操作 | 复制到草稿、失效，不展示编辑入口 |
| 已失效操作 | 复制到草稿、恢复、删除，不展示编辑入口 |
| 编辑模式 | 默认只读，草稿点击编辑后可修改 |
| 入参 Tab | HTTP 请求头、HTTP 请求体、URL 参数 |
| 出参 Tab | HTTP 响应头、HTTP 响应体 |
| 认证方式 | SOA、APIG、Cookie、数字签名 |
| 参数组件 | SchemaEditor 重构为受控编辑引擎，CarrierSchemaTabs 负责 Tab 拆分合并 |
| 参数校验 | object、array 保存前递归校验基础类型子参数，并限制最多 5 层 |
| Cookie 注入 | 接口配置页只定义 Cookie 参数，连接流中引用上游参数注入运行时值 |
| 数字签名 | 参数名称默认空且可编辑；`X-Signature` 展示在签名值来源；密钥默认掩码，显示/隐藏由独立按钮控制 |

#### 6.3.1 HTTP 方法整改

接口配置页的协议类型需要和后端当前能力保持一致。页面中所有协议选择入口只保留 GET、POST、PUT、DELETE，避免用户创建后端不支持的接口配置。

| 项目 | 方案 |
|---|---|
| 常量配置 | 从 HTTP 方法选项中移除 PATCH |
| 页面展示 | 协议类型按钮组、下拉项或单选项不再展示 PATCH |
| 默认值处理 | 新建或无配置时默认选择 GET |
| 历史数据 | 如果后端返回 PATCH，页面不提供继续选择入口，联调时按后端规则处理 |
| 保存数据 | 保存时只允许提交 GET、POST、PUT、DELETE |

关键代码设计：

```jsx
/**
 * HTTP 方法选项。
 * 接口配置页协议类型只允许选择后端当前支持的四种方法。
 */
export const HTTP_METHOD_OPTIONS = ['GET', 'POST', 'PUT', 'DELETE'];

/**
 * 接口配置默认值。
 * 新建或无配置时默认使用 GET，避免空值导致协议按钮无选中态。
 */
export const DEFAULT_API_CONFIG = {
  protocolType: 'GET',
  protocolAddress: '',
  authType: '',
  authRequestSchema: [],
  requestSchema: [],
  responseSchema: [],
};
```

#### 6.3.2 版本状态枚举和操作矩阵

接口配置从单份配置改为多版本配置后，页面展示和按钮权限均由版本状态驱动。状态统一抽象为草稿、已发布、已失效三类。

| 状态 | 展示文案 | 页面可编辑 | 主要操作 |
|---|---|---|---|
| 无版本 | 暂无版本 | 不可编辑 | 创建草稿 |
| 草稿 | 草稿 | 点击编辑后可编辑 | 编辑、保存、发布 |
| 已发布 | 已发布 | 不可编辑，不展示编辑入口 | 复制到草稿、失效 |
| 已失效 | 已失效 | 不可编辑，不展示编辑入口 | 复制到草稿、恢复、删除 |

版本状态枚举建议放在公共常量中，版本选择卡和页面主体共同使用，保证文案、颜色和按钮权限一致。

```js
/**
 * 连接器版本状态枚举。
 * 页面展示状态标签和操作按钮时统一使用该配置。
 */
export const CONNECTOR_VERSION_STATUS = {
  DRAFT: 0,
  PUBLISHED: 4,
  INVALID: 5,
};

/**
 * 连接器版本状态展示映射。
 * 用于版本选择卡中的状态文案和颜色。
 */
export const CONNECTOR_VERSION_STATUS_MAP = {
  [CONNECTOR_VERSION_STATUS.DRAFT]: { text: '草稿', color: 'default' },
  [CONNECTOR_VERSION_STATUS.PUBLISHED]: { text: '已发布', color: 'green' },
  [CONNECTOR_VERSION_STATUS.INVALID]: { text: '已失效', color: 'red' },
};
```

#### 6.3.3 版本选择卡

版本选择卡是接口配置页的主控区域，负责版本切换、状态展示和版本操作。页面主体只展示当前选中版本的数据，不直接关心版本列表如何加载和切换。

| 功能 | 方案 |
|---|---|
| 版本下拉 | 展示当前连接器下的所有版本，切换后加载对应版本详情；版本列表为空时展示空占位 |
| 空版本操作 | 当版本列表为空时，版本选择卡右侧仅展示“创建草稿”按钮，点击后新增一个草稿版本并切换为当前版本 |
| 状态标签 | 根据当前版本状态展示草稿、已发布、已失效 |
| 草稿操作 | 默认只读；点击编辑进入编辑态；编辑态中点击保存提交当前版本配置；点击发布触发发布操作 |
| 已发布操作 | 不展示编辑入口；提供复制到草稿和失效，失效需要二次确认 |
| 已失效操作 | 不展示编辑入口；提供复制到草稿、恢复、删除，删除需要二次确认 |
| 切换保护 | 编辑态切换版本时提示当前修改未保存，确认后再切换 |

交互规则：

1. 页面首次进入时加载版本列表，并默认选中后端返回的默认版本或最新版本。
2. 如果版本列表为空，版本选择卡仅展示“创建草稿”按钮，不展示编辑、发布、复制到草稿、失效、恢复、删除等按钮。
3. 点击“创建草稿”后新增草稿版本，刷新版本下拉和按钮状态，并将新草稿设为当前版本。
4. 切换版本后退出编辑态，页面恢复只读。
5. 复制到草稿成功后刷新版本列表，并切换到新草稿版本。
6. 发布、失效、恢复、删除成功后刷新版本列表和当前版本详情；删除最后一个版本后留在当前页展示“创建草稿”。
7. 版本操作失败时保留当前配置，不清空页面数据。

关键代码设计：

```jsx
/**
 * 根据版本状态生成版本操作按钮。
 * 版本选择卡只负责抛出操作事件，具体接口调用由页面容器处理。
 */
const getVersionActionButtons = (options) => {
  const { status, versions = [] } = options;

  if (versions.length === 0) {
    return [{ key: 'createDraft', label: '创建草稿', type: 'primary' }];
  }

  const actionMap = {
    [CONNECTOR_VERSION_STATUS.DRAFT]: [
      { key: 'edit', label: '编辑' },
      { key: 'publish', label: '发布', type: 'primary' },
    ],
    [CONNECTOR_VERSION_STATUS.PUBLISHED]: [
      { key: 'copy', label: '复制到草稿' },
      { key: 'invalidate', label: '失效', danger: true },
    ],
    [CONNECTOR_VERSION_STATUS.INVALID]: [
      { key: 'copy', label: '复制到草稿' },
      { key: 'restore', label: '恢复' },
      { key: 'delete', label: '删除', danger: true },
    ],
  };

  return actionMap[status] || [];
};

/**
 * 版本选择卡接入示例。
 * 页面通过 currentVersion 控制版本详情展示和表单可编辑范围。
 */
<VersionSelectorCard
  versions={versionList}
  value={currentVersionId}
  status={currentVersion?.status}
  onChange={handleVersionChange}
  onAction={handleVersionAction}
/>;
```

#### 6.3.4 编辑 / 只读切换

接口配置页默认只读，避免误改已发布或已失效版本。只有草稿版本允许进入编辑态，编辑态中才允许修改基础配置、认证方式、入参和出参。

| 场景 | 页面状态 |
|---|---|
| 首次进入页面 | 默认只读 |
| 当前版本为草稿 | 展示编辑入口 |
| 点击编辑 | 保存当前配置快照，表单控件启用，入参和出参操作按钮显示 |
| 点击保存 | 先执行参数校验，再提交数据；保存成功后退出编辑态 |
| 切换版本 | 如果处于编辑态，先提示是否放弃修改；确认后切换并恢复只读 |
| 已发布 / 已失效版本 | 始终只读，不展示编辑入口 |

控件禁用范围包括协议类型、协议地址、认证方式选择、认证参数配置、入参/出参新增编辑删除按钮。

```jsx
/**
 * 判断当前版本是否允许进入编辑态。
 * 只有草稿版本可以编辑，已发布和已失效版本均保持只读。
 */
const canEditCurrentVersion = currentVersion?.status === CONNECTOR_VERSION_STATUS.DRAFT;

/**
 * 进入编辑态。
 * 进入前保存快照，用于切换版本或取消编辑时恢复。
 */
const handleEnterEdit = () => {
  setEditingSnapshot(JSON.parse(JSON.stringify(apiConfig)));
  setIsEditing(true);
};

/**
 * 接口基础配置接入编辑态。
 * disabled 由版本状态和编辑态共同控制。
 */
<Radio.Group
  value={apiConfig.protocolType}
  disabled={!isEditing}
  onChange={(event) => handleConfigChange('protocolType', event.target.value)}
>
  {HTTP_METHOD_OPTIONS.map((method) => (
    <Radio.Button key={method} value={method}>{method}</Radio.Button>
  ))}
</Radio.Group>;
```

#### 6.3.5 认证方式扩展

认证方式由原来的 SOA、APIG 扩展为 SOA、APIG、Cookie、数字签名。认证方式可以多选，勾选后展示对应参数配置区域。认证参数同样受编辑态控制，非编辑态只读展示。

| 认证类型 | 配置内容 | 编辑规则 |
|---|---|---|
| SOA | 参数名称、参数类型、参数位置、值来源 | 编辑态可维护，非编辑态只读 |
| APIG | 参数名称、参数类型、参数位置、值来源 | 编辑态可维护，非编辑态只读 |
| Cookie | 参数名称、参数类型、参数位置、值来源说明 | 接口配置页不允许手填运行时取值，运行时值由连接流编辑器引用上游参数注入 |
| 数字签名 | 参数名称、参数类型、参数位置、签名值来源、签名密钥 | 参数名称默认空且可编辑；`X-Signature` 默认放到签名值来源；签名密钥是否掩码由独立按钮控制，不和编辑态绑定 |

Cookie 认证在接口配置页只定义“需要哪些 Cookie 参数”，不配置实际运行时值。连接流中使用该连接器时，在连接器节点配置面板中为 Cookie 参数选择上游参数来源，包括触发器入参、上游连接器出参、数据处理节点出参、数据输出节点出参。

#### 6.3.6 参数位置 Tab 和 CarrierSchemaTabs 设计

入参和出参配置需要按参数位置拆分展示，避免所有参数混在一个列表中导致位置语义不清。Tab 只负责展示当前参数位置的数据，保存时仍然需要把各 Tab 下的数据合并为完整配置提交。

| 区域 | Tab | carrier | 数据范围 |
|---|---|---|---|
| 入参配置 | HTTP 请求头 | header | 请求 Header 参数 |
| 入参配置 | HTTP 请求体 | body | 请求 Body 参数 |
| 入参配置 | URL 参数 | query | URL Query 参数 |
| 出参配置 | HTTP 响应头 | header | 响应 Header 参数 |

#### 6.3.7 SchemaEditor 重构设计

当前 `SchemaEditor` 已支持参数名称、参数类型、参数位置、说明、引用参数、object/array 嵌套、编辑态和只读态。但现有问题是与 form 强耦合，内部通过 `form.getFieldValue('apiConfig')` 读取数据，并通过 `form.setFieldValue` 写回，导致 Tab 包装层无法准确控制数据流。

重构目标是让 `SchemaEditor` 成为纯粹的参数编辑引擎，不再耦合业务层 form、apiConfig 和 Tab 切换。

| 方面 | 重构前 | 重构后 |
|---|---|---|
| 数据来源 | `form.getFieldValue` + `value` prop 双重来源 | 纯 `value` prop |
| 数据写回 | `form.setFieldValue` + `onChange` | 纯 `onChange` |
| 内部 state | `schemaData` + `internalChange` 标志 | 直接使用受控 value |
| carrier 写入 | 用户手动选择 | Tab 模式下由 `fixedCarrier` 自动写入 |
| carrier 展示 | 始终显示下拉 | `showCarrier=false` 时隐藏 |
| Tab 感知 | 无 | 继续不感知 Tab，Tab 由 CarrierSchemaTabs 管理 |

外部接口设计：

#### 6.3.9 object / array 参数校验

复杂类型参数如果没有基础类型子参数，运行时无法形成明确的数据结构。保存前需要递归校验所有入参和出参，确保 object / array 类型下至少包含 string、number、boolean 中的一种基础类型子参数。

| 校验对象 | 校验规则 |
|---|---|
| object 参数 | 必须至少包含一个基础类型子参数 |
| array 参数 | 必须至少包含一个基础类型子参数 |
| 嵌套 object / array | 递归校验每一层 |
| 层级上限 | object / array 嵌套最多支持 10 层，达到第 10 层后不再允许添加子节点 |
| 空 children | 不允许保存 |
| 仅嵌套复杂类型 | 不允许保存，直到某一层包含基础类型子参数 |

保存流程：

1. 点击保存后收集接口基础配置、认证配置、入参配置、出参配置。
2. 对入参所有 Tab 数据执行复杂类型校验和 10 层层级校验。
3. 对出参所有 Tab 数据执行复杂类型校验和 10 层层级校验。
4. 对连接流相关节点保存时，校验触发器输入契约、连接器输入映射、数据输出映射、Cookie 认证值来源。
5. 如果存在错误，提示具体参数路径并阻止保存。
6. 校验通过后再调用保存接口。

错误提示示例：

```text
入参配置-HTTP 请求体-用户信息：object 类型参数必须包含基础类型子参数
出参配置-HTTP 响应头-result：array 类型参数必须包含基础类型子参数
连接器节点「获取用户信息」Cookie 认证参数「SESSION」未设置值来源
参数「用户信息.地址.扩展信息」层级最多支持 10 层
```

校验代码设计：

```js
/**
 * 基础参数类型集合。
 * object / array 的子参数中必须至少包含一种基础类型。
 */
const BASIC_PARAM_TYPES = ['string', 'number', 'boolean'];

/**
 * 参数嵌套层级上限。
 * object / array 最多支持 10 层，超过后不允许保存。
 */
const MAX_SCHEMA_DEPTH = 10;

/**
 * 校验单个参数节点。
 * 复杂类型必须包含至少一个基础类型子参数，并递归校验子层级。
 */
const validateParamNode = (options) => {
  const { param, path, depth } = options;
  const currentPath = path ? `${path}.${param.paramName}` : param.paramName;
  const isComplexType = param.paramType === 'object' || param.paramType === 'array';

  if (depth > MAX_SCHEMA_DEPTH) {
    return `参数「${currentPath}」层级最多支持 ${MAX_SCHEMA_DEPTH} 层`;
  }

  if (!isComplexType) {
    return null;
  }

  const children = Array.isArray(param.children) ? param.children : [];
  const hasBasicChild = children.some((child) => BASIC_PARAM_TYPES.includes(child.paramType));

  if (!hasBasicChild) {
    return `参数「${currentPath}」必须包含至少一个基础类型子参数`;
  }

  for (const child of children) {
    const errorMessage = validateParamNode({ param: child, path: currentPath, depth: depth + 1 });

    if (errorMessage) {
      return errorMessage;
    }
  }

  return null;
};

/**
 * 保存前校验接口配置参数。
 * 入参和出参都需要执行复杂类型校验。
 */
export const validateApiConfigBeforeSave = (apiConfig) => {
  const schemas = [
    ...(apiConfig.requestSchema || []),
    ...(apiConfig.responseSchema || []),
  ];

  for (const param of schemas) {
    const errorMessage = validateParamNode({ param, path: '', depth: 1 });

    if (errorMessage) {
      return errorMessage;
    }
  }

  return null;
};
```

```js
 * 参数项数据结构。
 * children 用于描述 object / array 的递归子参数。
 */
const schemaItem = {
  paramName: 'userId',
  paramType: 'string',
  carrier: 'body',
  description: '',
  sourceType: 'static',
  paramValue: '',
  referencePath: '',
  children: [],
};
```

`carrier` 含义：

| carrier | 接口入参含义 | 接口出参含义 |
|---|---|---|
| header | HTTP 请求头 | HTTP 响应头 |
| body | HTTP 请求体 | HTTP 响应体 |
| query | URL 查询参数 | 不适用 |

后端提交结构示例：

```js
/**
 * 后端接收的参数结构。
 * 转换层负责将数组结构按 carrier 聚合为 header、body、query 对象。
 */
const backendContract = {
  header: {
    type: 'object',
    properties: {
      'X-Custom': { type: 'string' },
    },
  },
  query: {
    type: 'object',
    properties: {},
  },
  body: {
    type: 'object',
    properties: {
      userId: { type: 'string' },
    },
  },
};
```

## 7 系统级非功能设计

### 7.1 FMEA 影响分析

| 风险 | 影响 | 措施 |
|---|---|---|
| 空版本无入口 | 用户无法创建首个版本 | 版本列表为空时仅展示“创建草稿”按钮 |
| 切换版本丢失编辑 | 用户修改丢失 | 编辑态切换前提示 |
| Tab 合并丢参数 | 保存数据不完整 | 只替换当前 carrier，保留其他位置 |
| 复杂参数非法 | 运行结构不明确 | 保存前递归校验基础类型子参数和 10 层层级上限 |
| 版本接口失败 | 页面异常 | 保留当前配置并提示 |

### 7.2 安全影响分析

数字签名密钥默认掩码展示，并通过独立按钮控制显示或隐藏；Cookie 运行时值不在连接器配置页维护；发布、失效、删除和创建草稿依赖后端鉴权。

### 7.3 兼容性

历史参数缺少 carrier 时默认归入 body；后续认证方式和状态可通过枚举扩展；历史复杂参数如果超过 10 层，保存时需要按层级上限提示并阻止提交。

### 7.4 可运维

版本操作成功后刷新列表和详情；版本列表为空时保留当前页面并展示创建草稿入口；校验失败提示参数路径和层级上限。

### 7.5 资料

更新连接器配置、版本状态、创建草稿、数字签名、参数配置和层级校验说明。
