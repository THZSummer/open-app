## 1 需求价值和概述

连接流编辑器当前画布可能为空，节点新增依赖左侧拖拽，节点类型不足，缺少连线加号插入、删除自动重连、数据处理、Cookie 参数引用、版本详情和调试能力。本次整改目标是提升编排连续性、配置能力和发布前验证能力。

价值：降低新建门槛；保证链路连续；支持错误处理、并行、数据处理等复杂编排；支持 Cookie 运行时参数注入；支持版本化管理和调试验证。

## 6 功能设计

### 6.1 业界方案实现

流程编排工具通常通过默认起止节点保证流程有入口和出口，通过连线插入保持链路连续，通过节点属性面板配置参数，通过版本管理隔离草稿与发布态。本次整改采用“默认起止节点 + 连线加号插入 + 节点配置抽屉 + 版本头部操作区”的组合方式，减少拖拽依赖，让用户按主链路顺序完成编排。

### 6.2 功能实现整体设计方案

页面由头部版本区、画布区、节点配置抽屉、节点选择弹窗、版本详情抽屉、版本对比抽屉和调试抽屉组成。新增节点统一从连线加号进入，节点配置在右侧抽屉完成，版本操作在头部完成。所有节点保存配置时均校验节点名称必填。

### 6.3 节点体系设计

本期连接流编辑器共涉及 9 种节点类型。其中触发器、连接器、数据输出为已有节点；错误处理、循环、并行、条件、文本、数据处理为新增节点。循环和条件只补充类型定义，本期不在节点选择弹窗开放添加。

| 节点类型 | 状态 | 定义与作用 | 实现方式 |
|---|---|---|---|
| 触发器节点 | 开放 | 连接流入口，定义 HTTP 请求触发方式。每个连接流只允许一个触发器。 | 保留 `trigger` 类型，默认初始化画布时自动生成。 |
| 连接器节点 | 开放 | 调用远程连接器动作，支持 SOA、APIG、Cookie、数字签名等认证方式。 | 保留 `connector` 类型，在节点配置抽屉中配置连接器、版本、超时时间，并展示连接器配置页已配置的认证参数；除 Cookie 值来源外均只读。 |
| 数据输出节点 | 开放 | 定义连接流输出结构，包括 HTTP 响应体、HTTP 响应头等返回结果。 | 保留 `exit` 类型，复用 SchemaEditor 按 HTTP 响应体和 HTTP 响应头配置输出映射。 |
| 错误处理节点 | 开放 | 捕获流程异常，按错误类型配置忽视、重试、中止等处理策略；右侧分支有且只能添加一个连接器节点。 | `insertMode` 为 `group`，插入时生成错误处理节点组，并限制右侧分支节点类型和数量。 |
| 循环 | 不开放 | 定义重复执行逻辑。 | 仅保留类型和元数据，`enabled` 为 `false`。 |
| 并行节点 | 开放 | 定义多分支并行处理逻辑，最多支持 8 个分支。 | `insertMode` 为 `group`，插入时生成完整并行节点组，添加分支时校验上限。 |
| 条件 | 不开放 | 定义条件分支逻辑。 | 仅保留类型和元数据，`enabled` 为 `false`。 |
| 文本节点 | 开放 | 用于结束占位、分支说明、错误说明等。 | 新增 `text` 类型，不参与实际编排执行。 |
| 数据处理节点 | 开放 | 对上游参数进行函数处理，再输出给下游节点。 | 新增 `data_process` 类型，复用 SchemaEditor 并扩展函数处理字段。 |

节点元数据同时服务节点选择弹窗、节点渲染、配置抽屉和后端转换。保留已有 `trigger`、`connector`、`exit`，避免影响历史编排数据。

### 6.4 默认初始化画布

#### 6.4.1 整改目标

当后端未返回编排配置、编排配置为空、或节点列表为空时，前端自动创建 HTTP 触发器节点和文本节点，并建立默认连线。文本节点文案固定为“结束”，用于表达流程结束占位；结束节点不再作为单独节点类型存在。

#### 6.4.2 实现思路

默认初始化只在没有有效编排数据时执行，不能覆盖后端已有节点。默认连线使用 `plusEdge`，这样用户进入页面后可以直接点击连线中间的“+”继续插入节点。

```js
/**
 * 初始化默认连接流画布。
 * 仅在后端没有有效编排配置时调用。
 * @returns {{ nodes: Array, edges: Array }} 默认节点和连线
 */
export const initializeDefaultFlowGraph = () => {
  const triggerId = generateNodeId('trigger');
  const endTextId = generateNodeId('text');

  return {
    nodes: [
      { id: triggerId, type: 'trigger', position: { x: 180, y: 220 }, data: { label: 'HTTP 触发器' } },
      { id: endTextId, type: 'text', position: { x: 620, y: 220 }, data: { label: '结束' } },
    ],
    edges: [
      { id: generateEdgeId(), source: triggerId, target: endTextId, type: 'plusEdge' },
    ],
  };
};
```

### 6.5 隐藏左侧节点库

本期不再通过左侧节点库拖拽新增节点，新增入口统一改为连线“+”。`NodeLibrary` 文件可暂时保留，但编辑器页面不再渲染该组件，避免出现两套新增入口。

### 6.6 连线“+”插入节点

#### 6.6.1 整改目标

新增 `PlusEdge` 和 `NodePickerModal`。用户点击连线中点“+”后打开节点选择弹窗。选择普通节点时，拆分原连线并插入单节点；选择错误处理或并行节点时，插入节点组。

#### 6.6.2 普通节点插入

普通节点包括连接器、数据输出、文本、数据处理等。插入时删除原连线，新增 `source -> newNode` 和 `newNode -> target` 两条连线。

```js
/**
 * 在指定连线上插入普通单节点。
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {string} params.edgeId - 目标连线 ID
 * @param {string} params.newNodeType - 新节点类型
 * @returns {{ nodes: Array, edges: Array }} 更新后的节点和连线
 */
export const insertSingleNodeOnEdge = (params) => {
  const { nodes, edges, edgeId, newNodeType } = params;
  const targetEdge = edges.find((edge) => edge.id === edgeId);

  if (!targetEdge) {
    return { nodes, edges };
  }

  const sourceNode = nodes.find((node) => node.id === targetEdge.source);
  const targetNode = nodes.find((node) => node.id === targetEdge.target);
  const newNode = createDefaultFlowNode({ newNodeType, sourceNode, targetNode });
  const nextEdges = createSplitEdges({ edges, targetEdge, insertedNodeIds: [newNode.id] });

  return { nodes: nodes.concat(newNode), edges: nextEdges };
};
```

#### 6.6.3 节点组插入

错误处理和并行节点属于结构化节点组。节点选择弹窗返回 `error_handler` 或 `parallel` 后，工具函数根据类型生成主节点、文本节点和内部连线。

```js
/**
 * 根据节点类型创建插入载荷。
 * 错误处理和并行节点返回多个节点，普通节点返回单节点。
 * @param {Object} params - 参数对象
 * @param {string} params.newNodeType - 新节点类型
 * @param {Object} params.sourceNode - 原连线起点节点
 * @param {Object} params.targetNode - 原连线终点节点
 * @returns {{ nodes: Array, internalEdges: Array }} 新增节点和内部连线
 */
export const createInsertNodePayload = (params) => {
  const { newNodeType, sourceNode, targetNode } = params;

  if (newNodeType === 'error_handler') {
    return createErrorHandlerNodeGroup({ sourceNode, targetNode });
  }

  if (newNodeType === 'parallel') {
    return createParallelNodeGroup({ sourceNode, targetNode });
  }

  return createSingleNodePayload({ newNodeType, sourceNode, targetNode });
};
```

错误处理节点组包含错误处理主节点、异常说明文本节点、兜底处理文本节点。主节点进入主链路，两个文本节点作为异常说明和兜底处理配置入口。错误处理右侧分支有且只能添加一个连接器节点；未添加连接器时，右侧分支连线加号仅展示连接器节点，已添加后再次点击提示“错误处理分支只能添加一个连接器节点”。

```js
/**
 * 创建错误处理节点组。
 * 节点组包含错误处理主节点、异常说明文本节点、兜底处理文本节点。
 * @param {Object} params - 参数对象
 * @param {Object} params.sourceNode - 原连线起点节点
 * @param {Object} params.targetNode - 原连线终点节点
 * @returns {{ nodes: Array, internalEdges: Array }} 错误处理节点组
 */
export const createErrorHandlerNodeGroup = (params) => {
  const { sourceNode, targetNode } = params;
  const handlerId = generateNodeId('error_handler');
  const errorTextId = generateNodeId('text');
  const fallbackTextId = generateNodeId('text');

  return {
    nodes: [
      { id: handlerId, type: 'error_handler', position: getGroupMainPosition({ sourceNode, targetNode }), data: { label: '错误处理' } },
      { id: errorTextId, type: 'text', position: getGroupBranchPosition({ sourceNode, targetNode, index: 0 }), data: { label: '异常说明' } },
      { id: fallbackTextId, type: 'text', position: getGroupBranchPosition({ sourceNode, targetNode, index: 1 }), data: { label: '兜底处理' } },
    ],
    internalEdges: [
      { id: generateEdgeId(), source: handlerId, target: errorTextId, type: 'plusEdge' },
      { id: generateEdgeId(), source: handlerId, target: fallbackTextId, type: 'plusEdge' },
    ],
  };
};
```

并行节点组默认生成完整结构：并行主节点、两个分支开始文本、两个分支结束文本、一个汇合文本。分支内部的“分支开始 -> 分支结束”连线允许继续插入节点；分支结束后统一连接到汇合文本，再回到主链路。

#### 6.6.4 错误处理节点配置

错误处理节点插入后，点击节点本体打开节点配置抽屉。配置项如下：

| 字段 | 类型 | 说明 |
|---|---|---|
| `errorConfigs` | Array | 错误配置列表，可配置多个错误类型 |

9. 保存后写入 `node.data.errorConfigs`。
10. 画布节点卡片同步展示当前错误处理摘要。

#### 6.6.5 并行节点分支交互

并行节点插入后默认生成 6 个节点和 8 条连线。

| 元素 | 默认数量 | 说明 |
|---|---:|---|
| 并行主节点 | 1 | 并行结构入口 |
| 分支开始文本 | 2 | 分支可编辑链路起点 |
| 分支结束文本 | 2 | 分支可编辑链路终点 |
| 汇合文本 | 1 | 所有分支统一出口 |
| 结构连线 | 8 | 包含上游到主节点、分支内部、分支汇合、汇合到下游 |

交互流程：

1. 用户点击连线“+”并选择并行节点。
2. 系统删除原连线，并插入完整并行结构。
3. 用户可以在分支内部连线上继续插入节点。
4. 用户点击并行主节点上的“添加分支”按钮后，若当前分支数小于 8，系统新增一个分支开始、一个分支结束和三条连线。
5. 当并行分支数已达到 8 个时，不再新增分支，并提示“并行分支最多支持8个”。
6. 用户删除分支开始文本节点时，系统删除该分支开始、分支结束及相关连线。
7. 用户点击并行主节点，配置抽屉中也可以增删分支或调整分支名称。
8. 画布重新布局时，按分支序号横向排列分支，并将汇合文本节点放在所有分支结束节点下方。

并行分支字段结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `parallelGroupId` | string | 并行结构分组 ID，通常等于并行主节点 ID |
| `parallelRole` | string | 节点角色，支持 `root`、`branch-start`、`branch-end`、`merge` |
| `parallelBranchId` | string | 分支唯一标识，分支开始和分支结束使用同一个 ID |
| `parallelBranchIndex` | number | 分支序号，用于分支显示和布局排序 |

#### 6.6.6 插入位置重排

连线“+”插入节点时，除拆分连线外，还需要同步调整画布位置，避免新增节点和原下游节点重叠。位置处理原则：只影响目标边后续链路，不移动原 `source` 节点，不移动与目标边无关的其他分支。

| 插入类型 | 位置规则 |
|---|---|
| 普通节点 | 新节点放在原连线中点；如果间距不足，target 及其下游节点整体下移 |
| 错误处理节点组 | 主节点放在原连线中点；异常说明和兜底处理放在右侧；target 及其下游节点整体下移 |
| 并行节点组 | 并行主节点放在原连线中点；分支横向展开；汇合节点在分支下方；target 及其下游节点整体下移 |

当在某条并行分支内部插入嵌套并行或条件结构时，嵌套结构会横向展开多条子分支。为了避免覆盖右侧兄弟分支，必须将被插入分支右侧的兄弟分支整体右移。

```js
/**
 * 计算插入节点后的布局结果。
 * 只处理目标连线后方节点，避免影响无关节点位置。
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {Object} params.targetEdge - 被插入的目标连线
 * @param {Array} params.insertedNodes - 本次新增节点列表
 * @param {number} params.requiredHeight - 本次插入需要预留的高度
 * @returns {Array} 调整位置后的节点列表
 */
export const getNodesAfterInsertLayout = (params) => {
  const { nodes, edges, targetEdge, insertedNodes, requiredHeight } = params;
  const targetNodeIds = getDownstreamNodeIds({ edges, startNodeId: targetEdge.target });
  const insertedNodeIds = insertedNodes.map((node) => node.id);

  return nodes.concat(insertedNodes).map((node) => {
    // 新增节点使用创建时已经计算好的位置，不再二次移动。
    if (insertedNodeIds.includes(node.id)) {
      return node;
    }

    // 用户锁定过位置的节点不参与自动重排。
    if (node.data?.positionLocked) {
      return node;
    }

    // 只有目标节点及其下游节点需要整体下移。
    if (targetNodeIds.includes(node.id)) {
      return {
        ...node,
        position: {
          ...node.position,
          y: node.position.y + requiredHeight,
        },
      };
    }

    return node;
  });
};
```

自动重排约束：

- 只移动目标边 target 节点及其下游节点。
- 不移动 source 节点和 source 的其他上游节点。
- 不移动与目标边无连通关系的节点。
- 若节点存在 `positionLocked: true`，自动重排时跳过该节点。
- 每次插入只做局部重排，不做全画布重新布局，减少对用户手动排版的影响。

### 6.7 删除节点自动重连

#### 6.7.1 整改目标

删除中间节点时，如果该节点存在入边和出边，则自动新增入边来源到出边目标的连线，保证主链路不断开。删除节点组内文本节点时，只删除相关连线，不强制重连到其他分支。

#### 6.7.2 删除交互流程

1. 用户点击节点，右侧节点配置抽屉打开。
2. 用户点击“删除节点”。
3. 系统弹出二次确认，提示删除后会移除当前节点及相关连线。
4. 用户确认后，前端先判断节点类型。
5. 如果是触发器节点，直接阻止删除。
6. 如果是普通中间节点，删除节点和相关连线，并自动连接前置节点和后置节点。
7. 如果是节点组主节点，删除整个节点组，并将主链路前后节点重新连接。
8. 如果是节点组内部文本节点，只删除该文本节点和相关内部连线，不强制重连到其他分支。
9. 删除完成后关闭配置抽屉，并按规则上移后续节点。

#### 6.7.3 删除策略

| 删除对象 | 删除范围 | 重连方式 | 位置处理 |
|---|---|---|---|
| 普通节点 | 仅删除当前节点和相关边 | 入边来源连接到出边目标 | 后续主链路节点上移一个节点间距 |
| 文本节点 | 删除当前文本节点和相关边 | 普通文本可按普通节点重连；节点组内部文本不重连 | 仅移除当前文本占位 |
| 错误处理主节点 | 删除错误处理主节点、异常说明、兜底处理及内部边 | 原 source 直接连接原 target | target 和下游节点上移节点组高度 |
| 并行主节点 | 删除并行主节点、分支文本及内部边 | 原 source 直接连接原 target | target 和下游节点上移节点组高度 |
| 循环主节点 | 本期不开放，仅预留 | 后续删除循环体和回流边后重连主链路 | target 和下游节点上移循环区域高度 |
| 条件主节点 | 本期不开放，仅预留 | 后续删除 true / false 分支和汇聚边后重连主链路 | target 和下游节点上移条件区域高度 |

```js
/**
 * 删除节点后自动重连。
 * @param {Object} params - 参数对象
 * @param {Array} params.edges - 当前连线列表
 * @param {string} params.removedNodeId - 被删除节点 ID
 * @param {Object} params.removedNode - 被删除节点对象
 * @param {Array} params.removedGroupNodeIds - 同时删除的节点组 ID 列表
 * @returns {Array} 重连后的连线列表
 */
export const getReconnectedEdges = (params) => {
  const { edges, removedNodeId, removedNode, removedGroupNodeIds = [] } = params;
  const removedNodeIds = [removedNodeId].concat(removedGroupNodeIds);
  const incoming = edges.filter((edge) => edge.target === removedNodeId && !removedNodeIds.includes(edge.source));
  const outgoing = edges.filter((edge) => edge.source === removedNodeId && !removedNodeIds.includes(edge.target));
  const remaining = edges.filter((edge) => !removedNodeIds.includes(edge.source) && !removedNodeIds.includes(edge.target));

  // 节点组内部文本节点只删除自身相关边，不强制重连。
  if (removedNode?.data?.groupRole === 'branchText') {
    return remaining;
  }

  if (incoming.length === 0 || outgoing.length === 0) {
    return remaining;
  }

  const reconnected = [];
  incoming.forEach((inEdge) => {
    outgoing.forEach((outEdge) => {
      reconnected.push({
        id: generateEdgeId(),
        source: inEdge.source,
        target: outEdge.target,
        type: 'plusEdge',
      });
    });
  });

  return remaining.concat(reconnected);
};
```

### 6.8 数据处理节点

#### 6.8.1 整改目标

数据处理节点用于对上游参数进行函数处理后再输出给下游节点。它的配置体验类似数据输出节点，但在新增或编辑参数时增加“处理函数”配置。

1. 用户点击画布连线上的“+”。

 * 将数据处理参数转换为后端编排字段。

export const transformDataProcessFieldToBackend = (params) => {
  const { field } = params;

  if (field.sourceType === 'function') {
    return {
      type: field.paramType || 'string',
      description: field.description || '',
      value: buildFunctionProcessValue({ functionConfig: field.functionConfig }),
      functionConfig: field.functionConfig,
    };

    return {
      type: field.paramType || 'string',
      description: field.description || '',
      value: `\${$.node.${field.referencePath}}`,
    };

### 6.9 Cookie 认证参数注入

#### 6.9.1 整改目标

连接器节点配置抽屉在选择连接器后，展示该连接器在连接器配置页配置的认证方式和认证参数明细。参数名称、类型、位置、值来源默认不可编辑；仅 Cookie 的值来源允许从当前节点上游所有节点出参中选择。

### 6.10 上游参数选择器

#### 6.10.1 整改目标

`UpstreamParamPicker` 用于统一选择当前节点可访问的上游参数。它不直接从全量节点中选参数，而是基于当前节点 ID 和连线关系向前追溯，只展示当前节点上游链路中的可引用参数。该选择器同时服务 Cookie 认证参数、连接器输入映射和数据处理节点参数。

#### 6.10.2 可引用参数范围

| 使用场景 | 可引用范围 | 说明 |
|---|---|---|
| Cookie 认证参数 | 上游所有节点出参 | 只展示当前连接器节点之前的节点出参，用于选择 Cookie 值来源 |
| 连接器输入映射 | 触发器入参、上游连接器出参、上游数据处理出参 | 用于构造连接器请求参数 |
| 数据处理节点参数 | 触发器入参、上游连接器出参、上游数据处理出参、上游数据输出出参 | 用于选择待处理参数 |

#### 6.10.3 数据流

1. 配置抽屉打开时，将当前节点 ID、节点列表、连线列表传给 `getUpstreamParams`。
2. `getUpstreamParams` 先通过连线关系递归查找所有上游节点。
3. 对每个上游节点调用 `getNodeOutputParams`，根据节点类型提取可引用参数。
4. `UpstreamParamPicker` 按节点分组展示参数。
5. 用户选择参数后，返回 `nodeId`、`nodeType`、`paramPath`、`carrier`、`paramType`。
6. 调用方将选择结果写入 `valueSource`、`referencePath` 或 `functionConfig.input`。

```js
/**
 * 获取当前节点的上游参数。
 * @param {Object} params - 参数对象
 * @param {string} params.nodeId - 当前节点 ID
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @returns {Array} 按上游节点分组的参数列表
 */
export const getUpstreamParams = (params) => {
  const { nodeId, nodes, edges } = params;
  const upstreamNodes = getUpstreamNodes({ nodeId, nodes, edges });

  return upstreamNodes.map((node) => ({
    nodeId: node.id,
    nodeName: node.data?.label || node.id,
    nodeType: node.type,
    params: getNodeOutputParams({ node }),
  }));
};
```

弹窗交互：左侧按上游节点分组展示，例如“HTTP 触发器”“获取用户信息”；右侧展示当前节点下的参数列表，支持按 header、body、query 筛选；参数行展示参数名、参数类型、参数路径、描述；点击确认后回填到调用方表单。

### 6.11 版本子系统

#### 6.11.1 整改目标

连接流编辑器头部新增版本下拉、状态标签、详情、版本操作、更多配置、保存并调试。版本操作按状态控制可用性。进入页面后默认加载版本列表，并优先选中草稿版本；若无草稿，则选中最新版本。

#### 6.11.2 页面头部结构

/**
 * 获取默认选中的版本。
 * 优先选择草稿版本；如果没有草稿，则选择列表中的第一条版本。

#### 6.11.4 数据流

1. 页面初始化时调用 `fetchFlowVersions({ flowId })`。
2. 根据版本列表选择默认版本：优先草稿，其次最新版本；若版本列表为空，则头部右侧展示“创建草稿”。
3. 调用 `fetchFlowVersionDetail({ flowId, versionId })` 获取当前版本详情和编排配置。
4. 将 `orchestrationConfig` 转换为 React Flow 的 `nodes` 和 `edges`。
5. 用户切换版本时重复第 3、4 步。
6. 用户执行发布、撤回、复制、失效、恢复、删除、创建草稿后，刷新版本列表和当前版本详情。

#### 6.11.5 更多配置

| 配置项 | 说明 |
|---|---|
| 缓存 Key | 支持添加多个缓存 Key，并展示每个缓存 Key 拼接后的结果 |
| 缓存时间 | 单位为秒，最大值为 1296000 秒（15 天），超过上限时提示“上限为 1296000 秒” |
| 限流配置 | 最大值为 1000，超过上限时提示用户调整 |

#### 6.11.6 只读与编辑控制

- 草稿、已驳回、已撤回版本允许编辑和保存。
- 已驳回、已撤回版本保存后状态改为草稿。
- 草稿、已驳回、已撤回、已失效版本允许删除。
- 审批中、已发布、已失效版本默认只读，其中已失效仅允许删除、恢复等状态操作。
- 版本列表为空时，头部右侧展示“创建草稿”按钮，点击后创建一个草稿版本。
- 保存并调试允许对当前版本配置发起调试，但只读版本不允许直接改写编排配置。

### 6.12 版本详情抽屉

#### 6.12.1 整改目标

右侧抽屉展示当前版本完整信息，包括基础信息、状态信息、审批信息、编排摘要和操作记录。抽屉使用 `mask={false}`，避免遮挡画布。

#### 6.12.2 展示内容

| 区块 | 字段 | 说明 |
|---|---|---|
| 基础信息 | 版本号、版本名称、版本状态、创建人、创建时间、更新人、更新时间 | 所有状态都展示 |
| 审批信息 | 审批单号、审批状态、提交人、提交时间、审批人 | 审批中、已驳回、已撤回等状态展示 |
| 驳回 / 撤回信息 | 驳回原因、撤回原因、处理时间 | 有值时展示 |
| 编排摘要 | 节点数量、连线数量、连接器节点数量、数据处理节点数量 | 从 `orchestrationConfig` 解析生成 |
| 操作入口 | 详情对比、复制到草稿、发布、失效、恢复、删除等 | 根据版本状态控制 |

```js
/**
 * 构建版本编排摘要。
 * @param {Object} params - 参数对象
 * @param {Object} params.orchestrationConfig - 版本编排配置
 * @returns {Object} 编排摘要
 */
export const buildVersionFlowSummary = (params) => {
  const { orchestrationConfig } = params;
  const nodes = orchestrationConfig?.nodes || [];
  const edges = orchestrationConfig?.edges || [];

  return {
    nodeCount: nodes.length,
    edgeCount: edges.length,
    connectorCount: nodes.filter((node) => node.type === 'connector').length,
    dataProcessCount: nodes.filter((node) => node.type === 'data_process').length,
  };
};
```

状态差异展示：草稿展示编辑/保存、发布、删除入口；审批中展示审批单信息、撤回、催办入口；已驳回展示驳回原因、编辑/保存、发布、删除入口；已撤回展示撤回原因、编辑/保存、发布、删除入口；已发布展示复制到草稿和失效入口；已失效展示恢复和删除入口。

### 6.13 调试抽屉

#### 6.13.1 整改目标

点击“保存并调试”后，先保存当前编排配置，再打开调试抽屉。调试抽屉负责构造触发器入参、发起调试请求、展示执行结果和错误信息。调试不改变版本状态，只用于当前版本配置验证。

#### 6.13.2 抽屉布局

| 区域 | 内容 | 说明 |
|---|---|---|
| 顶部信息 | 当前连接流名称、版本号、版本状态 | 只读展示 |
| 入参区域 | header、body、query 三个 Tab | 根据触发器节点 `inputContract` 自动生成模板，允许用户填写调试值 |
| 操作区 | 开始调试、清空入参、复制结果 | 开始调试前校验入参 JSON 格式 |
| 输出区域 | 状态、耗时、响应 header、响应 body、错误信息 | 调试接口返回后展示 |
| 执行日志 | 节点执行顺序、每个节点耗时、失败节点 | 后端返回时展示；无数据时隐藏 |

#### 6.13.3 执行流程

1. 用户点击“保存并调试”。
2. 前端先执行当前画布保存逻辑，将 `nodes` 和 `edges` 转换为后端编排结构。
3. 保存成功后，根据触发器节点的 `inputContract` 生成调试入参模板。
4. 打开 `DebugDrawer`，用户补充或修改 header、body、query。
5. 用户点击“开始调试”。
6. 前端调用 `/flows/{flowId}/versions/{versionId}/debug`。
7. 调试完成后展示响应 JSON、状态、耗时、错误信息和执行日志。

```js
/**
 * 保存并打开调试抽屉。
 * @param {Object} params - 参数对象
 * @param {string} params.flowId - 连接流 ID
 * @param {string} params.versionId - 版本 ID
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {Function} params.openDebugDrawer - 打开调试抽屉回调
 * @returns {Promise<void>}
 */
export const saveAndOpenDebugDrawer = async (params) => {
  const { flowId, versionId, nodes, edges, openDebugDrawer } = params;
  const orchestrationConfig = transformToBackend(nodes, edges);

  await saveFlowVersionConfig({ flowId, versionId, orchestrationConfig });

  openDebugDrawer({
    flowId,
    versionId,
    inputTemplate: buildDebugInputTemplate({ nodes }),
  });
};
```

## 7 系统级非功能设计

### 7.1 FMEA 影响分析

| 风险 | 影响 | 措施 |
|---|---|---|
| 节点插入失败 | 链路异常 | 保留原画布数据 |
| 删除节点断链 | 编排不可用 | 自动重连前后节点 |
| Cookie 来源未配置 | 运行认证失败 | 保存前校验 |
| 保存失败仍调试 | 调试数据不一致 | 保存失败不打开调试抽屉 |
| 只读版本误编辑 | 版本配置被错误修改 | 审批中、已发布、已失效默认只读；已驳回和已撤回允许编辑保存为草稿 |
| 并行布局重叠 | 画布可读性下降 | 插入节点时局部重排，嵌套结构右移兄弟分支 |

### 7.2 安全影响分析

版本发布、删除、状态变更依赖后端鉴权；Cookie 值只保存引用来源，不在配置页手填敏感值；调试入参仅用于当前调试请求，不改变版本状态。

### 7.3 兼容性

已有 trigger、connector、exit 类型保持不变；循环和条件仅定义不开放，便于后续扩展；旧版本编排数据加载时仍按已有节点和连线渲染，只有空配置才初始化默认画布。

### 7.4 可运维

调试抽屉展示状态、耗时、响应和错误信息；版本详情展示编排摘要；执行日志展示节点执行顺序、每个节点耗时和失败节点，便于定位运行问题。

### 7.5 资料

更新连接流编辑器、节点类型、版本和调试使用说明。新增使用说明应覆盖：默认画布、连线加号插入、节点组、数据处理函数、Cookie 参数引用、版本详情和保存并调试。
