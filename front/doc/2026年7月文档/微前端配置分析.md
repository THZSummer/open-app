# qiankun 微前端工程配置分析

## 一、工程总览

| 工程 | 技术栈 | 构建工具 | 端口 | 角色 |
|------|--------|---------|------|------|
| main-app | React 18 | Webpack 5 | 8080 | 主应用（基座） |
| sub-app-b | React 18 | Vite 5 | 5173 | 子应用 B |
| sub-app-c | React 18 | Webpack 5 | 8081 | 子应用 C |
| sub-app-d | Vue 3 | Webpack 5 | 8082 | 子应用 D |
| sub-app-e | Vue 3 | Vite 5 | 5174 | 子应用 E |

## 二、五个工程的关系

```
┌──────────────────────────────────────────────────────┐
│                main-app (主应用/基座)                   │
│                React + Webpack :8080                  │
│                                                        │
│  ┌──────────┐  ┌───────────────────────────────────┐ │
│  │ 侧边导航   │  │  #sub-app-viewport (挂载容器)       │ │
│  │ /        │  │                                   │ │
│  │ /sub-b ──┼──┤  -> sub-app-b 渲染于此              │ │
│  │ /sub-c ──┼──┤  -> sub-app-c 渲染于此              │ │
│  │ /sub-d ──┼──┤  -> sub-app-d 渲染于此              │ │
│  │ /sub-e ──┼──┤  -> sub-app-e 渲染于此              │ │
│  └──────────┘  └───────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
        │ fetch entry          │ fetch entry          │ fetch entry          │ fetch entry
        ▼                      ▼                      ▼                      ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│ sub-app-b     │     │ sub-app-c     │     │ sub-app-d     │     │ sub-app-e     │
│ React + Vite  │     │ React+Webpack │     │ Vue3 +Webpack │     │ Vue3 + Vite   │
│ :5173         │     │ :8081         │     │ :8082         │     │ :5174         │
└───────────────┘     └───────────────┘     └───────────────┘     └───────────────┘
```

**关系说明：**

- **main-app** 是基座（主应用），负责侧边导航、路由分发、子应用注册与生命周期管理
- 四个子应用各自独立运行在独立端口，彼此不依赖，互相不知道对方存在
- 主应用通过 qiankun 的 `registerMicroApps` 注册子应用，根据浏览器 URL 自动匹配 `activeRule`，拉取对应子应用的 entry 资源并渲染到 `#sub-app-viewport` 容器中
- 四个子应用也可脱离主应用独立运行（直接访问各自端口），得益于各自的独立运行检测逻辑

## 三、主应用（main-app）的微前端配置

### 3.1 安装 qiankun 依赖

在 `package.json` 中引入：

```json
"qiankun": "^2.10.5"
```

**作用：** 提供 `registerMicroApps`、`start` 等 API，是整个微前端框架的核心库。

### 3.2 子应用注册列表（microApps.js）

```js
const microApps = [
  { name: 'sub-app-b', entry: '//localhost:5173', container: '#sub-app-viewport', activeRule: '/sub-b' },
  { name: 'sub-app-c', entry: '//localhost:8081', container: '#sub-app-viewport', activeRule: '/sub-c' },
  { name: 'sub-app-d', entry: '//localhost:8082', container: '#sub-app-viewport', activeRule: '/sub-d' },
  { name: 'sub-app-e', entry: '//localhost:5174', container: '#sub-app-viewport', activeRule: '/sub-e' }
];
```

| 字段 | 作用 |
|------|------|
| `name` | 子应用唯一标识，必须与子应用导出的名称一致，qiankun 据此匹配 |
| `entry` | 子应用资源入口地址，qiankun 会 fetch 这个地址拉取 HTML 和 JS |
| `container` | 子应用挂载的 DOM 容器选择器，所有子应用共用 `#sub-app-viewport` |
| `activeRule` | 路由激活规则，浏览器 URL 匹配此规则时自动加载对应子应用 |

### 3.3 注册并启动 qiankun（index.jsx）

```js
registerMicroApps(microApps);
start({
  prefetch: true,
  sandbox: { experimentalStyleIsolation: true }
});
```

| 配置 | 作用 |
|------|------|
| `registerMicroApps(microApps)` | 将子应用配置注册到 qiankun，qiankun 内部监听路由变化 |
| `start()` | 启动 qiankun 框架 |
| `prefetch: true` | 在浏览器空闲时预加载子应用资源，切换时更快 |
| `sandbox.experimentalStyleIsolation` | 开启实验性样式隔离，给子应用样式加 `div[data-qiankun]` 前缀作用域，防止主子应用 CSS 互相污染 |

### 3.4 子应用挂载容器（App.jsx）

```jsx
<div id="sub-app-viewport"></div>
```

**作用：** qiankun 加载子应用后，会将子应用内容渲染到这个 DOM 节点内。对应 `microApps.js` 中 `container: '#sub-app-viewport'`。

### 3.5 导航路由（App.jsx）

```jsx
<Link to="/sub-b">子应用 B</Link>
<Link to="/sub-c">子应用 C</Link>
<Link to="/sub-d">子应用 D</Link>
<Link to="/sub-e">子应用 E</Link>
```

**作用：** 点击导航切换 URL，qiankun 监听到 URL 变化后匹配 `activeRule` 自动加载/卸载子应用。

## 四、子应用 B（sub-app-b, React + Vite）的微前端配置

Vite 子应用的接入方式与 Webpack 子应用完全不同，使用 `vite-plugin-qiankun` 插件，不需要手动导出 UMD 生命周期。

### 4.1 安装 vite-plugin-qiankun

在 `package.json` 中引入：

```json
"vite-plugin-qiankun": "^1.0.15"
```

**作用：** Vite 的构建产物是 ESM 格式，qiankun 默认无法直接获取生命周期钩子。此插件在 Vite 构建时注入适配层，使 qiankun 能识别并接管 Vite 子应用。

### 4.2 Vite 配置（vite.config.js）

```js
plugins: [
  qiankun('sub-app-b', { useDevMode: true })
],
esbuild: {
  jsx: 'automatic'
},
server: {
  port: 5173,
  cors: true,
  origin: 'http://localhost:5173'
}
```

| 配置 | 作用 |
|------|------|
| `qiankun('sub-app-b', ...)` | 第一个参数是子应用名称，必须与主应用 `microApps.js` 中的 `name` 一致。`useDevMode: true` 开启开发模式适配 |
| `esbuild.jsx: 'automatic'` | 使用 React 18+ 的 automatic JSX runtime，不使用 `@vitejs/plugin-react`。避免 `react-refresh` preamble 与 qiankun 沙箱 `window` 不一致的冲突 |
| `server.cors: true` | 允许主应用（localhost:8080）跨域请求子应用（localhost:5173）的资源 |
| `server.origin` | 固定静态资源的 origin 为子应用地址。Vite 默认会用相对路径，被主应用加载时会导致资源 404，必须固定为绝对地址 |

### 4.3 入口文件生命周期导出（main.jsx）

```js
import { renderWithQiankun, qiankunWindow } from 'vite-plugin-qiankun/dist/helper';

renderWithQiankun({
  bootstrap() { console.log('子应用 B bootstrap'); },
  mount(props) { render(props); },
  unmount(props) { if (root) { root.unmount(); root = null; } },
  update() {}
});

if (!qiankunWindow.__POWERED_BY_QIANKUN__) {
  render({});
}
```

| 配置 | 作用 |
|------|------|
| `renderWithQiankun()` | `vite-plugin-qiankun` 提供的辅助函数，替代手动 `export` 生命周期。内部将 `bootstrap/mount/unmount` 注册到 qiankun 能识别的全局变量上 |
| `qiankunWindow.__POWERED_BY_QIANKUN__` | Vite 子应用不能直接访问 `window.__POWERED_BY_QIANKUN__`（沙箱隔离），需通过 `qiankunWindow` 获取。用于判断是否被主应用加载 |
| `render(props)` 中的 `container` | 被主应用加载时，qiankun 传入 `container`（即 `#sub-app-viewport` 的 DOM），子应用在其内部查找 `#root` 挂载；独立运行时直接用 `document.getElementById('root')` |

> **注意：Vite 子应用不需要 `public-path.js`**，因为 `vite-plugin-qiankun` 和 `server.origin` 配置已解决了资源路径问题。

## 五、子应用 C（sub-app-c, React + Webpack）的微前端配置

Webpack 子应用采用 qiankun 官方推荐的手动导出生命周期 + UMD 打包方式。

### 5.1 public-path.js 动态资源路径

```js
if (window.__POWERED_BY_QIANKUN__) {
  __webpack_public_path__ = window.__INJECTED_PUBLIC_PATH_BY_QIANKUN__;
}
```

**作用：** Webpack 的 `__webpack_public_path__` 是全局变量，用于动态设置静态资源加载的基础路径。独立运行时为默认值（`/`）；被主应用加载时，qiankun 会注入 `__INJECTED_PUBLIC_PATH_BY_QIANKUN__`（值为子应用 entry 地址），确保子应用的图片、字体、chunk 等资源从子应用自己的服务器加载，而不是主应用服务器，避免 404。

### 5.2 Webpack 配置 UMD 导出 + CORS

```js
output: {
  library: 'sub-app-c',
  libraryTarget: 'umd',
  chunkLoadingGlobal: 'webpackJsonp_sub_app_c'
},
devServer: {
  headers: { 'Access-Control-Allow-Origin': '*' }
}
```

| 配置 | 作用 |
|------|------|
| `libraryTarget: 'umd'` | 将打包产物导出为 UMD 格式，挂载到 `window['sub-app-c']` 上。qiankun 主应用通过 `import-html-entry` 拉取子应用 JS 后，从该全局变量上获取 `bootstrap/mount/unmount` 生命周期函数 |
| `library: 'sub-app-c'` | UMD 库的名称，必须与主应用注册的 `name` 一致 |
| `chunkLoadingGlobal` | Webpack 按需加载 chunk 时使用的全局变量名。多个子应用默认都用 `webpackJsonp`，会互相覆盖，必须改为唯一名称 |
| `headers: CORS` | 主应用（8080）需要 fetch 子应用（8081）的 JS 资源，属于跨域，必须设置 `Access-Control-Allow-Origin: *` |

### 5.3 入口文件导出生命周期（main.jsx）

```js
import './public-path';  // 必须在最前面引入

if (!window.__POWERED_BY_QIANKUN__) {
  render({});
}

export async function bootstrap() { /* ... */ }
export async function mount(props) { render(props); }
export async function unmount(props) { /* ... */ }
```

| 配置 | 作用 |
|------|------|
| `import './public-path'` 置顶 | 必须在所有其他模块之前执行，确保 Webpack 的 `publicPath` 在任何资源加载前就被正确设置 |
| `window.__POWERED_BY_QIANKUN__` | qiankun 在 `mount` 前注入的全局变量，值为 `true` 表示当前被主应用加载。独立运行时为 `undefined`，走 `render({})` 独立渲染 |
| `export bootstrap/mount/unmount` | qiankun 要求的三个生命周期函数，通过 UMD 导出后挂到 `window['sub-app-c']` 上，主应用通过 `import-html-entry` 获取 |

### 5.4 三个生命周期的作用

| 生命周期 | 调用时机 | 作用 |
|---------|---------|------|
| `bootstrap` | 子应用首次激活时调用一次 | 初始化全局配置，通常只做 console 或空操作 |
| `mount` | 每次进入子应用路由时调用 | 创建应用实例并渲染到 `container` 内的 DOM 节点 |
| `unmount` | 每次离开子应用路由时调用 | 销毁应用实例，清理 DOM 和事件，释放内存 |

## 六、子应用 D（sub-app-d, Vue 3 + Webpack）的微前端配置

与 sub-app-c 的接入方式完全一致，只是框架从 React 换成 Vue 3，实例管理方式不同。

### 6.1 public-path.js

与 sub-app-c 完全相同。

### 6.2 Webpack 配置

```js
output: {
  library: 'sub-app-d',
  libraryTarget: 'umd',
  chunkLoadingGlobal: 'webpackJsonp_sub_app_d'
},
devServer: {
  headers: { 'Access-Control-Allow-Origin': '*' }
}
```

与 sub-app-c 的差异仅为 `library` 和 `chunkLoadingGlobal` 名称不同，确保多子应用之间全局变量不冲突。

### 6.3 入口文件生命周期（main.js）

```js
import './public-path';
import { createApp } from 'vue';

function render(props) {
  const { container } = props || {};
  const mountNode = container ? container.querySelector('#app') : document.getElementById('app');
  app = createApp(App);
  app.mount(mountNode);
}

if (!window.__POWERED_BY_QIANKUN__) {
  render({});
}

export async function bootstrap() {}
export async function mount(props) { render(props); }
export async function unmount(props) {
  app.unmount();
  app = null;
}
```

### 6.4 与 sub-app-c 的差异对比

| 差异点 | sub-app-c (React) | sub-app-d (Vue 3) |
|--------|-------------------|-------------------|
| 挂载节点 ID | `#root` | `#app` |
| 创建实例 | `createRoot(mountNode).render(<App />)` | `createApp(App).mount(mountNode)` |
| 卸载实例 | `root.unmount()` | `app.unmount()` |

## 七、子应用 E（sub-app-e, Vue 3 + Vite）的微前端配置

与 sub-app-b 同为 Vite 子应用，使用 `vite-plugin-qiankun` 插件接入。但因 Vue 不使用 JSX，无需配置 `esbuild.jsx`，转而需要 `@vitejs/plugin-vue` 处理 `.vue` 文件。

### 7.1 安装依赖

在 `package.json` 中引入：

```json
"vite-plugin-qiankun": "^1.0.15",
"@vitejs/plugin-vue": "^5.0.0"
```

**作用：** `vite-plugin-qiankun` 用于 qiankun 生命周期适配；`@vitejs/plugin-vue` 用于编译 Vue 单文件组件（.vue）。

### 7.2 Vite 配置（vite.config.js）

```js
plugins: [
  vue(),
  qiankun('sub-app-e', { useDevMode: true })
],
server: {
  port: 5174,
  cors: true,
  origin: 'http://localhost:5174'
}
```

| 配置 | 作用 |
|------|------|
| `vue()` | `@vitejs/plugin-vue` 插件，处理 `.vue` 单文件组件的编译 |
| `qiankun('sub-app-e', ...)` | 第一个参数是子应用名称，必须与主应用 `microApps.js` 中的 `name` 一致。`useDevMode: true` 开启开发模式适配 |
| `server.cors: true` | 允许主应用（localhost:8080）跨域请求子应用（localhost:5174）的资源 |
| `server.origin` | 固定静态资源的 origin 为子应用地址。Vite 默认会用相对路径，被主应用加载时会导致资源 404，必须固定为绝对地址 |

> **与 sub-app-b 的关键区别：** sub-app-b 需要配置 `esbuild.jsx: 'automatic'` 来规避 React 的 `react-refresh` 与 qiankun 沙箱冲突；sub-app-e 使用 Vue，不涉及 JSX，无需此配置，但需要 `vue()` 插件处理 `.vue` 文件。

### 7.3 入口文件生命周期导出（main.js）

```js
import { createApp } from 'vue';
import { renderWithQiankun, qiankunWindow } from 'vite-plugin-qiankun/dist/helper';

renderWithQiankun({
  bootstrap() { console.log('子应用 E bootstrap'); },
  mount(props) { render(props); },
  unmount(props) { if (app) { app.unmount(); app = null; } },
  update() {}
});

if (!qiankunWindow.__POWERED_BY_QIANKUN__) {
  render({});
}
```

| 配置 | 作用 |
|------|------|
| `renderWithQiankun()` | `vite-plugin-qiankun` 提供的辅助函数，替代手动 `export` 生命周期。内部将 `bootstrap/mount/unmount` 注册到 qiankun 能识别的全局变量上 |
| `qiankunWindow.__POWERED_BY_QIANKUN__` | Vite 子应用不能直接访问 `window.__POWERED_BY_QIANKUN__`（沙箱隔离），需通过 `qiankunWindow` 获取。用于判断是否被主应用加载 |
| `render(props)` 中的 `container` | 被主应用加载时，qiankun 传入 `container`（即 `#sub-app-viewport` 的 DOM），子应用在其内部查找 `#app` 挂载；独立运行时直接用 `document.getElementById('app')` |

### 7.4 与 sub-app-b 的差异对比

| 差异点 | sub-app-b (React + Vite) | sub-app-e (Vue 3 + Vite) |
|--------|--------------------------|--------------------------|
| 框架插件 | 无（使用 esbuild 内置 JSX 转换） | `@vitejs/plugin-vue` |
| JSX 配置 | `esbuild.jsx: 'automatic'`（规避 react-refresh 冲突） | 不需要（Vue 不使用 JSX） |
| 挂载节点 ID | `#root` | `#app` |
| 创建实例 | `createRoot(mountNode).render(<App />)` | `createApp(App).mount(mountNode)` |
| 卸载实例 | `root.unmount()` | `app.unmount()` |

> **注意：Vite 子应用均不需要 `public-path.js`**，因为 `vite-plugin-qiankun` 和 `server.origin` 配置已解决了资源路径问题。

## 八、B / C / D / E 四个子工程微前端配置完整对比

### 8.1 接入方式总览

| 对比维度 | sub-app-b (React+Vite) | sub-app-c (React+Webpack) | sub-app-d (Vue3+Webpack) | sub-app-e (Vue3+Vite) |
|---------|------------------------|---------------------------|--------------------------|------------------------|
| 生命周期导出方式 | `renderWithQiankun()` 辅助函数 | 手动 `export` 三个函数 | 手动 `export` 三个函数 | `renderWithQiankun()` 辅助函数 |
| 打包格式 | ESM（Vite 默认） | UMD（`libraryTarget: 'umd'`） | UMD（`libraryTarget: 'umd'`） | ESM（Vite 默认） |
| 资源路径处理 | `server.origin` 固定绝对地址 | `public-path.js` 动态设置 | `public-path.js` 动态设置 | `server.origin` 固定绝对地址 |
| 环境检测变量 | `qiankunWindow.__POWERED_BY_QIANKUN__` | `window.__POWERED_BY_QIANKUN__` | `window.__POWERED_BY_QIANKUN__` | `qiankunWindow.__POWERED_BY_QIANKUN__` |
| 跨域配置 | `server.cors: true` | `devServer.headers: CORS` | `devServer.headers: CORS` | `server.cors: true` |
| 核心额外依赖 | `vite-plugin-qiankun` | 无 | 无 | `vite-plugin-qiankun` + `@vitejs/plugin-vue` |
| 需要 public-path.js | 否 | 是 | 是 | 否 |
| 需要 UMD library 配置 | 否 | 是 | 是 | 否 |
| 需要 chunkLoadingGlobal | 否 | 是 | 是 | 否 |

### 8.2 框架差异对比

| 对比维度 | sub-app-b (React) | sub-app-c (React) | sub-app-d (Vue 3) | sub-app-e (Vue 3) |
|---------|-------------------|-------------------|--------------------|--------------------|
| 挂载节点 ID | `#root` | `#root` | `#app` | `#app` |
| 创建实例 | `createRoot(mountNode).render(<App />)` | `createRoot(mountNode).render(<App />)` | `createApp(App).mount(mountNode)` | `createApp(App).mount(mountNode)` |
| 卸载实例 | `root.unmount()` | `root.unmount()` | `app.unmount()` | `app.unmount()` |
| 组件格式 | `.jsx` | `.jsx` | `.vue` SFC | `.vue` SFC |

### 8.3 Vite vs Webpack 接入差异的根本原因

| 对比维度 | Vite 子应用（B、E） | Webpack 子应用（C、D） |
|---------|--------------------|-----------------------|
| 开发模式产物 | 原生 ESM + 浏览器 `import` | UMD 格式，挂载到 `window` |
| 生命周期获取 | `vite-plugin-qiankun` 适配层注入 | qiankun 直接从 `window[library]` 读取 |
| 资源路径 | `server.origin` 固定绝对地址 | `public-path.js` 动态修改 `__webpack_public_path__` |
| 全局变量冲突风险 | 无（ESM 无全局变量） | 有（需唯一 `chunkLoadingGlobal`） |

**根本原因：** Vite 开发模式使用原生 ESM + 浏览器原生 `import`，构建产物没有全局变量挂载机制，qiankun 默认的 `import-html-entry` 无法直接获取生命周期函数，因此需要 `vite-plugin-qiankun` 做适配。而 Webpack 的 UMD 格式天然支持将导出挂到 `window` 上，qiankun 可以直接读取。

### 8.4 React vs Vue 接入差异

| 对比维度 | React 子应用（B、C） | Vue 3 子应用（D、E） |
|---------|---------------------|----------------------|
| 挂载节点 ID | `#root` | `#app` |
| 实例创建 API | `createRoot` from `react-dom/client` | `createApp` from `vue` |
| 实例渲染方式 | `.render(<App />)` | `.mount(mountNode)` |
| 实例卸载 API | `root.unmount()` | `app.unmount()` |
| 组件文件格式 | `.jsx` | `.vue`（SFC） |
| Vite 下额外注意 | 需 `esbuild.jsx: 'automatic'` 规避 react-refresh 冲突 | 需 `@vitejs/plugin-vue` 编译 SFC，无 JSX 冲突问题 |

---

## 九、总结：从普通工程到微前端需要添加的配置

### 9.1 主应用需要添加

1. 安装 `qiankun` 依赖
2. 创建子应用配置文件（`microApps.js`），声明每个子应用的 name / entry / container / activeRule
3. 调用 `registerMicroApps()` + `start()` 注册并启动
4. 在布局中提供子应用挂载容器 DOM 节点

### 9.2 Webpack 子应用需要添加（适用 C、D）

1. 新建 `public-path.js` 并在入口最前面引入
2. Webpack `output` 配置 `library` + `libraryTarget: 'umd'` + 唯一的 `chunkLoadingGlobal`
3. Webpack devServer 配置 CORS headers
4. 入口文件导出 `bootstrap` / `mount` / `unmount` 三个生命周期函数
5. 入口文件添加 `window.__POWERED_BY_QIANKUN__` 独立运行检测

### 9.3 Vite + React 子应用需要添加（适用 B）

1. 安装 `vite-plugin-qiankun` 依赖
2. Vite 配置中引入 `qiankun()` 插件，传入子应用名称
3. Vite `server` 配置 `cors: true` + `origin` 固定地址
4. Vite `esbuild` 配置 `jsx: 'automatic'`（避免 react-refresh 与 qiankun 沙箱冲突）
5. 入口文件使用 `renderWithQiankun()` 注册生命周期 + `qiankunWindow.__POWERED_BY_QIANKUN__` 独立运行检测

### 9.4 Vite + Vue 子应用需要添加（适用 E）

1. 安装 `vite-plugin-qiankun` + `@vitejs/plugin-vue` 依赖
2. Vite 配置中引入 `vue()` 和 `qiankun()` 插件，传入子应用名称
3. Vite `server` 配置 `cors: true` + `origin` 固定地址
4. 不需要 `esbuild.jsx` 配置（Vue 不使用 JSX，无 react-refresh 冲突）
5. 入口文件使用 `renderWithQiankun()` 注册生命周期 + `qiankunWindow.__POWERED_BY_QIANKUN__` 独立运行检测
