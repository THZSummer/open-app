# qiankun 微前端工程配置分析

## 一、工程总览

| 工程 | 技术栈 | 构建工具 | 端口 | host 域名 | 角色 |
|------|--------|---------|------|----------|------|
| main-app | React 18 | Webpack 5 | 8080 | localhost.uat.com | 主应用（基座） |
| sub-app-b | React 18 | Vite 5 | 5174 | localhost.uat.com | 子应用 B |
| sub-app-c | React 18 | Webpack 5 | 8082 | localhost.uat.com | 子应用 C |
| sub-app-d | Vue 3 | Webpack 5 | 8083 | localhost.uat.com | 子应用 D |
| sub-app-e | Vue 3 | Vite 5 | 5175 | localhost.uat.com | 子应用 E |

> **说明：** 所有工程统一使用 `localhost.uat.com` 作为开发服务器 host，而非默认的 `localhost`。这样做便于跨子应用的 cookie / 域名隔离调试，同时避免某些浏览器对 `localhost` 的特殊安全策略干扰 qiankun 资源加载。

## 二、五个工程的关系

```
┌──────────────────────────────────────────────────────┐
│                main-app (主应用/基座)                   │
│                React + Webpack :8080                  │
│                host: localhost.uat.com                │
│                                                        │
│  ┌──────────┐  ┌───────────────────────────────────┐ │
│  │ 侧边导航   │  │  #sub-app-viewport (挂载容器)       │ │
│  │ /        │  │                                   │ │
│  │ /qiankun/sub-b ─┤  -> sub-app-b 渲染于此          │ │
│  │ /qiankun/sub-c ─┤  -> sub-app-c 渲染于此          │ │
│  │ /qiankun/sub-d ─┤  -> sub-app-d 渲染于此          │ │
│  │ /qiankun/sub-e ─┤  -> sub-app-e 渲染于此          │ │
│  └──────────┘  └───────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
        │ fetch entry          │ fetch entry          │ fetch entry          │ fetch entry
        ▼                      ▼                      ▼                      ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│ sub-app-b     │     │ sub-app-c     │     │ sub-app-d     │     │ sub-app-e     │
│ React + Vite  │     │ React+Webpack │     │ Vue3 +Webpack │     │ Vue3 + Vite   │
│ :5174         │     │ :8082         │     │ :8083         │     │ :5175         │
│ localhost.uat │     │ localhost.uat │     │ localhost.uat │     │ localhost.uat │
└───────────────┘     └───────────────┘     └───────────────┘     └───────────────┘
```

**关系说明：**

- **main-app** 是基座（主应用），负责侧边导航、路由分发、子应用注册与生命周期管理
- 主应用使用 **HashRouter**，所有子应用的 `activeRule` 统一以 `/qiankun/` 为前缀（如 `/qiankun/sub-b`），通过 hash 匹配激活
- 四个子应用各自独立运行在独立端口，彼此不依赖，互相不知道对方存在
- 主应用通过 qiankun 的 `registerMicroApps` 注册子应用，根据浏览器 URL hash 自动匹配 `activeRule`，拉取对应子应用的 entry 资源并渲染到 `#sub-app-viewport` 容器中
- 四个子应用也可脱离主应用独立运行（直接访问各自端口），得益于各自的独立运行检测逻辑
- 所有工程共享同一个自定义域名 `localhost.uat.com`，需在系统 hosts 文件中添加 `127.0.0.1 localhost.uat.com`

## 三、主应用（main-app）的微前端配置

### 3.1 安装 qiankun 依赖

在 `package.json` 中引入：

```json
"qiankun": "^2.10.5"
```

**作用：** 提供 `registerMicroApps`、`start` 等 API，是整个微前端框架的核心库。

### 3.2 子应用注册列表（microApps.js）

```js
/**
 * 生成基于 hash 的 activeRule 匹配函数
 * @param {string} rule - 路由规则，如 '/qiankun/sub-b'
 * @returns {function} 返回匹配函数，供 qiankun 判断子应用是否激活
 */
function hashRule(rule) {
  return (location) => location.hash.startsWith('#' + rule);
}

const microApps = [
  {
    // 子应用 B（React + Vite）
    name: 'sub-app-b',
    entry: '//localhost.uat.com:5174',
    container: '#sub-app-viewport',
    activeRule: hashRule('/qiankun/sub-b')
  },
  {
    // 子应用 C（React + Webpack）
    name: 'sub-app-c',
    entry: '//localhost.uat.com:8082',
    container: '#sub-app-viewport',
    activeRule: hashRule('/qiankun/sub-c')
  },
  {
    // 子应用 D（Vue3 + Webpack）
    name: 'sub-app-d',
    entry: '//localhost.uat.com:8083',
    container: '#sub-app-viewport',
    activeRule: hashRule('/qiankun/sub-d')
  },
  {
    // 子应用 E（Vue3 + Vite）
    name: 'sub-app-e',
    entry: '//localhost.uat.com:5175',
    container: '#sub-app-viewport',
    activeRule: hashRule('/qiankun/sub-e')
  }
];
```

| 字段 | 作用 |
|------|------|
| `name` | 子应用唯一标识，必须与子应用导出的名称一致，qiankun 据此匹配 |
| `entry` | 子应用资源入口地址（含自定义域名和端口），qiankun 会 fetch 这个地址拉取 HTML 和 JS |
| `container` | 子应用挂载的 DOM 容器选择器，所有子应用共用 `#sub-app-viewport` |
| `activeRule` | **函数形式**，从 `location.hash` 中匹配路由前缀。主应用使用 HashRouter，qiankun 默认匹配 `location.pathname`（始终为 `/`），因此必须改为函数从 hash 中匹配 |

> **关键点：为什么 activeRule 用函数而非字符串？**
>
> 主应用使用 `HashRouter`，URL 形如 `http://localhost.uat.com:8080/#/qiankun/sub-b`。此时 `location.pathname` 始终为 `/`，qiankun 默认的字符串匹配方式无法工作。改用函数形式，通过 `location.hash.startsWith('#/qiankun/sub-b')` 判断子应用是否激活。

### 3.3 注册并启动 qiankun（index.jsx）

```js
import { registerMicroApps, start } from 'qiankun';
import { createRoot } from 'react-dom/client';
import React from 'react';
import App from './App';
import microApps from './microApps';

/**
 * 注册所有子应用
 * qiankun 会根据 activeRule 自动加载对应子应用并渲染到 container
 */
registerMicroApps(microApps);

/**
 * 启动 qiankun
 * prefetch  开启子应用预加载，提升切换速度
 * sandbox   开启样式隔离，避免主子应用样式互相污染
 */
start({
  prefetch: true,
  sandbox: { experimentalStyleIsolation: true }
});

// 渲染主应用根组件
const root = createRoot(document.getElementById('root'));
root.render(<App />);
```

| 配置 | 作用 |
|------|------|
| `registerMicroApps(microApps)` | 将子应用配置注册到 qiankun，qiankun 内部监听路由变化 |
| `start()` | 启动 qiankun 框架 |
| `prefetch: true` | 在浏览器空闲时预加载子应用资源，切换时更快 |
| `sandbox.experimentalStyleIsolation` | 开启实验性样式隔离，给子应用样式加 `div[data-qiankun]` 前缀作用域，防止主子应用 CSS 互相污染 |

### 3.4 主应用布局与路由（App.jsx）

```jsx
import React from 'react';
import { HashRouter, Routes, Route, Link } from 'react-router-dom';
import './index.css';

/**
 * 主应用布局组件
 * 包含侧边导航栏和内容区域
 * 使用 HashRouter，qiankun 子应用从 location.hash 中匹配 activeRule
 */
function App() {
  return (
    <HashRouter>
      <div className="layout">
        {/* 侧边导航栏 */}
        <aside className="sidebar">
          <h1 className="logo">微前端基座</h1>
          <nav>
            <Link to="/" className="nav-link">首页</Link>
            <Link to="/qiankun/sub-b?source=main-app&userId=1001" className="nav-link">子应用 B</Link>
            <Link to="/qiankun/sub-c?source=main-app&userId=1001" className="nav-link">子应用 C</Link>
            <Link to="/qiankun/sub-d?source=main-app&userId=1001" className="nav-link">子应用 D</Link>
            <Link to="/qiankun/sub-e?source=main-app&userId=1001" className="nav-link">子应用 E</Link>
          </nav>
        </aside>
        {/* 内容区域：主应用自身页面 + 子应用挂载容器 */}
        <main className="content">
          {/* 主应用自身路由页面 */}
          <Routes>
            <Route path="/" element={<div className="home">主应用首页</div>} />
            {/* qiankun 子应用路由占位，实际渲染由 #sub-app-viewport 容器接管 */}
            <Route path="qiankun/*" element={null} />
          </Routes>
          {/* 子应用挂载容器：qiankun 根据 activeRule 自动把对应子应用渲染到这里 */}
          <div id="sub-app-viewport"></div>
        </main>
      </div>
    </HashRouter>
  );
}

export default App;
```

| 配置 | 作用 |
|------|------|
| `<HashRouter>` | 主应用使用 HashRouter，URL 以 `#/` 路由。这是 activeRule 必须用函数匹配 hash 的根本原因 |
| `<Link to="/qiankun/sub-b?...">` | 导航链接统一以 `/qiankun/` 为前缀，可附带查询参数传递给子应用 |
| `<Route path="qiankun/*" element={null} />` | 路由占位符。匹配 qiankun 子应用路由时主应用自身不渲染内容（element 为 null），实际渲染由 `#sub-app-viewport` 容器接管 |
| `<div id="sub-app-viewport"></div>` | 子应用挂载容器，对应 `microApps.js` 中 `container: '#sub-app-viewport'` |

## 四、子应用 B（sub-app-b, React + Vite）的微前端配置

Vite 子应用的接入方式与 Webpack 子应用完全不同，使用 `vite-plugin-qiankun` 插件，不需要手动导出 UMD 生命周期。

### 4.1 安装 vite-plugin-qiankun

在 `package.json` 中引入：

```json
"vite-plugin-qiankun": "^1.0.15"
```

**作用：** Vite 的构建产物是 ESM 格式，qiankun 默认无法直接获取生命周期钩子。此插件在 Vite 构建时注入适配层，使 qiankun 能识别并接管 Vite 子应用。

> **注意：** 虽然 `package.json` 中也安装了 `@vitejs/plugin-react`，但 `vite.config.js` 中并未启用它。这是因为 React 的 `react-refresh` preamble 会与 qiankun 沙箱的 `window` 不一致产生冲突，改用 esbuild 内置的 JSX 转换来规避此问题。

### 4.2 Vite 配置（vite.config.js）

```js
import { defineConfig } from 'vite';
import qiankun from 'vite-plugin-qiankun';

// Vite 子应用接入 qiankun 配置
// 说明：不使用 @vitejs/plugin-react，改用 esbuild 内置的 JSX 转换（automatic runtime），
//       从根源避免 react-refresh preamble 与 qiankun 沙箱的 window 不一致问题
export default defineConfig({
  plugins: [
    // 注入子应用名称，qiankun 据此识别并接管子应用生命周期
    qiankun('sub-app-b', { useDevMode: true })
  ],
  // 使用 esbuild 的 automatic JSX runtime 转换 JSX（React 18+），
  // 不注入 react-refresh，无 preamble、无 HMR 检测代码
  esbuild: {
    jsx: 'automatic'
  },
  server: {
    // 使用自定义域名，便于跨子应用 cookie / 域名隔离
    host: 'localhost.uat.com',
    port: 5174,
    // 允许主应用跨域拉取子应用资源
    cors: true,
    origin: 'http://localhost.uat.com:5174'
  }
});
```

| 配置 | 作用 |
|------|------|
| `qiankun('sub-app-b', ...)` | 第一个参数是子应用名称，必须与主应用 `microApps.js` 中的 `name` 一致。`useDevMode: true` 开启开发模式适配 |
| `esbuild.jsx: 'automatic'` | 使用 React 18+ 的 automatic JSX runtime，不使用 `@vitejs/plugin-react`。避免 `react-refresh` preamble 与 qiankun 沙箱 `window` 不一致的冲突 |
| `server.host: 'localhost.uat.com'` | 使用自定义域名，与主应用保持一致 |
| `server.cors: true` | 允许主应用（localhost.uat.com:8080）跨域请求子应用（localhost.uat.com:5174）的资源 |
| `server.origin` | 固定静态资源的 origin 为子应用地址。Vite 默认会用相对路径，被主应用加载时会导致资源 404，必须固定为绝对地址 |

### 4.3 入口文件生命周期导出（main.jsx）

```jsx
import React from 'react';
import { createRoot } from 'react-dom/client';
import { BrowserRouter, HashRouter } from 'react-router-dom';
import { renderWithQiankun, qiankunWindow } from 'vite-plugin-qiankun/dist/helper';
import App from './App';

// 子应用根实例引用
let root = null;

/**
 * 渲染子应用到挂载节点
 * @param {Object} props qiankun 注入的参数对象（可含 container 等）
 */
function render(props) {
  // 解构出挂载容器
  const { container } = props || {};
  // 被主应用加载时用 container 内的 #root，独立运行时用 document 的 #root
  const mountNode = container ? container.querySelector('#root') : document.getElementById('root');
  // 根据运行环境设置路由 basename，qiankun 环境下用 activeRule 作为 basename
  const isQiankun = qiankunWindow.__POWERED_BY_QIANKUN__;
  const basename = isQiankun ? '/qiankun/sub-b' : '/';
  // qiankun 环境下使用 HashRouter，从 hash 解析路由
  const Router = isQiankun ? HashRouter : BrowserRouter;
  root = createRoot(mountNode);
  root.render(
    <Router basename={basename}>
      <App />
    </Router>
  );
}

// 通过 vite-plugin-qiankun 导出生命周期，并兼容独立运行
renderWithQiankun({
  /**
   * bootstrap：子应用首次激活时调用一次
   */
  bootstrap() {
    console.log('子应用 B bootstrap');
  },
  /**
   * mount：每次进入子应用时调用
   * @param {Object} props qiankun 注入的参数对象
   */
  mount(props) {
    render(props);
  },
  /**
   * unmount：每次离开子应用时调用
   * @param {Object} props qiankun 注入的参数对象
   */
  unmount(props) {
    if (root) {
      root.unmount();
      root = null;
    }
  },
  update() {}
});

// 独立运行（非 qiankun 环境）时直接渲染
if (!qiankunWindow.__POWERED_BY_QIANKUN__) {
  render({});
}
```

| 配置 | 作用 |
|------|------|
| `renderWithQiankun()` | `vite-plugin-qiankun` 提供的辅助函数，替代手动 `export` 生命周期。内部将 `bootstrap/mount/unmount` 注册到 qiankun 能识别的全局变量上 |
| `qiankunWindow.__POWERED_BY_QIANKUN__` | Vite 子应用不能直接访问 `window.__POWERED_BY_QIANKUN__`（沙箱隔离），需通过 `qiankunWindow` 获取。用于判断是否被主应用加载 |
| `container` | 被主应用加载时，qiankun 传入 `container`（即 `#sub-app-viewport` 的 DOM），子应用在其内部查找 `#root` 挂载；独立运行时直接用 `document.getElementById('root')` |
| `basename` | qiankun 环境下设为 `/qiankun/sub-b`（与主应用 activeRule 一致），独立运行时为 `/` |
| `Router` 切换 | qiankun 环境下用 `HashRouter`（与主应用一致，从 hash 解析路由），独立运行时用 `BrowserRouter` |

> **注意：Vite 子应用不需要 `public-path.js`**，因为 `vite-plugin-qiankun` 和 `server.origin` 配置已解决了资源路径问题。

## 五、子应用 C（sub-app-c, React + Webpack）的微前端配置

Webpack 子应用采用 qiankun 官方推荐的手动导出生命周期 + UMD 打包方式。

### 5.1 public-path.js 动态资源路径

```js
/**
 * 动态设置 publicPath
 * 被主应用加载时，qiankun 会注入子应用资源地址，避免静态资源加载 404
 * 注意：此文件必须在入口最前面引入
 */
if (window.__POWERED_BY_QIANKUN__) {
  // eslint-disable-next-line no-undef
  __webpack_public_path__ = window.__INJECTED_PUBLIC_PATH_BY_QIANKUN__;
}
```

**作用：** Webpack 的 `__webpack_public_path__` 是全局变量，用于动态设置静态资源加载的基础路径。独立运行时为默认值（`/`）；被主应用加载时，qiankun 会注入 `__INJECTED_PUBLIC_PATH_BY_QIANKUN__`（值为子应用 entry 地址），确保子应用的图片、字体、chunk 等资源从子应用自己的服务器加载，而不是主应用服务器，避免 404。

### 5.2 Webpack 配置 UMD 导出 + CORS

```js
module.exports = {
  output: {
    // 导出为 umd 格式，供主应用 qiankun 获取子应用导出的生命周期
    library: 'sub-app-c',
    libraryTarget: 'umd',
    // 避免与其它子应用的 chunk 加载全局变量冲突
    chunkLoadingGlobal: 'webpackJsonp_sub_app_c'
  },
  devServer: {
    // 使用自定义域名，便于跨子应用 cookie / 域名隔离
    host: 'localhost.uat.com',
    port: 8082,
    // 允许主应用跨域拉取子应用资源
    headers: {
      'Access-Control-Allow-Origin': '*'
    },
    historyApiFallback: true,
    hot: true
  }
};
```

| 配置 | 作用 |
|------|------|
| `libraryTarget: 'umd'` | 将打包产物导出为 UMD 格式，挂载到 `window['sub-app-c']` 上。qiankun 主应用通过 `import-html-entry` 拉取子应用 JS 后，从该全局变量上获取 `bootstrap/mount/unmount` 生命周期函数 |
| `library: 'sub-app-c'` | UMD 库的名称，必须与主应用注册的 `name` 一致 |
| `chunkLoadingGlobal` | Webpack 按需加载 chunk 时使用的全局变量名。多个子应用默认都用 `webpackJsonp`，会互相覆盖，必须改为唯一名称 |
| `host: 'localhost.uat.com'` | 使用自定义域名，与主应用保持一致 |
| `headers: CORS` | 主应用（8080）需要 fetch 子应用（8082）的 JS 资源，属于跨域，必须设置 `Access-Control-Allow-Origin: *` |

### 5.3 入口文件导出生命周期（main.jsx）

```jsx
// 必须在最前面引入，用于动态设置 publicPath
import './public-path';
import React from 'react';
import { createRoot } from 'react-dom/client';
import { BrowserRouter, HashRouter } from 'react-router-dom';
import App from './App';

// 子应用根实例引用
let root = null;

/**
 * 渲染子应用到挂载节点
 * @param {Object} props qiankun 注入的参数对象（可含 container 等）
 */
function render(props) {
  // 解构出挂载容器
  const { container } = props || {};
  // 被主应用加载时用 container 内的 #root，独立运行时用 document 的 #root
  const mountNode = container ? container.querySelector('#root') : document.getElementById('root');
  // 根据运行环境设置路由 basename，qiankun 环境下用 activeRule 作为 basename
  const isQiankun = window.__POWERED_BY_QIANKUN__;
  const basename = isQiankun ? '/qiankun/sub-c' : '/';
  // qiankun 环境下使用 HashRouter，从 hash 解析路由
  const Router = isQiankun ? HashRouter : BrowserRouter;
  root = createRoot(mountNode);
  root.render(
    <Router basename={basename}>
      <App />
    </Router>
  );
}

// 独立运行（非 qiankun 环境）时直接渲染
if (!window.__POWERED_BY_QIANKUN__) {
  render({});
}

/**
 * bootstrap：子应用首次激活时调用，只执行一次
 */
export async function bootstrap() {
  console.log('子应用 C bootstrap');
}

/**
 * mount：每次进入子应用时调用
 * @param {Object} props qiankun 注入的参数对象
 */
export async function mount(props) {
  render(props);
}

/**
 * unmount：每次离开子应用时调用
 * @param {Object} props qiankun 注入的参数对象
 */
export async function unmount(props) {
  if (root) {
    root.unmount();
    root = null;
  }

```

| 配置 | 作用 |
|------|------|
| `import './public-path'` 置顶 | 必须在所有其他模块之前执行，确保 Webpack 的 `publicPath` 在任何资源加载前就被正确设置 |
| `window.__POWERED_BY_QIANKUN__` | qiankun 在 `mount` 前注入的全局变量，值为 `true` 表示当前被主应用加载。独立运行时为 `undefined`，走 `render({})` 独立渲染 |
| `basename` | qiankun 环境下设为 `/qiankun/sub-c`，独立运行时为 `/` |
| `Router` 切换 | qiankun 环境下用 `HashRouter`，独立运行时用 `BrowserRouter` |
| `export bootstrap/mount/unmount` | qiankun 要求的三个生命周期函数，通过 UMD 导出后挂到 `window['sub-app-c']` 上，主应用通过 `import-html-entry` 获取 |

### 5.4 三个生命周期的作用

| 生命周期 | 调用时机 | 作用 |
|---------|---------|------|
| `bootstrap` | 子应用首次激活时调用一次 | 初始化全局配置，通常只做 console 或空操作 |
| `mount` | 每次进入子应用路由时调用 | 创建应用实例并渲染到 `container` 内的 DOM 节点 |
| `unmount` | 每次离开子应用路由时调用 | 销毁应用实例，清理 DOM 和事件，释放内存 |

### 6.4 路由配置（router/index.js）

Vue 子应用的路由处理与 React 子应用有本质区别：

```js
import { createRouter, createWebHistory, createMemoryHistory } from 'vue-router';
import List from '../pages/List.vue';
import Detail from '../pages/Detail.vue';
import Edit from '../pages/Edit.vue';

// 基础路由配置表（独立运行时使用，路径不含前缀）
const baseRoutes = [
  // 列表页 - 默认首页
  { path: '/', component: List },
  // 详情页
  { path: '/detail', component: Detail },
  // 编辑页
  { path: '/edit', component: Edit }
];

/**
 * 创建路由实例
 * @param {string} basename 路由基础路径
 * @returns {Object} 路由实例
 */
function createAppRouter(basename) {
  const isQiankun = window.__POWERED_BY_QIANKUN__;

  if (isQiankun) {
    // 主应用使用 HashRouter，hash 已被主应用路由占用（如 #/qiankun/sub-d），
    // Vue 的 createWebHashHistory 无法像 React HashRouter 那样从 hash 中剥离 basename，
    // 改用 createMemoryHistory 不操作 URL，子应用内部导航在内存中进行
    return createRouter({
      history: createMemoryHistory(),
      routes: baseRoutes,
    });
  }

  // 独立运行时使用 createWebHistory，basename 由 Vue Router 自动剥离
  return createRouter({
    history: createWebHistory(basename),
    routes: baseRoutes,
  });
}

export default createAppRouter;
```

> **关键区别：Vue 子应用在 qiankun 环境下使用 `createMemoryHistory` 而非 HashRouter。**
>
> 主应用使用 HashRouter，URL 的 hash 已被主应用路由占用（如 `#/qiankun/sub-d`）。React 的 `HashRouter` 可以通过 `basename` 从 hash 中剥离前缀，但 Vue 的 `createWebHashHistory` 没有类似机制。因此 Vue 子应用改用 `createMemoryHistory`，不操作 URL，子应用内部导航完全在内存中进行，避免与主应用 hash 路由冲突。

### 6.5 与 sub-app-c 的差异对比

| 差异点 | sub-app-c (React) | sub-app-d (Vue 3) |
|--------|-------------------|-------------------|
| 挂载节点 ID | `#root` | `#app` |
| 创建实例 | `createRoot(mountNode).render(<App />)` | `createApp(App).mount(mountNode)` |
| 卸载实例 | `root.unmount()` | `app.unmount()` |
| qiankun 环境路由 | `HashRouter` + `basename` | `createMemoryHistory`（内存路由） |
| 独立运行路由 | `BrowserRouter` + `basename` | `createWebHistory(basename)` |

```jsx
function LayoutInner() {
  const location = useLocation();

  // 判断是否为微前端子应用页面（qiankun 子应用统一以 /qiankun/ 为前缀）
  const isMicroAppPage = location.pathname.startsWith('/qiankun/');

  // 根据当前路由匹配对应的微前端子应用名称
  const currentMicroAppName = isMicroAppPage
    ? microApps.find((app) => location.pathname.startsWith('/' + app.menuKey))?.name
    : null;

  // 手动加载/卸载微前端子应用（兼容 hash 路由模式）
  useEffect(() => {
    if (!currentMicroAppName) return;

    const matchedApp = microApps.find((app) => app.name === currentMicroAppName);
    if (!matchedApp) return;

    // loadMicroApp 返回 MicroApp 句柄对象（非 Promise），包含 mount/unmount 方法
    const handle = loadMicroApp(
      {
        name: matchedApp.name,
        entry: matchedApp.entry,
        container: '#sub-app-viewport',
      },
      {
        sandbox: { experimentalStyleIsolation: true },
      }
    );

    // 清理函数：离开子应用路由时卸载子应用
    return () => {
      handle.unmount();
    };
  }, [currentMicroAppName]);

  return (
    <AntLayout>
      <Content>
        {/* 微前端子应用页面：隐藏 Outlet，显示 qiankun 挂载容器 */}
        {isMicroAppPage ? (
          <div id="sub-app-viewport" style={{ minHeight: '100%' }} />
        ) : (
          <Outlet />
        )}
      </Content>
    </AntLayout>
  );
}
```

| 配置 | 作用 |
|------|------|
| `loadMicroApp()` | qiankun 提供的手动加载子应用 API，返回 MicroApp 句柄对象（非 Promise），包含 `mount`/`unmount` 方法 |
| `useEffect` + 依赖 `currentMicroAppName` | 当路由变化导致匹配的子应用名称变化时，自动卸载旧子应用并加载新子应用 |
| `handle.unmount()` | 在 `useEffect` 清理函数中调用，确保离开子应用路由时正确卸载子应用，避免内存泄漏 |
| `sandbox` 在 `loadMicroApp` 第二个参数中配置 | 手动模式下 sandbox 配置在每次调用时传入，而非全局 `start()` |
| `isMicroAppPage` 条件渲染 | 子应用页面时显示 `#sub-app-viewport` 容器并隐藏 `Outlet`；非子应用页面时显示 `Outlet` |

### 9.3 两种模式的适用场景

| 模式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| `registerMicroApps` + `start` | 自动路由匹配，支持 prefetch 预加载，代码简洁 | 全局配置，灵活性较低 | 标准微前端场景，子应用与路由一一对应 |
| `loadMicroApp` | 手动控制加载/卸载时机，灵活度高；可在同页面加载多个子应用 | 无自动 prefetch，需手动管理生命周期；代码更复杂 | 需要精细控制子应用加载时机的场景，或同页面多子应用共存 |

## 十、其他微前端框架/方案对比

除了 qiankun 之外，业界还有多种成熟的微前端方案，各有不同的设计理念和适用场景。

### 10.1 主流方案总览

| 框架/方案 | 开发团队 | 技术基础 | JS 沙箱 | CSS 隔离 | Vite 支持 | 子应用保活 | 多实例 |
|-----------|---------|---------|---------|---------|-----------|-----------|--------|
| **qiankun** | 蚂蚁金服 | single-spa + HTML entry | Proxy 沙箱 | Shadow DOM / Scoped | 需插件 | 不支持 | 不支持 |
| **single-spa** | 开源社区 | 路由调度 + 生命周期 | 无 | 无 | 原生支持 | 不支持 | 支持 |
| **micro-app** | 京东 | WebComponent + qiankun 沙箱 | Proxy 沙箱 | Shadow DOM | 需插件 | 支持 | 支持 |
| **wujie（无界）** | 腾讯 | WebComponent + iframe | iframe 天然隔离 | iframe 天然隔离 | 原生支持 | 支持 | 支持 |
| **Module Federation** | Webpack 官方 | 构建时模块共享 | 无 | 无 | 有 Vite 插件 | 不支持 | 不支持 |
| **Garfish** | 字节跳动 | Proxy 沙箱 + 动态样式 | Proxy 沙箱 | 动态隔离 | 支持 | 支持 | 支持 |
| **EMP** | 欢聚集团 | Module Federation 封装 | 无 | 无 | 不支持 | 不支持 | 不支持 |
| **iframe** | 浏览器原生 | iframe 标签 | 天然隔离 | 天然隔离 | 原生支持 | 不适用 | 支持 |

### 10.2 各方案详细分析

#### 10.2.1 single-spa（微前端"元框架"）

qiankun 就是基于 single-spa 的上层封装。single-spa 只提供路由调度和生命周期管理，不提供沙箱和样式隔离。

- **与 qiankun 的区别**：qiankun 在 single-spa 基础上增加了 HTML entry、JS 沙箱、CSS 隔离、资源预加载等能力；single-spa 需要手动实现这些
- **优点**：框架无关，灵活度最高；社区活跃，有 `single-spa-react`/`single-spa-vue` 等辅助库；原生支持 Vite ESM
- **缺点**：不提供沙箱/样式隔离/通信机制，需全部手动实现；配置繁琐，学习曲线陡峭
- **适用**：需要深度自定义微前端体系的场景，或已有成熟隔离方案的项目

#### 10.2.2 micro-app（京东）

使用 WebComponent（自定义元素 `<micro-app>`）加载子应用，而非 qiankun 的 HTML entry + 路由监听。

- **与 qiankun 的区别**：组件式 API（`<micro-app src="...">`），更符合组件化思维；支持子应用保活和多实例；复用了 qiankun 的沙箱机制
- **优点**：子应用侵入性更低，接入成本更低；支持子应用保活（切走不销毁，切回不重新渲染）；支持多实例（同页面多个子应用）
- **缺点**：沙箱仍用 Proxy，有性能损耗；Vite 支持需插件且 JS 无法沙箱隔离；IE 不兼容
- **适用**：渐进式迁移存量项目，需要子应用保活的场景

#### 10.2.3 wujie / 无界（腾讯）

用 iframe 做 JS 沙箱（天然隔离），用 WebComponent 做 CSS 隔离容器。

- **与 qiankun 的区别**：隔离方式从 Proxy 沙箱改为 iframe 天然隔离，彻底解决 JS 隔离问题；CSS 隔离从 Shadow DOM/Scoped 改为 WebComponent 容器
- **优点**：隔离最彻底（iframe 天然隔离，无 Proxy 性能损耗）；原生支持 Vite ESM；支持子应用保活和多实例；预加载能力强
- **缺点**：iframe 历史包袱（弹窗挂载到 body 的问题、滚动条问题）；React 16 兼容性差；跨 iframe 通信成本较高
- **适用**：对隔离性和安全性要求极高的场景（金融、医疗）；以 Vite 为主的项目

#### 10.2.4 Module Federation（Webpack 5 / Rspack 原生功能）

基于构建时模块共享，运行时动态加载远程模块，而非运行时加载整个子应用。

- **与 qiankun 的区别**：粒度不同——Module Federation 是模块级共享（可共享单个组件/函数），qiankun 是应用级加载（加载整个子应用）；Module Federation 天然支持依赖去重（React/Vue 只加载一次）
- **优点**：模块级共享，代码复用率最高；依赖去重，性能最优（无重复加载 React/Vue）；Rspack 版本构建速度极快
- **缺点**：强依赖 Webpack 5 / Rspack；无沙箱和样式隔离（需自行实现或靠约定）；不支持子应用保活；要求技术栈统一（同版本构建工具）
- **适用**：技术栈统一的新项目，追求极致性能和模块共享的多团队协作

#### 10.2.5 Garfish（字节跳动）

类似 qiankun 的运行时方案，但在多实例和 Vite 支持方面更完善。

- **与 qiankun 的区别**：原生支持多实例和子应用保活；原生支持 Vite；资源调度更高效（并行加载、缓存优化）
- **优点**：高效资源管理，支持并行加载和缓存优化；强大的扩展性；原生 Vite 支持
- **缺点**：文档和社区活跃度不如 qiankun；对构建工具链的整合需一定学习成本
- **适用**：大型应用跨团队协作，需高效资源调度，技术栈混合且追求性能

#### 10.2.6 iframe（浏览器原生方案）

最简单直接的隔离方案，每个子应用运行在独立 iframe 中。

- **与 qiankun 的区别**：无需任何框架，浏览器原生隔离
- **优点**：完全隔离（JS/CSS/路由），安全性最高；技术栈完全无关；实现最简单
- **缺点**：性能差（每个 iframe 独立浏览器上下文，资源消耗大）；交互受限（全局弹窗、拖拽等跨应用交互困难）；路由问题（前进/后退按钮异常）；SEO 不友好；通信复杂（需 postMessage）
- **适用**：需要完全隔离的第三方应用嵌入；对性能要求不高的简单场景

### 10.3 选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 多技术栈共存 + 存量项目迁移 | **qiankun** 或 **micro-app** | 成熟稳定，社区大，多框架支持好 |
| 对隔离性/安全性要求极高 | **wujie** | iframe 天然隔离，最安全 |
| 技术栈统一 + 追求极致性能 | **Module Federation** | 模块共享，依赖去重，性能最优 |
| 需要子应用保活 + 多实例 | **wujie** 或 **micro-app** | qiankun 不支持保活和多实例 |
| Vite 子应用为主 | **wujie** 或 **Garfish** | 原生支持 Vite ESM，无需额外插件 |
| 从零搭建新项目（统一技术栈） | **Module Federation (Rspack)** | 构建快，模块共享，架构更现代 |
| 需要完全隔离的第三方嵌入 | **iframe** | 最简单，最安全，无需框架 |
| 需要深度自定义微前端体系 | **single-spa** | 灵活度最高，可完全控制 |

## 十一、总结：从普通工程到微前端需要添加的配置

### 11.1 主应用需要添加

1. 安装 `qiankun` 依赖
2. 创建子应用配置文件（`microApps.js`），声明每个子应用的 name / entry / container / activeRule
   - 若主应用使用 HashRouter，`activeRule` 必须用函数形式从 `location.hash` 中匹配
3. 调用 `registerMicroApps()` + `start()` 注册并启动（或使用 `loadMicroApp` 手动模式）
4. 在布局中提供子应用挂载容器 DOM 节点（`#sub-app-viewport`）
5. 添加路由占位符 `<Route path="qiankun/*" element={null} />`
6. 所有工程统一配置 `host: 'localhost.uat.com'`（需在系统 hosts 文件中添加映射）

### 11.2 Webpack 子应用需要添加（适用 C、D）

1. 新建 `public-path.js` 并在入口最前面引入
2. Webpack `output` 配置 `library` + `libraryTarget: 'umd'` + 唯一的 `chunkLoadingGlobal`
3. Webpack devServer 配置 `host` + CORS headers
4. 入口文件导出 `bootstrap` / `mount` / `unmount` 三个生命周期函数
5. 入口文件添加 `window.__POWERED_BY_QIANKUN__` 独立运行检测
6. 入口文件添加路由切换逻辑：qiankun 环境用 HashRouter（React）或 createMemoryHistory（Vue），独立运行用 BrowserRouter / createWebHistory

### 11.3 Vite + React 子应用需要添加（适用 B）

1. 安装 `vite-plugin-qiankun` 依赖
2. Vite 配置中引入 `qiankun()` 插件，传入子应用名称
3. Vite `server` 配置 `host` + `cors: true` + `origin` 固定地址
4. Vite `esbuild` 配置 `jsx: 'automatic'`（避免 react-refresh 与 qiankun 沙箱冲突）
5. 入口文件使用 `renderWithQiankun()` 注册生命周期 + `qiankunWindow.__POWERED_BY_QIANKUN__` 独立运行检测
6. 入口文件添加路由切换逻辑：qiankun 环境用 HashRouter + basename，独立运行用 BrowserRouter

### 11.4 Vite + Vue 子应用需要添加（适用 E）

1. 安装 `vite-plugin-qiankun` + `@vitejs/plugin-vue` 依赖
2. Vite 配置中引入 `vue()` 和 `qiankun()` 插件，传入子应用名称
3. Vite `server` 配置 `host` + `cors: true` + `origin` 固定地址
4. 不需要 `esbuild.jsx` 配置（Vue 不使用 JSX，无 react-refresh 冲突）
5. 入口文件使用 `renderWithQiankun()` 注册生命周期 + `qiankunWindow.__POWERED_BY_QIANKUN__` 独立运行检测
6. 路由配置中使用 `createMemoryHistory`（qiankun 环境）或 `createWebHistory`（独立运行），Vue 无法像 React 那样用 HashRouter + basename

## 十二、问题分析

### 12.1 子应用不用 hash 路由时的路由重复问题

**问题背景：** 如果子应用（如 sub-app-b 和 sub-app-c）不是 hash 路由，之前会出现主工程要在 `#` 前多添加一个对应路径才能正常访问。分析如果子应用不用 hash 路由，主应用应该如何修改才能解决路由重复问题。

当前主应用使用 HashRouter，子应用在 qiankun 环境下也使用 HashRouter + basename。如果子应用改用 BrowserRouter，子应用会从 `location.pathname` 读取路由，但主应用 HashRouter 下 `location.pathname` 始终为 `/`，导致子应用无法匹配内部路由。

之前出现的"主工程要在 `#` 前多添加路径"就是因为需要把子应用路径放到 pathname 中，形成类似 `http://localhost.uat.com:8080/qiankun/sub-b#/qiankun/sub-b` 的 URL 结构：pathname 部分给子应用 BrowserRouter 使用，hash 部分给主应用 HashRouter 使用，导致路径重复且维护困难。

**如果子应用不用 hash 路由，主应用的修改方案：**

主应用改用 BrowserRouter，`activeRule` 从函数匹配 hash 改为字符串匹配 pathname。URL 变为 `http://localhost.uat.com:8080/qiankun/sub-b`，不再需要 `#`，路径不再重复。但需要服务器配置 `historyApiFallback` 以支持前端路由的刷新访问。

---

### 12.2 主应用固定 HashRouter，子应用混合 Router 类型时的加载方案

**问题背景：** 如果有多个子应用，一些用 HashRouter，一些用 BrowserRouter，且主应用就需要使用 HashRouter 的情况下，有什么方法能让主应用同时都能加载对应的子应用。

主应用固定使用 HashRouter，URL 形如 `http://host:8080/#/qiankun/sub-b`，`location.pathname` 始终为 `/`。

#### HashRouter 子应用

通过 `basename` 从 hash 中剥离前缀，当前项目中 sub-app-b/c 已采用此方式，无需额外处理。

#### BrowserRouter 子应用

BrowserRouter 从 `location.pathname` 读取路由，而 HashRouter 下 pathname 恒为 `/`，子应用无法匹配内部路由。解决方案有以下几种：

**方案1：子应用在 qiankun 环境下切换为 MemoryRouter（推荐，简单）**

子应用检测 `__POWERED_BY_QIANKUN__`，为 true 时使用 MemoryRouter 替代 BrowserRouter，内部导航在内存中进行，不依赖 URL。这与 Vue 子应用使用 `createMemoryHistory` 的思路一致。缺点是子应用内部路由变化不会反映在浏览器地址栏，且存在路由参数获取问题（详见 12.4 节）。

**方案2：自定义 history 适配层（推荐，功能完整）**

使用 React Router 的基础 `<Router>` 组件，传入一个自定义 history 对象。该 history 对象从 `location.hash` 中解析出 pathname 部分（如从 `#/qiankun/sub-b/detail` 中提取 `/detail`），使 BrowserRouter 的路由逻辑能正常工作。此方案保留了 URL 驱动导航，实现详见 12.4 节。

**方案3：通过 qiankun props 传递自定义路由信息**

主应用在 `registerMicroApps` 时通过 props 传递当前路由路径，子应用 mount 时据此初始化 BrowserRouter 的 basename 和初始路径。但 BrowserRouter 仍依赖 `popstate` 事件，HashRouter 下 `popstate` 不会携带 hash 路由变化，需要额外监听 `hashchange` 事件做转换，实现较复杂。

**方案对比：**

| 方案 | 实现复杂度 | URL 反映路由 | 页面刷新保持 | 直接 URL 访问 |
|------|-----------|-------------|-------------|--------------|
| MemoryRouter | 低 | 否 | 否 | 否 |
| 自定义 history 适配层 | 中 | 是 | 是 | 是 |
| qiankun props 传递 | 高 | 否 | 需额外处理 | 需额外处理 |

---

### 12.3 UMD 打包格式的影响

**问题背景：** 子工程使用 webpack 打包时，添加打包产物导出为 UMD 格式，这个格式有什么影响，对之前没有添加 qiankun 子工程配置时的打包产物又有什么影响。

UMD（Universal Module Definition）格式不影响打包产物的运行逻辑。UMD 只是模块定义格式的包装层，它在不改变代码执行逻辑的前提下，将入口文件的 `exports` 挂载到全局变量 `window['sub-app-c']` 上，同时兼容 CommonJS 和 AMD 环境。

对比未添加 qiankun 配置时的普通打包产物：

| 对比维度 | 未添加 qiankun 配置 | 添加 UMD 配置后 |
|---------|--------------------|-----------------|
| 代码执行逻辑 | 不受影响 | 不受影响 |
| 模块导出方式 | IIFE，不暴露全局变量 | 入口 exports 挂载到 `window['sub-app-c']` |
| 独立运行 | 正常 | 正常（额外暴露的全局变量不影响功能） |
| chunk 加载全局变量 | 默认 `webpackJsonp` | 改为唯一名称 `webpackJsonp_sub_app_c` |

`chunkLoadingGlobal` 从默认的 `webpackJsonp` 改为唯一名称，仅影响 webpack 按需加载 chunk 时使用的全局队列变量名，不影响功能，但防止多个 webpack 子应用共存时队列互相覆盖。

**结论：** UMD 格式不影响最后打包产物的运行，只是 UMD 格式能将入口文件的 exports 挂到 window 上而已，供 qiankun 的 `import-html-entry` 拉取 JS 执行后读取生命周期函数。

### 12.4 部署后不同域名的跨域问题

**问题背景：** 当前在 qiankunProject 文件内，项目都是在 localhost 中运行，当主应用和子应用都部署后，如果域名不同，会出现跨域问题吗，主应用该如何访问子应用资源。

会产生跨域。主应用需要通过 `fetch` 拉取子应用的 entry HTML 和 JS 资源，当主应用和子应用域名不同时，浏览器同源策略会阻止跨域请求。

**开发环境的解决方式：**

| 构建工具 | 跨域配置 |
|---------|---------|
| Webpack | `devServer.headers: { 'Access-Control-Allow-Origin': '*' }` |
| Vite | `server.cors: true` |

**生产环境的解决方式：**

1. **子应用服务器配置 CORS**：在子应用的 Nginx 或服务器配置中，添加 CORS 响应头，允许主应用域名访问：
   ```
   # Nginx 配置示例
   add_header Access-Control-Allow-Origin "https://main-app.example.com";
   add_header Access-Control-Allow-Methods "GET, OPTIONS";
   ```

2. **Nginx 反向代理（推荐）**：通过 Nginx 反向代理，将子应用资源代理到主应用域名下的某个路径，浏览器看到所有请求来自同一域名，避免跨域：
   ```
   # Nginx 配置示例
   location /sub-app-b/ {
       proxy_pass http://sub-app-b-server:5174/;
   }
   ```
   主应用的 `entry` 配置改为 `//main-app.example.com/sub-app-b`。

### 12.5 HashRouter 与 BrowserRouter 的区别及切换影响

**问题背景：** HashRouter 和 BrowserRouter 有什么区别。如果子应用之前使用 BrowserRouter，要嵌到主应用中，能否直接修改为 HashRouter，修改后会出现什么影响。

#### HashRouter 与 BrowserRouter 的区别

| 对比维度 | HashRouter | BrowserRouter |
|---------|-----------|--------------|
| URL 形式 | `http://host/#/path` | `http://host/path` |
| 路由数据来源 | `location.hash` | `location.pathname` |
| 服务器配置 | 不需要（hash 不发送到服务器） | 需要 `historyApiFallback` 配置 |
| 刷新行为 | 正常（服务器返回 index.html，hash 由前端处理） | 需服务器配合，否则 404 |
| SEO | 不友好（搜索引擎可能忽略 hash 部分） | 友好 |
| History API | 使用 `hashchange` 事件 | 使用 `pushState`/`replaceState` + `popstate` 事件 |

#### 能否直接从 BrowserRouter 修改为 HashRouter

可以修改，但会出现以下影响：

1. **URL 结构变化**：所有路由 URL 从 `/detail/123` 变为 `/#/detail/123`，已有的外部链接、书签全部失效
2. **与主应用 hash 路由冲突**：主应用 HashRouter 已占用 hash（如 `#/qiankun/sub-b`），子应用也用 HashRouter 会争抢同一个 hash 空间。React 可通过 `basename` 从 hash 中剥离前缀解决，但 Vue 的 `createWebHashHistory` 无法做到
3. **路由参数获取**：从 `location.pathname` 改为从 `location.hash` 解析，原有的 `useParams`、`useSearchParams` 等仍可工作（React Router 内部处理），但第三方库直接读取 `location.pathname` 的会失效

#### 不修改 Router 类型嵌入子应用的方法

子应用使用 BrowserRouter，主应用使用 HashRouter，根本冲突在于 BrowserRouter 从 `location.pathname` 读取路由，而 HashRouter 下 pathname 恒为 `/`。在不修改双方 Router 类型的前提下，可通过以下方法实现嵌入：

**方法1：自定义 history 适配层（详细方案）**

`<BrowserRouter>` 本质上是 `<Router>` + `createBrowserHistory()` 的封装。自定义 history 适配层的思路是：创建一个自定义 history 对象，从 `window.location.hash` 中解析出 pathname 和 search，伪装成 browser history 的行为，让 React Router 的路由逻辑正常工作。

**URL 结构映射关系：**

主应用 HashRouter 下，浏览器 URL 为：
```
http://localhost.uat.com:8080/#/qiankun/sub-b/detail/123?foo=bar
```

需要映射为子应用 BrowserRouter 能理解的 location：
```
pathname: /detail/123
search: ?foo=bar
```

映射过程：从 `location.hash`（`#/qiankun/sub-b/detail/123?foo=bar`）中剥离主应用前缀 `/qiankun/sub-b`，剩余部分 `/detail/123?foo=bar` 拆分为 pathname 和 search。

**工作原理流程图：**

```
浏览器 URL: http://host:8080/#/qiankun/sub-b/detail/123?foo=bar
                                    ↑
                          location.hash = "#/qiankun/sub-b/detail/123?foo=bar"
                                    │
                          ┌─────────▼──────────┐
                          │  自定义 history 适配层 │
                          │                    │
                          │  1. 读取 hash       │
                          │  2. 剥离 basename   │
                          │  3. 拆分 pathname/search │
                          │  4. 返回 location 对象  │
                          └────────┬───────────┘
                                   │
                          location = {
                            pathname: "/detail/123",
                            search: "?foo=bar",
                            hash: ""
                          }
                                   │
                          ┌────────▼───────────┐
                          │  React Router v6    │
                          │  (BrowserRouter语义) │
                          │                    │
                          │  匹配 /detail/:id   │
                          │  useParams() => {id: "123"} │
                          │  useSearchParams() => {foo: "bar"} │
                          └────────────────────┘
```

反向操作（子应用内部导航时）：子应用调用 `history.push('/detail/123?foo=bar')`，适配层将其转换为 hash 格式 `#/qiankun/sub-b/detail/123?foo=bar`，更新 `window.location.hash`。

**实现步骤：**

第1步：创建自定义 history 对象

```js
// hashBridgeHistory.js
import { createBrowserHistory } from 'history';

/**
 * 创建 hash 桥接 history 对象
 * 从 location.hash 中解析路由信息，使 BrowserRouter 能在 HashRouter 主应用中工作
 * @param {string} basename - 子应用的基础路径，如 '/qiankun/sub-b'
 * @returns {History} 自定义 history 对象，接口与 createBrowserHistory 返回值一致
 */
function createHashBridgeHistory(basename) {
  // 创建底层 browser history，仅用于获取 listen 的基础能力
  const browserHistory = createBrowserHistory();

  /**
   * 从 location.hash 中解析出子应用的 location 对象
   * @returns {Object} location 对象 { pathname, search, hash, state, key }
   */
  function parseHashLocation() {
    // 获取当前 hash，如 "#/qiankun/sub-b/detail/123?foo=bar"
    const hash = window.location.hash || '#/';

    // 去掉 # 前缀
    const hashPath = hash.slice(1);

    // 剥离 basename 前缀，得到子应用路由部分
    // 如 "/qiankun/sub-b/detail/123?foo=bar" -> "/detail/123?foo=bar"
    let appPath = hashPath;
    if (hashPath.startsWith(basename)) {
      appPath = hashPath.slice(basename.length);
    }

    // 如果剥离后为空，则为根路径
    if (!appPath || appPath === '') {
      appPath = '/';
    }

    // 拆分 pathname 和 search
    const searchIndex = appPath.indexOf('?');
    let pathname = appPath;
    let search = '';
    if (searchIndex !== -1) {
      pathname = appPath.slice(0, searchIndex);
      search = appPath.slice(searchIndex);
    }

    return {
      pathname,
      search,
      hash: '',
      state: browserHistory.location.state,
      key: browserHistory.location.key,
    };
  }

  // 自定义 history 对象
  const customHistory = {
    // 当前 location（从 hash 解析）
    get location() {
      return parseHashLocation();
    },
    // 监听路由变化（监听 hashchange 事件）
    listen(callback) {
      return browserHistory.listen(() => {
        callback({ location: parseHashLocation(), action: 'POP' });
      });
    },
    // 导航：将子应用路径转换为完整 hash 路径后更新
    push(to) {
      const path = typeof to === 'string' ? to : to.pathname;
      const fullPath = basename + path;
      window.location.hash = '#' + fullPath;
    },
    // 替换：与 push 类似但使用 replaceState
    replace(to) {
      const path = typeof to === 'string' ? to : to.pathname;
      const fullPath = basename + path;
      const url = window.location.origin + window.location.pathname + '#' + fullPath;
      window.history.replaceState(null, '', url);
    },
    createHref(to) {
      const path = typeof to === 'string' ? to : to.pathname;
      return '#' + basename + path;
    },
    action: 'POP',
  };

  return customHistory;
}

export default createHashBridgeHistory;
```

第2步：子应用入口文件使用自定义 history

```jsx
// main.jsx（子应用入口）
import { unstable_HistoryRouter } from 'react-router-dom';
import createHashBridgeHistory from './hashBridgeHistory';

/**
 * 渲染子应用到挂载节点
 * @param {Object} props qiankun 注入的参数对象（可含 container 等）
 */
function render(props) {
  // 解构出挂载容器
  const { container } = props || {};
  // 被主应用加载时用 container 内的 #root，独立运行时用 document 的 #root
  const mountNode = container ? container.querySelector('#root') : document.getElementById('root');
  const isQiankun = window.__POWERED_BY_QIANKUN__;

  if (isQiankun) {
    // qiankun 环境：使用自定义 hash 桥接 history
    // Router 类型不变（仍是 browser history 语义），仅改变 location 数据来源
    const history = createHashBridgeHistory('/qiankun/sub-b');
    root = createRoot(mountNode);
    root.render(
      <unstable_HistoryRouter history={history}>
        <App />
      </unstable_HistoryRouter>
    );
  } else {
    // 独立运行：正常使用 BrowserRouter
    root = createRoot(mountNode);
    root.render(
      <BrowserRouter basename="/">
        <App />
      </BrowserRouter>
    );
  }
}
```

**数据流完整路径：**


#### MemoryRouter 的路由参数问题

子应用在 qiankun 环境下使用 MemoryRouter，独立运行使用 BrowserRouter，存在以下路由参数获取问题：

**1. 页面刷新丢失内部路由状态**

MemoryRouter 的路由历史保存在 JavaScript 内存中，不写入 URL。当用户在子应用内部导航到 `/detail/123` 后刷新页面，MemoryRouter 会重新初始化，回到初始路由，`/detail/123` 的状态丢失。用户无法通过刷新保持当前页面。

**2. 直接访问 / 书签 / 分享链接失效**

MemoryRouter 下子应用内部的路由不反映在浏览器 URL 中，用户无法直接通过 URL 访问子应用的某个具体页面。例如无法通过 `http://host/#/qiankun/sub-b/detail/123` 直接打开详情页，刷新后只会回到子应用首页。

**3. 浏览器前进 / 后退按钮异常**

MemoryRouter 维护自己的内存历史栈，与浏览器的历史栈不同步。浏览器的前进/后退按钮不会触发子应用内部的路由变化，用户体验不一致。

**4. 查询参数获取问题**

主应用通过 hash URL 传递查询参数（如 `#/qiankun/sub-b?source=main-app&userId=1001`），BrowserRouter 环境下子应用可从 `location.search` 读取。但 MemoryRouter 下 `location.search` 为空，需要从 qiankun props 或手动解析 hash 中提取参数。

**结论：** MemoryRouter 适合子应用内部路由层级浅、不需要直接 URL 访问的场景。对于需要保持 URL 状态、支持刷新和分享的子应用，MemoryRouter 存在实际问题，应优先考虑自定义 history 适配层方案。

#### 方案优缺点对比

| 对比维度 | MemoryRouter 方案 | 自定义 history 适配层方案 |
|---------|-------------------|--------------------------|
| 实现复杂度 | 低（直接替换组件） | 中（需实现自定义 history） |
| URL 反映子应用路由 | 否（内存中，不可见） | 是（反映在 hash 中） |
| 页面刷新保持路由 | 否（回到初始页） | 是（从 hash 重新解析） |
| 直接 URL 访问 | 否 | 是 |
| 浏览器前进/后退 | 不同步（内存栈与浏览器栈不一致） | 同步（hash 变化驱动浏览器历史） |
| 分享链接 | 否 | 是 |
| 查询参数 | 需手动从 props/hash 提取 | 自动解析（useSearchParams 正常工作） |
| 子应用代码改动 | 需切换 Router 组件 | 入口文件替换 history 即可，路由定义不变 |

### 12.6 bootstrap 和 mount 生命周期函数的触发顺序

**问题背景：** 子应用中的 bootstrap 和 mount 生命周期函数，哪个先触发。

**bootstrap 先触发。**

qiankun 的生命周期调用顺序如下：

| 阶段 | 调用时机 | 调用次数 | 作用 |
|------|---------|---------|------|
| `bootstrap` | 子应用**首次**被激活时调用 | 整个生命周期仅一次 | 初始化全局配置，通常只做 console 或空操作 |
| `mount` | 每次**进入**子应用路由时调用 | 每次进入都调用 | 创建应用实例并渲染到 container 内的 DOM 节点 |
| `unmount` | 每次**离开**子应用路由时调用 | 每次离开都调用 | 销毁应用实例，清理 DOM 和事件，释放内存 |

**首次进入子应用的执行流程：**

```
用户首次访问 /qiankun/sub-b
    │
    ├── qiankun 匹配 activeRule
    ├── fetch 子应用 entry HTML
    ├── 解析 HTML，加载 JS 资源
    ├── 执行子应用 JS，注册生命周期函数
    │
    ├── 调用 bootstrap()    ← 首次且仅此一次
    ├── 调用 mount(props)   ← 每次进入都调用
    └── 子应用渲染到 #sub-app-viewport
```

**后续再次进入子应用的执行流程：**

```
用户离开 /qiankun/sub-b，再次访问 /qiankun/sub-b
    │
    ├── 离开时调用 unmount(props)
    ├── 再次进入时 qiankun 匹配 activeRule
    │
    ├── 跳过 bootstrap()    ← 不再调用（仅首次一次）
    ├── 调用 mount(props)   ← 每次进入都调用
    └── 子应用渲染到 #sub-app-viewport
```
