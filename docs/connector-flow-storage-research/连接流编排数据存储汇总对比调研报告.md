# 连接流编排数据存储汇总对比调研报告

**版本**：V1.0
**日期**：2026年5月
**调研范围**：Zapier / Make / Power Automate / MuleSoft / 钉钉

---

## 一、执行摘要

本报告对 **Zapier、Make、Power Automate、MuleSoft、钉钉** 五个平台的连接流编排数据存储方案进行了深度调研，从**编排定义存储、运行时状态存储、执行历史存储、凭证存储**四个维度进行对比分析，提炼适用于我们的连接器平台数据存储设计的最佳实践。

### 核心结论

| 对比维度 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|---------|--------|------|---------------|----------|------|
| **编排定义格式** | JSONB | JSONB | Workflow JSON | XML | JSON TEXT |
| **编排模型** | 线性+嵌套分支 | DAG（模块+路由） | 线性+作用域 | 图（Flow+SubFlow） | 线性+有限分支 |
| **定义存储位置** | PostgreSQL JSONB | PostgreSQL JSONB | Dataverse/Blob | 文件系统/Git | MySQL TEXT |
| **运行时状态** | Redis + MySQL | Redis + Firestore | Cosmos DB + Blob | ObjectStore + 可选DB | Redis + MySQL |
| **执行历史** | PostgreSQL + S3 | PostgreSQL + BigQuery + GCS | Cosmos DB + Blob | 自建/CloudHub | MySQL + OSS |
| **凭证加密** | AES-256 + KMS | AES-256 + KMS | Azure Key Vault | Secure Properties / Vault | 阿里云 KMS |
| **版本管理** | 自动快照 | 场景克隆 | Solution ALM | Git + Maven | 发布版本 |
| **内置存储** | 无 | Data Store（Firestore） | 无 | ObjectStore | 无 |
| **图数据库** | 不需要 | 不需要 | 不需要 | 不需要 | 不需要 |

## 二、编排定义存储对比

### 2.1 编排模型对比

| 模型维度 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|---------|--------|------|---------------|----------|------|
| **数据结构** | 有序 Step 列表 | 模块数组 + 独立路由表 | 嵌套 Actions 对象 | Flow XML 元素树 | 有序 Step 列表 |
| **分支表示** | Paths 嵌套步骤 | Router 模块 + 独立 routes | Condition/Switch 嵌套 | choice 元素分支 | condition 步骤嵌套 |
| **循环表示** | Looping 步骤（有限） | Iterator + Aggregator 模块 | Apply_to_each 嵌套 | ForEach 元素 | 有限支持 |
| **错误处理** | 步骤内重试配置 | Error Handler 独立模块 | runAfter + Scope | error-handler 元素 | 步骤内配置 |
| **参数引用** | `source + field` | 表达式 + 模块路径 | `@{outputs()}` 表达式 | DataWeave / MEL | `source + path / template` |
| **Schema 需求** | 无严格 Schema | Data Structure 可选 | JSON Schema 定义 | 强类型 Type System | 字段映射配置 |

### 2.2 定义存储技术对比

| 技术维度 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|---------|--------|------|---------------|----------|------|
| **数据库** | PostgreSQL | PostgreSQL | Dataverse（Azure SQL） | 文件系统 / Git / 可选DB | RDS MySQL |
| **JSON 支持** | JSONB | JSONB | 大文本字段 / Blob | 不适用（XML） | TEXT / LONGTEXT |
| **大字段处理** | JSONB 字段内 | JSONB 字段内 | Blob 外置 + Link 引用 | 独立 .dwl 文件 | TEXT 字段内 / OSS 外置 |
| **Schema 灵活性** | 极高（无 Schema 约束） | 高（可定义 Data Structure） | 中（需定义操作 Schema） | 高（XML Schema + Java Type） | 高（无 Schema 约束） |
| **查询能力** | JSONB 索引查询 | JSONB 索引查询 | 元数据字段查询 | Git grep / XPath | 仅元数据查询 |
| **扩展性** | PostgreSQL 分片 | PostgreSQL 分片 | Dataverse 横向扩展 | 文件系统 + Git LFS | MySQL 分库分表 |

### 2.3 推荐结论：定义存储选型

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **MVP 快速验证** | PostgreSQL JSONB | 灵活，无需预定义 Schema，查询能力强 |
| **企业级 CI/CD** | 文件系统 + Git（MuleSoft 模式） | 源码管理友好，支持代码审查和 CI/CD |
| **大规模 SaaS** | JSONB + 按 Tenant 分区 | 关系数据库成熟，运维成本低 |
| **内聚型平台** | MySQL TEXT（钉钉模式） | 架构简单，内聚型场景查询复杂度有限 |

## 三、运行时状态存储对比

### 3.1 核心组件对比

| 运行时组件 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|-----------|--------|------|---------------|----------|------|
| **执行队列** | Redis（消息队列） | Redis Pub/Sub | Azure Service Bus | VM / JMS / Anypoint MQ | RocketMQ |
| **并发控制** | Redis 分布式锁 | Redis 场景级锁 | Redis Cache 锁 | 线程池 + 集群锁 | Redis 分布式锁 |
| **轮询游标** | Redis + PostgreSQL | Data Store | Blob + Table | ObjectStore | MySQL |
| **Webhook 路由** | Redis + PostgreSQL | Redis + PostgreSQL | API Management | HTTP Listener | Stream / HTTP |
| **内置存储** | 无 | Data Store（Firestore） | 无 | ObjectStore | 无 |
| **缓存** | Redis | Redis | Redis Cache | Hazelcast | Redis |

### 3.2 内置数据存储方案对比

| 能力维度 | Make Data Store | MuleSoft ObjectStore | Zapier/Power Automate/钉钉 |
|---------|----------------|---------------------|--------------------------|
| **存储引擎** | Firestore（NoSQL） | 内存 / Hazelcast / DB | 无内置存储 |
| **访问方式** | Data Store 模块 CRUD | Java API / Mule 组件 | 依赖外部存储 |
| **跨场景共享** | 同组织内共享 | 同应用内共享 | — |
| **容量限制** | 按订阅计划（1MB-500MB+） | CloudHub: 10MB/应用 | — |
| **数据持久化** | 永久（Firestore 托管） | 可选持久化到数据库 | — |
| **典型用途** | 增量游标/去重/状态共享 | 游标/去重/临时缓存 | — |

### 3.3 推荐结论：运行时状态存储

| 需求 | 推荐方案 | 理由 |
|------|---------|------|
| **轻量级（MVP）** | Redis + PostgreSQL | 成熟方案，运维成本低 |
| **大规模生产** | Redis + 持久化（可选 MySQL/TiDB） | Redis 承载高吞吐，持久化保障不丢数据 |
| **内置存储** | 加分项非必选项（借鉴 Make Data Store） | 增量/去重是刚需，内置可降低外部依赖 |

## 四、执行历史存储对比

### 4.1 存储架构对比

| 架构维度 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|---------|--------|------|---------------|----------|------|
| **热存储** | PostgreSQL | PostgreSQL | Cosmos DB | 自建 / CloudHub | MySQL |
| **温存储** | — | BigQuery | — | ELK / Splunk | — |
| **冷存储** | S3 | GCS | Blob Storage | 自建归档 | OSS |
| **I/O 外置** | S3（大字段） | GCS（大执行） | Blob（全部 I/O 默认外置） | 日志系统配置 | OSS（大数据） |
| **数据量级** | 月均百万级 | 月均百万级 | 月均千万级 | 企业级万级 | 月均万级 |
| **保留期** | 1-6月（按计划） | 30-90天（按计划） | 28天（可延长） | 可配置 | 30天 |

### 4.2 关键设计差异

| 设计决策 | Power Automate（推荐模式） | Zapier/Make（常见模式） | 说明 |
|---------|--------------------------|----------------------|------|
| **I/O 存储** | **默认外置**到 Blob | 混合（小字段内嵌 + 大字段外置） | 默认外置模式更简洁，避免单行数据膨胀 |
| **数据引用** | URI Link + ContentVersion | 内嵌 + 条件外置 | Link 模式天然支持大字段，ContentVersion 保证一致性 |
| **分片策略** | Cosmos DB 自动分片 | PostgreSQL 按时间分区 | 自动分片降低运维复杂度 |
| **计量数据** | 内置 billingMetrics | 需额外统计 | 从第一天内置计费字段可避免后期加字段的迁移成本 |

### 4.3 推荐结论：执行历史存储

| 层次 | 存储选型 | 保留期 | 用途 |
|------|---------|--------|------|
| **热数据** | PostgreSQL（按日期分区） | 7-30天 | 近期运行查询、重试操作 |
| **温数据** | ClickHouse / TiDB | 1-6个月 | 报表分析、趋势查询 |
| **冷数据** | 对象存储（S3/OSS/MinIO） | 6个月-永久 | 合规审计、历史回溯 |
| **I/O 快照** | 对象存储（引用机制） | 同对应执行历史 | 步骤输入/输出详情 |

## 五、凭证与安全存储对比

### 5.1 加密方案对比

| 安全维度 | Zapier | Make | Power Automate | MuleSoft | 钉钉 |
|---------|--------|------|---------------|----------|------|
| **加密算法** | AES-256-GCM | AES-256-GCM | AES-256 | AES-256 | AES-256 |
| **密钥管理** | AWS KMS | GCP Cloud KMS | Azure Key Vault | Anypoint Vault / 外部KMS | 阿里云 KMS |
| **传输加密** | TLS 1.2+ | TLS 1.2+ | TLS 1.2+ | TLS 1.3 | TLS 1.2+ |
| **字段级加密** | 仅凭证字段 | 仅凭证字段 | 仅凭证字段 | 支持任意属性加密 | 凭证字段 |
| **加密上下文** | 绑定 Connection | 绑定 Connection | 绑定 Environment | 绑定 Application | 绑定企业 |
| **审计日志** | 基础审计 | 标准审计 | Office 365 审计 | 深度审计 | 标准审计 |

### 5.2 最佳实践总结

| 最佳实践 | 说明 | 优先级 |
|---------|------|--------|
| **KMS 托管密钥** | 使用云 KMS 而非内置密钥，支持密钥轮换和审计 | P0 |
| **加密上下文隔离** | 每个 Connection 的加密上下文不同，避免跨租户破解 | P0 |
| **字段级加密** | 仅加密敏感字段，明文字段正常索引和查询 | P0 |
| **TLS 传输加密** | 所有 API 通信强制 TLS 1.2+ | P0 |
| **Token 自动刷新** | 持久化 Refresh Token，运行时自动获取新 Access Token | P0 |
| **密钥定期轮换** | KMS 支持自动密钥轮换（推荐 1 年） | P1 |
| **执行历史脱敏** | 执行历史中凭证和敏感数据自动脱敏 | P1 |

## 七、编排定义 Schema 设计建议

基于五个平台的调研，我们的连接流编排定义建议采用以下数据模型：

### 7.1 核心实体关系

```
ConnectionFlow (连接流)
├── id: UUID
├── name: string
├── description: string
├── status: draft | published | disabled
├── version: int
├── trigger: Trigger (JSONB)
├── nodes: Node[] (JSONB)  ← 核心编排定义
└── metadata: { createdBy, createdAt, updatedAt, ... }

Node (编排节点)
├── id: string (uuid)
├── type: trigger | connector_action | condition | loop | data_transform | error_handler | sub_flow | flow_exit
├── config: JSONB (节点类型相关配置)
├── connections: { next: string[], error: string[] }  ← 出边
└── position: { x, y } (可视化编排坐标)

Trigger (入口节点)
├── type: manual | schedule | webhook | event
└── config: JSONB (不同类型配置不同)

Connection (连接器实例)
├── id: UUID
├── connector_key: string
├── credential_id: UUID
└── params: JSONB

FlowExecution (执行记录)
├── id: UUID
├── flow_id: UUID
├── flow_version: int
├── status: running | success | failed | partial
├── trigger_data: JSONB (快照)
├── node_results: ExecutionNodeResult[] (JSONB)
├── started_at, completed_at, duration_ms
└── billing: { operations_consumed, ... }
```

### 7.2 存储技术选型建议

| 存储对象 | 技术选型 | 理由 |
|---------|---------|------|
| 编排定义 | PostgreSQL JSONB | 灵活、可查询、事务保障 |
| 执行历史 | PostgreSQL（热） + 对象存储（冷） | 冷热分离控制成本 |
| 运行时状态 | Redis + PostgreSQL | 高吞吐 + 持久化 |
| 凭证 | PostgreSQL 加密字段 + KMS | 成熟可靠 |
| 执行队列 | Redis Stream / 消息队列 | 异步解耦 Worker |
| 版本快照 | 同编排定义表 + version 字段 | 简单统一 |

### 7.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **编排定义格式** | JSON（非 XML） | 开发效率高，JS/TS 原生支持 |
| **编排模型** | 有向图（Nodes + Connections） | DAG 表达能力最强，覆盖所有编排场景 |
| **节点类型** | 枚举扩展（非可插拔） | MVP 确定 7 种节点类型，后续可扩展 |
| **定义存储** | 单一 JSONB 字段（非分表） | MVP 定义规模有限，单字段减少 JOIN 查询 |
| **版本管理** | flow_id + version 复合主键 | 简单可靠，每个版本存储完整定义快照 |
| **I/O 存储** | 默认外置对象存储 | 避免单行数据膨胀，简化数据库运维 |

## 九、MVP 存储方案核心建议

基于调研结论，对我们的连接器平台 MVP 阶段数据存储设计建议：

| 序号 | 建议 | 来源借鉴 | 说明 |
|------|------|---------|------|
| 1 | 编排定义使用 PostgreSQL JSONB | Zapier/Make | 灵活、成熟、低运维成本 |
| 2 | 编排模型采用有向图（Nodes + Connections） | Make/MuleSoft | DAG 覆盖线性/分支/循环/并行全部场景 |
| 3 | 执行历史采用冷热分离 | 全部平台共识 | 热 PostgreSQL + 冷对象存储 |
| 4 | 凭证使用字段级加密 + KMS | 全部平台共识 | 安全基线，无替代方案 |
| 5 | 执行历史 I/O 默认外置 | Power Automate | 避免单行数据膨胀 |
| 6 | 内置版本快照 + 一键回滚 | Zapier | MVP 不引入 Git，版本快照最轻量 |
| 7 | 运行时状态用 Redis + 持久化 | 全部平台共识 | Redis 承担高吞吐，持久化保障不丢数据 |
| 8 | 参数映射支持表达式和模板 | Make/Power Automate | 声明式映射是编排配置的核心表达能力 |
| 9 | 节点类型枚举预定义 7 种 | 全部平台 | 覆盖 trigger/action/condition/loop/data_transform/error_handler/sub_flow/flow_exit |
| 10 | 预留计量字段（operations_consumed） | Make/Power Automate | 从第一天设计，避免后期加字段的迁移成本 |
