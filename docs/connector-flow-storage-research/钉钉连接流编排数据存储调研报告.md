# 钉钉连接流编排数据存储调研报告

**版本**：V1.0
**日期**：2026年5月
**调研维度**：连接流编排数据存储（编排定义存储 / 运行时状态 / 执行历史 / Stream 模式）

---

## 一、概述

钉钉连接平台是**内聚型连接平台**的典型代表，其连接流（Connection Flow）编排能力是钉钉低代码生态的重要组成部分。与 Zapier/Make 等中立 iPaaS 不同，钉钉的连接流深度绑定钉钉自身能力（审批、IM、通讯录、考勤等），外部系统通过**预置连接器**和**自定义连接器**接入。钉钉连接流的数据存储依托阿里云基础设施（RDS MySQL、Redis、OSS、Tablestore），整体架构强调**高可用**和**内网低延迟**。

## 二、编排定义存储（连接流配置存储）

### 2.1 核心实体模型

| 实体 | 职责 | 存储位置 |
|------|------|---------|
| **连接流（Flow）** | 连接流整体定义 | RDS MySQL（核心表） |
| **触发器（Trigger）** | 流入口配置 | MySQL + JSONB 字段 |
| **动作（Action）** | 流节点配置 | MySQL + JSONB 字段 |
| **条件（Condition）** | 分支逻辑配置 | 嵌套在 Action 的 JSON 中 |
| **连接凭证** | 认证信息 | MySQL（加密字段）+ KMS |
| **连接器定义** | 预置/自定义连接器元数据 | MySQL + OSS（Schema 大字段） |
| **宜搭集成** | 宜搭表单触发配置 | 宜搭数据源 + 连接流引用 |

### 2.3 钉钉特有实体

| 实体 | 说明 | 存储特点 |
|------|------|---------|
| **审批流定义** | 钉钉原生审批流程模板 | 独立审批引擎，通过事件与连接流联动 |
| **互动卡片** | 钉钉消息卡片配置 | 卡片模板存储在 OSS，连接流中引用卡片 ID |
| **Stream 订阅** | WebSocket 长连接订阅关系 | Redis 管理活跃连接，MySQL 持久化订阅配置 |
| **机器人配置** | 机器人消息收发配置 | 机器人配置表，关联连接流触发器 |

### 2.4 存储技术

| 存储对象 | 技术选型 | 说明 |
|---------|---------|------|
| 连接流元数据 | RDS MySQL | 核心表，含基本信息和版本 |
| 连接流定义 JSON | RDS MySQL TEXT 字段 | 完整编排定义以 JSON 存入 TEXT 字段 |
| 连接器定义 | RDS MySQL + OSS | 元数据在 MySQL，大字段 Schema 在 OSS |
| 审批流模板 | 独立审批数据库 | 与连接流数据库分离 |
| 互动卡片模板 | OSS | 卡片 JSON 模板存储在 OSS |
| Stream 订阅 | Redis + MySQL | 活跃连接在 Redis，持久化配置在 MySQL |
| 版本快照 | RDS MySQL | 每次发布生成版本记录 |

### 2.5 与 Zapier/Make 的关键差异

| 维度 | Zapier/Make | 钉钉连接流 |
|------|------------|-----------|
| **编排复杂度** | 支持 DAG、Router、Iterator | 线性为主，有限条件分支 |
| **定义存储** | JSONB 在 PostgreSQL | JSON TEXT 在 MySQL |
| **触发器** | Webhook/轮询为主 | 深度绑定钉钉事件（审批/通讯录/消息） |
| **外部系统** | 标准化接入 | 通过预置/自定义连接器 |
| **编排能力定位** | 通用 iPaaS 完整编排 | 轻量级场景化编排 |

## 三、运行时状态存储

### 3.1 执行调度模型

| 调度类型 | 存储方式 | 说明 |
|---------|---------|------|
| **事件触发** | RocketMQ（阿里云消息队列） | 钉钉事件推送到 RocketMQ，连接流消费 |
| **Stream 触发** | Redis Pub/Sub | WebSocket 消息通过 Redis 分发到连接流 Worker |
| **定时触发** | MySQL + 定时任务 | Cron 表达式存储在连接流定义中，调度器轮询执行 |
| **手动触发** | API 触发 | 用户点击触发，Worker 即时消费 |

### 3.2 执行上下文

钉钉连接流执行时上下文信息包括：

| 上下文 | 来源 | 说明 |
|--------|------|------|
| 组织 ID | 触发上下文 | 钉钉企业组织标识 |
| 用户 ID | 触发上下文 | 操作人或事件关联人 |
| 应用信息 | 触发上下文 | 触发源的钉钉应用信息 |
| 事件数据 | 触发器输出 | 钉钉事件的结构化数据 |
| 步骤间数据 | 流程上下文 | 上游步骤输出的引用 |
| 凭证 Token | 连接管理 | 运行时从加密存储解密加载 |

### 3.3 失败重试

| 重试层级 | 策略 | 存储机制 |
|---------|------|---------|
| 步骤重试 | 最多 3 次，指数退避 | Redis 记录重试计数 + MySQL 持久化 |
| 流级重试 | 整体重试或跳过失败步骤 | 流执行状态表记录重试策略 |
| 最终失败 | 记录错误，触发告警 | MySQL 错误记录 + IM 告警推送 |

## 四、执行历史存储

### 4.1 执行记录结构

```
flow_execution = {
  "executionId": "exec_xyz789",
  "flowId": "flow_abc123",
  "flowVersion": 3,
  "status": "success" | "failed" | "running" | "partial_failed",
  "trigger": {
    "type": "event",
    "eventType": "approval.finished",
    "sourceData": { /* 触发数据快照（含敏感字段脱敏） */ }
  },
  "startedAt": "2026-05-14T10:00:00Z",
  "completedAt": "2026-05-14T10:00:03Z",
  "durationMs": 3456,
  "stepResults": [
    {
      "stepId": "step_1",
      "type": "action",
      "connector": "dingtalk.im",
      "status": "success",
      "input": { /* 输入参数快照（脱敏） */ },
      "output": { /* 输出结果 */ },
      "error": null,
      "durationMs": 1200,
      "retryCount": 0
    }
  ],
  "errorSummary": null,
  "executedBy": "auto" | "user_123"
}
```

### 4.2 存储策略

| 存储对象 | 技术选型 | 保留策略 |
|---------|---------|---------|
| 执行历史元数据 | RDS MySQL | 30 天（连接流标准版） |
| 步骤输入/输出快照 | RDS MySQL + OSS | 小数据在 MySQL，大数据在 OSS |
| 运行日志 | 阿里云日志服务（SLS） | 30 天，可投递到 OSS 长期保存 |
| 统计聚合 | RDS MySQL（聚合表） | 按月汇总，永久保留 |
| 失败详情 | RDS MySQL + SLS | 同执行历史保留期 |

### 4.3 数据量参考

| 指标 | 典型值 | 存储策略 |
|------|-------|---------|
| 单次执行记录 | ~2-8 KB（简单流程） | MySQL 行存储 |
| 含大数据的步骤 | ~100-500 KB | 输出外置到 OSS，MySQL 存引用 |
| 企业月均执行量 | ~1000-50000 次 | MySQL 分区表按时间自动分片 |

## 五、Stream 模式下的状态存储

钉钉的 **Stream 模式**（WebSocket 长连接）是其独特的事件推送机制，相关状态存储：

| 状态 | 存储 | 说明 |
|------|------|------|
| 连接状态 | Redis | 活跃连接的心跳检测、断线重连状态 |
| 订阅关系 | Redis + MySQL | Redis 快速路由，MySQL 持久化订阅配置 |
| 消息缓冲 | Redis List | 消息队列缓冲，保证消息不丢失 |
| 重连偏移 | Redis + MySQL | 记录最后成功处理的消息偏移量 |
| 消费进度 | Redis | Stream 连接的消费进度追踪 |

## 六、连接凭证存储

| 凭证类型 | 加密方案 | 存储位置 |
|---------|---------|---------|
| OAuth 2.0 Token | 阿里云 KMS（AES-256）+ 应用层加密 | MySQL `credentials` 表加密字段 |
| API Key | 阿里云 KMS + 应用层加密 | MySQL 加密字段 |
| 钉钉 AccessToken | 运行时动态获取（不持久化） | Redis 缓存（2 小时 TTL） |
| 企业 Corp Secret | 阿里云 KMS 加密 | 企业配置表加密存储 |

## 七、关键设计模式总结

| 设计模式 | 钉钉做法 | 对我们的启示 |
|---------|---------|------------|
| **内聚型事件驱动** | 连接流触发器深度绑定钉钉内部事件 | 内聚型平台的编排定义天然与平台事件耦合 |
| **流定义 JSON** | 完整编排定义以 JSON 存入 MySQL TEXT | 简单场景下 JSON TEXT 比 JSONB 更轻量，但查询能力弱 |
| **Stream 双通道** | Redis + MySQL 双写维护 Stream 连接状态 | 实时状态用 Redis，持久化配置用 MySQL 是标准模式 |
| **审批流 + 连接流协同** | 审批流独立引擎，通过事件联动连接流 | 多引擎协作时，事件是引擎间的唯一耦合点 |
| **配置脱敏存储** | 执行历史中输入参数进行脱敏处理 | 执行历史必须支持字段级别的脱敏策略 |
| **轻量编排** | 线性为主，条件分支有限 | 简单场景不需要图数据库，JSON 序列化即可满足 |
