# 钉钉连接器运行时调度逻辑及技术框架调研报告

**版本**：V1.0
**日期**：2026年5月
**调研维度**：连接器平台运行时调度逻辑及技术框架（触发模型 / 任务队列 / 执行引擎 / 工作节点 / 错误编排 / 框架选型 / 高可用）

---

## 一、概述

钉钉连接平台作为**内聚型连接平台**的代表，其运行时调度架构与 Zapier/Make 等中立 iPaaS 有本质差异。钉钉的连接流引擎深度绑定钉钉自身能力（审批、IM、通讯录、考勤等），调度体系以**阿里云基础设施**为底座（RocketMQ、Tablestore、Redis、OSS），强调**内网低延迟**和**与钉钉生态的原生联动**。与商业 iPaaS 不同，钉钉的编排调度能力相对轻量——连接流主要处理线性 + 简单条件分支的场景，复杂流程依赖独立的审批流引擎。

## 二、触发模型

### 2.1 触发类型

| 类型 | 机制 | 延迟 | 适用场景 |
|------|------|------|---------|
| **事件触发** | 钉钉内部事件 → RocketMQ → 连接流消费 | 秒级 | 审批完成、通讯录变更、消息接收 |
| **Webhook 触发** | 外部 HTTP 请求 | 秒级 | 外部系统事件推送 |
| **定时触发** | Cron 表达式调度 | 分钟级 | 每日同步、定时报告 |
| **Stream 触发** | WebSocket 长连接事件推送 | 毫秒级 | 实时消息推送、低延迟场景 |
| **手动触发** | 用户点击执行 | 即时 | 调试测试 |
| **机器人触发** | 群内交互卡片按钮 | 秒级 | 审批操作、命令交互 |

### 2.2 事件触发架构（核心模式）

钉钉的连接流**深度依赖钉钉内部事件**作为触发源，这是内聚型平台的典型特征：

```
钉钉业务系统 → 事件总线 → RocketMQ Topic → 连接流 Worker → 执行连接流
```

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **事件源** | 钉钉内部事件总线 | 50+ 事件类型（审批/通讯录/消息/考勤/日程等） |
| **消息中间件** | RocketMQ | 高吞吐、低延迟、消息持久化 |
| **事件过滤** | 连接流配置中的 eventFilter | 按事件类型、字段值过滤 |
| **消费确认** | RocketMQ ACK 机制 | 消费成功才确认，保证至少一次投递 |

### 2.3 Stream 模式（WebSocket 长连接）

钉钉独特的 **Stream 模式** 是专门为内网系统设计的低延迟触发方案：

```
钉钉事件 → Stream Server → WebSocket 推送 → Consumer 内网应用
```

| 组件 | 说明 |
|------|------|
| **连接建立** | Consumer 通过 SDK 建立 WebSocket 连接到钉钉 Stream Server |
| **心跳** | 每 30 秒心跳检测，断线自动重连 |
| **消息格式** | JSON 格式，包含事件类型和数据 |
| **ACK 确认** | Consumer 处理完后发送 ACK，未 ACK 的消息重新投递 |
| **重连偏移** | 记录最后成功处理的消息偏移量，重连后从断点继续消费 |

### 2.4 定时触发

| 属性 | 说明 |
|------|------|
| **调度存储** | 连接流定义的 `trigger.config` 中 Cron 表达式 |
| **调度器** | 基于 Quartz（可嵌入在连接流服务中） |
| **精度** | 分钟级（最小间隔 1 分钟） |
| **调度方式** | 调度器定期扫描 MySQL 中待触发的定时连接流 |

## 三、任务队列架构

### 3.1 消息与队列

| 层级 | 组件 | 说明 |
|------|------|------|
| **事件入口** | RocketMQ | 接收钉钉内部事件，按 Topic 分发 |
| **执行队列** | Redis List | 待执行的连接流任务队列，Worker 消费 |
| **延迟队列** | RocketMQ（定时消息） | 需要延迟执行的任务，利用 RocketMQ 定时消息 |
| **死信队列** | RocketMQ DLQ | 超过最大重试次数的消息 |

### 3.2 任务模型

```
ExecutionTask = {
  executionId: string,
  flowId: string,
  flowVersion: number,
  triggerType: "event" | "webhook" | "schedule" | "manual",
  triggerData: any,
  stepIndex: number,     // 当前执行步骤
  retryCount: number,
  context: {
    orgId: string,        // 钉钉企业 ID
    userId: string,        // 触发用户
    appInfo: any,          // 触发应用信息
  },
  createdAt: timestamp
}
```

钉钉的任务粒度为**单步骤级别**，与 Zapier 类似。

### 3.3 并发控制

| 级别 | 机制 |
|------|------|
| **连接流级别** | Redis 分布式锁，同一连接流同时最多 1 次执行 |
| **事件级别** | RocketMQ 消费组内负载均衡，消息按 flowId 分区保证顺序 |
| **企业级别** | 按钉钉企业 ID 的资源配额控制总并发 |

## 四、执行引擎

### 4.1 引擎架构

| 维度 | 钉钉方案 | 说明 |
|------|---------|------|
| **引擎类型** | 分布式 Worker + 调度器 | Worker 从队列拉取任务，调度器负责定时触发 |
| **执行模型** | 线性（步进式） | 按步骤列表顺序执行，有限条件分支 |
| **隔离** | 线程级隔离（线程池） | 每个连接流执行在独立线程中，通过线程池隔离 |
| **运行时** | Java + Spring Boot | 后端技术栈 |
| **凭证注入** | 运行时从 KMS 解密加载 | Token 缓存到 Redis（2h TTL） |

### 4.2 步骤执行流水线

```
事件到达 → [过滤条件判断] → [凭证注入] → [参数映射] → [连接器调用]
  → [响应解析] → [条件分支判断] → [下一组步骤入队] → [步骤循环]
```

| 阶段 | 说明 |
|------|------|
| **过滤判断** | 检查 trigger 的 eventFilter 是否匹配当前事件 |
| **凭证注入** | 从加密存储解密 Connection 凭证 |
| **参数映射** | 将 data 和步骤输出按映射规则注入参数（支持 template + source 模式） |
| **连接器调用** | 调用预置连接器或自定义连接器的 HTTP API |
| **条件分支** | 判断 condition 步骤的 expression，决定走哪个分支 |

### 4.3 数据传递

钉钉连接流的步骤间数据传递通过**流程上下文**对象：

```
FlowContext = {
  triggerData: {},       // 触发数据（事件源数据）
  variables: {},         // 流程变量（步骤间传递）
  stepResults: {         // 各步骤的执行结果
    "step_1": { output: {}, status: "success", ... },
    "step_2": { output: {}, status: "success", ... }
  },
  orgId: "",
  userId: "",
  executedBy: "auto" | "user_id"
}
```

上下文会持久化到 MySQL（小数据）或 OSS（大数据）。

## 五、工作节点与扩缩容

### 5.1 Worker 架构

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **Worker 运行时** | ECS (Elastic Compute Service) | 阿里云 ECS 虚拟机集群 |
| **容器化** | 阿里云 ACK (Kubernetes) | 部分新服务已迁移到 K8s |
| **线程池** | 自研线程池管理 | 每个 Worker 多线程并发处理 |
| **任务拉取** | RocketMQ Push Consumer | 消息主动推送到 Worker |

### 5.2 弹性扩缩

| 维度 | 机制 |
|------|------|
| **ECS 伸缩** | 基于 RocketMQ 队列深度 + CPU 负载自动伸缩 |
| **ACK K8s 伸缩** | HPA（Horizontal Pod Autoscaler）+ 自定义 Metrics |
| **冷备切换** | 多可用区部署，ACK 跨可用区调度 |
| **资源预留** | 企业版专享资源池 |

### 5.3 资源隔离

| 租户层级 | 隔离方式 |
|---------|---------|
| **企业（Org）** | 按 orgId 分区 |
| **连接流** | 流级别分布式锁 + 并发限制 |
| **消费者** | RocketMQ 消费组隔离 |

## 六、错误编排与重试

### 6.1 重试策略

| 重试层级 | 策略 | 存储 |
|---------|------|------|
| **步骤重试** | 最多 3 次，指数退避 | Redis 重试计数 + MySQL 持久化 |
| **流级重试** | 整体重试或跳过失败步骤 | 连接流状态表记录重试策略 |
| **最终失败** | 记录错误，触发告警 | MySQL 错误记录 + IM 告警推送 |

### 6.2 错误分类

| 错误类别 | 处理方式 |
|---------|---------|
| **API 超时** | 自动重试 |
| **限流（429）** | 退避重试，解析 `Retry-After` |
| **网络错误** | 重试 2-3 次 |
| **认证失败** | 不重试，触发告警 |
| **业务参数错误** | 不重试，记录错误详情 |

### 6.3 错误处理能力边界

钉钉的连接流在错误编排方面**较为基础**：

| 能力 | 支持程度 | 说明 |
|------|---------|------|
| 自动重试 | ✅ | 指数退避 |
| 错误通知 | ✅ | 通过 IM 推送告警 |
| 条件分支 | ✅ | 简单的 if/else |
| 错误处理节点 | ❌ | 无 Error Handler 模块 |
| 部分成功 | ❌ | 无 Compensate/回滚机制 |
| 死信处理 | ❌ | 仅记录错误日志 |

## 七、编排执行框架选型

### 7.1 框架选择

钉钉连接流引擎**自研**，未采用第三方工作流引擎：

| 框架 | 未采用原因 |
|------|-----------|
| **Camunda** | 钉钉连接流的编排能力较简单（线性为主），不需要 BPMN |
| **Temporal** | 钉钉连接流诞生时间较早，当时 Temporal 未被广泛采用 |
| **Quartz** | 仅用于定时调度，连接流整体执行引擎是自研的 |

### 7.2 自研引擎核心组件

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **流程定义** | MySQL TEXT JSON | 连接流完整 JSON 定义 |
| **调度执行** | 自研 Step Executor | 按步骤顺序执行，支持条件判断 |
| **定时调度** | Quartz + MySQL | Cron 表达式调度 |
| **事件消费** | RocketMQ Client | 消息拉取和消费 |
| **状态管理** | MySQL + Redis | 运行时状态管理 |
| **凭证管理** | KMS + Redis Cache | 凭证加解密和缓存 |

### 7.3 引擎能力对比

| 能力 | 钉钉连接流 | Zapier | Make | Power Automate |
|------|----------|--------|------|----------------|
| **线性执行** | ✅ | ✅ | ✅ | ✅ |
| **条件分支** | ⚠️ 简单 | ⚠️ 有限 | ✅ Router | ✅ Condition/Switch |
| **循环迭代** | ❌ | ❌ | ✅ Iterator | ✅ Apply_to_each |
| **并行执行** | ❌ | ❌ | ⚠️ 有限 | ✅ Scope + 并行 |
| **错误路由** | ❌ | ❌ | ✅ Error Handler | ✅ runAfter |
| **审批集成** | ✅ 原生 | ❌ | ❌ | ⚠️ 需自定义 |

## 八、高可用与后端框架

### 8.1 物理部署

| 维度 | 方案 |
|------|------|
| **云厂商** | 阿里云 |
| **多 AZ** | 多可用区部署（杭州/上海/深圳等） |
| **数据库** | RDS MySQL（主备 + 跨 AZ 灾备） |
| **缓存** | Redis（主从 + 跨 AZ 哨兵） |
| **消息** | RocketMQ（多 AZ 集群） |
| **计算** | ECS + ACK（K8s 跨 AZ 调度） |
| **对象存储** | OSS（同城冗余） |
| **Stream 长连接** | 独立 WebSocket 集群（多 AZ 负载均衡） |

### 8.2 后端框架

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **Web API** | Spring Boot (Java) | 管理面 API，连接流 CRUD |
| **连接流执行** | 自研 Java 引擎 | 步骤执行引擎 |
| **事件消费** | RocketMQ Client (Java) | 消息消费 |
| **定时调度** | Quartz + Spring Boot | 定时任务 |
| **Stream 服务** | Netty (WebSocket) | 长连接服务，高并发 I/O |
| **数据层** | MyBatis + Redis | 持久化 |
| **认证** | JWT + OAuth 2.0 | 应用认证 |
| **日志** | 阿里云 SLS (Log Service) | 日志收集和分析 |

### 8.3 高可用机制

| 机制 | 实现 | 说明 |
|------|------|------|
| **多 AZ 部署** | ECS + ACK 跨可用区调度 | AZ 级故障自动迁移 |
| **数据库主备** | RDS MySQL 主备切换 | 秒级故障转移 |
| **消息持久化** | RocketMQ 同步刷盘 | 消息不丢失 |
| **Stream 断线重连** | SDK 自动重连 + 偏移量恢复 | 连接恢复后从断点继续 |
| **优雅降级** | 非核心路径熔断 | 优先保障核心功能 |
| **全链路监控** | SLS + 阿里云监控 | 实时告警和问题排查 |

### 8.4 关键技术特点

| 特点 | 说明 |
|------|------|
| **Netty 长连接** | Stream 模式使用 Netty 实现高并发 WebSocket，支持数十万连接 |
| **内网优先** | 所有服务部署在同一阿里云 VPC，内网通信低延迟 |
| **钉钉生态深度耦合** | 事件源、认证体系、通知渠道等深度绑定钉钉 SDK |
| **轻量编排** | 执行引擎简洁，不适合复杂 DAG 场景 |

## 九、关键设计模式总结

| 设计模式 | 钉钉做法 | 对我们的启示 |
|---------|---------|------------|
| **Stream 长连接触发** | WebSocket 实时推送事件，降低触发延迟 | 内网场景下 WebSocket 长连接比 HTTP 回调更可靠 |
| **内聚型事件驱动** | 连接流触发器深度绑定钉钉内部事件 | 内聚型平台的事件源天然丰富，无需外部轮询 |
| **RocketMQ 异步解耦** | 事件和生产消费完全解耦 | 高吞吐场景下消息队列是标准模式 |
| **审批流 + 连接流分离** | 简单自动化用连接流，复杂审批走独立审批引擎 | 多引擎各自负责擅长的场景，通过事件联动 |
| **Netty WebSocket** | 长连接服务使用 Netty 而非传统 Servlet 容器 | 高并发长连接场景应选用 NIO 框架 |
| **轻量级连接流引擎** | 线性 + 有限分支，不追求 DAG 复杂度 | 内聚型场景的编排需求相对简单，引擎可以轻量 |
