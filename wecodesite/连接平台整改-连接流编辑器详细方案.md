# 连接平台整改-连接流编辑器详细方案

> 本文档描述连接流编辑器页面的详细整改方案，包含默认节点、连线加号插入、节点体系扩展、节点组插入、数据处理节点、Cookie 认证参数注入、版本子系统、详情抽屉、详情对比和调试抽屉。

---

## 一、页面现状

- 进入编辑器时画布可能为空。
- 用户通过左侧节点库拖拽新增节点。
- 节点类型较少，仅覆盖触发器、连接器、数据输出等基础场景。
- 连线中间无“+”按钮，不能在连线上插入节点。
- 删除节点后不会自动重连前后节点。
- 缺少错误处理、并行、文本、数据处理等节点。
- 错误处理、并行等复杂节点缺少一次插入多个节点的节点组能力。
- 数据处理节点缺少参数函数处理配置。
- 连接器节点缺少 Cookie 认证参数值来源配置。
- 无版本选择、版本状态、版本详情、详情对比和调试抽屉。

---

## 二、整改目标

- 默认初始化 HTTP 触发器节点和文本节点，并将文本节点文案设置为“结束”。
- 隐藏左侧节点库，仅通过连线“+”新增节点。
- 节点体系调整为触发器节点、连接器节点、数据输出节点、错误处理节点、循环、并行节点、条件、文本节点、数据处理节点。
- 本期开放触发器、连接器、数据输出、错误处理、并行、文本、数据处理；循环和条件仅补充类型定义，本期不开放添加。
- 插入普通节点时自动拆分原连线。
- 插入错误处理、并行节点时生成节点组，并自动生成节点组内部连线。
- 错误处理节点右侧分支有且只能添加一个连接器节点。
- 并行节点分支数量最多支持 8 个。
- 删除中间节点时自动重连前后节点。
- 数据处理节点支持类似数据输出节点的参数配置，并在添加参数时选择函数处理。
- 数据输出节点参数组装配置按 HTTP 响应体和 HTTP 响应头分 Tab 展示。
- 连接器节点展示当前连接器配置页已配置的认证方式和认证参数明细，除 Cookie 值来源外均只读；Cookie 值来源从上游所有节点出参中选择。
- 连接器节点支持配置超时时间。
- 异常错误配置支持按错误类型配置处理策略，错误类型包含全部、超时错误、连接错误，处理策略包含忽视、重试、中止。
- 节点名称保存时必填校验。
- 增加连接流版本子系统，版本列表为空时支持创建草稿。
- 增加版本详情抽屉、详情对比和调试抽屉。
- 更多配置支持多个缓存 Key、缓存时间上限 15 天、限流上限 1000。

## 四、整改点与实现思路

### 4.1 节点类型

本期连接流编辑器共涉及 9 种节点类型。其中 3 种为已有节点，6 种为新增节点（含 2 种本期仅定义不开放添加）。以下逐一说明节点定义、作用和实现方式。

| 节点类型 | 状态 | 定义与作用 | 实现方式 |
|----------|------|-----------|----------|
| 触发器节点 | 已有 | 连接流的入口节点，定义流程触发方式（如 HTTP 请求触发）。每个连接流只能有一个触发器节点，不允许重复添加。 | 保留现有 `trigger` 类型，`NodeProperties` 中展示输入契约配置面板。默认初始化画布时自动生成。 |
| 连接器节点 | 已有 | 调用远程连接器的执行动作，是连接流的核心处理节点。支持 SOA、APIG、Cookie、数字签名等认证方式。 | 保留现有 `connector` 类型。`NodeProperties` 中展示连接器选择、连接器版本、超时时间、认证参数明细和出参配置；认证参数来自连接器配置页，除 Cookie 值来源外均只读。通过连线“+”插入。 |
| 数据输出节点 | 已有 | 定义连接流的输出数据结构（HTTP 响应体、HTTP 响应头），作为流程的返回结果描述节点。 | 保留现有 `exit` 类型。`NodeProperties` 中使用 SchemaEditor 按 HTTP 响应体和 HTTP 响应头配置输出映射，可通过连线“+”插入。 |
| 错误处理节点 | 新增 | 用于捕获流程执行过程中的异常，并执行兜底逻辑。插入时自动生成节点组（错误处理主节点 + 异常说明文本节点 + 兜底处理文本节点）；右侧分支有且只能添加一个连接器节点。 | `FLOW_NODE_META` 中标记 `insertMode: 'group'`，通过 [`createInsertNodePayload`](#节点组插入思路) 生成节点组，并在右侧分支插入时限制节点类型和数量。 |
| 循环 | 新增（本期不添加） | 定义可重复执行的逻辑块，支持循环次数或条件控制。本期仅补充类型定义和节点元数据，为后续迭代预留。 | `FLOW_NODE_META` 中标记 `enabled: false`，不在连线加号弹窗中展示；节点类型定义保留在常量中。 |
| 并行节点 | 新增 | 定义可同时执行的分支逻辑。插入时自动生成完整并行节点组（并行主节点 + 分支开始文本 + 分支结束文本 + 汇合文本）。分支数量可通过主节点按钮或配置弹窗动态增删，最多支持 8 个分支。 | `FLOW_NODE_META` 中标记 `insertMode: 'group'`，通过 [`createInsertNodePayload`](#节点组插入思路) 生成节点组，并在添加分支时校验 8 个上限。 |
| 条件 | 新增（本期不添加） | 定义条件分支逻辑，根据表达式结果选择执行路径。本期仅补充类型定义和节点元数据，为后续迭代预留。 | `FLOW_NODE_META` 中标记 `enabled: false`，不在连线加号弹窗中展示；节点类型定义保留在常量中。 |
| 文本节点 | 新增 | 用于分支说明、错误说明、占位说明等场景。默认结束节点也是一种文本节点，文案为“结束”。文本节点不参与实际编排逻辑，仅为流程可读性服务。 | `FLOW_NODE_META` 中标记 `type: 'text'`，删除时不会触发自动重连。可随节点组自动生成，也可通过连线“+”单独插入。 |
| 数据处理节点 | 新增 | 对上游节点的输出参数进行函数处理后再输出给下游节点。类似数据输出节点，但在参数配置时增加“处理函数”选择。 | 新增 `data_process` 类型。`NodeProperties` 中复用 SchemaEditor 配置参数结构，在每条参数上增加 `functionConfig` 字段。通过连线“+”插入。 |

#### 实现思路

节点元数据需要同时服务节点选择弹窗、节点渲染、配置抽屉和后端转换。保留现有 `trigger`、`connector`、`exit` 类型避免影响已有数据；新增节点使用明确类型值，循环和条件标记为 `enabled: false` 以控制可见性。

```js
  error_handler: { label: '错误处理节点', color: '#f5222d', enabled: true, insertMode: 'group' },
  loop: { label: '循环', color: '#722ed1', enabled: false, insertMode: 'single' },
  parallel: { label: '并行节点', color: '#13c2c2', enabled: true, insertMode: 'group' },
  condition: { label: '条件', color: '#faad14', enabled: false, insertMode: 'single' },
  text: { label: '文本节点', color: '#8c8c8c', enabled: true, insertMode: 'single' },

  .map(([type, meta]) => ({ type, ...meta }));
```

### 4.2 默认初始化画布

#### 整改目标

当后端未返回编排配置、编排配置为空、或节点列表为空时，自动创建触发器节点和文本节点，并建立默认连线。文本节点文案为“结束”，用于表达流程结束占位；结束节点不再作为单独节点类型存在。

#### 改动范围

- `src/utils/flowUtils.js`：新增 `initializeDefaultFlowGraph`。
- `src/pages/ConnectPlatform/FlowEditor/index.jsx`：加载流程详情后判断是否需要初始化默认画布。
- `src/pages/ConnectPlatform/FlowEditor/constants.js`：统一默认节点 label 和默认连线类型。

#### 实现思路

默认初始化只在无有效编排数据时执行，不能覆盖后端已有节点。默认连线使用后续新增的 `plusEdge`，这样初始化后用户可以直接点击连线“+”插入节点。

```js
/**
 * 初始化默认连接流画布
 * 仅在后端没有有效编排配置时调用。
 * @returns {{ nodes: Array, edges: Array }} 默认节点和连线
 */
export const initializeDefaultFlowGraph = () => {
  const triggerId = generateNodeId('trigger');
  const endTextId = generateNodeId('text');

  return {
    nodes: [
      { id: triggerId, type: 'trigger', position: { x: 180, y: 220 }, data: { label: 'HTTP 触发器' } },
      { id: endTextId, type: 'text', position: { x: 620, y: 220 }, data: { label: '结束' } },
    ],
    edges: [
      { id: generateEdgeId(), source: triggerId, target: endTextId, type: 'plusEdge' },
    ],
  };
};
```

### 4.4 连线“+”插入节点

#### 整改目标

新增 `PlusEdge` 和 `NodePickerModal`。用户点击连线中点“+”后打开节点选择弹窗；选择普通节点时拆分原连线并插入单节点；选择错误处理或并行节点时插入节点组，而不是只插入一个节点。

#### 改动范围

- `src/utils/flowUtils.js`：新增普通节点插入、节点组插入、连线拆分工具。

#### 普通节点插入思路

普通节点包括连接器、数据输出、文本、数据处理等。插入时删除原连线，新增 `source → newNode` 和 `newNode → target` 两条连线。

```js
/**
 * 在指定连线上插入普通单节点
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {string} params.edgeId - 目标连线 ID
 * @param {string} params.newNodeType - 新节点类型
 * @returns {{ nodes: Array, edges: Array }} 更新后的节点和连线
 */
export const insertSingleNodeOnEdge = (params) => {
  const { nodes, edges, edgeId, newNodeType } = params;
  const targetEdge = edges.find((edge) => edge.id === edgeId);

  if (!targetEdge) {
    return { nodes, edges };
  }

  const sourceNode = nodes.find((node) => node.id === targetEdge.source);
  const targetNode = nodes.find((node) => node.id === targetEdge.target);
  const newNode = createDefaultFlowNode({ newNodeType, sourceNode, targetNode });
  const nextEdges = createSplitEdges({ edges, targetEdge, insertedNodeIds: [newNode.id] });

  return { nodes: nodes.concat(newNode), edges: nextEdges };
};
```

#### 节点组插入思路

错误处理和并行节点属于结构化节点组。节点选择弹窗返回 `error_handler` 或 `parallel` 后，工具函数根据类型生成主节点和多个文本节点，并生成内部连线。错误处理节点右侧分支只能作为兜底处理动作入口，有且只能插入一个连接器节点。

```js
/**
 * 根据节点类型创建插入载荷
 * 错误处理和并行节点会返回多个节点，普通节点返回单节点。
 * @param {Object} params - 参数对象
 * @param {string} params.newNodeType - 新节点类型
 * @param {Object} params.sourceNode - 原连线起点节点
 * @param {Object} params.targetNode - 原连线终点节点
 * @returns {{ nodes: Array, internalEdges: Array }} 新增节点和内部连线
 */
export const createInsertNodePayload = (params) => {
  const { newNodeType, sourceNode, targetNode } = params;

  if (newNodeType === 'error_handler') {
    return createErrorHandlerNodeGroup({ sourceNode, targetNode });
  }

  if (newNodeType === 'parallel') {
    return createParallelNodeGroup({ sourceNode, targetNode });
  }

  return createSingleNodePayload({ newNodeType, sourceNode, targetNode });
};
```

```js
/**
 * 创建错误处理节点组
 * 节点组包含错误处理主节点、异常说明文本节点、兜底处理文本节点。
 * @param {Object} params - 参数对象
 * @param {Object} params.sourceNode - 原连线起点节点
 * @param {Object} params.targetNode - 原连线终点节点
 * @returns {{ nodes: Array, internalEdges: Array }} 错误处理节点组
 */
export const createErrorHandlerNodeGroup = (params) => {
  const { sourceNode, targetNode } = params;
  const handlerId = generateNodeId('error_handler');
  const errorTextId = generateNodeId('text');
  const fallbackTextId = generateNodeId('text');

  return {
    nodes: [
      { id: handlerId, type: 'error_handler', position: getGroupMainPosition({ sourceNode, targetNode }), data: { label: '错误处理' } },
      { id: errorTextId, type: 'text', position: getGroupBranchPosition({ sourceNode, targetNode, index: 0 }), data: { label: '异常说明' } },
      { id: fallbackTextId, type: 'text', position: getGroupBranchPosition({ sourceNode, targetNode, index: 1 }), data: { label: '兜底处理' } },
    ],
    internalEdges: [
      { id: generateEdgeId(), source: handlerId, target: errorTextId, type: 'plusEdge' },
      { id: generateEdgeId(), source: handlerId, target: fallbackTextId, type: 'plusEdge' },
    ],
  };
};
```

#### 错误处理节点配置交互

错误处理节点插入后，用户点击节点本体打开节点配置弹窗。配置弹窗中展示错误类型和错误处理策略配置。错误类型支持全部、超时错误、连接错误；处理策略支持忽视、重试、中止。选择重试时展示重试间隔。可以配置多个错误类型，但每个错误类型只能添加一次；选择“全部”后不能添加其他错误类型，已选择其他错误类型后不能再选择“全部”。

交互流程如下：

1. 用户点击错误处理节点。
2. 右侧打开节点配置弹窗，弹窗标题仍为“节点配置”。
3. 弹窗展示节点名称、节点描述、错误配置列表。
4. 每条错误配置包含错误类型、错误处理策略。
5. 错误类型默认为“全部”，保存值为 `all`。
6. 错误处理策略默认为“忽视”，保存值为 `ignore`。
7. 用户将错误处理策略切换为“重试”后，展示“重试间隔”输入框。
8. 用户点击添加错误类型时，系统过滤已选择的错误类型，并处理“全部”和其他错误类型的互斥关系。
9. 用户点击保存后，将错误配置列表写入 `node.data.errorConfigs`。
10. 画布节点卡片同步展示当前错误处理摘要。

字段结构如下：

| 字段 | 类型 | 说明 |
|------|------|------|
| `errorConfigs` | Array | 错误配置列表，可配置多个错误类型 |
| `errorConfigs[].errorType` | string | 错误类型，支持 `all`、`timeout`、`connection`；每个错误类型只能添加一次，`all` 与其他类型互斥 |
| `errorConfigs[].strategy` | string | 错误处理策略，支持 `ignore`、`retry`、`abort` |
| `errorConfigs[].retryInterval` | number | 重试间隔，仅当 `strategy` 为 `retry` 时生效 |

```js
/**
 * 错误处理节点默认配置
 * 用于创建错误处理节点时初始化错误类型和处理策略。
 */
export const ERROR_HANDLER_DEFAULT_CONFIG = {
  errorConfigs: [
    {
      errorType: 'all',
      strategy: 'ignore',
      retryInterval: 1,
    },
  ],
};
```

```js
/**
 * 保存错误处理节点配置
 * @param {Object} params - 参数对象
 * @param {Object} params.node - 当前错误处理节点
 * @param {Array} params.errorConfigs - 错误配置列表
 * @returns {Object} 更新后的错误处理节点
 */
export const saveErrorHandlerConfig = (params) => {
  const { node, errorConfigs } = params;

  return {
    ...node,
    data: {
      ...node.data,
      errorConfigs,
    },
  };
};
```

```js
/**
 * 获取错误处理节点展示文案
 * @param {Object} params - 参数对象
 * @param {Object} params.node - 当前错误处理节点
```

#### 并行节点分支交互

并行节点插入后默认生成完整的并行结构，而不是只生成两个分支文本节点。默认结构包含 1 个并行主节点、2 个分支开始文本节点、2 个分支结束文本节点和 1 个汇合文本节点。每条分支内部的“分支开始 -> 分支结束”连线允许继续插入普通节点、错误处理节点、循环节点、并行节点或条件节点；分支结束后统一连接到汇合文本节点，再从汇合文本节点回到主链路下游节点。

交互流程如下：

1. 用户点击连线“+”并选择并行节点。
2. 系统删除原连线，并插入完整并行结构。
3. 系统默认创建 6 个节点：并行主节点、分支1开始、分支1结束、分支2开始、分支2结束、并行合并。
4. 系统默认创建 8 条连线：上游到并行主节点、并行主节点到两个分支开始、两个分支开始到对应分支结束、两个分支结束到并行合并、并行合并到原下游节点。
5. 用户可以在“分支开始 -> 分支结束”这条分支内部连线上继续插入节点。
6. 用户点击并行主节点上的“添加分支”按钮后，若当前分支数小于 8，系统新增 1 个分支开始文本节点、1 个分支结束文本节点和 3 条连线。
7. 当并行分支数已达到 8 个时，不再新增分支，并提示“并行分支最多支持8个”。
8. 用户点击分支开始文本节点上的删除按钮，系统删除该分支开始节点、分支结束节点及相关连线。
9. 用户点击并行主节点，打开节点配置弹窗，也可以在弹窗中增删分支或调整分支名称。
10. 画布重新布局时，按分支序号横向排列分支，并将汇合文本节点放在所有分支结束节点下方。

字段结构如下：

| 字段 | 类型 | 说明 |
|------|------|------|
| `parallelGroupId` | string | 并行结构分组 ID，通常等于并行主节点 ID |
| `parallelRole` | string | 节点角色，支持 `root`、`branch-start`、`branch-end`、`merge` |
| `parallelBranchId` | string | 分支唯一标识，分支开始和分支结束使用相同 ID |
| `parallelBranchIndex` | number | 分支序号，用于分支显示和布局排序 |

#### 插入节点时的位置重排方案

连线“+”插入节点时，除了拆分连线，还需要同步处理画布节点位置，避免新增节点和原有下游节点重叠。位置处理以“只影响目标边后续链路”为原则：不移动原连线 source 节点，不移动与目标边无关的其他分支，只移动 target 节点及从 target 继续向下游可达的节点。

##### 普通节点位置处理

普通节点包括连接器、数据输出、文本、数据处理。处理顺序如下：

1. 找到目标边 `source -> target`。
2. 计算 source 与 target 的中点，作为新增节点初始位置。
3. 判断 source 与 target 的垂直间距是否足够容纳新增节点。
4. 如果间距不足，将 target 和 target 的所有下游节点整体下移。
5. 删除原连线，新增 `source -> newNode` 和 `newNode -> target`。

示例：

```text
插入前：HTTP 触发器 -> 结束
插入后：HTTP 触发器 -> 数据处理节点 -> 结束
位置：结束节点整体下移，为数据处理节点腾出空间
```

##### 错误处理节点组位置处理

错误处理节点不是单节点插入，而是节点组插入。节点组包含错误处理主节点、异常说明文本节点、兜底处理文本节点。

位置规则：

- 错误处理主节点放在原连线中点，参与主链路。
- 异常说明文本节点放在错误处理主节点右上方或右侧第一行。
- 兜底处理文本节点放在错误处理主节点右下方或右侧第二行。
- 原 target 节点及其所有下游节点整体下移，预留节点组高度。
- 内部文本节点只作为说明或兜底配置入口，不直接改变主链路走向。

示例：

```text
插入前：A -> C
插入后：A -> 错误处理 -> C
               ├-> 异常说明
               └-> 兜底处理
位置：C 和 C 的下游节点整体下移，避免覆盖错误处理节点组
```

##### 并行节点组位置处理

并行节点插入时默认生成完整并行结构。并行主节点作为结构入口，分支开始文本和分支结束文本组成每条分支的可编辑链路，汇合文本节点作为所有分支的统一出口，再连接到原 target 节点。

位置规则：

- 并行主节点放在原连线中点。
- 默认两个分支开始文本节点分别放在并行主节点下方，并按分支序号横向展开，同层分支之间的水平间距固定（例如 260px）。
- 每个分支结束文本节点放在对应分支开始文本节点下方。
- 汇合文本节点放在所有分支结束文本节点下方，用于承接所有分支。
- 如果后续增加分支，新增分支开始和分支结束按分支序号向右侧展开。
- 原 target 节点及其所有下游节点整体下移，预留完整并行结构高度。
- 并行结构的出口是汇合文本节点，不是并行主节点直接连接原 target。

##### 并行分支内插入嵌套结构时右侧分支右移

当在某条并行分支内部插入并行节点或条件分支节点（即嵌套结构）时，该嵌套结构会横向展开多条子分支。为了避免当前分支的嵌套子分支与右侧其他分支重叠，必须将被插入分支**右侧的所有兄弟分支整体右移**。

具体来说，在并行节点 A 的 3 条分支（分支1、分支2、分支3）中，如果在分支2内部插入了一个并行节点（生成分支2-1、分支2-2），那么：

- 分支3及分支3内的所有节点的 X 坐标向右移动。
- 右移距离 = 新插入嵌套结构的子分支占用的宽度。
- 汇合文本节点如果跨所有分支下方居中，也需要相应右移或展宽。

示例：

```text
插入前（并行节点 A 有 3 条分支）：
  A -> 分支1开始 -> ... -> 分支1结束 \
  A -> 分支2开始 -> ... -> 分支2结束  → 并行合并 -> D
  A -> 分支3开始 -> ... -> 分支3结束 /

插入后（在分支2中插入并行节点 B，B 有 2 条子分支）：
  A -> 分支1开始 -> ... -> 分支1结束 \
  A -> 分支2开始 -> 并行节点 B -> 分支2结束 -> 并行合并 -> D
                     ├-> 子分支2-1 ->
                     └-> 子分支2-2 ->
  A -> 分支3开始 -> ... -> 分支3结束 /

位置：分支3 及其内部所有节点水平右移（右侧空出嵌套结构宽度），避免 B 的子分支和分支3重叠
```

```js
/**
 * 计算并行分支内插入嵌套结构后的布局
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {string} params.parallelGroupId - 并行结构分组 ID
 * @param {number} params.insertBranchIndex - 被插入的分支序号（从 1 开始）
 * @param {number} params.nestedStructureWidth - 嵌套结构整体占用的宽度
 * @returns {Array} 调整位置后的节点列表
 */
export const getNodesAfterInsertNestedParallelLayout = (params) => {
  const { nodes, parallelGroupId, insertBranchIndex, nestedStructureWidth } = params;

  return nodes.map((node) => {
    const nodeConfig = node.data?.config || node.data || {};
    const currentBranchIndex = nodeConfig.parallelBranchIndex || nodeConfig.parentParallelBranchIndex || 0;
    const isCurrentParallelBranchNode = nodeConfig.parallelGroupId === parallelGroupId;
    const isCurrentParallelBranchInnerNode = nodeConfig.parentParallelGroupId === parallelGroupId;
    const shouldMoveRight = (isCurrentParallelBranchNode || isCurrentParallelBranchInnerNode)
      && currentBranchIndex > insertBranchIndex;

    // 被插入分支右侧的兄弟分支节点，以及这些分支内部的节点，都需要整体右移。
    if (!shouldMoveRight) {
      return node;
    }

    return {
      ...node,
      position: {
        ...node.position,
        x: node.position.x + nestedStructureWidth,
      },
    };
  });
};
```

##### 循环和条件节点位置预案

循环和条件本期仅补充类型定义，不在节点选择弹窗中开放添加。为了避免后续实现时重新设计位置体系，本期文档先约定位置预案。

| 节点类型 | 本期状态 | 后续位置处理预案 |
|----------|----------|------------------|
| 循环 | 不开放添加 | 循环主节点插入主链路，循环体占位节点放在主节点下方，回流边从循环体返回循环主节点；原 target 和下游节点整体下移。 |
| 条件 | 不开放添加 | 条件主节点插入主链路，true / false 分支左右展开，分支末端汇聚后再连接原 target；原 target 和下游节点整体下移。 |

##### 自动重排约束

- 只移动目标边 target 节点及其下游节点。
- 不移动 source 节点和 source 的其他上游节点。
- 不移动与目标边无连通关系的节点。
- 若节点存在 `positionLocked: true`，表示用户手动固定过位置，自动重排时跳过该节点。
- 每次插入只做局部重排，不做全画布重新布局，降低对用户手动排版的影响。

```js
/**
 * 计算插入节点后的布局结果
 * 只处理目标连线后方节点，避免影响无关节点位置。
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {Object} params.targetEdge - 被插入的目标连线
 * @param {Array} params.insertedNodes - 本次新增节点列表
 * @param {number} params.requiredHeight - 本次插入需要预留的高度
 * @returns {Array} 调整位置后的节点列表
 */
export const getNodesAfterInsertLayout = (params) => {
  const { nodes, edges, targetEdge, insertedNodes, requiredHeight } = params;
  const targetNodeIds = getDownstreamNodeIds({ edges, startNodeId: targetEdge.target });
  const insertedNodeIds = insertedNodes.map((node) => node.id);

  return nodes.concat(insertedNodes).map((node) => {
    // 新增节点使用创建时已经计算好的位置，不再二次移动。
    if (insertedNodeIds.includes(node.id)) {
      return node;
    }

    // 用户锁定过位置的节点不参与自动重排。
    if (node.data?.positionLocked) {
      return node;
    }

    // 只有目标节点及其下游节点需要整体下移。
    if (targetNodeIds.includes(node.id)) {
      return {
        ...node,
        position: {
          ...node.position,
          y: node.position.y + requiredHeight,
        },
      };
    }

    return node;
  });
};
```

```js
/**
 * 获取指定节点的所有下游节点 ID
 * 用于插入节点时只移动目标链路后方节点。
 * @param {Object} params - 参数对象
 * @param {Array} params.edges - 当前连线列表
 * @param {string} params.startNodeId - 起始节点 ID
 * @returns {Array} 下游节点 ID 列表
 */
export const getDownstreamNodeIds = (params) => {
  const { edges, startNodeId } = params;
  const visited = new Set();

  /**
   * 递归收集下游节点
   * @param {string} currentNodeId - 当前节点 ID
   */
  const collect = (currentNodeId) => {
    if (visited.has(currentNodeId)) {
      return;
    }

    visited.add(currentNodeId);

    edges
      .filter((edge) => edge.source === currentNodeId)
      .forEach((edge) => {
        collect(edge.target);
      });
  };

  collect(startNodeId);

  return Array.from(visited);
};
```

### 4.5 删除节点自动重连

#### 整改目标

删除中间节点时，如果该节点存在入边和出边，则自动新增入边来源到出边目标的连线，保证主链路不断开。删除节点组内文本节点时，只删除相关连线，不强制重连到其他分支。

#### 改动范围

- `src/pages/ConnectPlatform/FlowEditor/index.jsx`：删除节点时调用重连工具。
- `src/utils/flowUtils.js`：新增 `getReconnectedEdges`、`getNodesAfterDeleteLayout`、`getDeletedGroupNodeIds`，并识别节点组场景。

#### 删除交互实录

1. 用户点击节点，右侧节点配置抽屉打开。
2. 用户点击“删除节点”。
3. 系统弹出二次确认，提示删除后会移除当前节点及相关连线。
4. 用户确认删除后，前端先判断节点类型。
5. 如果是触发器节点，直接阻止删除。
6. 如果是普通中间节点，删除节点和相关连线，并自动连接前置节点和后置节点。
7. 如果是节点组主节点，删除整个节点组，并将主链路前后节点重新连接。
8. 如果是节点组内部文本节点，只删除该文本节点和相关内部连线，不强制重连到其他分支。
9. 删除完成后关闭配置抽屉，并按规则上移后续节点，填补空位。

#### 删除策略

| 删除对象 | 删除范围 | 重连方式 | 位置处理 |
|----------|----------|----------|----------|
| 普通节点 | 仅删除当前节点和相关边 | 入边来源连接到出边目标 | 后续主链路节点上移一个节点间距 |
| 文本节点 | 删除当前文本节点和相关边 | 若为普通文本节点可按普通节点重连；若为节点组内部文本不重连 | 仅移除当前文本占位，不移动主链路 |
| 错误处理主节点 | 删除错误处理主节点、异常说明、兜底处理及内部边 | 原 source 直接连接原 target | target 和下游节点上移节点组高度 |
| 并行主节点 | 删除并行主节点、默认分支文本及内部边 | 原 source 直接连接原 target | target 和下游节点上移节点组高度 |
| 循环主节点 | 本期不开放，仅预留 | 后续删除循环体和回流边后重连主链路 | target 和下游节点上移循环区域高度 |
| 条件主节点 | 本期不开放，仅预留 | 后续删除 true / false 分支和汇聚边后重连主链路 | target 和下游节点上移条件区域高度 |

#### 普通节点删除示例

```text
删除前：A -> B -> C
删除后：A -> C
位置：C 和 C 的下游节点整体上移，填补 B 原来的位置
```

#### 错误处理节点组删除示例

```text
删除前：A -> 错误处理 -> C
              ├-> 异常说明
              └-> 兜底处理
删除后：A -> C
位置：C 和 C 的下游节点整体上移，填补错误处理节点组区域
```

#### 并行节点组删除示例

```text
删除前：A -> 并行节点 -> C
              ├-> 并行分支 1
              └-> 并行分支 2
删除后：A -> C
位置：C 和 C 的下游节点整体上移，填补并行节点组区域
```

```js
/**
 * 删除节点后自动重连
 * @param {Object} params - 参数对象
 * @param {Array} params.edges - 当前连线列表
 * @param {string} params.removedNodeId - 被删除节点 ID
 * @param {Object} params.removedNode - 被删除节点对象
 * @param {Array} params.removedGroupNodeIds - 同时删除的节点组 ID 列表
 * @returns {Array} 重连后的连线列表
 */
export const getReconnectedEdges = (params) => {
  const { edges, removedNodeId, removedNode, removedGroupNodeIds = [] } = params;
  const removedNodeIds = [removedNodeId].concat(removedGroupNodeIds);
  const incoming = edges.filter((edge) => edge.target === removedNodeId && !removedNodeIds.includes(edge.source));
  const outgoing = edges.filter((edge) => edge.source === removedNodeId && !removedNodeIds.includes(edge.target));
  const remaining = edges.filter((edge) => !removedNodeIds.includes(edge.source) && !removedNodeIds.includes(edge.target));

  // 节点组内部文本节点只删除自身相关边，不强制重连。
  if (removedNode?.data?.groupRole === 'branchText') {
    return remaining;
  }

  if (incoming.length === 0 || outgoing.length === 0) {
    return remaining;
  }

  const reconnected = [];
  incoming.forEach((inEdge) => {
    outgoing.forEach((outEdge) => {
      reconnected.push({
        id: generateEdgeId(),
        source: inEdge.source,
        target: outEdge.target,
        type: 'plusEdge',
      });
    });
  });

  return remaining.concat(reconnected);
};
```

```js
/**
 * 获取删除节点时需要一并删除的节点组成员
 * 错误处理和并行主节点删除时，需要同步删除其组内文本节点。
 * @param {Object} params - 参数对象
 * @param {Array} nodes - 当前节点列表
 * @param {Object} removedNode - 被删除节点对象
 * @returns {Array} 需要同时删除的节点 ID 列表
 */
export const getDeletedGroupNodeIds = (params) => {
  const { nodes, removedNode } = params;

  if (!removedNode?.data?.groupId) {
    return [];
  }

  // 只有节点组主节点删除时，才级联删除组内成员。
  if (removedNode.data.groupRole !== 'groupMain') {
    return [];
  }

  return nodes
    .filter((node) => node.data?.groupId === removedNode.data.groupId && node.id !== removedNode.id)
    .map((node) => node.id);
};
```

```js
/**
 * 计算删除节点后的布局结果
 * 普通节点删除后上移后续节点，节点组删除后上移更多距离。
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {Object} params.removedNode - 被删除节点对象
 * @param {Array} params.removedNodeIds - 本次删除的节点 ID 列表
 * @param {number} params.releaseHeight - 本次删除释放的高度
 * @returns {Array} 调整位置后的节点列表
 */
export const getNodesAfterDeleteLayout = (params) => {
  const { nodes, edges, removedNode, removedNodeIds, releaseHeight } = params;
  const outgoing = edges.find((edge) => edge.source === removedNode.id && !removedNodeIds.includes(edge.target));
  const downstreamNodeIds = outgoing ? getDownstreamNodeIds({ edges, startNodeId: outgoing.target }) : [];

  return nodes
    .filter((node) => !removedNodeIds.includes(node.id))
    .map((node) => {
      // 用户锁定过位置的节点不参与自动重排。
      if (node.data?.positionLocked) {
        return node;
      }

      // 删除节点后，只上移被删除节点后方的主链路节点。
      if (downstreamNodeIds.includes(node.id)) {
        return {
          ...node,
          position: {
            ...node.position,
            y: Math.max(0, node.position.y - releaseHeight),
          },
        };
      }

      return node;
    });
};
```

### 4.6 数据处理节点

#### 整改目标

数据处理节点用于对上游参数进行函数处理后再输出给下游节点。它的配置体验类似数据输出节点，但在新增或编辑参数时增加“处理函数”配置。

#### 改动范围

- `src/components/CustomFlowNodes/DataProcessNode.jsx`：新增数据处理节点展示组件。
- `src/pages/ConnectPlatform/FlowEditor/constants.js`：注册 `data_process` 节点和转换配置。
- `src/pages/ConnectPlatform/FlowEditor/NodeProperties.jsx`：新增数据处理配置区。
- `src/components/SchemaEditor/SchemaEditor.jsx`：参数项支持函数处理配置字段。
- `src/utils/flowUtils.js`：后端转换时保留函数处理配置。

#### 配置字段

| 字段 | 说明 |
|------|------|
| `paramName` | 输出参数名 |
| `paramType` | 输出参数类型 |
| `carrier` | 输出参数位置，支持 header、body、query |
| `sourceType` | 参数来源，支持 static、reference、function |
| `paramValue` | 静态值来源时填写的固定值 |
| `referencePath` | 引用参数来源时选择的上游参数路径 |
| `functionConfig.input` | 函数处理的输入参数，来自上游参数选择器 |
| `functionConfig.type` | 函数分类，如 string、array、object、date、json |
| `functionConfig.name` | 函数名称，如 trim、map、filter、formatDate、parseJson |
| `functionConfig.args` | 函数参数，不同函数展示不同参数项 |

#### 函数处理交互实录

1. 用户点击画布连线上的“+”。
2. 系统打开节点选择弹窗。
3. 用户选择“数据处理节点”。
4. 系统在当前连线上插入数据处理节点，原连线拆分为 `source -> data_process -> target`。
5. 用户点击数据处理节点，右侧配置抽屉打开。
6. 用户在“参数组装配置”中点击“添加参数”。
7. 系统新增一行参数，默认来源类型为“引用参数”。
8. 用户将来源类型切换为“函数处理”。
9. 参数行中的配置输入框变为只读状态，展示占位文案“点击选择函数处理”。
10. 用户点击该输入框后，系统打开“函数处理配置”弹窗。
11. 弹窗展示当前参数名称，并提供“选择函数”下拉框。
12. 用户选择函数，例如 `trim`、`concat`、`formatDate`、`parseJson`。
13. 系统根据所选函数动态渲染函数参数区。
14. 如果函数不需要额外参数，函数参数区展示“该函数无需额外参数”。
15. 如果函数需要额外参数，函数参数区展示对应参数项，例如日期格式、拼接字符串、属性路径等。
16. 函数参数值可从当前节点可引用的上游参数列表中选择，参数来源来自 `availableReferenceParams`。
17. 用户点击确定后，系统将函数名称和函数参数保存到当前参数行的 `functionConfig`。
18. 当前参数行回填函数展示文案，例如 `trim()`、`formatDate(createTime)`。
19. 用户保存数据处理节点配置后，函数处理配置跟随参数组装配置一起写入节点数据。

#### 函数选择弹窗交互设计

函数选择不直接内嵌在参数行中，而是通过独立弹窗完成，避免参数表格过宽，也便于不同函数展示不同参数项。

弹窗包含以下区域：

| 区域 | 说明 |
|------|------|
| 参数名称 | 只读展示当前正在配置的参数名，帮助用户确认配置对象 |
| 选择函数 | 下拉选择具体处理函数，如 map、filter、trim、formatDate |
| 函数参数 | 根据函数动态展示参数项；无参数时展示提示文案 |
| 底部操作 | 取消关闭弹窗，确定保存函数配置 |

交互细节如下：

- 打开弹窗时，系统记录当前配置上下文，包括 `paramId` 和 `containerId`。
- 如果当前参数已经配置过函数，则弹窗回显已选择的函数和函数参数。
- 切换函数时，旧函数参数不继续沿用，避免参数含义错配。
- 函数参数下拉的数据来源为当前节点上游可引用参数，不能选择下游节点参数。
- 点击确定时必须选择函数，否则提示“请选择函数”。
- 保存后调用参数行更新逻辑，将配置方式设置为 `function`。

```js
 * @param {Object} params - 参数对象
 * @param {Object} params.field - 当前数据处理参数
 * @returns {Object} 后端编排字段
 */
export const transformDataProcessFieldToBackend = (params) => {
  const { field } = params;

  if (field.sourceType === 'function') {
    return {
      type: field.paramType || 'string',
      description: field.description || '',
      value: buildFunctionProcessValue({ functionConfig: field.functionConfig }),
      functionConfig: field.functionConfig,
    };
  }

  if (field.sourceType === 'reference') {
    return {
      type: field.paramType || 'string',
      description: field.description || '',
      value: `\${$.node.${field.referencePath}}`,
    };
  }

  return {
    type: field.paramType || 'string',
    description: field.description || '',
    value: `\${$.constant:${field.paramValue || ''}}`,
  };
};
```

### 4.7 Cookie 认证参数注入

#### 整改目标

连接器节点配置抽屉在选择连接器后，展示当前连接器在连接器配置页配置的所有认证方式和认证参数明细。认证参数包含参数名称、类型、位置、值来源四列，参数名称、类型、位置和值来源默认不可编辑；仅 Cookie 认证的值来源允许在连接流节点中选择，且只能从当前节点上游所有节点的出参中选择。

#### 展示条件

| 条件 | 处理方式 |
|------|----------|
| 当前节点不是连接器节点 | 不展示认证参数明细 |
| 当前连接器未选择 | 不展示认证参数明细 |
| 当前连接器配置未启用任何认证方式 | 展示“未配置认证方式” |
| 当前连接器配置启用 SOA、APIG、Cookie、数字签名等认证方式 | 按认证类型分组展示认证参数明细 |
| 当前连接器配置启用 Cookie 认证 | Cookie 参数的值来源展示为上游出参下拉选择，其余字段只读 |

#### 可引用来源

| 来源 | 说明 | 示例 |
|------|------|------|
| 上游触发器出参 | 当前连接器节点上游链路中的触发器输出或入参结构 | `trigger_001.output.header.token` |
| 上游连接器出参 | 当前连接器节点之前的连接器节点输出参数，包括 HTTP 响应体、HTTP 响应头 | `connector_001.output.body.sessionId` |
| 上游数据处理出参 | 当前连接器节点之前的数据处理节点输出参数 | `data_process_001.output.body.cookieValue` |
| 上游数据输出出参 | 若链路中存在数据输出节点，也可作为普通上游节点读取其输出结构 | `exit_001.output.body.result` |

#### 改动范围

- `src/pages/ConnectPlatform/FlowEditor/NodeProperties.jsx`：在连接器节点配置区域判断认证方式，按需渲染 Cookie 面板。
- `src/components/AuthFields/CookieAuthFields.jsx`：展示 Cookie 参数列表、当前值来源、选择上游参数入口。
- `src/components/UpstreamParamPicker/UpstreamParamPicker.jsx`：仅传入当前连接器节点可访问的上游参数。
- `src/utils/flowUtils.js`：扩展上游参数提取和 Cookie 参数保存前校验。

#### 实现思路

连接器节点切换连接器后，先获取连接器配置页已配置的认证方式。配置抽屉按 SOA、APIG、Cookie、数字签名等认证类型分组展示参数明细：参数名称、类型、位置和值来源默认只读；Cookie 认证的值来源使用下拉框展示当前节点上游所有节点出参。用户保存连接器节点时，仅写入 Cookie 值来源和超时时间，其他认证参数继续以连接器配置页数据为准。

```jsx
/**
 * 判断是否需要展示 Cookie 认证参数面板
 * @param {Object} params - 参数对象
 * @param {Object} params.selectedNode - 当前选中的节点
 * @param {Object} params.connectorConfig - 当前连接器配置

  const { selectedNode, connectorConfig } = params;

  if (selectedNode?.type !== 'connector') {
    return false;
  }

  return connectorConfig?.authType === 'cookie';
```

```jsx

 * 渲染连接器节点的 Cookie 认证参数面板
 * @param {Object} params - 参数对象
 * @param {Object} params.selectedNode - 当前连接器节点
 * @param {Object} params.connectorConfig - 当前连接器配置
```

```js
/**
 * 校验连接器节点 Cookie 认证参数
 * @param {Object} node - 连接器节点

export const validateConnectorCookieAuthFields = (node) => {
```

### 4.8 上游参数选择器

#### 整改目标

`UpstreamParamPicker` 用于统一选择当前节点可访问的上游参数。它不直接从全量节点中选参数，而是基于当前节点 ID 和连线关系向前追溯，只展示当前节点上游链路中的可引用参数。该选择器同时服务 Cookie 认证参数、连接器输入映射和数据处理节点参数。

#### 可引用参数范围

| 当前使用场景 | 可引用范围 | 说明 |
|--------------|------------|------|
| Cookie 认证参数 | 触发器入参、上游连接器出参、上游数据处理出参、上游数据输出出参 | 只展示当前连接器节点之前的参数 |
| 连接器输入映射 | 触发器入参、上游连接器出参、上游数据处理出参 | 用于构造连接器请求参数 |
| 数据处理节点参数 | 触发器入参、上游连接器出参、上游数据处理出参、上游数据输出出参 | 用于选择待处理参数 |

#### 改动范围

- `src/components/UpstreamParamPicker/UpstreamParamPicker.jsx`：新增上游参数选择弹窗，支持按节点分组、按参数位置筛选、关键字搜索。
- `src/utils/flowUtils.js`：扩展 `getUpstreamNodes`、`getNodeOutputParams`、`getUpstreamParams`。
- `src/pages/ConnectPlatform/FlowEditor/NodeProperties.jsx`：Cookie 参数、连接器输入映射、数据处理参数中复用选择器。

#### 数据流

1. 配置抽屉打开时，将当前节点 ID、节点列表、连线列表传给 `getUpstreamParams`。
2. `getUpstreamParams` 先通过连线关系递归查找所有上游节点。
3. 对每个上游节点调用 `getNodeOutputParams`，根据节点类型提取可引用参数。
4. `UpstreamParamPicker` 按节点分组展示参数。
5. 用户选择参数后，返回 `nodeId`、`nodeType`、`paramPath`、`carrier`、`paramType`。
6. 调用方将选择结果写入 `valueSource`、`referencePath` 或 `functionConfig.input`。

```js
/**
 * 获取当前节点的上游参数
 * @param {Object} params - 参数对象
 * @param {string} params.nodeId - 当前节点 ID
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @returns {Array} 按上游节点分组的参数列表
 */
export const getUpstreamParams = (params) => {
  const { nodeId, nodes, edges } = params;
  const upstreamNodes = getUpstreamNodes({ nodeId, nodes, edges });

  return upstreamNodes.map((node) => ({
    nodeId: node.id,
    nodeName: node.data?.label || node.id,
    nodeType: node.type,
    params: getNodeOutputParams({ node }),
  }));
};
```

```js
/**
 * 根据节点类型提取可被下游引用的参数
 * @param {Object} params - 参数对象
 * @param {Object} params.node - 当前上游节点
 * @returns {Array} 可引用参数列表
 */
export const getNodeOutputParams = (params) => {
  const { node } = params;

  if (node.type === 'trigger') {
    return normalizeTriggerInputParams({ inputContract: node.data?.inputContract });
  }

  if (node.type === 'connector') {
    return normalizeConnectorOutputParams({ outputParams: node.data?.outputParams });
  }

  if (node.type === 'data_process') {
    return normalizeDataProcessOutputParams({ processMapping: node.data?.processMapping });
  }

  if (node.type === 'exit') {
    return normalizeOutputMappingParams({ outputMapping: node.data?.outputMapping });
  }

  return [];
};
```

```js
/**
 * 构建上游参数值来源
 * @param {Object} params - 参数对象
 * @param {string} params.nodeId - 上游节点 ID
 * @param {string} params.paramPath - 参数路径
 * @returns {string} 参数值来源表达式
 */
export const buildValueSource = (params) => {
  const { nodeId, paramPath } = params;

  return `${nodeId}.${paramPath}`;
};
```

1. 用户点击“保存并调试”。
2. 前端先执行当前画布保存逻辑，将 `nodes` 和 `edges` 转换为后端编排结构。
3. 保存成功后，根据触发器节点的 `inputContract` 生成调试入参模板。
4. 打开 `DebugDrawer`，用户补充或修改 header、body、query。
5. 用户点击“开始调试”。
6. 前端调用 `/flows/{flowId}/versions/{versionId}/debug`。
7. 调试完成后展示响应 JSON、状态、耗时、错误信息和执行日志。

```js
/**
 * 保存并打开调试抽屉
 * @param {Object} params - 参数对象
 * @param {string} params.flowId - 连接流 ID
 * @param {string} params.versionId - 版本 ID
 * @param {Array} params.nodes - 当前节点列表
 * @param {Array} params.edges - 当前连线列表
 * @param {Function} params.openDebugDrawer - 打开调试抽屉回调
 * @returns {Promise<void>}
 */
export const saveAndOpenDebugDrawer = async (params) => {
  const { flowId, versionId, nodes, edges, openDebugDrawer } = params;
  const orchestrationConfig = transformToBackend(nodes, edges);

  await saveFlowVersionConfig({ flowId, versionId, orchestrationConfig });

  openDebugDrawer({
    flowId,
    versionId,
    inputTemplate: buildDebugInputTemplate({ nodes }),
  });
};
```

```js
/**
 * 构建调试入参模板
 * @param {Object} params - 参数对象
 * @param {Array} params.nodes - 当前节点列表
 * @returns {{ header: Object, body: Object, query: Object }} 调试入参模板
 */
export const buildDebugInputTemplate = (params) => {
  const { nodes } = params;
  const triggerNode = nodes.find((node) => node.type === 'trigger');
  const inputContract = triggerNode?.data?.inputContract || {};

  return {
    header: buildTemplateByCarrier({ inputContract, carrier: 'header' }),
    body: buildTemplateByCarrier({ inputContract, carrier: 'body' }),
    query: buildTemplateByCarrier({ inputContract, carrier: 'query' }),
  };
};
```

```js
/**
 * 发起连接流调试
 * @param {Object} params - 参数对象
 * @param {string} params.flowId - 连接流 ID
 * @param {string} params.versionId - 版本 ID
 * @param {Object} params.debugInput - 调试入参
 * @returns {Promise<Object>} 调试结果
 */
export const debugFlowVersion = async (params) => {
  const { flowId, versionId, debugInput } = params;

  return request.post(`/flows/${flowId}/versions/${versionId}/debug`, {
    header: debugInput.header || {},
    body: debugInput.body || {},
    query: debugInput.query || {},
  });
};
```

#### 异常处理

- 保存失败：不打开调试抽屉，提示保存失败原因。
- 入参 JSON 格式错误：阻止调试，定位到错误 Tab。
- 调试接口失败：保留入参，输出区展示错误信息。
- 后端返回节点执行日志为空：仅展示响应结果，不展示执行日志区。

## 七、验证点

- 空配置进入编辑器时自动出现触发器节点和文本节点，文本节点文案为“结束”。
- 左侧节点库不再展示。
- 连线“+”可以打开节点选择弹窗。
- 循环、条件在节点选择弹窗中不开放添加。
- 选择连接器、数据输出、文本、数据处理等普通节点后，原连线被拆分为两条连线，target 及其下游节点按需下移。
- 选择错误处理节点后，自动生成错误处理主节点和多个文本节点，并为节点组预留画布空间。
- 选择并行节点后，自动生成并行主节点、分支开始文本、分支结束文本和汇合文本；分支链路可继续插入节点，汇合文本连接回主链路。
- 循环和条件只存在类型定义，不在节点选择弹窗中开放添加。
- 删除普通中间节点后前后节点自动重连，后续节点按需上移。
- 删除错误处理或并行主节点后，节点组成员同步删除，主链路自动重连。
- 数据处理节点新增参数时可以选择函数处理，并能保存输入参数、函数分类、函数名称和函数参数。
- 连接器节点选择连接器后展示连接器配置页中的认证方式和认证参数明细。
- Cookie 认证参数值来源只能从上游所有节点出参中选择。
- 进入编辑页面后默认加载版本列表，并优先选中草稿版本；版本列表为空时展示创建草稿按钮。
- 草稿、已驳回、已撤回版本可编辑保存，已驳回和已撤回保存后状态改为草稿。
- 版本下拉、详情抽屉、调试抽屉可正常打开。
- 保存并调试后输出区展示 JSON、状态、耗时和错误信息。
