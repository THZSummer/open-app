# 连接平台整改-参数配置适配方案

> 本文档基于 `wecodesite` 当前实现、接口配置整改方案、连接流编辑器整改方案以及 `connector-editor.html` 原型交互，重新设计一套适配现状的参数配置方案。
>
> **核心矛盾：** 用户编辑时需要按位置分 Tab，但传给后端时必须将参数聚合为 `{ header, body, query }` 对象再由转换层处理。
> **解决策略：** 前端内部继续使用数组结构 + `carrier` 标识位置；展示层用 `CarrierSchemaTabs` 拆分 Tab；保存时仍然走现有转换逻辑。

---

## 一、全量变更清单

在开始方案前，先明确改什么、不改什么，避免实施过程中边界模糊。

### 1.1 改的部分

| 模块 | 改什么 | 原因 |
|---|---|---|
| `SchemaEditor.jsx` | 重构成纯"编辑引擎"：移除 `form` 依赖，改为受控组件；新增 `fixedCarrier` / `hideCarrier` / `value` / `onChange` 对外接口；删掉内部 `form.setFieldValue` 副作用 | 当前 SchemaEditor 和 form 强耦合，Tab 包装层无法控制数据流 |
| `CarrierSchemaTabs.jsx` | 新增组件，负责 Tab 切换、筛选 carrier、合并 carrier | 这是整改的核心新组件 |
| `ConnectorEditor/index.jsx` | 入参/出参区域改用 `CarrierSchemaTabs` 替换直接 SchemaEditor | 交互改造 |
| `NodeProperties.jsx` | 触发器的 inputContract、连接器的 inputMapping、数据输出的 outputMapping 改用 `CarrierSchemaTabs` | 交互改造 |
| 认证方式 | `AUTH_TYPE_OPTIONS` 扩展 Cookie、数字签名新增 `AuthFields/CookieAuthFields.jsx`、`AuthFields/SignatureAuthFields.jsx` | 新增认证能力 |
| 版本相关 | 接口配置页新增版本选择、编辑/只读切换 | 版本化改造 |
| `ConnectorEditor/constants.jsx` | 移除 PATCH；新增 Cookie、Signature 的 `AUTH_SCHEMA_MAP`；新增 `REQUEST_PARAM_TABS` / `RESPONSE_PARAM_TABS` | 配置更新 |
| `web.config.js` | 新增版本相关端点 | 后端接口 |

### 1.2 不改的部分

| 模块 | 不改什么 | 原因 |
|---|---|---|
| 前端内部数据结构 | 继续使用 `[{ paramName, paramType, carrier, children }]` 数组，不改成 `{ header: [], body: [], query: [] }` | 改格式会影响 SchemaEditor 递归、转换层、flowUtils 全部联动 |
| `transformToBackend`（flowUtils） | 不变。它仍然遍历 nodes，将数组通过 `processMappingItem` 按 carrier 聚合为 `{ header: { type: 'object', properties: {...} }, body: {...} }` | 后端需要的格式不变 |
| `transformFromBackend`（flowUtils） | 不变。仍然将后端的 `{ header: properties, body: properties }` 转为数组 | 前端需要的格式不变 |
| `transformToSchemaFormat` / `transformFromSchemaFormat`（ConnectorEditor/thunk.js） | 不变。仍然按 carrier 聚合/拆分 | 接口配置页的后端格式不变 |
| `getNodeOutputParams`（flowUtils） | 不变。仍然从各个 carrier 提取参数供上游引用 | 上游参数选择机制不变 |
| `getUpstreamParams`（flowUtils） | 不变 | 上游参数回溯机制不变 |
| 基础类型选项 | 继续使用 `['string', 'number', 'boolean', 'object', 'array']` | 类型系统不变 |

> **结论：** 只变视图层（编辑组件 + 交互），不变数据层和转换层。

## 二、现状说明

### 2.1 当前参数编辑组件

当前项目中参数配置统一使用 `SchemaEditor` 组件。

该组件已支持：

- 参数名称编辑。
- 参数类型编辑。
- 参数位置 `carrier` 编辑。
- 参数描述或字段映射编辑。
- 引用参数模式。
- object / array 嵌套参数。
- 编辑态和只读态。
- 增加、删除参数。
- 锁定参数名、参数类型、参数位置等字段。

**当前 SchemaEditor 的问题：**

1. 强耦合 `form` 实例，内部通过 `form.getFieldValue('apiConfig')` 读取数据、通过 `form.setFieldValue` 写回。
2. 它不是纯粹受控组件，外部无法通过单纯的 `value` + `onChange` 控制。
3. 内部有 `internalChange` 标志和多个 `useEffect` 用于协调 form 和本地 state，逻辑复杂。
4. 入参/出参不分 Tab，所有位置的参数混在一起展示，用户需要在 carrier 下拉手动选择位置。

### 2.2 当前参数数据结构

前端维护的数组格式：

```js
[
  {
    paramName: 'userId',
    paramType: 'string',
    carrier: 'body',       // 标识参数位置
    description: '',
    sourceType: 'static',
    paramValue: '',
    referencePath: '',
    children: [
      // 子参数，递归结构
    ]
  }
]
```

| carrier | 含义 |
|---|---|
| `header` | HTTP 请求头或响应头 |
| `body` | HTTP 请求体或响应体 |
| `query` | URL 查询参数 |

传给后端时的转换后格式（已在 flowUtils 中实现）：

```js
{
  header: { type: 'object', properties: { 'X-Custom': { type: 'string', ... } } },
  query: { type: 'object', properties: { ... } },
  body: { type: 'object', properties: { userId: { type: 'string', ... } } }
}
```

### 2.3 当前完整数据流

```
用户编辑 Tab → CarrierSchemaTabs 筛选当前 carrier → SchemaEditor 渲染
                                                        ↓
                                        用户增删改参数
                                                        ↓
                              CarrierSchemaTabs 合并回完整数组
                                                        ↓
                             页面容器持有完整数组（requestSchema / inputMapping etc.）
                                                        ↓
                             保存时：flowUtils.transformToBackend / ConnectorEditor thunk
                              遍历数组，按 carrier 聚合为 { header, body, query }
                                                        ↓
                             后端接收 carrier 聚合后的数据结构
```

## 三、整改目标

### 3.1 交互目标

接口配置页：

| 区域 | Tab | carrier |
|---|---|---|
| 入参配置 | HTTP 请求头 | `header` |
| 入参配置 | HTTP 请求体 | `body` |
| 入参配置 | URL 查询参数 | `query` |
| 出参配置 | HTTP 响应头 | `header` |
| 出参配置 | HTTP 响应体 | `body` |

连接流编辑器：

| 节点 | 配置项 | Tab |
|---|---|---|
| 触发器节点 | 连接流入参配置 | header / body / query |
| 连接器节点 | 连接器入参映射 | header / body / query |
| 数据输出节点 | 数据组装 | header / body |
| 数据处理节点 | 输出或处理参数配置 | 按实际参数位置配置 |

### 3.2 技术目标

- 将 SchemaEditor 从非受控表单耦合组件，重构为纯受控编辑组件。
- 新增 CarrierSchemaTabs 作为 Tab 包装层，不把 Tab 写进 SchemaEditor 内部。
- 不改动前端数据结构（数组 + carrier），不改动转换层。
- 保证 Tab 切换不丢失其他位置参数。
- 保证新增参数自动写入当前 Tab 对应的 carrier。
- 保证保存前校验覆盖所有 carrier 参数，而非只校验当前 Tab。

## 四、整体方案

### 4.1 组件结构

```text
ConnectorEditor / NodeProperties  （页面容器，持有完整数组）
         │
         ├── CarrierSchemaTabs      （新增，按 carrier 拆分/合并）
         │     └── SchemaEditor     （重构，纯受控编辑引擎）
         │
         ├── AuthFields             （认证参数，不按 Tab）
         ├── CookieAuthFields       （Cookie 专用字段）
         └── SignatureAuthFields    （数字签名专用字段）
```

### 4.2 各层职责

| 层 | 职责 | 关键约束 |
|---|---|---|
| 页面容器 | 持有完整 schema 数组；决定使用哪些 Tabs；控制编辑/只读态；触保存和校验 | 不直接操作 carrier 筛选逻辑 |
| `CarrierSchemaTabs` | Tab 切换；按 carrier 筛选当前数据；合并当前 Tab 修改的数据回完整数组；管理空状态 | 不改动数组结构，只做筛选和合并 |
| `SchemaEditor` | 参数行渲染；参数增删改；嵌套参数树；引用参数选择；字段锁定；只读态 | 不感知 Tab，只编辑收到的数组 |
| 转换层（flowUtils / thunk） | 数组 ↔ `{ header, body, query }` 对象 | 逻辑不变 |

## 五、SchemaEditor 重构设计

这是本次整改的核心改动。重构目标是让 SchemaEditor 变成纯粹的"参数编辑引擎"，不再耦合业务层的 form、apiConfig、Tab 切换。

### 5.1 当前 SchemaEditor 的主要问题

```
问题 1：强耦合 form
  - 内部读 form.getFieldValue('apiConfig')
  - 内部写 form.setFieldValue('apiConfig')
  → CarrierSchemaTabs 无法控制数据何时写回完整数组

问题 2：内部 state 和外部 prop 竞争
  - schemaData 内部 state
  - value 外部 prop
  - internalChange 标志位协调
  → 多个 useEffect 相互覆盖，难以调试

问题 3：carrier 展示和编辑耦合
  - 入参、出参、认证参数都共用同一套 carrier 展示
  - 没有“Tab 模式下隐藏 carrier”的开关
```

### 5.2 重构后 SchemaEditor 的外部接口

```
SchemaEditor
  value: []                                  // 当前参数数组（受控，由 CarrierSchemaTabs 筛选后传入）
  onChange: ([]) => void                     // 参数变化时回传（CarrierSchemaTabs 接收后合并回完整数组）
  editable: boolean                          // 是否可编辑
  scene: string                              // 当前使用场景，如 connectorContract、connectorMapping 等
  fieldVisibility: {}                        // 字段显隐配置
  sourceOptions: []                          // 参数来源选项，如固定值、引用、函数
  functionOptions: []                        // 数据处理节点可选择的函数列表
  mode: 'default' | 'reference'
  showCarrier: boolean                       // Tab 模式下 false，认证参数模式下 true
  fixedCarrier: string                       // Tab 模式下固定写入当前 Tab key
  typeOptions: []
  upstreamParams: []
  lockedFields: {}
  showActionButtons: boolean
```

说明：

- `carrierOptions` 不再由外部传入。当 `showCarrier=true` 时内部固定使用 `['header', 'body', 'query']`。
- `showCarrier=true` 且 `showCarrier=false` 时均不展示 carrier 下拉（Tab 模式下 carrier 由 Tab key 在 CarrierSchemaTabs 层写入）。
- `showCarrier=true` 且非 Tab 模式（认证方式配置）时展示 carrier 下拉。

### 5.3 重构后 SchemaEditor 的内部行为

| 方面 | 重构前 | 重构后 |
|---|---|---|
| 数据来源 | `form.getFieldValue` + `value` prop，双重来源 | 纯 `value` prop，无 form 依赖 |
| 数据写回 | `form.setFieldValue` + `onChange` | 纯 `onChange` |
| 内部 state | `schemaData` + `internalChange` 标志 | 无内部数据 state，直接用 `value` |
| carrier 写入 | 用户手动选择 | Tab 模式下由 `fixedCarrier` 自动写入 |
| carrier 展示 | 始终显示下拉 | `showCarrier=false` 时隐藏 |

### 5.4 重构后 SchemaEditor 不负责的事情

- 不负责 Tab 切换。
- 不负责按 carrier 筛选数据。
- 不负责完整数组的合并。
- 不负责 form 同步。
- 不负责保存校验。
- 不负责空状态。

### 5.5 字段展示控制方案

重构后的 `SchemaEditor` 通过 `scene`、`fieldVisibility`、`sourceOptions` 控制字段展示，不在组件内部写死业务判断。

`CarrierSchemaTabs` 只负责参数位置 Tab，不负责控制参数名称、参数类型、参数说明、参数来源等字段的显隐。

字段展示基础规则：

1. 参数名称、参数类型是基础字段，所有场景都展示。
2. 参数位置只在认证方式配置中展示；入参、出参、节点配置中由 Tab 表达参数位置，不再展示参数位置下拉。
3. 参数说明只在参数定义场景展示，包括连接器配置页入参/出参、触发器节点入参配置。
4. 参数来源只在参数映射或数据组装场景展示，包括连接器节点、数据处理节点、数据输出节点。
5. 数据处理节点的参数来源比其他节点多一个“函数”选项。

字段展示矩阵：

| 场景 | 参数名称 | 参数类型 | 参数位置 | 参数说明 | 参数来源 | 固定值输入 | 引用选择 | 函数选择 |
|---|---|---|---|---|---|---|---|---|
| 接口配置页入参 | 展示 | 展示 | 不展示 | 展示 | 不展示 | 不展示 | 不展示 | 不展示 |
| 接口配置页出参 | 展示 | 展示 | 不展示 | 展示 | 不展示 | 不展示 | 不展示 | 不展示 |
| 认证方式配置 | 展示 | 展示 | 展示 | 可选 | 不展示 | 不展示 | 不展示 | 不展示 |
| 触发器节点入参 | 展示 | 展示 | 不展示 | 展示 | 不展示 | 不展示 | 不展示 | 不展示 |
| 连接器节点入参映射 | 展示 | 展示 | 不展示 | 不展示 | 展示 | 展示 | 展示 | 不展示 |
| 数据处理节点配置 | 展示 | 展示 | 不展示 | 不展示 | 展示 | 展示 | 展示 | 展示 |
| 数据输出节点配置 | 展示 | 展示 | 不展示 | 不展示 | 展示 | 展示 | 展示 | 不展示 |

## 六、CarrierSchemaTabs 设计

### 6.1 组件定位

`CarrierSchemaTabs` 是一个无业务侵入的通用包装组件，用于将同一组 schema 参数按 `carrier` 拆分到不同 Tab 中展示。

它不关心参数是什么业务含义，只关心：

- 完整参数数组 `value`。
- Tab 配置 `tabs`。
- 当前 Tab 的参数变化后如何合并回完整数组。

### 6.2 外部接口

```
CarrierSchemaTabs
  value: []                    // 完整 schema 数组
  tabs: [{ key, label }]      // Tab 配置
  editable: boolean            // 是否可编辑
  schemaEditorProps: {}        // 透传给 SchemaEditor 的配置
  onChange: ([]) => void       // 回传合并后的完整数组
```

### 6.3 内部逻辑

```text
// 数据筛选
currentSchema = value.filter(param => param.carrier === activeTab.key)

// 数据合并
otherParams = value.filter(param => param.carrier !== activeTab.key)
// 当前 Tab 参数统一写入 activeTab.key
patchedParams = currentSchema.map(param => ({ ...param, carrier: activeTab.key }))
// 递归写入子参数
nextValue = [...otherParams, ...patchedParams]
onChange(nextValue)
```

### 6.4 Tabs 常量配置

```js
// 接口入参
export const REQUEST_PARAM_TABS = [
  { key: 'header', label: 'HTTP 请求头' },
  { key: 'body', label: 'HTTP 请求体' },
  { key: 'query', label: 'URL 查询参数' },
];

// 接口出参
export const RESPONSE_PARAM_TABS = [
  { key: 'header', label: 'HTTP 响应头' },
  { key: 'body', label: 'HTTP 响应体' },
];

// 连接流输出
export const OUTPUT_PARAM_TABS = [
  { key: 'header', label: 'HTTP 响应头' },
  { key: 'body', label: 'HTTP 响应体' },
];
```

### 6.5 空状态规则

| 场景 | 展示 |
|---|---|
| 编辑态且当前 Tab 无参数 | 展示空状态 + "添加参数"按钮 |
| 只读态且当前 Tab 无参数 | 展示空状态，不展示添加按钮 |
| 连接器映射锁定态且当前 Tab 无参数 | 展示"当前连接器无该位置参数" |

### 6.6 历史数据兼容

对于后端返回数据中缺少 `carrier` 的参数：

- 入参默认归入 `body`。
- 出参默认归入 `body`。
- 第一次编辑保存时自动补全 carrier。

## 七、接口配置页接入方案

### 7.1 入参配置

```jsx
<CarrierSchemaTabs
  value={apiConfig.requestSchema}
  tabs={REQUEST_PARAM_TABS}
  editable={isEditing}
  schemaEditorProps={{
    editable: isEditing,
    showCarrier: false,
    fixedCarrier: activeTab,   // 由 CarrierSchemaTabs 内部控制
    mode: 'default',
    showActionButtons: isEditing,
  }}
  onChange={(nextSchema) => handleConfigChange('requestSchema', nextSchema)}
/>
```

交互规则：

1. 默认选中 `HTTP 请求体`。
2. 切换 Tab 不丢失其他 Tab 数据（CarrierSchemaTabs 合并时只替换同 carrier 参数）。
3. 新增参数自动写入当前 Tab 对应的 carrier。
4. 不展示 carrier 下拉。
5. 保存时提交完整 `requestSchema` 数组，转换层通过 `buildContractFromSchema` 按 carrier 聚合。

### 7.2 出参配置

```jsx
<CarrierSchemaTabs
  value={apiConfig.responseSchema}
  tabs={RESPONSE_PARAM_TABS}
  editable={isEditing}
  schemaEditorProps={{
    editable: isEditing,
    showCarrier: false,
    fixedCarrier: activeTab,
    mode: 'default',
    showActionButtons: isEditing,
  }}
  onChange={(nextSchema) => handleConfigChange('responseSchema', nextSchema)}
/>
```

交互规则：

1. 默认选中 `HTTP 响应体`。
2. 出参不支持 query，Tab 只包含 header 和 body。
3. 新增参数只允许写入 header 或 body（由 CarrierSchemaTabs 的 Tab key 控制）。

### 7.3 认证参数配置

认证参数不按 Tab 展示。

| 认证方式 | 配置方式 |
|---|---|
| SOA | 保持现有 SchemaEditor，显示 carrier 下拉（新增参数时可选手动位置） |
| APIG | 保持现有 SchemaEditor，显示 carrier 下拉 |
| Cookie | 接口配置页只定义参数名列表，展示只读说明"连接流中引用上游参数配置" |
| 数字签名 | 使用专用字段展示签名参数（参数名、参数位置、签名密钥），密钥在只读态掩码 |

认证参数配置不使用 `CarrierSchemaTabs`，不使用 `fixedCarrier`，保持 `showCarrier=true`。

## 八、连接流编辑器接入方案

### 8.1 触发器节点 - 连接流入参配置

```jsx
<CarrierSchemaTabs
  value={inputContractArray}   // 数组格式
  tabs={REQUEST_PARAM_TABS}
  editable={true}
  schemaEditorProps={SCHEMA_EDITOR_CONFIG.inputContractTab}
  onChange={(nextSchema) => handleConfigChange('inputContract', nextSchema)}
/>
```

规则：

- 用户在不同 Tab 中定义连接流入口参数。
- 保存时通过 `transformToBackend` 中 `NODE_TYPE_TO_MAPPING_KEY` 处理 `inputContract` 字段。
- 后续节点引用上游参数时，能看到参数位置和路径。

### 8.2 连接器节点 - 入参映射

```jsx
<CarrierSchemaTabs
  value={inputMappingArray}
  tabs={REQUEST_PARAM_TABS}
  editable={true}
  schemaEditorProps={{
    ...SCHEMA_EDITOR_CONFIG.inputMapping,
    showCarrier: false,
    fixedCarrier: activeTab,
    showActionButtons: false,   // 连接器映射不允许增删参数
  }}
  onChange={(nextSchema) => handleConfigChange('inputMapping', nextSchema)}
/>
```

规则：

- 参数来源于连接器接口配置定义。
- 参数名、参数类型、参数位置保持锁定。
- 用户只配置值来源（sourceType + referencePath / paramValue）。
- 不允许新增或删除连接器定义参数。
- 某个位置无参数时展示"当前连接器无该位置参数"空状态。

### 8.3 数据输出节点 - 数据组装

```jsx
<CarrierSchemaTabs
  value={outputMappingArray}
  tabs={OUTPUT_PARAM_TABS}     // 只有 header, body
  editable={true}
  schemaEditorProps={{
    ...SCHEMA_EDITOR_CONFIG.outputMapping,
    showCarrier: false,
    fixedCarrier: activeTab,
  }}
  onChange={(nextSchema) => handleConfigChange('outputMapping', nextSchema)}
/>
```

规则：

- 响应头和响应体分开 Tab 配置。
- 保存时通过 `transformToBackend` 中 `NODE_TYPE_TO_MAPPING_KEY` 处理 `outputMapping` 字段。

### 8.4 数据处理节点（后续）

后续新增数据处理节点后，复用同一方案，Tabs 按节点语义选择：

- 输出给下游连接器：`REQUEST_PARAM_TABS`（header / body / query）。
- 输出给最终响应：`OUTPUT_PARAM_TABS`（header / body）。

## 十一、参数校验方案

### 11.1 校验范围

保存前校验完整 schema 数组：

接口配置页：

- 入参 header。
- 入参 body。
- 入参 query。
- 出参 header。
- 出参 body。

连接流编辑器：

- 触发器输入契约。
- 连接器输入映射。
- 数据输出映射。
- 数据处理节点参数（后续）。
- Cookie 认证值来源。

### 11.2 object / array 校验规则

| 参数类型 | 规则 |
|---|---|
| object | 必须至少包含一个基础类型子参数 |
| array | 必须至少包含一个基础类型子参数 |
| 嵌套 object / array | 递归校验每一层 |
| 空 children | 不允许保存 |
| 仅包含复杂类型子参数 | 不允许保存 |

基础类型：`string`、`number`、`boolean`。

### 11.3 错误提示格式

```text
入参配置-HTTP 请求体-用户信息：object 类型参数必须包含基础类型子参数
出参配置-HTTP 响应头-result：array 类型参数必须包含基础类型子参数
连接器节点「获取用户信息」Cookie 认证参数「SESSION」未设置值来源
```

错误提示包含参数位置和参数路径，便于定位。

## 十二、改前后对照

### 12.1 SchemaEditor 改前后对比

| 维度 | 改前 | 改后 |
|---|---|---|
| 数据来源 | `form` + `value` prop 双重来源 | 纯 `value` prop 受控 |

## 十三、实施建议

### 13.1 实施顺序

1. **扩展常量配置** — `AUTH_TYPE_OPTIONS` 新增 Cookie / 数字签名；`HTTP_METHOD_OPTIONS` 移除 PATCH；新增 `REQUEST_PARAM_TABS` / `RESPONSE_PARAM_TABS` / `OUTPUT_PARAM_TABS`。
2. **重构 SchemaEditor** — 改为纯受控组件（`value` + `onChange`），新增 `fixedCarrier` / `showCarrier`，移除 form 耦合。
3. **新增 CarrierSchemaTabs** — Tab 切换、筛选 carrier、合并 carrier、空状态。
4. **接口配置页接入** — 入参、出参改用 CarrierSchemaTabs；移除 carrier 下拉；接入编辑/只读态。
5. **连接流编辑器节点配置接入** — 触发器、连接器、数据输出节点改用 CarrierSchemaTabs。
6. **扩展认证方式** — 新增 Cookie、数字签名字段；Cookie 在接口配置页只定义参数，连接流连接器节点中配置值来源。
7. **参数校验增强** — object / array 校验、保存前完整 schema 校验、错误提示带位置路径。
8. **联调验证** — 保存、回显、Tab 切换、只读态、认证方式切换、版本切换。

### 13.2 风险与规避

| 风险 | 规避方式 |
|---|---|
| SchemaEditor 重构影响现有页面 | 先确认所有调用方，确保 `value` + `onChange` 能覆盖所有场景 |
| Tab 切换导致其他 carrier 参数丢失 | CarrierSchemaTabs 合并时只替换当前 carrier |
| 历史数据缺少 carrier | 默认归入 body，保存时自动补齐 |
| 连接器映射参数被误删 | `showActionButtons=false`，不允许增删 |
| 子参数 carrier 不一致 | 子参数递归继承父级 carrier |
| 保存只校验当前 Tab | 校验函数接收完整数组，不感知 Tab |

## 十四、验证点

### 14.1 接口配置页

- 入参展示 HTTP 请求头、HTTP 请求体、URL 查询参数三个 Tab。
- 出参展示 HTTP 响应头、HTTP 响应体两个 Tab。
- Tab 切换不丢失其他 carrier 参数。
- 新增参数自动写入当前 Tab 的 carrier。
- Tab 模式下 carrier 下拉隐藏。
- 非 Tab 模式下（认证参数）carrier 下拉正常显示。
- 保存后后端结构按 header、body、query 聚合。
- 回显时能根据 carrier 分配到对应 Tab。
- 只读态下可切换 Tab，但不能编辑。

### 14.2 连接流编辑器

- 触发器入参按 header、body、query 分 Tab。
- 连接器入参映射按 header、body、query 分 Tab，参数名/类型/位置锁定。
- 数据输出按 header、body 分 Tab。
- 保存后节点配置结构不丢失 carrier 信息。
- 回显时能恢复到对应的 Tab。
- 上游参数选择时显示参数位置和路径。

### 14.3 Cookie 认证

- 接口配置页 Cookie 不填写运行时值。
- 连接流连接器节点中展示 Cookie 值来源配置。
- Cookie 参数只能选择上游参数。
- 未配置 Cookie 值来源时保存拦截。

### 14.4 校验

- object 无基础类型子参数时保存失败。
- array 无基础类型子参数时保存失败。
- 嵌套复杂类型递归校验生效。
- 错误提示包含参数位置和参数路径。

## 十五、最终结论

本方案以 **"重构 SchemaEditor + 新增 CarrierSchemaTabs + 不变动转换层"** 为核心策略：

1. **SchemaEditor** 回归为纯编辑引擎，不再耦合 form 和 apiConfig。
2. **CarrierSchemaTabs** 统一管理参数位置 Tab，各页面复用。
3. **转换层和数据结构** 完全不变，降低实施风险。

整体改动集中在视图层，数据层和转换层零改动，在满足新交互需求的同时将影响范围控制到最小。
