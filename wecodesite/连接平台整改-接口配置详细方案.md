# 连接平台整改-接口配置详细方案

> 本文档描述接口配置页面的详细整改方案，包含版本选择、编辑/只读切换、认证方式扩展、出入参位置 Tab、参数校验和版本接口规划。

---

## 一、页面现状

- 当前是单份接口配置编辑页，无版本概念。
- 页面默认可编辑，没有只读态。
- HTTP 方法包含 PATCH。
- 认证方式仅支持 SOA、APIG。
- 缺 Cookie 认证和数字签名认证。
- 入参、出参配置缺少按参数位置区分的 Tab 说明。
- object / array 参数缺少保存前校验。

---

## 二、整改目标

- 顶部增加版本选择卡。
- 页面默认只读，草稿版本可编辑。
- 版本状态支持草稿、已发布、已失效。
- 移除 PATCH 协议。
- 新增 Cookie 和数字签名认证方式。
- 入参配置按 HTTP 请求头、HTTP 请求体、URL 参数区分 Tab。
- 出参配置按 HTTP 响应头、HTTP 响应体区分 Tab。
- object / array 参数必须包含基础类型子参数，且嵌套层级最多支持 10 层。
- 当连接器版本列表为空时，仅展示“创建草稿”按钮，点击后新增草稿版本。
- 数字签名参数名称默认空且可编辑，`X-Signature` 放在签名值来源中展示。
- 数字签名密钥掩码通过独立按钮控制，不和编辑态绑定。

## 四、整体页面方案

接口配置页由“页面头部、版本选择卡、接口基础配置、认证方式配置、入参配置、出参配置”组成。

| 区域 | 调整前 | 调整后 |
|------|--------|--------|
| 页面头部 | 仅展示接口配置入口信息 | 保留返回与标题，页面主体由版本驱动展示 |
| 版本选择卡 | 无 | 统一承载版本切换、版本状态和版本操作 |
| 接口基础配置 | 默认可编辑 | 默认只读，草稿点击编辑后可修改 |
| 认证方式配置 | 仅 SOA、APIG | 扩展 Cookie、数字签名，并跟随编辑态禁用或启用 |
| 入参配置 | 未明确参数位置 Tab | 按 HTTP 请求头、HTTP 请求体、URL 参数分 Tab 管理 |
| 出参配置 | 未明确参数位置 Tab | 按 HTTP 响应头、HTTP 响应体分 Tab 管理 |

## 五、整改点、详细思路与整改方案

### 5.1 移除 PATCH

#### 整改思路

接口配置页的协议类型需要和本期确认的后端能力保持一致。页面中所有协议选择入口都只保留 GET、POST、PUT、DELETE，避免用户创建后端不支持的接口配置。

#### 整改方案

| 项目 | 方案 |
|------|------|
| 常量配置 | 从 HTTP 方法选项中移除 PATCH，只保留 GET、POST、PUT、DELETE。 |
| 页面展示 | 协议类型按钮组、下拉项或单选项不再展示 PATCH。 |
| 默认值处理 | 新建或无数据时默认选择 GET；历史数据如果返回 PATCH，页面不提供继续选择入口，联调时按后端规则处理。 |
| 保存数据 | 保存时只允许提交 GET、POST、PUT、DELETE。 |
| 验证点 | 页面、默认值、保存数据中均不再出现 PATCH。 |

关键代码片段：

```jsx
/**
 * HTTP 方法选项。
 * 接口配置页协议类型只允许选择后端当前支持的四种方法。
 */
export const HTTP_METHOD_OPTIONS = ['GET', 'POST', 'PUT', 'DELETE'];

/**
 * 接口配置默认值。
 * 新建或无配置时默认使用 GET，避免空值导致协议按钮无选中态。
 */
export const DEFAULT_API_CONFIG = {
  protocolType: 'GET',
  protocolAddress: '',
  authType: '',
  authRequestSchema: [],
  requestSchema: [],
  responseSchema: [],
};
```

### 5.2 版本状态枚举

#### 整改思路

接口配置从单份配置改为多版本配置后，页面展示和按钮权限都需要由版本状态驱动。版本状态统一抽象为草稿、已发布、已失效三类，页面只根据状态决定可编辑性和可执行操作。

#### 整改方案

| 状态 | 展示文案 | 页面可编辑 | 主要操作 |
|------|----------|------------|----------|
| 草稿 | 草稿 | 点击编辑后可编辑 | 编辑、保存、发布 |
| 已发布 | 已发布 | 不可编辑 | 复制到草稿、失效 |
| 已失效 | 已失效 | 不可编辑 | 复制到草稿、恢复、删除 |

补充说明：

- 版本状态枚举放在公共常量中，便于版本选择卡和页面主体共同使用。
- 状态颜色与文案保持统一，避免不同区域展示不一致。
- 页面是否显示“编辑”按钮、是否禁用表单控件，均以当前版本状态为判断基础。

关键代码片段：

```js
/**
 * 连接器版本状态枚举。
 * 页面展示状态标签和操作按钮时统一使用该配置。
 */
export const CONNECTOR_VERSION_STATUS = {
  DRAFT: 0,
  PUBLISHED: 4,
  INVALID: 5,
};

/**
 * 连接器版本状态展示映射。
 * 用于版本选择卡中的状态文案和颜色。
 */
export const CONNECTOR_VERSION_STATUS_MAP = {
  [CONNECTOR_VERSION_STATUS.DRAFT]: { text: '草稿', color: 'default' },
  [CONNECTOR_VERSION_STATUS.PUBLISHED]: { text: '已发布', color: 'green' },
  [CONNECTOR_VERSION_STATUS.INVALID]: { text: '已失效', color: 'red' },
};
```

### 5.3 版本选择卡

#### 整改思路

版本选择卡是接口配置页的主控区域，负责版本切换、状态展示和版本操作。页面主体只展示当前选中版本的数据，不直接关心版本列表如何加载和切换。

#### 整改方案

| 功能 | 方案 |
|------|------|
| 版本下拉 | 展示当前连接器下的所有版本，切换后加载对应版本详情；版本列表为空时展示空占位。 |
| 空版本操作 | 当版本列表为空时，版本列表右侧仅展示“创建草稿”按钮，点击后新增一个草稿版本并切换为当前版本。 |
| 状态标签 | 根据当前版本状态展示草稿、已发布、已失效。 |
| 草稿操作 | 默认只读；点击编辑进入编辑态；编辑态中点击保存提交当前版本配置；点击发布触发发布操作。 |
| 已发布操作 | 不展示编辑入口；提供复制到草稿和失效，失效需要二次确认。 |
| 已失效操作 | 不展示编辑入口；提供复制到草稿、恢复、删除，删除需要二次确认。 |
| 切换保护 | 编辑态切换版本时需要提示用户当前修改未保存，确认后再切换。 |

交互规则：

- 页面首次进入时加载版本列表，并默认选中后端返回的默认版本或最新版本。
- 如果版本列表为空，版本选择卡仅展示“创建草稿”按钮，不展示编辑、发布、复制到草稿、失效、恢复、删除等按钮。
- 点击“创建草稿”后新增草稿版本，刷新版本下拉和按钮状态，并将新草稿设为当前版本。
- 切换版本后退出编辑态，页面恢复只读。
- 复制到草稿成功后刷新版本列表，并切换到新草稿版本。
- 发布、失效、恢复、删除成功后刷新版本列表和当前版本详情；删除最后一个版本后留在当前页展示“创建草稿”。

关键代码片段：

```jsx
/**
 * 根据版本状态生成版本操作按钮。
 * 版本选择卡只负责抛出操作事件，具体接口调用由页面容器处理。
 */
const getVersionActionButtons = (options) => {
  const { status, versions = [] } = options;

  if (versions.length === 0) {
    return [{ key: 'createDraft', label: '创建草稿', type: 'primary' }];
  }

  const actionMap = {
    [CONNECTOR_VERSION_STATUS.DRAFT]: [
      { key: 'edit', label: '编辑' },
      { key: 'publish', label: '发布', type: 'primary' },
    ],
    [CONNECTOR_VERSION_STATUS.PUBLISHED]: [
      { key: 'copy', label: '复制到草稿' },
      { key: 'invalidate', label: '失效', danger: true },
    ],
    [CONNECTOR_VERSION_STATUS.INVALID]: [
      { key: 'copy', label: '复制到草稿' },
      { key: 'restore', label: '恢复' },
      { key: 'delete', label: '删除', danger: true },
    ],
  };

  return actionMap[status] || [];
};

/**
 * 版本选择卡接入示例。
 * 页面通过 currentVersion 控制版本详情展示和表单可编辑范围。
 */
<VersionSelectorCard
  versions={versionList}
  value={currentVersionId}
  status={currentVersion?.status}
  onChange={handleVersionChange}
  onAction={handleVersionAction}
/>;
```

### 5.4 编辑 / 只读切换

#### 整改思路

接口配置页默认只读，避免用户误改已发布或已失效版本。只有草稿版本允许进入编辑态，编辑态中才允许修改基础配置、认证方式、入参和出参。

#### 整改方案

| 场景 | 页面状态 |
|------|----------|
| 首次进入页面 | 默认只读。 |
| 当前版本为草稿 | 展示编辑入口。 |
| 点击编辑 | 保存当前配置快照，表单控件启用，入参和出参操作按钮显示。 |
| 点击保存 | 先执行参数校验，再提交数据；保存成功后退出编辑态。 |
| 切换版本 | 如果处于编辑态，先提示是否放弃修改；确认后切换并恢复只读。 |
| 已发布 / 已失效版本 | 始终只读，不展示编辑入口。 |

控件禁用范围：

- 协议类型。
- 协议地址。
- 认证方式选择。
- 认证参数配置。
- 入参、出参的新增、编辑、删除操作。

关键代码片段：

```jsx
/**
 * 判断当前版本是否允许进入编辑态。
 * 只有草稿版本可以编辑，已发布和已失效版本均保持只读。
 */
const canEditCurrentVersion = currentVersion?.status === CONNECTOR_VERSION_STATUS.DRAFT;

/**
 * 进入编辑态。
 * 进入前保存快照，用于切换版本或取消编辑时恢复。
 */
const handleEnterEdit = () => {
  setEditingSnapshot(JSON.parse(JSON.stringify(apiConfig)));
  setIsEditing(true);
};

/**
 * 接口基础配置接入编辑态。
 * disabled 由版本状态和编辑态共同控制。
 */
<Radio.Group
  value={apiConfig.protocolType}
  disabled={!isEditing}
  onChange={(event) => handleConfigChange('protocolType', event.target.value)}
>
  {HTTP_METHOD_OPTIONS.map((method) => (
    <Radio.Button key={method} value={method}>{method}</Radio.Button>
  ))}
</Radio.Group>;

/**
 * SchemaEditor 接入编辑态。
 * 只读态隐藏新增、删除等操作按钮。
 */
<SchemaEditor
  form={form}
  apiConfig={apiConfig}
  editable={isEditing}
  showActionButtons={isEditing}
  {...REQUEST_SCHEMA_CONFIG}
/>;
```

### 5.5 认证方式扩展

#### 整改思路

认证方式由原来的 SOA、APIG 扩展为 SOA、APIG、Cookie、数字签名。认证方式可以多选，勾选后展示对应参数配置区域。认证参数同样受编辑态控制，非编辑态只读展示。

#### 整改方案

| 认证类型 | 配置内容 | 编辑规则 |
|----------|----------|----------|
| SOA | 参数名称、参数类型、参数位置、值来源 | 编辑态可维护，非编辑态只读。 |
| APIG | 参数名称、参数类型、参数位置、值来源 | 编辑态可维护，非编辑态只读。 |
| Cookie | 参数名称、参数类型、参数位置、值来源说明 | 接口配置页不允许手填运行时取值，运行时值由连接流编辑器引用上游参数注入。 |
| 数字签名 | 参数名称、参数类型、参数位置、签名值来源、签名密钥 | 参数名称默认空且可编辑；`X-Signature` 默认放到签名值来源；签名密钥是否掩码由独立按钮控制，不和编辑态绑定。 |

补充说明：

- Cookie 的值来源在接口配置页只表达“需要从连接流引用参数配置”，不在此页面直接填写具体上游参数。
- 数字签名的参数名称默认不填，由用户按实际接口要求维护。
- 数字签名的 `X-Signature` 作为签名值来源展示，参数名称和签名值来源不再使用同一个字段。
- 数字签名密钥属于敏感字段，默认掩码展示；用户可通过独立按钮切换显示或隐藏，该操作不依赖页面是否处于编辑态。
- 认证方式取消勾选时，对应配置不参与保存。

关键代码片段：

```jsx
/**
 * 认证方式选项。
 * 扩展 Cookie 和数字签名认证类型。
 */
export const AUTH_TYPE_OPTIONS = [
  { value: 'SOA', label: 'SOA' },
  { value: 'APIG', label: 'APIG' },
  { value: 'Cookie', label: 'Cookie' },
  { value: 'SIGNATURE', label: '数字签名' },
];

/**
 * 认证参数默认配置。
 * Cookie 的运行时取值不在接口配置页手填，仅保留 valueSource 占位说明。
 */
export const AUTH_SCHEMA_MAP = {
  SOA: [{ paramName: 'X-Huawei-Auth', paramType: 'string', carrier: 'header', children: [] }],
  APIG: [{ paramName: 'X-HW-ID', paramType: 'string', carrier: 'header', children: [] }],
  Cookie: [{ paramName: 'Cookie', paramType: 'string', carrier: 'header', valueSource: 'FLOW_REFERENCE', children: [] }],
  SIGNATURE: [{ paramName: '', paramType: 'string', carrier: 'header', signatureValueSource: 'X-Signature', secret: '', children: [] }],
};

/**
 * 数字签名密钥展示。
 * 密钥默认掩码展示，显示/隐藏由独立按钮控制，不依赖编辑态。
 */
const SignatureSecretField = (props) => {
  const { value, masked, disabled, onChange, onToggleMask } = props;

  return (
    <Input
      value={value}
      type={masked ? 'password' : 'text'}
      disabled={disabled}
      onChange={(event) => onChange(event.target.value)}
      suffix={<Button onClick={onToggleMask}>{masked ? '显示' : '隐藏'}</Button>}
    />
  );
};
```

### 5.6 入参 / 出参参数位置 Tab

#### 整改思路

入参和出参配置需要按参数位置拆分展示，避免所有参数混在一个列表中导致位置语义不清。Tab 只负责展示当前参数位置的数据，保存时仍然需要把各 Tab 下的数据合并为完整配置提交。

#### 整改方案

| 区域 | Tab | 参数位置含义 | 数据范围 |
|------|-----|--------------|----------|
| 入参配置 | HTTP 请求头 | 请求 Header 参数 | 仅展示和维护 header 入参。 |
| 入参配置 | HTTP 请求体 | 请求 Body 参数 | 仅展示和维护 body 入参。 |
| 入参配置 | URL 参数 | URL Query 参数 | 仅展示和维护 query 入参。 |
| 出参配置 | HTTP 响应头 | 响应 Header 参数 | 仅展示和维护 header 出参。 |
| 出参配置 | HTTP 响应体 | 响应 Body 参数 | 仅展示和维护 body 出参。 |

交互规则：

- 点击 Tab 时切换当前参数列表，不丢失其他 Tab 已编辑的数据。
- 每个 Tab 内支持新增、编辑、删除参数。
- 参数新增时自动带入当前 Tab 对应的位置，不要求用户重复选择参数位置。
- 保存时按入参、出参分别聚合所有 Tab 的数据。
- 非编辑态下 Tab 可切换查看，但参数输入框和操作按钮禁用。

数据组织建议：

| 数据层级 | 说明 |
|----------|------|
| 入参 | 按 header、body、query 三组维护。 |
| 出参 | 按 header、body 两组维护。 |
| 参数项 | 保留参数名称、参数类型、参数描述、子参数、参数位置等字段。 |

关键代码片段：

```jsx
/**
 * 入参位置 Tab 配置。
 * 每个 Tab 对应一个 carrier，新增参数时自动写入当前 carrier。
 */
export const REQUEST_PARAM_TABS = [
  { key: 'header', label: 'HTTP 请求头' },
  { key: 'body', label: 'HTTP 请求体' },
  { key: 'query', label: 'URL 参数' },
];

/**
 * 出参位置 Tab 配置。
 * 出参只支持响应头和响应体，不支持 query。
 */
export const RESPONSE_PARAM_TABS = [
  { key: 'header', label: 'HTTP 响应头' },
  { key: 'body', label: 'HTTP 响应体' },
];

/**
 * 按参数位置筛选当前 Tab 数据。
 * SchemaEditor 只接收当前 Tab 下的参数列表。
 */
const getSchemaByCarrier = (params, carrier) => {
  return (params || []).filter((param) => param.carrier === carrier);
};

/**
 * 更新当前 Tab 参数数据。
 * 保存当前 Tab 时只替换同 carrier 数据，其他 Tab 数据保持不变。
 */
const updateSchemaByCarrier = (options) => {
  const { allParams = [], carrier, nextParams = [] } = options;
  const otherParams = allParams.filter((param) => param.carrier !== carrier);
  const patchedParams = nextParams.map((param) => ({ ...param, carrier }));

  return [...otherParams, ...patchedParams];
};

/**
 * 入参 Tab 接入示例。
 * activeRequestCarrier 用于控制当前展示的入参位置。
 */
<Tabs
  activeKey={activeRequestCarrier}
  items={REQUEST_PARAM_TABS.map((tab) => ({ key: tab.key, label: tab.label }))}
  onChange={setActiveRequestCarrier}
/>
<SchemaEditor
  form={form}
  apiConfig={{
    ...apiConfig,
    requestSchema: getSchemaByCarrier(apiConfig.requestSchema, activeRequestCarrier),
  }}
  editable={isEditing}
  fixedCarrier={activeRequestCarrier}
  onSchemaChange={(nextParams) => handleRequestSchemaChange({ carrier: activeRequestCarrier, nextParams })}
/>;
```

### 5.7 object / array 参数校验

#### 整改思路

复杂类型参数如果没有基础类型子参数，后续运行时无法形成明确的数据结构。保存前需要递归校验所有入参和出参，确保 object / array 类型下至少包含 string、number、boolean 中的一种基础类型子参数。

#### 整改方案

| 校验对象 | 校验规则 |
|----------|----------|
| object 参数 | 必须至少包含一个基础类型子参数。 |
| array 参数 | 必须至少包含一个基础类型子参数。 |
| 嵌套复杂参数 | 递归校验每一层 object / array。 |
| 层级上限 | object / array 嵌套最多支持 10 层，达到第 10 层后不再允许添加子节点。 |
| 空 children | 不允许保存。 |
| 仅嵌套复杂类型 | 不允许保存，直到某一层包含基础类型子参数。 |

保存流程：

1. 点击保存后先收集接口基础配置、认证配置、入参配置、出参配置。
2. 对入参所有 Tab 数据执行复杂类型校验和 10 层层级校验。
3. 对出参所有 Tab 数据执行复杂类型校验和 10 层层级校验。
4. 如果存在错误，提示具体参数路径并阻止保存。
5. 校验通过后再调用保存接口。

错误提示要求：

- 提示信息需要包含参数路径，方便用户定位问题。
- 优先提示第一个错误，避免一次弹出过多提示影响操作。

关键代码片段：

```js
/**
 * 基础参数类型集合。
 * object / array 的子参数中必须至少包含一种基础类型。
 */
const BASIC_PARAM_TYPES = ['string', 'number', 'boolean'];

/**
 * 参数嵌套层级上限。
 * object / array 最多支持 10 层，超过后不允许保存。
 */
const MAX_SCHEMA_DEPTH = 10;

/**
 * 校验单个参数节点。
 * 复杂类型必须包含至少一个基础类型子参数，并递归校验子层级。
 */
const validateParamNode = (options) => {
  const { param, path, depth } = options;
  const currentPath = path ? `${path}.${param.paramName}` : param.paramName;
  const isComplexType = param.paramType === 'object' || param.paramType === 'array';

  if (depth > MAX_SCHEMA_DEPTH) {
    return `参数「${currentPath}」层级最多支持 ${MAX_SCHEMA_DEPTH} 层`;
  }

  if (!isComplexType) {
    return null;
  }

  const children = Array.isArray(param.children) ? param.children : [];
  const hasBasicChild = children.some((child) => BASIC_PARAM_TYPES.includes(child.paramType));

  if (!hasBasicChild) {
    return `参数「${currentPath}」必须包含至少一个基础类型子参数`;
  }

  for (const child of children) {
    const errorMessage = validateParamNode({ param: child, path: currentPath, depth: depth + 1 });

    if (errorMessage) {
      return errorMessage;
    }
  }

  return null;
};

/**
 * 保存前校验接口配置参数。
 * 入参和出参都需要执行复杂类型校验。
 */
export const validateApiConfigBeforeSave = (apiConfig) => {
  const schemas = [
    ...(apiConfig.requestSchema || []),
    ...(apiConfig.responseSchema || []),
  ];

  for (const param of schemas) {
    const errorMessage = validateParamNode({ param, path: '', depth: 1 });

    if (errorMessage) {
      return errorMessage;
    }
  }

  return null;
};

/**
 * 保存入口接入校验。
 * 校验不通过时阻止保存，并提示第一个错误。
 */
const handleSubmit = async () => {
  const apiConfig = form.getFieldValue('apiConfig') || {};
  const errorMessage = validateApiConfigBeforeSave(apiConfig);

  if (errorMessage) {
    message.error(errorMessage);
    return;
  }

  await saveConnectorVersionConfig({ connectorId, versionId: currentVersionId, apiConfig });
};
```

### 5.8 版本端点

#### 整改思路

版本相关操作统一通过连接器 ID 和版本 ID 定位。页面只调用版本列表、版本详情、发布、复制到草稿、状态变更、删除等端点。

#### 整改方案

| 操作 | 方法 | 地址 | 页面行为 |
|------|------|------|----------|
| 版本列表 | GET | `/connectors/{connectorId}/versions` | 页面初始化和版本操作成功后刷新版本下拉；为空时展示创建草稿入口。 |
| 创建草稿 | POST | `/connectors/{connectorId}/versions/draft` | 当版本列表为空时创建首个草稿版本。 |
| 版本详情 | GET | `/connectors/{connectorId}/versions/{versionId}` | 切换版本后加载页面主体配置。 |
| 发布 | POST | `/connectors/{connectorId}/versions/{versionId}/publish` | 草稿版本发布为已发布。 |
| 复制到草稿 | POST | `/connectors/{connectorId}/versions/{versionId}/copy` | 基于已发布或已失效版本生成草稿。 |
| 状态变更 | PUT | `/connectors/{connectorId}/versions/{versionId}/status` | 用于失效和恢复。 |
| 删除 | DELETE | `/connectors/{connectorId}/versions/{versionId}` | 删除已失效版本。 |

补充说明：

- 所有版本操作成功后都需要刷新版本列表。
- 当前版本被删除后，应切换到后端返回的默认版本或列表中的第一个可用版本。
- 如果删除后版本列表为空，页面留在接口配置页，版本选择卡仅展示“创建草稿”按钮。
- 接口失败时保留当前页面状态，不清空已有配置。
