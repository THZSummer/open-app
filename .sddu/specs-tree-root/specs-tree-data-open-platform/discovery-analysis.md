# 需求挖掘分析 - 数据开放平台

**会话 ID**: feature-session-001  
**分析创建时间**: 2026-04-03  
**状态**: 进行中

---

## 一、需求概述

### 1.1 核心定位
**数据开放平台**是 open-app 体系下的子平台，聚焦 XX 通讯平台的数据开放管理，将企业内部 XX 平台的数据开放给企业内部其它三方平台消费使用。

### 1.2 目标用户
| 角色 | 职责 |
|------|------|
| 数据 Owner | 注册数据、生产数据 |
| 开放平台管理员 | 审批数据注册信息 |
| 三方平台业务方 | 订阅数据、消费数据 |

---

## 二、核心痛点

### 2.1 现状描述
- AI 大行其道，企业内部 XX 通讯平台的 AI 能力较为薄弱
- 平台本身的能力多数局限在平台内部使用
- 企业内其它三方平台没有很好的把业务融进来，利用平台的资源能力开展他们自己的业务

### 2.2 核心问题
**能力封闭**：XX 平台的数据和能力无法被企业内部其他三方平台有效利用，形成数据孤岛和能力孤岛。

### 2.3 目标状态
**生态开放**：让三方平台能利用 XX 平台资源开展业务，形成企业内部的能力生态。

---

## 三、关键决策记录

### 3.1 数据范围
**决策**: 待定，暂无明确优先级与范围

**洞察**:
- 平台需要具备通用性和扩展性，能够适配不同类型的数据域
- 建设思路是先搭平台框架，数据域后续逐步接入

### 3.2 数据开放形式
**决策**: 多种形式均有可能（API 查询、事件订阅、批量导出、数据同步、数据服务）

**洞察**:
- 平台需要支持多种开放形式，不能局限于单一模式
- 可复用企业内部消息平台，降低建设成本
- 参考竞品：飞书、钉钉

### 3.3 核心流程
**决策**: 确认流程框架
```
数据 Owner 注册 → 管理员审批 → 数据上架 → 三方平台订阅 → 消费数据 → 审计日志
```

**详细决策**:
| 确认点 | 决策 |
|--------|------|
| 注册信息 | 参考竞品 |
| 审批触发 | 注册和更新都需要审批（取决于业务决策） |
| 消费凭证 | 三方平台需创建应用，获取 AK/SK；消费个人数据可能需用户授权 |
| 数据生产 | 主要是系统自动生产，不依赖人工推送 |

### 3.4 与 open-app 的关系
**决策**: 
- 业务归属：子平台，归于 open-app 体系
- 技术架构：技术上单独的服务
- 实现方式：业务上增强

**现状**:
- open-app 现有能力（应用管理、权限、OAuth 2.0）有代码但服务化程度低
- 数据开放平台是 open-app 的能力增强，不是从零开始

## 四、待深入问题

### 4.1 数据 Owner 视角（已完成）
- **谁是数据 Owner？** ✅ 业务模块负责人（如 IM 业务模块负责人拥有群、聊天等消息数据对象）
- **审批权限？** ✅ 平台审批只对技术负责（数据符合平台规范即可），业务上是否开放由线下沟通决议
- **数量关系？** ✅ 通常是一个，不必过于纠结
- **开放动机？** ✅ 价值驱动（对本业务模块的价值、对 XX 通讯平台的价值、对企业整体的价值、对用户的价值、对企业内三方平台的价值）

### 4.2 消费方视角（部分完成）

- **想用数据做什么？** ⏳ 待业务调研

- **现状？** ✅ 可能已经在消费，只是没有标准统一的通道（所以需要数据开放平台做这个事情）
- **具体 AI 应用规划？** ⏳ 待调研

### 4.4 数据分类与敏感度（部分完成）
- **敏感度分级？** ✅ 逻辑或事实上肯定有

- **数据对象清单？** ❌ XX 平台内部有哪些数据对象，也没有一个统一的清单
- **开放策略差异？** ✅ 不同级别自然存在开放策略的差异

- 数据开放平台可能需要承担**数据资产目录**的职责
- 数据分级标准需要**可配置、可演进**，不是硬编码

- **运营角色？** ✅ 应该有专门的平台运营人员

**关键洞察**:
- 线下流程 + 线上平台结合

## 八、阶段三需求挖掘：权限关系（2026-04-09）

### 8.1 核心设计原则

**双重权限模型**：
```
消费权限 = 通道权限 ⊗ 数据对象权限
```

| 维度 | 设计决策 |
|------|---------|
| **数据对象定义** | 业务逻辑层面概念（如：员工信息对象、文件对象、群组对象），不是数据库表级别 |
| **通道类型** | P0: REST API、Webhook、消息队列 |
| **权限主体** | 开放平台的应用（Application），永久有效，主动回收 |
| **审批流程** | 一次申请=一次审批，动态生成审批链 |
| **审批角色** | 数据 Owner、通道 Owner、平台管理员、上级/CTO（L3+）、安全合规（L3+） |
| **权限回收** | 低频谨慎，提前通知，平滑过渡 |
| **审计日志** | 以通道为入口记录，多用途支持 |
| **应用隔离** | 无共享/继承，岗位稳定，凭证无感 |

### 8.2 关键洞察

**1. 数据对象是业务概念，不是技术概念**
- 数据对象是业务逻辑层面的输出单元
- 示例：员工信息对象、文件对象、群组对象
- 不对应数据库实体，简化了权限管理复杂度

**2. 双重权限模型解决通道与数据的解耦**
- 通道权限和数据对象权限独立管理
- 应用需同时拥有两种权限才能消费
- 支持同一数据对象不同通道权限独立控制

**3. 应用级授权简化权限管理**
- 权限绑定在应用 ID 上，不是 AK/SK
- AK/SK 重置不影响权限
- 数据 Owner 人员变更不影响已授权权限（针对岗位）

**4. 动态审批链平衡效率与安全**
- 根据数据敏感度自动生成审批链
- L1-L2：2-3 级审批（数据 Owner+ 通道 Owner+ 平台管理员）
- L3-L4：4-5 级审批（增加上级/CTO+ 安全合规）
- 一次申请完成所有审批，用户体验友好

**5. 权限回收低频谨慎**
- 回收是严肃的决策，不轻易执行
- 必须提前通知应用方，给缓冲时间
- 逻辑数据对象层面变动频率低

**6. 审计以通道为入口**
- 不单独记录"权限使用日志"
- API 调用日志、事件推送日志关联权限信息
- 支持安全审计、使用量统计、性能优化、问题排查、合规要求

**7. 模块协同复用优先**
- 应用管理：复用现有能力
- API 开放：复用现有 API 订阅管理
- OAuth2：暂不涉及，仅管理 AK/SK
- 审计日志：复用统一审计能力

### 8.3 与前期决策的关系

| 前期决策 | 阶段三深化 |
|---------|-----------|
| 应用管理（AK/SK） | 明确权限基于应用 ID，AK/SK 仅是凭证 |
| 数据敏感度分级 | 明确审批链根据敏感度动态生成 |
| 全流程线上审批 | 明确一次申请=一次审批，动态生成审批链 |
| 审计日志 | 明确以通道为入口记录，多用途支持 |
| 复用现有能力 | 明确应用管理、API 开放、审计日志复用策略 |

### 8.4 待确认事项

| 事项 | 说明 |
|------|------|
| 可申请目录过滤 | 是否需要根据应用资质过滤可申请的权限 |
| 回收提前通知天数 | 提前 7 天、15 天还是 30 天通知 |
| 变更程度判定 | 什么程度的变更需要重新审批 |

## 九、阶段四需求挖掘：字段权限（2026-04-13）

### 9.1 核心设计原则

**抹除数据对象概念**：
- 不再区分「员工信息对象」、「群组对象」等业务概念
- 直接面向 API/事件通道，管控每个通道暴露的数据字段
- 权限控制粒度从「对象级」下沉到「字段级」

**新权限模型**：
```
消费权限 = 通道权限 + 字段权限（必须同时拥有）
```

| 维度 | 设计决策 |
|------|---------|
| **字段粒度** | 逻辑字段（如：用户名数据字段），不是具体数据库字段 |
| **平台定位** | 平台不碰数据，不管理具体数据含义，仅管控权限 |
| **字段申请** | 申请 API 时提示包含敏感字段，同时申请全部字段 |
| **默认策略** | 通道权限默认附带基础字段权限，敏感字段需额外申请 |
| **字段定义者** | API Owner 在注册通道时附带定义字段权限 |
| **审批设计** | 两条审批单（通道 + 字段），并行审批，取并集去重 |
| **审批人视角** | 看权限，不看数据；可展示技术元数据（如 API 地址） |
| **权限有效期** | 永久有效，除非主动回收 |
| **低频场景** | 权限回收、变更暂不考虑，优先保证核心功能 |

### 9.2 与飞书/钉钉的对比

| 维度 | 飞书/钉钉 | 我们（阶段四） | 未来优化 |
|------|----------|---------------|---------|
| 字段粒度 | 具体字段 | 逻辑字段 | 保持逻辑字段 |
| 申请方式 | 按需申请敏感字段 | 同时申请全部字段 | 按需申请 |
| 审批流程 | 一起审批 | 两条审批单，并行审批 | 保持并行审批 |
| 默认策略 | 基础字段默认开放 | 基础字段默认开放 | 基础字段默认开放 |
| 平台定位 | 管理字段权限 | 平台不碰数据，仅管控权限 | 保持 |

**核心驱动**：
- D - 对标竞品，避免走竞品未走过的路遇到新的问题
- 跟随飞书/钉钉已验证的模式，降低试错成本

### 9.3 与阶段三的对比

| 维度 | 阶段三（已备份） | 阶段四（当前） |
|------|-----------------|---------------|
| 权限模型 | 通道权限 + 数据对象权限 | 通道权限 + 字段权限 |
| 数据概念 | 业务逻辑对象（员工信息对象等） | 抹除，直接面向逻辑字段 |
| 字段粒度 | 不做细粒度控制 | 逻辑字段级别管控 |
| 字段申请 | 不适用 | 必须同时申请（如涉及） |
| 平台定位 | 管理数据对象 | 平台不碰数据，仅管控权限 |
| 对标策略 | 自主设计 | 跟随飞书/钉钉已验证模式 |
| 审批设计 | 一条审批单，动态审批链 | 两条审批单，并行审批 |
| 可见性 | 明确的权限清单和目录 | 待定，后续设计 |
| 回收变更 | 低频谨慎，提前通知 | 暂不考虑 |

### 9.4 关键洞察

**1. 抹除数据对象简化设计**
- 避免业务对象的复杂抽象
- 直接面向技术实现（API/事件通道）
- 降低理解和实现成本

**2. 逻辑字段管控，平台不碰数据**
- 字段是业务逻辑概念，不是具体数据库字段
- 平台不管理具体数据含义
- 通道提供方如有需求，在通道文档中体现字段细节

**3. 两条审批单，并行审批**
- 简化设计，降低复杂度
- 通道审批单 + 字段审批单，独立审批
- 取并集去重，支持并行审批，提升效率

**4. 默认策略平衡体验与安全**
- 基础字段默认开放，降低申请门槛
- 敏感字段需额外申请，确保安全管控
- 申请 API 时提示包含敏感字段，同时申请

**5. 对标竞品降低试错成本**
- 跟随飞书/钉钉已验证的模式
- 避免自主设计遇到新的问题
- 降低学习和接入成本

## 十、阶段五需求挖掘：权限思维（2026-04-13）

### 10.1 核心设计原则

**网关定位**：
- 数据开放平台作为数据开放网关，不触碰具体数据
- 管控权限元数据，不管理业务数据
- 类似 API Gateway，但专注于数据开放场景

**权限思维**：
- 永远申请的是权限，不是对象
- 消费权限 = 通道权限 + 数据对象权限
- 平台内部有数据对象概念，消费方无感知（或可选感知）

**隔离设计**：
- 两条审批单：通道权限审批单 + 数据对象权限审批单
- 独立审批，各自动态生成审批链
- 鉴权时同时检查两个权限

| 维度 | 设计决策 |
|------|---------|
| **数据对象** | 数据 Owner 注册时感知，消费方无感知（或可选提示） |
| **通道关系** | 通道 Owner 注册时关联数据对象权限（多对多，不做限制） |
| **申请流程** | 申请 API 权限，后台自动生成数据对象权限申请单 |
| **用户感知** | 尽可能向仅申请通道权限靠拢，提示依赖关系 |
| **权限清单** | 展示 API 列表，背后关联数据对象（平台内部） |
| **审批设计** | 两条审批单，隔离设计更清晰，并行审批 |
| **鉴权逻辑** | 检查通道权限 + 通道依赖的数据对象权限 |
| **权限有效期** | 永久有效，除非主动回收 |
| **低频场景** | 权限回收、变更暂不考虑，优先保证核心功能 |

### 10.2 与阶段三/四的对比

| 维度 | 阶段三 | 阶段四 | 阶段五（当前） |
|------|--------|--------|---------------|
| **核心理念** | 对象思维 | 字段思维 | **权限思维** |
| **申请对象** | 通道 + 数据对象 | API+ 字段 | 通道权限 + 数据对象权限 |
| **数据对象** | 用户明确感知 | 抹除 | 数据 Owner 感知，消费方无感知 |
| **平台定位** | 管理数据对象 | 管控字段权限 | 网关角色，管控权限元数据 |
| **审批单设计** | 一条审批单 | 两条审批单 | 两条审批单（隔离） |
| **审批链生成** | 动态生成 | 取并集 | 动态生成（各自独立） |
| **鉴权逻辑** | 检查通道 + 对象 | 检查 API+ 字段 | 检查通道权限 + 对象权限 |
| **关系定义** | 数据 Owner 定义 | API Owner 定义 | 通道 Owner 定义 |
| **关系模式** | 待定 | 不适用 | 多对多，不做限制 |
| **权限清单** | 通道 + 对象组合 | 通道 + 字段组合 | API 列表（背后关联对象） |

### 10.3 关键洞察

**1. 网关定位清晰边界**
- 平台不触碰具体数据，只管控权限元数据
- 降低平台复杂度和责任边界
- 聚焦核心能力：路由、限流、鉴权、审计、监控

**2. 权限思维简化认知**
- 永远申请的是权限，不是对象
- 降低用户认知负担
- 消费方无感知数据对象，或可选提示

**3. 隔离设计提升清晰度**
- 两条审批单，独立审批
- 鉴权时同时检查两个权限
- 降低审批链组合复杂度

**4. 多对多关系灵活扩展**
- 通道 Owner 自主关联数据对象权限
- 不做限制，适应不同业务场景
- 数据 Owner 不需要关心通道

**5. 用户体验优化**
- 申请流程向仅申请通道权限靠拢
- 后台自动生成数据对象权限申请单
- 用户仅需要关注审批进度
