# ADR-001: 轻量顺序执行引擎技术方案

## 状态
ACCEPTED

## 背景
连接器平台 MVP 需要支持连接流的运行时调度执行。规范 (spec.md §3.3) 定义了四种触发方式（事件/Webhook/定时/手动）和线性编排执行（入口节点→连接器节点→出口节点）。在 Plan 阶段，面临以下技术方案选择：

**方案 A（推荐）— 轻量顺序执行引擎**：基于现有 open-server 扩展，不引入额外框架。编排配置以 JSON 存储，运行时引擎按节点顺序依次执行。使用消息队列进行异步调度，执行上下文存储在 Redis 中，执行记录持久化到 MySQL。

**方案 B — Spring StateMachine 状态机引擎**：引入 Spring StateMachine 作为流程引擎核心，将连接流执行抽象为状态转换。

**方案 C — 消息驱动引擎**：以消息队列为核心，每个连接器节点封装为独立的消息消费者，流程执行为消息在消费者间的流转。

前置调研文档：`front/doc/2026-05-11-连接器平台-三方案对比分析.md` 已对上述三种方案从架构设计、核心能力、开发成本、运维复杂度、性能与扩展性五个维度进行了全面对比分析。

## 决策
我们决定采用**方案 A：轻量顺序执行引擎**。

### 决策内容
1. **顺序执行器**：基于 Spring Boot 自研顺序执行器，从入口节点开始，按拓扑顺序依次执行每个连接器节点，最后执行出口节点
2. **编排配置格式**：FlowVersion.orchestration_config 以 JSON 格式存储完整编排信息
3. **异步调度**：触发事件写入 MQS，执行器异步消费处理
4. **执行上下文**：Redis 存储运行时上下文（触发数据、中间结果、执行状态），MySQL 持久化最终执行记录
5. **MVP 范围**：仅支持线性编排（入口节点→连接器节点→出口节点），无分支/循环/并行
6. **V1 演进路径**：通过增加节点类型处理逻辑（区分 connector/control/data/error/async/subflow 节点）扩展分支/循环/子流程能力，执行器从顺序执行升级为调度执行

### 理由
1. **MVP 范围精确匹配**：MVP 明确仅需线性编排，方案 A 是满足需求的最简方案，不做过度设计
2. **最小化技术债务**：不引入 Spring StateMachine 或自研 BPMN 引擎，与现有架构完全一致
3. **开发效率最优**：可在现有 open-server 中扩展，复用 MyBatis/MySQL/Redis/MQS 基础设施
4. **调试友好**：线性执行的每步输入/输出清晰，测试运行时可逐步验证
5. **渐进式演进**：MVP→V1 无需推翻重来，通过扩展节点类型处理逻辑即可
6. **团队无学习成本**：不引入新技术栈

### 拒绝的方案
- **方案 B (Spring StateMachine)**：状态机对于仅需线性执行的 MVP 场景过度设计；学习曲线增加；嵌套状态机复杂度会随分支/循环快速上升
- **方案 C (消息驱动)**：过早引入分布式复杂度；线性流程的场景中消息传递延迟高于同步调用；调试和运维复杂度增加

## 后果
### 正面
- MVP 可在 10-14 周内交付，满足快速验证业务价值的需要
- 架构清晰简单，团队成员可快速上手开发和维护
- 运行时与 CRUD 在同一进程中，部署运维简单

### 负面
- 高并发场景下单实例执行器可能成为瓶颈（需增加线程池隔离和资源配额）
- 非 BPMN 标准格式，与其他流程工具的互操作性受限
- 复杂编排（并行/分支/循环）需要 V1 重构执行器为调度模式

### 风险缓解
- 线程池隔离 + 资源配额限制：单连接流独立线程池，防止相互影响
- JSON 编排格式预留 node_type 字段（connector/control/data/error/async/subflow），V1 扩展时无需变更数据模型
- 执行器接口定义 `NodeExecutor` 抽象，V1 新增节点类型只需新增实现类
