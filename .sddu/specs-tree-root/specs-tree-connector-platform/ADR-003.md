# ADR-003: 运行时架构 — 独立部署 `connector-api` 服务（v2.0 修订）

## 状态
ACCEPTED（v2.0 修订版，2026-05-22）

> ⚠️ **本 ADR 已于 v2.0 修订**：原决策「单体嵌入 + 模块化隔离」已被取代（见下方「v2.0 修订决策」段落）。原决策正文保留在文末「附录：v1.x 原决策（SUPERSEDED）」以供追溯。

## v2.0 修订背景
随着 spec v4.0 将执行模型从异步改为同步，并在与团队评审中确认以下诉求：
- 管理类操作（CRUD）与运行时执行（高并发、长耗时同步调用）资源特征差异显著，需要**独立扩缩容**
- 运行时阻塞或异常**不能**影响 open-server 管理后台的可用性
- V1 阶段计划引入异步执行/MQS，独立服务后改造范围更小
- 手动调试、测试运行等流程需要**收口在同一执行引擎实现**，避免运行时逻辑在 open-server 与运行时服务两边重复

## v2.0 修订决策
将运行时**独立部署为 `connector-api` 服务**，与 open-server 分离。

### 决策内容
1. **部署形态**：
   - `open-server` 承载所有管理类模块：connector / flow / monitor / debug（调试代理）
   - `connector-api` **新增独立 Spring Boot 工程**，承载运行时模块：runtime / http-trigger（对外触发）/ debug-api（对内调试接口）
   - 两个服务**共享同一 MySQL / Redis 实例**，通过数据库实现状态共享（编排配置、执行记录）
2. **调试/测试运行通道**：
   - 前端（wecodesite）的手动调试、测试运行请求 → open-server `debug` 模块 → 内部 HTTP 调用 → connector-api `debug-api` → 复用 runtime 执行引擎
   - 运行时逻辑收口在 connector-api，open-server 仅做请求代理与权限校验
3. **对外 HTTP 触发**：
   - 外部消费方直接调用 connector-api 的 `http-trigger` 端点，链路最短
4. **内部通信安全**：
   - open-server → connector-api 走内网 HTTP，加共享 token 鉴权（`InternalAuthFilter`）
   - 设置超时（默认 30s）+ 1 次重试 + 熔断降级（调试不可用时前端提示，不影响管理面 CRUD）
5. **前端归属**：
   - 连接器平台的所有前端页面统一放在 `wecodesite`（不再有 open-web 等其它前端入口）

### 理由
1. **职责清晰**：管理与执行物理隔离，资源/故障/扩缩容互不影响
2. **演进友好**：V1 引入异步执行/MQS 时只需改造 connector-api，open-server 保持稳定
3. **调试通道收口**：避免「open-server 内嵌运行时」与「未来独立运行时」两套实现并存
4. **运维成本可控**：复用 open-server 的部署流水线模板、MySQL/Redis 实例、监控告警体系

### 拒绝的方案
- **保留 v1.x 单体嵌入**：随同步执行落地，运行时与管理类操作资源/故障耦合的弊端将显化；调试通道也需要在两处实现执行逻辑
- **运行时再拆为多个微服务（trigger / executor / debug 分服务）**：MVP 阶段无必要，过度设计；当前单一 connector-api 已满足职责隔离需求

## v2.0 修订后的后果

### 正面
- 运行时阻塞/异常完全不影响管理面 CRUD，可用性显著提升
- connector-api 可独立水平扩容，应对高并发触发场景
- 前端入口统一在 wecodesite，用户侧只需对接一个前端项目
- 调试通道收口，杜绝逻辑重复

### 负面
- 新增一个独立服务实例（运维 +1）
- 新增 open-server → connector-api 一条内部 HTTP 链路（调试/测试运行）
- 工程脚手架与共享代码（entity/mapper）需要单独维护
- 预估增加 0.5-1 周「迭代 0」用于 connector-api 工程初始化与联调

### 风险缓解
- 内部 HTTP 加共享 token 鉴权 + 超时 + 重试 + 熔断
- connector-api 复用 open-server 的部署流水线模板与监控告警接入
- 共享代码（如 entity/mapper、加解密）按需在 connector-api 内复制或抽取为内部公共依赖（视团队成本评估）

## 附录：v1.x 原决策（SUPERSEDED）

> 以下为 v1.x 版（基于 spec v3.x）的原决策内容，已于 v2.0 被取代，仅作历史追溯。

### 原状态
SUPERSEDED（被 v2.0 修订版取代）

### 原背景
连接器平台的运行时模块（spec.md §3.3）负责连接流的调度执行、执行上下文管理、错误处理和运行监控。运行时包括以下功能：
- 四种触发方式：事件触发、Webhook 触发、定时触发、手动触发（FR-022 ~ FR-025）
- 顺序执行引擎（从入口节点到出口节点依次执行）
- 执行上下文创建与管理（FR-026）
- 步骤间数据传递（FR-027）
- 资源配额与隔离（FR-028）
- 节点级错误处理与重试（FR-029 ~ FR-030）

在架构设计时面临两种选择：
1. **单体嵌入**：运行时作为 open-server 的一个模块（package），与 CRUD 管理在同一进程中
2. **独立服务**：运行时作为独立服务部署（runtime-server），通过 RPC/MQS 与 open-server 通信

## 决策
我们决定在 **MVP 阶段采用单体嵌入 + 模块化隔离** 的架构，预留独立服务的抽取路径。（**已被 v2.0 修订版取代**）

### 原决策内容
1. **部署形态**：运行时作为 open-server 的一个独立 Package（`modules.runtime`），与 connector/flow 等 CRUD 模块在同一进程中
2. **隔离机制**：
   - 线程池隔离：运行时使用独立的线程池（`flow-executor`），不与 CRUD 请求共用 Tomcat 线程池
   - 资源配额：单连接流独立配置超时限制（默认 5 分钟）和并发上限（默认 10）
   - 数据库表独立：运行时表（execution_record, execution_step）使用独立表前缀 `cp_runtime_`
3. **代码边界**：运行时模块不直接调用 CRUD 模块的内部方法，统一通过 service 接口交互
4. **预留抽取路径**：
   - 运行时模块的所有依赖注入使用接口，不依赖具体实现
   - 执行上下文通过 `ExecutionContextManager` 抽象，Redis → MySQL 双写确保未来可独立部署
   - MQS 消息生产/消费通过独立 Topic（`cp_runtime_*`）
5. **抽取条件**：当出现以下任一情况时，将运行时抽取为独立服务：
   - 单连接流执行时间超过 30 分钟（长时间占用线程池资源）
   - 运行时 CPU/内存使用率超过 open-server 总资源的 30%
   - 需要独立扩展运行时实例（水平扩展）

### 原理由
1. **MVP 阶段避免过度拆分**：独立服务需要额外部署运维（注册发现、配置中心、健康检查、链路追踪），MVP 阶段不划算
2. **复用现有基础设施**：运行时与 open-server 共享 MySQL、Redis、MQS 连接池，无需额外配置
3. **简化开发和调试**：单体进程中调试执行链路更便捷，可在 IDE 中单步跟踪执行过程
4. **MVP 负载可预估**：MVP 上线初期连接流数量和触发频率有限，单体架构完全可承受
5. **抽取成本可控**：通过接口抽象和独立 Topic 预留，抽取为独立服务的改造成本预估 1-2 周

### 原拒绝的方案
- **独立服务 (runtime-server)**：MVP 阶段增加运维复杂度；需要额外的服务注册发现、配置管理、健康检查等基础设施；团队需维护两个 Spring Boot 应用的 CI/CD 流水线
- 注：v2.0 修订版重新评估了该方案，认为同步执行落地后职责隔离收益大于运维成本，故采纳「独立服务」方向

### 原后果
#### 原正面
- MVP 阶段架构简洁，开发和部署效率高
- 运行时与 CRUD 在同一进程中，调试和问题排查方便
- 无需引入服务间通信的额外延迟（RPC 序列化/网络开销）
- 复用现有配置和监控体系

#### 原负面
- 运行时故障可能影响 CRUD 管理功能的可用性（通过线程池隔离缓解）
- 运行时资源消耗与 CRUD 资源共享（通过资源配额限制缓解）
- 后期抽取时需要一定的重构工作

#### 原风险缓解
- 线程池隔离：运行时线程池设置独立参数（corePoolSize=10, maxPoolSize=50），与 Tomcat 线程池分离
- 资源配额：单连接流超时 5 分钟强制终止，单流并发上限 10
- 健康检查：运行时模块提供独立健康检查端点 `actuator/health/runtime`
- 监控告警：运行时执行失败/超时/线程池满载时触发告警
