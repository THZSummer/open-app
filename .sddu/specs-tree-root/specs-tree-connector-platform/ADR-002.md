# ADR-002: React Flow 可视化编排画布

## 状态
ACCEPTED

## 背景
连接器平台需要提供可视化拖拽编排连接流的能力（spec.md FR-017）。根据行业调研（钉钉连接平台使用 Ant Design 自研 SVG 画布、企业微信使用自研画布、腾讯云 HiFlow 使用自研 React+SVG），主流的连接器平台均采用自研或基于开源框架的流程图编辑器。

在 Plan 阶段，需要考虑可视化编排画布的技术选型。关键评估维度：
- 与现有前端技术栈的兼容性（React 18 + TypeScript）
- MVP 仅需线性编排的适配度
- 社区活跃度和长期维护
- V1 支持分支/循环/子流程的扩展能力

## 决策
我们决定采用 **React Flow (@xyflow/react)** 作为可视化编排画布的基础框架。

### 决策内容
1. **核心框架**：`@xyflow/react` (v12.x) — 成熟的 React 流程图库
2. **节点类型**：基于 React Flow 的 custom node 机制，实现三类节点（入口节点/连接器节点/出口节点）
3. **MVP 限制**：使用 React Flow 的受控模式，限制仅支持线性节点连接（禁止循环连接、禁止分支连接）
4. **配置面板**：节点选中后，侧边弹出配置面板（Sidebar/Panel），配置参数使用 Ant Design 表单组件
5. **数据映射**：通过弹窗对话框（Modal）配置字段映射（源字段→目标字段）

### 理由
1. **React-native**：与现有 React 18 + TypeScript 技术栈完全一致
2. **轻量级**：@xyflow/react 仅约 30KB (gzip)，不会显著增加前端包体积
3. **完善的 TypeScript 支持**：类型定义完整，开发体验好
4. **受控模式**：可通过受控模式限制编排行为（禁止循环、限制节点类型数量），完美适配 MVP 线性编排需求
5. **扩展性**：V1 需要分支/循环时，React Flow 原生支持多个输入/输出连接点，无需更换框架
6. **社区活跃**：GitHub 25k+ stars，npm 周下载量 500k+，文档完善
7. **现有经验**：团队已有 React 开发经验，React Flow 的 API 设计符合 React 惯例

### 拒绝的方案
- **AntV X6**：基于 Canvas 渲染，与 React 组件树集成不如 React Flow 自然；需要额外学习 X6 的图形渲染体系
- **自研 SVG/Canvas 画布**：开发周期长（预计 4-6 周），MVP 阶段不建议自研
- **自研 HTML/CSS 拖拽**：功能过于基础，无法满足节点连接/缩放/平移等需求

## 后果
### 正面
- 前端团队可在熟悉的技术栈上快速上手，降低学习成本
- React Flow 的受控模式使得 MVP 限制线性编排易于实现
- 社区生态丰富（已有大量自定义节点示例和插件）
- V1 扩展分支/循环时无需更换框架

### 负面
- 需要额外引入 ~30KB 的依赖
- 高度自定义的 UI 需求（如节点底部的字段映射连接点）可能需要深入定制 React Flow
- 图表渲染性能在大规模流程（100+ 节点）时需关注，MVP 场景（<20 节点）无此问题

### 风险缓解
- MVP 阶段流程节点数限制在 20 个以内，确保渲染性能
- 复杂自定义节点优先参考 React Flow 官方示例和社区最佳实践
- 预留回退方案：如果 React Flow 自定义节点无法满足需求，可降级为简单表单配置模式
