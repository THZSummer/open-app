## 强制要求

| # | 要求 | 说明 |
|:-:|------|------|
| 1 | **所有操作必须通过浏览器操作** | 禁止使用 `axios` / `fetch` / `request.newContext()` / `request.post()` 等任何直接调 HTTP 接口的方式模拟用户行为，**包括 §10 审计日志测试在内**，必须通过 Playwright 页面交互（点击、输入、导航等）执行测试。唯一允许的辅助手段是 DB 直连读取审计日志表做验证 |
| 2 | **测试过程发现 BUG 记录到 BUG.md** | 测试过程中发现的任何 BUG 必须记录到 `.sddu/specs-tree-root/specs-tree-app-list/BUG.md`，包含：用例ID、重现步骤、期望结果、实际结果、严重程度 |
| 3 | **不允许私自修复 BUG** | 测试人员的职责是发现问题并记录，不允许在测试阶段自行修改任何代码或配置 |

### 测试执行准则（必须遵守）

以下准则是铁律，违反任何一条等同于该用例未执行：

| # | 准则 | 说明 |
|:-:|------|------|
| A1 | **每条用例必须真实操作** | 禁止以"由前端处理"、"需mock接口"、"已在前序验证"、"代码审查确认"等理由跳过实际操作。没有实际浏览器操作 = 该用例未测。 |
| A2 | **写操作必须验证数据库** | 任何涉及数据写入的用例（创建/编辑/删除/添加/转移/发布/撤回），操作完成后必须通过 DB 查询确认数据真实持久化，不能只看 API 返回 200 或 toast 提示。 |
| A3 | **标 PASS 必须有操作证据** | 每条 PASS 的用例必须附带至少一项证据：浏览器快照文字、DB 查询结果、或网络请求响应。无证据的 PASS 无效。 |
| A4 | **遇到异常立刻调查** | 操作过程中出现任何异常（控制台报错、页面空白、数据缺失、跳转异常），必须立刻停止并排查根因，不能假装没看见继续标 PASS。 |
| A5 | **不能跨用例推断** | 每条用例独立执行和判定。A 用例通过不能推断 B 用例也通过，即使逻辑相似。 |
| A6 | **PASS/FAIL/BUG 三态** | 只有实际操作过的用例才能标 PASS 或 FAIL。发现的功能缺陷标 BUG 并记录到 BUG.md。未执行的用例标"未测"，不得标 PASS。 |

### Page Object Model（POM）约定

**每个页面一个类**，封装 locator 与操作方法，禁止在 spec 文件中直接使用 `page.locator('xxx')` 写死选择器。

```js
// pages/AppListPage.js
export class AppListPage {
  constructor(page) {
    this.page = page;
    this.heroTitle = page.locator('.app-list-hero-title');
    this.appCard = (appName) => page.locator(`[data-testid="app-card-${appName}"]`);
    this.createAppBtn = page.locator('[data-testid="hero-create-btn"]');
  }

  async goto() { await this.page.goto('/'); }
  async clickAppCard(name) { await this.appCard(name).click(); }
  async openCreateAppModal() { await this.createAppBtn.click(); }
}
```

### 测试用例 ID 与文件命名

- 测试用例 ID 与本文件 `TC-X-YZ` 编号一一对应
- spec 文件用 `s{X}-*.spec.js` 命名，内部 `test()` 描述前缀用 `[TC-X-YZ]`
- 用例与文档的双向追溯：spec 用例注释里写 `// ref: TC-X-YZ`

```js
// s1-app-list.spec.js
import { test, expect } from '@playwright/test';
import { AppListPage } from '../pages/AppListPage';

test.describe('S1 浏览应用列表', () => {
  test('[TC-1-01] 正常加载应用列表', async ({ page, ownerAuth }) => {
    const appList = new AppListPage(page);
    await appList.goto();
    await expect(appList.heroTitle).toHaveText('应用管理');
  });
});
```

### Locator 选择器规范

> 避免依赖 CSS class / 层级（实现改动易碎），**优先使用 `data-testid` 属性**。

| 选择器类型 | 优先级 | 适用场景 |
|-----------|:------:|---------|
| `[data-testid="xxx"]` | ⭐⭐⭐ | **首选**，与实现解耦 |
| `getByRole('button', { name: '提交' })` | ⭐⭐ | 按钮、表单元素（语义化）|
| `getByText('立即创建')` | ⭐ | 静态文本（兜底）|
| `page.locator('.xxx')` | ❌ | **禁止**（CSS class 易碎）|

**前端配合**：在 `wecodesite/src/components/` 与 `pages/` 的关键交互元素上加 `data-testid` 属性。


### 用例数据隔离与清理

- **每个 `test()` 独立创建/清理应用**（用 `app.fixture.js` 的 `beforeEach` / `afterEach`）
- 测试应用命名规范：`e2e-{timestamp}-{role}`，便于失败定位
- 数据库级清理：测试结束后自动删除 `e2e-*` 前缀的应用及关联数据
