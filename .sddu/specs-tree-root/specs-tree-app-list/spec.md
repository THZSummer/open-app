## 1. 功能概述

### 背景

开放平台应用管理是企业内部开发者使用 WeLink 开放平台的核心入口。开发者通过此模块创建应用、配置能力、发布版本，实现与 WeLink 平台的功能集成。

### 目标

为开发者提供完整的应用生命周期管理能力，包括应用的创建与配置、团队成员协作、应用能力扩展以及版本发布审核流程。

### 2.3 成员管理

> 可添加企业内的人员为协作人员,并根据权限为成员设置不同角色

#### 成员管理权限矩阵（Viewer × Target）

> 本矩阵为 FR-006 ~ FR-009 的**权限总表**。viewer 为当前登录用户角色，target 为列表中目标成员行的角色。
> 表中符号说明：
> - ✅ **可执行** — 按钮可见且 enabled，点击可触发对应操作
> - ❌ **置灰** — 按钮可见但 disabled，不可点击（鼠标悬浮提示"无操作权限"）
> - 🚫 **不展示** — 按钮不渲染

| Viewer（当前用户） | 添加成员（页面顶部） | Owner 行（操作列） | 管理员 行（操作列） | 开发者 行（操作列） |
|--------------------|:--------------------:|:------------------:|:-------------------:|:----------------------:|
| **Owner** | ✅ 可执行 | ✅ 转移（可执行） | ✅ 删除（可执行） | ✅ 删除（可执行） |
| **管理员** | ✅ 可执行 | ❌ 转移（置灰） | ❌ 删除（置灰） | ✅ 删除（可执行） |
| **开发者** | 🚫 不展示 | ❌ 转移（置灰） | ❌ 删除（置灰） | ❌ 删除（置灰） |

**补充规则**（不在矩阵内，由后端兜底）：

1. **添加成员时**（FR-007）：Owner 可加 管理员 / 开发者；管理员 仅可加 开发者；开发者 不可加任何角色；**Owner 角色不能通过"添加成员"产生**，仅能通过"转移 Owner"产生（详见 FR-009）
2. **转移 Owner**（FR-009）：仅当前 Owner 可执行
3. **多角色共存**（v2.2 重要规则）：**同一成员可在多个角色下同时存在**（如张三可同时是 Owner / 管理员 / 开发者）；**同一角色下不能有相同成员**（如 Owner 角色下不能有两个张三）

#### FR-006: 成员列表

| 项目 | 内容 |
|------|------|
| 功能描述 | 展示应用的所有成员及其角色 |
| 展示内容 | 1. 成员姓名、工号、角色（开发者/Owner/管理员）<br>2. **同一成员可在多行中出现**（如张三可同时出现在 Owner 行 / 管理员 行 / 开发者 行，见 FR-007 多角色规则）<br>3. 操作列按钮根据「成员管理权限矩阵」动态决定（Owner 行只显示转移，管理员/开发者 行只显示删除）<br>4. **支持分页**（每页条数可选 10 / 20 / 50，默认 10），分页信息左侧显示总条数，右侧显示分页控件和条数选择器；**支持跳转到某页** |
| 操作能力 | 1. 查看成员列表（含分页 / 翻页 / 切换每页条数 / **跳转到某页**）<br>2. 按权限矩阵的规则执行对应操作（详见上方权限矩阵）<br>3. 可对**目标成员行**执行的操作：<br>&nbsp;&nbsp;· **Owner 行**：仅 Owner 可执行「转移」<br>&nbsp;&nbsp;· **管理员 行**：Owner 可执行「删除」<br>&nbsp;&nbsp;· **开发者 行**：Owner / 管理员 可执行「删除」 |
| 操作后效果 | 1. 点击删除后弹出确认框，确认后该成员**指定角色**下的记录被移除（不影响其他角色下的同成员记录）<br>2. 点击转移后弹出成员选择器，确认后完成 Owner 转移<br>3. 分页 / 翻页 / 切换每页条数 / **跳转到某页**后重新拉取数据 |
| 约束规则 | 1. **列表按成员姓名拼音字母顺序（A-Z 升序）排序；姓名相同时按 id 倒序排**<br>2. 权限规则严格遵循「成员管理权限矩阵」<br>3. 后端必须在每次写操作时再次校验角色权限（前端 disabled 不可作为唯一防线） |

#### FR-007: 添加成员

| 项目 | 内容 |

### 2.5 版本发布与审核

#### 版本状态机

> 版本生命周期共 **4 个状态**，状态机驱动版本流转；所有状态转换须严格遵循下表。
> 本节为**单一来源**，FR-011 ~ FR-015 中涉及状态流转的描述均引用本表。

**状态定义**：

| 状态码 | 状态名 | 说明 |
|:------:|--------|------|
| 1 | 待发布 | 初始创建后的状态，可编辑、可删除、不可直接发布 |
| 2 | 审批中 | 提交审批后的状态，等待审批人处理 |
| 3 | 审批未通过 | 审批拒绝后的状态，仅可查看或删除 |
| 4 | 已发布 | 审批通过后的状态，终态 |

**状态转换**：

| 当前状态 | 事件 | 目标状态 | 触发条件 |
|----------|------|----------|----------|
| 1（待发布） | 提交审批 | 2（审批中） | 用户点击「发布」按钮 |
| 2（审批中） | 审批通过 | 4（已发布） | 审批人通过（系统流程） |
| 2（审批中） | 审批拒绝 | 3（审批未通过） | 审批人拒绝（系统流程） |
| 2（审批中） | 撤回 | 1（待发布） | 用户点击「撤回」按钮 |
| 3（审批未通过） | 删除 | 终态 | 不可编辑，仅可查看或删除 |
| 4（已发布） | - | 终态 | 已发布版本不可重复发布 |

**流转路径**：

```
创建 → 1 待发布 ──提交审批──→ 2 审批中 ──审批通过──→ 4 已发布 [终]
            ↑                    │
            └─── 撤回 ────────────┤
                                 └──审批拒绝──→ 3 审批未通过 → 删除 [终]
```

### 2.7 EAMAP管理

> 管理应用与 EAMAP（应用服务）的绑定关系，包括存量个人应用的升级绑定、EAMAP 信息的展示等。

#### FR-018: 存量个人应用绑定 EAMAP 升级为业务应用

**业务背景**：历史上已创建但未绑定 EAMAP 的个人应用，类型标识为"存量个人应用"（`app_sub_type=0`）。

**业务规则**：

1. 存量个人应用可调用"绑定 EAMAP"接口绑定一个 EAMAP 服务
2. 绑定成功后，应用类型自动从"个人应用"升级为"业务应用"
3. 升级后，应用获得业务应用的全部菜单（成员、应用能力、版本发布与审核）
4. 业务应用创建时已强制绑定 EAMAP（创建流程中），不通过本接口更换 EAMAP
5. 一个 EAMAP 编码只能被一个应用绑定
6. 升级后系统会通知卡片服务，前端刷新页面即可看到完整的 4 个 Tab 菜单
7. 绑定成功后触发审计日志 `BIND_APP_EAMAP`（FR-019）

**约束规则**：

| 条件 | 结果 |
|------|------|
| 存量个人应用（`app_sub_type=0`）+ 绑定 EAMAP | 升级为业务应用（`app_type=1`、`app_sub_type=4`） |
| 业务应用（`app_type=1`）+ 调用本接口 | 拒绝（业务应用不通过本接口更换 EAMAP） |

#### FR-020: EAMAP 信息展示（应用详情页头部）

| 项目 | 内容 |
|------|------|

### 2.8 操作审计日志

#### FR-019: 操作审计日志

**业务背景**：开放平台应用承载企业内部 API 凭证、成员权限、能力订阅等核心资产，需对所有改动型操作留痕，用于安全审计、合规追溯与故障定位。

**业务规则**：

1. **覆盖范围**：应用管理模块的全部"增、删、改"操作均须记录审计日志；纯查询（GET）与文件上传不纳入审计
2. **审计字段**：每条日志须记录 操作人、操作时间、操作 IP、应用 ID、操作类型、操作对象、操作前快照（before_data）、操作后快照（after_data）、操作结果（成功/失败）、中英文描述
3. **失败也要记录**：操作执行失败时仍须记录一条 status=失败 的审计日志，便于追溯异常调用
4. **不阻塞主业务**：审计日志写入失败、异步队列拥塞均不得影响主业务接口的正常响应与状态

**纳入审计的操作清单**：

| 模块 | 操作 | 操作类型 |
|------|------|---------|
| 应用基础信息 | 新增应用（FR-002） | CREATE |
| 应用基础信息 | 更新应用基本信息（FR-004） | UPDATE |
| 认证方式 | 更新认证方式（FR-005） | UPDATE |
| 应用基础信息 | 绑定 EAMAP 升级业务应用（FR-018） | UPDATE |
| 成员管理 | 添加成员（FR-007） | CREATE |
| 成员管理 | 删除成员（FR-008） | DELETE |
| 成员管理 | 转移 Owner（FR-009） | UPDATE |
| 应用能力 | 添加应用能力（FR-010） | CREATE |
| 版本 | 创建版本（FR-012） | CREATE |
| 版本 | 更新版本（FR-013 衍生） | UPDATE |
| 版本 | 发布版本（FR-014） | UPDATE |
| 版本 | 撤回版本（FR-015） | UPDATE |
| 版本 | 删除版本（FR-013 衍生） | DELETE |

**约束规则**：

| 条件 | 结果 |
|------|------|
| 任一 GET 查询接口 | 不写审计日志 |
| 上传图片（FR-004 衍生） | 不写审计日志（非业务实体变更） |
| 主操作成功 + 审计写入失败 | 主操作仍返回成功 |
| 主操作失败 | 写一条 status=失败 的审计日志后抛出原异常 |
