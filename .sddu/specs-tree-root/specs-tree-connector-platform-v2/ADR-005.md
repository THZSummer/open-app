# ADR-005: Redis 令牌桶入站限流方案

## 状态
PROPOSED

## 背景

V2 新增连接流级入站限流能力（G8），限制连接流被触发的频率（QPS/并发数），防止自身过载。需要选择限流的实现方案。

限流配置存储在 `FlowVersion.flowConfig` 中：
- `rateLimitMode`: `QPS` | `CONCURRENCY`
- `rateLimitValue`: 正整数上限（0 = 关闭）
- 运行时使用**已部署版本**的限流配置

## 决策

使用 **Redis 令牌桶算法 + Lua 脚本**实现入站限流。

### 技术方案

```
┌──────────────┐     HTTP Request     ┌────────────────────┐
│   HTTP 触发器  │ ──────────────────▶  │ InboundRateLimiter  │
│  (WebFlux)    │                     │ (WebFilter)         │
└──────────────┘                     └─────────┬──────────┘
                                               │
                                    ┌──────────▼──────────┐
                                    │  Redis Lua Script    │
                                    │  - 令牌桶检查         │
                                    │  - 原子递减           │
                                    │  - 返回允许/拒绝      │
                                    └──────────────────────┘
                                               │
                              ┌────────────────┼────────────────┐
                              ▼                ▼                ▼
                        允许 (200 OK)    拒绝 (429)        Redis 不可用
                        继续执行流程     Too Many         → 降级：放行
                                        Requests         (availability > rate limit)
```

### 关键设计点

1. **QPS 模式**：使用 Redis 令牌桶，key = `cp:ratelimit:qps:{flowId}`，每秒补充 `rateLimitValue` 个令牌
2. **并发模式**：使用 Redis 计数器，key = `cp:ratelimit:concurrency:{flowId}`，请求开始 INCR，请求结束 DECR
3. **Lua 脚本原子操作**：令牌检查 + 扣减在同一 Lua 脚本中执行，避免竞态条件
4. **Redis 不可用降级**：限流器捕获 Redis 异常，降级为放行（可用性优先于限流精度）
5. **WebFilter 拦截**：在 Spring WebFlux 的过滤器链中尽早执行限流检查，拒绝时直接返回 429

### 配置生命周期

```
FlowVersion.flowConfig.rateLimit → 版本发布 → 版本部署 → 运行时读取
                                                        │
                                          版本切换/停止 → 清空限流计数器
```

## 后果

### 正面影响

1. **复用现有基础设施**：4 个后端服务均已集成 Redis（Lettuce），无需引入新组件
2. **分布式友好**：connector-api 多实例部署时钟摆一致，限流基于 Redis 中心化计数
3. **精度可控**：Lua 脚本实现亚毫秒级原子操作，QPS 精度在 1% 以内
4. **运维简单**：Redis 已有监控和告警体系，不增加运维负担

### 负面影响

1. **Redis 依赖**：限流能力依赖 Redis 可用性（已通过降级策略缓解）
2. **网络延迟**：每次请求额外一次 Redis 往返（~1ms），在 QPS 上限场景可接受
3. **QPS 精度**：令牌桶算法在高 QPS 下有轻微精度损失（<1%）

### 缓解措施

- Redis 不可用时自动降级放行，保证可用性
- 限流计数器设置 TTL（QPS 模式 2s，并发模式由连接流生命周期控制）
- 连接流停止/失效时主动清空限流计数器

## 替代方案

| 方案 | 描述 | 拒绝理由 |
|------|------|---------|
| 内存 Guava RateLimiter | 进程内存限流 | 多实例不共享，connector-api 多实例部署时总量不可控 |
| Sentinel | Alibaba Sentinel 限流 | 引入新组件，功能远超需求，运维复杂度增加 |
| 无 Redis Lua | Redis 简单 INCR + TTL | 非原子操作，竞态条件下 QPS 计数不精确 |
