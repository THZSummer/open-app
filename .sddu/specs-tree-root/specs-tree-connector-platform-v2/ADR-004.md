# ADR-004: 版本完整快照存储与递增整数版本号

## 状态
PROPOSED

## 背景

V1 采用单版本模型：每个 Connector/Flow 对应一条 ConnectorVersion/FlowVersion 记录，编辑即覆盖。V2 需要支持多版本并行共存，每个实体可以有最多 1000 个版本（含草稿、已发布、已失效等各种状态）。

需要决策：
1. **版本快照存储方式**：每个版本是存储完整配置快照，还是存储与上一版本的增量差异？
2. **版本号策略**：使用递增整数、全局递增，还是 SemVer？

## 决策

### 决策 1：完整快照存储

每个版本存储完整的 `connectionConfig` / `orchestrationConfig` JSON 快照，不做增量存储。

### 决策 2：实体内递增整数版本号

每个 Connector/Flow 实体的版本号独立从 1 开始递增。草稿创建时分配版本号（当前最大版本号 + 1），发布时沿用该版本号。

**版本号示例**：
```
连接器 A: v1(已发布), v2(已发布), v3(草稿)
连接器 B: v1(已失效), v2(已发布), v3(已发布), v4(草稿)
连接流 X: v1(已发布), v2(待审批), v3(草稿)
```

## 后果

### 正面影响

1. **版本自包含**：任意版本的配置可独立读取，不依赖其他版本数据；版本恢复、回滚、切换查看均为 O(1) 操作
2. **运维友好**：版本快照可直接用于调试、审计、迁移，无需重建算法
3. **实施简单**：V1 已有 `connectionConfig` / `orchestrationConfig` JSON 列，V2 复用此模式，仅需将 1:1 改为 1:N
4. **版本号直观**：用户无需理解 SemVer 语义，版本号即标识发布顺序
5. **迁移成本低**：V1 现有单版本数据直接标记为 v1「已发布」

### 负面影响

1. **存储开销**：每版本 ~100KB，1000 版本上限 ≈ 100MB/实体，极端场景下存储需关注
2. **无兼容性表达**：递增版本号不表达向后兼容性语义（但 V2 无需此语义）
3. **跨实体不可比较**：连接器 v5 与连接流 v5 无关联（符合业务直觉）

### 缓解措施

- 版本数量硬上限 1000 条，在「复制到草稿」操作前校验
- 物理删除真删除（非软删除），释放存储空间
- 监控版本表数据量，超过 800 条时告警

## 替代方案

| 方案 | 决策 | 拒绝理由 |
|------|------|---------|
| 增量存储 | 仅存储与上一版本的 diff | diff 算法复杂，读取需重建，回滚慢，故障恢复困难 |
| 混合存储 | 每 N 个版本一个完整快照 + 中间 diff | 实现复杂度最高，维护成本高，边界条件多 |
| SemVer | 语义化版本号 | V2 无需兼容性表达，引入概念复杂度 | 
| 全局递增 | 全平台统一递增版本号 | 用户不直观，连接器 v5 与连接流 v5 无关 |
