## 1. 概述

### 1.1 问题陈述

V1（CONN-PLAT-001）已验证了**零代码编排**的核心价值。但随着使用深入，V1 的能力边界暴露出以下痛点：

- **无版本管理**：连接器和连接流编辑即生效，无法保留多版本配置，变更无追溯
- **配置能力不足**：认证方式单一，超时不可配；编排仅支持串行，无并行分支和字段类型转换能力
- **安全防护薄弱**：缺少 URL 白名单和 SYSTOKEN 凭证白名单校验，连接器/连接流数据无应用级隔离，无应用白名单准入控制
- **发布缺少审批**：连接流版本发布无审批流程，关键变更缺少人工确认环节
- **运维不可见**：无运行记录监控，运行时缺乏版本配置解析和日志采集能力，变更操作无日志审计
- **调试效率低**：编排修改后必须发布、部署才能验证，迭代周期长
- **数据模型局限**：JSON Schema 参数传递和 HTTP 节点参数位置（header/query/body）支持不足

### 1.2 解决方案

V2 在 V1 基础上围绕**连接器增强、连接流增强、运行时增强、安全与准入、数据模型升级、调试体验、运维审计**七个方向升级：

- **连接器**：配置支持多版本管理（草稿→发布→失效→删除），认证类型扩展至数字签名、Cookie，支持多选组合
- **连接流**：配置支持多版本管理，生命周期增强为部署→启动→停止，版本发布需经过三级审批并支持一键催办，流程编排支持入站限流、超时控制、缓存、错误处理及并行分支，节点间数据支持字段类型转换，支持一键复制为独立连接流实体
- **运行时**：支持运行记录监控查看，运行时引擎升级以适配版本配置解析、日志采集等新特性
- **安全与准入**：连接器 URL 白名单校验，连接器/连接流数据按应用维度归属隔离，连接器平台能力按应用白名单逐步灰度开放
- **数据模型**：JSON Schema 增强，参数支持 input/output，HTTP 节点支持 header/query/body 参数位置
- **调试**：草稿版本和已发布版本均支持页面直接触发调用，无需部署
- **运维**：连接器、连接流的增删改、启停等变更操作支持记录操作日志

V2 架构核心变化：

| 层级 | V1 | V2 增强 |
|------|-----|--------|
| **版本层** | 单版本，运行时未校验 | 多版本管理（草稿→发布→失效→删除），运行时校验版本 |
| **外部依赖层** | 单一认证，超时不可配 | 多认证（数字签名、Cookie），支持多选，超时可配，URL 白名单校验 |
| **编排层** | 纯串行，3 种节点（触发器/连接器/数据输出） | 新增数据处理节点（字段类型转换），4 种节点；编排支持串行/并行 |
| **数据模型层** | 基础 JSON Schema，无类型展开约束 | JSON Schema 增强：input/output 参数支持、header/query/body 参数位置；数据结构必须递归展开到基本类型（string/number/boolean），设计态硬校验 |
| **执行层** | 调度执行 | 新增版本配置解析、运行记录监控、日志采集 |
| **安全层** | 无 | URL 白名单、应用数据隔离、应用白名单准入 |
| **审批层** | 无 | 版本发布审批（三级：应用级+平台连接流级+全局级）、审批一键催办 |
| **调试通道** | 无 | 草稿版本和已发布版本页面直调，无需部署 |
| **运维层** | 无 | 连接器/连接流增删改、启停等操作日志记录 |

### 1.4 Goals

| # | 目标 |
|---|------|
| **G1** | **连接器：配置多版本** — 支持多个已发布版本并行共存，可按需切换查看任意历史版本。生命周期：草稿 → 发布 → 失效 → 删除 |
| **G2** | **[已移除 — 并入 NG]** 连接器级限流配置 |
| **G3** | **连接器：认证类型增强** — 现有 SOA/APIG 基础上新增数字签名认证、Cookie 认证，支持认证多选组合，凭证支持配置放置位置（Header/Query） |
| **G4** | **连接流：配置多版本** — 支持多个已发布版本并行共存，可按需切换查看任意历史版本。生命周期：草稿 → 发布 → 失效 → 删除 |
| **G5** | **连接流：生命周期增强** — 部署 → 启动 → 停止，生命周状态：待部署 → 运行中 → 已停止 |
| **G6** | **连接流：版本发布审批** — 连接流版本发布需经过三级审批：应用级版本发布审批人、平台级连接流统一审批人、全局审批人，全部通过后版本生效。平台管理员统一配置审批人，复用开放平台现有三级审批能力 |
| **G7** | **连接流：版本发布审批一键催办** — 连接流版本发布审批支持一键催办，复用开放平台现有审批催办能力，拓展至版本发布审批场景 |
| **G8** | **连接流：流程配置增强** — 编排支持触发器节点 SYSTOKEN 凭证白名单、连接器节点超时可配、连接流自身触发限流、错误处理、串行/并行分支（仅并行，无条件和循环）；连接器节点可选引用版本；新增数据处理节点 |
| **G9** | **连接流：字段数据类型转换** — 数据处理节点支持字段级数据类型转换（string↔int、日期格式转换等）。本期仅支持函数处理类型，函数仅支持字段类型转换系列。字段映射已天然支持（节点间通过 `$.node.result.field` 引用），脚本/表达式/其他函数 → NG |
| **G18** | **连接流：一键复制** — 连接流列表支持一键复制，复制后生成独立连接流实体（含完整版本历史），名称自动追加 `_copy_xxxxx` 随机后缀，状态默认为「待部署」，仅限同应用内复制 |
| **G10** | **运行时：运行监控** — 连接流最近运行记录查看（触发时间、状态、耗时、触发方式） |
| **G11** | **运行时：运行时增强** — 运行时支持版本配置读取解析、日志采集记录、新增特性适配 |
| **G12** | **安全：连接器 URL 白名单校验** — 配置连接器时设置正则规则作为 URL 白名单，运行时按规则校验实际请求地址 |
| **G13** | **安全：数据按应用隔离** — 连接器、连接流数据按应用维度归属隔离，不同应用间资源互不可见 |
| **G14** | **安全：连接器平台应用白名单** — 平台管理员维护可开通连接器功能的应用白名单，白名单内应用才可使用连接器平台能力，支持逐步灰度 |
| **G15** | **数据模型：JSON Schema 增强** — 参数传递支持 input/output；HTTP 类型节点支持参数位置：header、query、body；全平台数据结构必须严格递归展开到基本数据类型（string/number/boolean），object/array 不可作为叶子类型；映射赋值时禁止非基本类型（object/array）整体引用，引用路径终点必须为基本类型且源目标类型一致 |
| **G16** | **调试：调试触发** — 连接流草稿版本和已发布版本均支持在页面直接触发调试调用，无需部署。失效版本不支持调试 |
| **G17** | **运维：操作日志** — 连接器、连接流的增删改、启停等变更操作支持记录操作日志 |

### 1.6 关键设计决策

| 维度 | 关键设计 | 保护对象 | 配置对象 | 归属 |
|------|---------|---------|---------|:--:|
| **超时** | 属于连接流，非连接器。超时是调用方诉求，不同流对同一连接器可设不同超时值 | 调用方（连接流）不长时间阻塞 | 应用管理员编排连接流时按节点配置 | G8 |
| **入站限流** | 属于连接流。限制连接流被触发的频率，防止自身过载 | 连接流自身 | 应用管理员在连接流编排时配置 | G8 |
| **缓存** | 属于连接流。通过缓存子图结果减少重复调用 | 后端系统 + 调用效率 | 应用管理员在连接流运行时配置中设定 | G8 |
| **引用稽核** | 被引用方的校验通过查询引用方确定，不把引用关系固化到被引用方的状态或字段中。被引用方只关心自身配置，不关心"被谁用"。细则：① FlowVersion「已部署」不设独立状态，失效前查 `flow.deployed_version_id` 即可（1:1，O(1)）；② ConnectorVersion 的编排引用通过 `connector_version_ref` 中间表显式管理（M:N），保存编排时同步维护 | — | 引用关系存于引用方或中间表 | — |

### 1.7 核心业务对象生命周期

V2 有四个核心业务对象承载状态流转，是功能需求的基础约束。生命周期决定了 FR 中「可/不可」操作的边界条件，以及 EC 的处理逻辑。

> 💡 **关键认知**：连接器和连接流是状态容器，自身状态极简；版本是真正的配置载体，承载完整的发布/失效/审批流转。

| 对象 | 状态数 | 状态列表 | 审批 | 多版本并存 |
|------|:---:|------|:---:|:---:|
| 连接器 (Connector) | 4 | 有效可用 → 有效不可用 → 已失效 → 物理删除 | ❌ | — |
| 连接器版本 (ConnectorVersion) | 4 | 草稿 → 已发布 → 已失效 → 物理删除 | ❌ | ✅ |
| 连接流 (Flow) | 5 | 待部署 → 运行中 ⇄ 已停止 → 已失效 → 物理删除 | ❌ | — |
| 连接流版本 (FlowVersion) | 7 | 草稿 → 待审批 → 已撤回 / 已驳回 → 已发布 → 已失效 → 物理删除 | ✅ 三级 | ✅ |

#### 1.7.5 四对象关系与约束总结

```mermaid
erDiagram
    Connector ||--o{ ConnectorVersion : "1 对 N 版本"
    Flow ||--o{ FlowVersion : "1 对 N 版本"
    ConnectorVersion ||--o{ ConnectorNode : "被编排引用"
    FlowVersion ||--|| Flow : "部署时绑定"
```

| 约束 | 涉及对象 | 说明 |
|------|---------|------|
| 删除连接器需检验无引用 | Connector → Flow (ConnectorNode) | 运行中流引用某连接器的任意版本，则该连接器不可删除 |
| 失效版本需检验无流引用 | ConnectorVersion → Flow (ConnectorNode) | 任何流引用该版本即禁止失效 |
| 失效流版本需检验未部署 | FlowVersion → Flow | 已部署到运行中/已停止流的版本禁止失效 |
| 删除流需已停止 | Flow | 仅已停止状态可删除 |
| 复制仅限同应用 | Flow → Application | 跨应用不可复制 |

## 3. 功能需求

### 3.1 连接器实体（G1）

> 💡 连接器实体是版本的载体，自身状态极简。本节约束连接器实体的创建操作。

| FR | 名称 | 描述 | 验收标准 |
|----|------|------|---------|
| **FR-001** | 创建连接器 | V2：创建连接器时自动生成一个空草稿版本作为初始版本，版本号从 1 开始递增 | ① 创建连接器时填写基本信息（名称、描述、协议等），提交后创建连接器实体，同时自动生成一个空草稿版本（版本号如 v1），连接器进入「有效不可用」状态（§1.7.1）<br>② 草稿版本随连接器创建自动生成，不单独触发 |
| **FR-002** | 恢复连接器 | V2：已失效的连接器可恢复 | ① 仅「已失效」状态可恢复<br>② 恢复后连接器进入「有效」状态，具体为「有效可用」还是「有效不可用」由当前是否存在已发布版本来决定——有已发布版本则为「有效可用」，无已发布版本则为「有效不可用」（与 §1.7.1 的判定逻辑一致）<br>③ 恢复操作记录操作日志（FR-046） |
| **FR-003** | 失效连接器 | V2：有效可用或有效不可用的连接器可标记为已失效 | ① 标记失效前校验无连接流引用该连接器的任何版本（§1.7.1）<br>② 标记失效后连接器状态变为「已失效」，不可编辑、不可发布版本 |
| **FR-004** | 删除连接器 | V2：已失效的连接器可物理删除 | ① 仅「已失效」状态可删除<br>② 删除前二次确认，删除后状态变为「物理删除」，不可恢复 |

### 3.2 连接器版本（G1）

> 💡 连接器版本生命周期见 §1.7.2：草稿 → 已发布 → 已失效 → 物理删除。V1 无版本概念，V2 新增完整多版本管理。

| FR | 名称 | 描述 | 验收标准 |
|----|------|------|---------|
| **FR-005** | 编辑草稿 | V1 编辑即生效。V2 新增：打开已有草稿版本进行编辑，保存覆盖当前草稿内容，不影响已发布版本。草稿版本由创建连接器时自动生成（FR-001）或通过复制到草稿（FR-006）创建 | ① 点击「编辑配置」打开已有草稿版本进行编辑<br>② 草稿保存覆盖当前草稿内容，不产生新版本号<br>③ 每连接器同时最多一个草稿，再次编辑复用已有草稿（EC-008）<br>④ 草稿不影响运行中的连接流——它们仍使用各自引用的已发布版本 |
| **FR-006** | 复制到草稿 | V2 新增：在已发布版本上点击「复制到草稿」，以该版本配置快照为内容生成或覆盖草稿版本 | ① 连接器版本列表的已发布版本行提供「复制到草稿」操作按钮<br>② 点击后以该已发布版本的完整配置快照为内容生成新草稿（新版本号递增），不影响原版本<br>③ 复制前校验版本总数：达 1000 上限时禁止复制，提示「版本数量已达上限（1000），请清理失效版本后再试」<br>④ 若当前已有草稿，以选中版本快照覆盖当前草稿内容；若无草稿，则新建草稿版本 |
| **FR-007** | 发布版本 | V2 新增：草稿发布为正式版本，同一数据记录状态由草稿变更为已发布，多已发布版本并行共存 | ① 仅草稿状态可发布；草稿配置为空时禁止发布（EC-009），提示「请先完善连接配置」<br>② 发布后版本状态变更为「已发布」，版本号沿用草稿时的版本号<br>③ 多已发布版本并行共存，互不影响<br>④ 首次发布时连接器状态从「有效不可用」→「有效可用」（§1.7.1） |
| **FR-008** | 版本查看 | V2 新增：查看已发布版本列表，可切换查看任意历史版本配置详情 | ① 版本列表展示：版本号、状态（已发布/已失效）、发布时间、发布人<br>② 点击版本行展开/跳转该版本的完整配置快照（只读）<br>③ 支持切换查看任意历史版本，当前选中行高亮<br>④ 无已发布版本时提示「暂无已发布版本」<br>⑤ 草稿版本在列表中单独展示，标记为「草稿」 |
| **FR-009** | 版本失效 | V2 新增：已发布版本可标记失效。已有连接流引用的版本不可标记失效，失效后不可被引用 | ① 仅「已发布」状态可标记失效，操作后状态变为「已失效」；草稿不可标记失效<br>② 失效前校验：查询引用关系（`connector_version_ref` 中间表），有连接流编排引用该版本时禁止失效（EC-001），提示「以下连接流引用了此版本：xxx」，列出引用流名称<br>③ 失效后该版本不可被新连接流编排引用<br>④ 若失效的是连接器最后一个已发布版本，连接器状态「有效可用」→「有效不可用」（§1.7.1） |
| **FR-010** | 版本删除 | V2 新增：失效版本可删除 | ① 仅「已失效」状态可删除，操作后状态变为「物理删除」<br>② 删除不可恢复，删除前二次确认「删除后不可恢复，确认删除？」<br>③ 「草稿」状态不可标记失效，不可删除<br>④ 「已发布」状态需先标记失效，再删除 |
| **FR-011** | 恢复版本 | V2 新增：已失效的连接器版本可恢复至「已发布」状态 | ① 仅「已失效」状态可恢复，操作后状态变为「已发布」<br>② 恢复后版本可重新被连接流编排引用<br>③ 若恢复的是连接器唯一已发布版本，连接器状态「有效不可用」→「有效可用」（§1.7.1） |

### 3.3 连接器配置（G3、G12）

| FR | 名称 | 描述 | 验收标准 |
|----|------|------|---------|
| **FR-012** | 认证类型 | V1 已支持 SOA、APIG。V2 新增：数字签名认证、Cookie 认证 | ① 连接器认证配置页面提供认证类型选择：SOA / APIG / 数字签名 / Cookie<br>② 选择数字签名后，仅需输入 Secret Key，加密存储，算法由平台统一处理<br>③ 选择 Cookie 后，仅需填写 Cookie 名称（如 `SESSION_ID`），Cookie 的具体值由连接流编排时手动映射，连接器侧不定义来源<br>④ 凭证加密存储，界面脱敏显示（同 V1 凭证安全要求 NFR-013）<br>⑤ 切换认证类型时提示「切换认证类型将清空当前凭证配置」，需用户确认 |

### 3.4 连接流实体（G5、G18）

> 💡 连接流生命周期见 §1.7.3：待部署 → 运行中 ⇄ 已停止 → 已失效 → 删除。部署即启动，部署和启动是同一动作。

| FR | 名称 | 描述 | 验收标准 |
|----|------|------|---------|
| **FR-016** | 创建连接流 | V2：创建连接流时自动生成一个空草稿版本作为初始版本，版本号从 1 开始递增 | ① 创建连接流时填写基本信息（名称、描述等），提交后创建连接流实体，同时自动生成一个空草稿版本（版本号如 v1），连接流进入「待部署」状态（§1.7.3）<br>② 草稿版本随连接流创建自动生成，不单独触发 |
| **FR-017** | 一键复制 | 连接流列表支持一键复制，复制后生成独立连接流实体（含完整版本历史），名称自动追加 `_copy_xxxxx` 随机后缀，状态默认为「待部署」。仅限同应用内复制 | ① 连接流列表每行提供「复制」操作按钮<br>② 点击复制后：创建新连接流实体，复制全部版本历史（含草稿/已发布/已失效等所有状态的版本）到新流<br>③ 新连接流名称 = 原名称 + `_copy_xxxxx`（xxxxx 为随机 4 位十六进制 0000~ffff）<br>④ 后端校验名称唯一性，碰撞自动重试新后缀（EC-017）<br>⑤ 新连接流状态默认为「待部署」（§1.7.3）<br>⑥ 仅限同应用内复制：源连接流所属应用 = 目标应用，跨应用不可复制<br>⑦ 复制时源连接流的状态不影响复制（运行中/已停止均可复制），复制生成的是独立实体（EC-016） |
| **FR-018** | 部署 | V1 编辑保存即视为部署。V2 增强：选择已发布版本部署并启动，状态「待部署」→「运行中」 | ① 在连接流详情页选择已发布版本，点击「部署」，系统绑定版本并立即启动<br>② 部署后连接流状态变为「运行中」，开始响应 HTTP 触发（§1.7.3）<br>③ 若连接流当前为「运行中」状态，部署操作替换当前运行版本为新选择的版本<br>④ 部署操作记录操作日志（FR-046） |
| **FR-019** | 启动 | V1 已支持。V2 增强：已停止的连接流可启动，恢复运行 | ① 仅「已停止」状态可启动，操作后状态变为「运行中」<br>② 启动后开始响应 HTTP 触发<br>③ 启动操作记录操作日志（FR-046）<br>④ 「已失效」状态不可启动 |
| **FR-020** | 停止 | V1 已支持。V2 增强：运行中的连接流可停止 | ① 仅「运行中」状态可停止，操作后状态变为「已停止」<br>② 停止后不再响应新触发；当前执行中的实例继续完成（EC-007）<br>③ 停止操作记录操作日志（FR-046）<br>④ 停止后仍可查看运行记录和版本历史 |
| **FR-021** | 恢复连接流 | V2：已失效的连接流可恢复 | ① 仅「已失效」状态可恢复<br>② 恢复后连接流状态统一变为「已停止」（安全中间态，不自动响应触发）<br>③ 恢复后管理员可手动启动（FR-019）<br>④ 恢复操作记录操作日志（FR-046） |
| **FR-022** | 失效连接流 | V2：待部署或已停止的连接流可标记为已失效 | ① 运行中的连接流不可直接失效，必须先停止（§1.7.3）<br>② 标记失效后连接流状态变为「已失效」，不可编辑、不可启动 |

### 3.5 连接流版本（G4）

### 3.8 连接流编排 — 编排画布（G8、G9）

| FR | 名称 | 描述 | 验收标准 |
|----|------|------|---------|
| **FR-047** | 数据结构定义与引用类型约束 | 全平台所有数据结构定义（连接器入参/出参 JSON Schema、连接流节点输入/输出结构、数据处理节点输出字段、HTTP 节点 header/query/body 参数等）必须递归展开到基本数据类型（string/number/boolean），object 和 array 不允许作为未展开的叶子类型 | ① 基本类型限定为 string、number、boolean 三种；null 不作为合法字段类型，number 不区分 integer/float<br>② object 类型字段必须定义子字段结构，每个子字段递归到基本类型；禁止无子结构的空 object<br>③ array 类型字段必须声明 items 元素类型；若 items 为 object，需继续递归展开子字段到基本类型<br>④ 设计态实时校验：Schema 编辑器或编排画布中，object 无子字段或 array 无 items 定义时标红，禁止保存或发布，提示「object 类型必须定义子字段结构」或「array 类型必须声明元素类型」<br>⑤ 连接器版本发布时校验入参/出参 Schema 合规性，不满足则禁止发布<br>⑥ 连接流编排保存时校验所有节点间数据结构定义的合规性<br>⑦ 数据处理节点中 object/array 类型不再作为终点类型，选后必须进一步展开子结构定义<br>⑧ 映射赋值时禁止非基本类型（object/array）通过 value 表达式整体引用赋值——即不允许出现 `{ "type": "object", "value": "${$.node.xxx.output.meta}" }` 这类写法；object/array 必须逐字段展开，每个叶子字段各自引用基本类型字段<br>⑨ 基本类型字段的 value 表达式引用的上游字段类型必须与当前字段声明的 type 一致——string→string、number→number、boolean→boolean，禁止隐式类型转换，严禁 string↔number 等跨类型赋值<br>⑩ 所有映射表达式（inputMapping/outputMapping 的 value、data_processor 的 source）的引用路径终点必须可解析到基本类型字段（string/number/boolean）；路径终点为 object 或 array 时，设计态标红禁止保存，提示「不支持引用非基本类型，请展开到具体字段」<br>⑪ 数组字段（type=array）的 items 内各子字段的 value 表达式，最多只能引用一个上游 array 类型字段（通过被引用字段的 Schema 定义判定其类型）；禁止同时引用两个不同 array 源的字段，避免数组长度不一致导致歧义；引用常量或 object 内基本类型字段无此限制；若 items 内所有 value 表达式均未引用任何 array 类型字段，则该数组最终长度为 1（单元素数组） |

## 6. 边界情况

| EC | 场景 | 处理方式 |
|----|------|---------|
| EC-001 | 连接器版本被标记失效时仍有连接流引用 | 已有引用的版本禁止失效，提示影响范围 |
| EC-002 | 连接流版本被标记失效时该版本正在运行 | 已部署的版本禁止失效，提示先停止 |
| EC-003 | 版本发布审批超时未处理 | 版本保持「待审批」状态，可催办 |
| EC-004 | 连接器版本引用被删除 | 连接流编排保存时校验，提示引用版本不可用 |
| EC-005 | HTTP 触发 URL 被非法调用 | 沿用 V1：签名失败 401 + 限流兜底 |
| EC-006 | 连接流执行超时 | 沿用 V1：强制终止，标记超时 |
| EC-007 | 连接流执行中被停止 | 沿用 V1：当前实例继续完成，新触发不响应 |
| EC-008 | 同一连接器多个草稿 | 每连接器仅一个草稿，再次编辑覆盖 |
| EC-009 | 草稿配置为空时发布 | 校验不通过，禁止发布 |
| EC-010 | URL 正则白名单规则语法错误 | 设计态实时校验正则合法性，不合法禁止保存 |
| EC-011 | SYSTOKEN 白名单为空 | 空即全部禁止，所有凭证不可触发，需至少配置一条白名单才能触发 |
| EC-012 | 缓存过期或未命中 | 正常执行 DAG，不中断流程 |
| EC-013 | 数据处理节点类型转换失败 | 标记节点失败，保留原始值和错误信息 |
| EC-014 | 调试触发时引用的连接器版本已失效 | 调试失败，提示引用版本不可用 |
| EC-015 | 应用被移出白名单 | 已开通的应用数据保留，新操作拒绝 |
| EC-016 | 复制时源连接流正在运行 | 不受影响，新流为独立实体，状态为待部署 |
| EC-017 | 复制后名称后缀碰撞 | 随机 4 位十六进制（0000~ffff），后端校验唯一性，碰撞自动重试 |
| EC-018 | 数据结构定义未展开到基本类型，或映射引用违反类型约束 | 设计态实时校验：① object 无子字段 / array 无 items → 标红，禁止保存/发布；② value 表达式引用路径终点为 object/array → 标红，提示「不支持引用非基本类型，请展开到具体字段」；③ value 表达式引用的上游字段类型与声明类型不一致 → 标红，提示「类型不匹配：字段 "xxx" 声明为 number，引用源 "yyy" 类型为 string」<br>④ 数组 items 内 value 表达式同时引用了两个不同 array 源的字段 → 标红，提示「数组元素引用冲突：同时引用了 "xxx" 和 "yyy" 两个不同数组，只能引用一个数组源或全部使用非数组字段」（FR-047） |
| EC-019 | 连接器「复制到草稿」时版本数已达上限（1000） | 禁止复制，提示「版本数量已达上限（1000），请清理失效版本后再试」（FR-006） |
| EC-020 | 连接流「复制到草稿」时版本数已达上限（1000） | 禁止复制，提示「版本数量已达上限（1000），请清理失效版本后再试」（FR-025） |
| EC-021 | 恢复连接器时所有版本均已删除 | 恢复成功但提示「无已发布版本，连接器处于有效不可用状态，请先发布版本」（FR-002） |

| 认证方式 | 已支持 SOA、APIG | 新增数字签名、Cookie，支持多选 |
| 入站限流 | 无 | 连接流级可配 |
| 编排模式 | 纯串行 | 串行 + 并行（边级） |
| 节点类型 | 触发器、连接器、数据输出 | 新增数据处理节点 |
| 版本发布审批 | 无 | 版本发布需审批，支持催办 |
| 运行监控 | 无 | 运行记录查看 |
| 运行日志 | 无 | 节点 I/O 日志采集 |
| 安全 | 无 | URL 正则白名单、SYSTOKEN 白名单、应用白名单 |
| 数据模型约束 | 无强制类型展开 | object/array 必须递归展开到基本类型（string/number/boolean），设计态硬校验 |
| 调试 | 必须发布部署后验证 | 草稿/已发布版本直接调试 |
| 审计 | 无 | 变更操作日志 |
| 角色 | 平台管理员 | 平台管理员 + 应用管理员 |
| 运行时 | HTTP 同步调度 | 完全复用 V1 |
