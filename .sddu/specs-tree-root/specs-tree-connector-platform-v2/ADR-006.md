# ADR-006: MySQL 主存储运行记录与日志

## 状态
PROPOSED

## 背景

V2 新增运行时监控能力（G10、G11），需要存储和查询：
1. **运行记录**（execution_record）：每次连接流触发的元数据（触发时间、状态、耗时、触发方式）
2. **运行日志**（execution_step）：每个节点执行的输入/输出数据快照、耗时、错误信息

V1 已预留 `execution_record_t`、`execution_step_t` 表结构，但**未经生产验证**（V1 MVP 未写入这两张表），V2 启用前需重新审视表设计。`storage_blob_ref_t` 经评估后移除——MEDIUMTEXT 16MB 对 JSON 日志足够，无需外置对象存储。

## 决策

使用 **MySQL 作为运行记录和日志的主存储**，配合定时清理任务控制数据量。

### 存储策略

| 数据 | 表 | 保留策略 | 清理方式 |
|------|-----|---------|---------|
| 运行记录 | `execution_record_t` | 保留最近 30 天 | 每天凌晨 3:00 定时任务清理 |
| 运行日志 | `execution_step_t` | 保留最近 30 天 | 每天凌晨 3:00 定时任务清理 |

### 写入策略

- **异步写入**：运行记录和日志在请求完成后异步写入，不阻塞请求响应
- **写入失败隔离**：日志写入失败不影响业务流程，仅记录错误
- **批量写入**：单个请求的多个节点日志合并为一次批量 INSERT

### 查询策略

- 运行记录列表：按 `flow_id` + 时间范围查询，默认倒序，分页加载
- 运行日志详情：按 `execution_record_id` 查询所有关联的 `execution_step` 记录
- 查询接口走 open-server（管理面），connector-api 仅负责写入

## 后果

### 正面影响

1. **复用现有表结构**：V1 已预留完整表结构（含索引），直接启用，零额外建表成本
2. **事务一致性**：运行记录和日志在同一 MySQL 实例，支持事务写入
3. **查询简单**：标准 SQL 查询，无需学习新的查询语言或 API
4. **运维简单**：不引入新存储组件，备份/恢复/监控体系复用现有 MySQL 运维

### 负面影响

1. **写入压力**：高 TPS 场景下异步写入可能积压（300 TPS × 平均 5 节点 = 1500 writes/s）
2. **存储空间**：每节点日志 ~2KB，30 天累计数据量需评估
3. **查询性能**：大时间范围查询可能慢，需合理索引

### 缓解措施

- 写入使用异步队列（内存缓冲 + 批量写入），削峰填谷
- 定时清理任务严格控制数据量
- 运行记录表按 `trigger_time` 分区（如按月分区），加速清理和查询
- 设置日志写入队列容量上限，超限时丢弃最旧日志（降级策略）
- 若后续数据量超预期，可平滑迁移至 MySQL + 对象存储混合方案

## ⚠️ V1 预留表启用前需修正的问题

V1 的三张运行时表未经生产验证，V2 启用前需逐表审视：

| # | 表 | 问题 | 修正 |
|:--:|-----|------|------|
| 1 | `execution_record_t` | `execution_status` 起始值偏移：V1 用 `1=pending`，与 V2 规范 `0=pending` 不一致 | MODIFY 为 `DEFAULT 0` |
| 2 | `execution_record_t` | `trigger_type` 列已存在，plan-db.md 误写 `ADD COLUMN` | 改为 `MODIFY COLUMN` |
| 3 | `execution_record_t` | `flow_version_id` 列已存在，同上 | 改为 `MODIFY COLUMN` |
| 4 | `execution_record_t` | 索引 `idx_flow_trigger_time` 引用不存在的 `trigger_time` 列（V1 仅有 `create_time`） | 改为 `idx_flow_create_time (flow_id, create_time)` |
| 5 | `execution_step_t` | `node_type` 为 `VARCHAR(32)` 字符串枚举，违反 V2 TINYINT 规范 | MODIFY 为 `TINYINT(10)` |
| 6 | `execution_step_t` | `step_status` 起始值偏移 | MODIFY 为 `DEFAULT 0` |
| 7 | 两表 | 审计字段 `VARCHAR(50)`，V2 规范要求 `VARCHAR(100)` | MODIFY 列宽 |

> 💡 详细 DDL 修正见 plan-db.md §3.9~3.10（待单独审视）。

## 替代方案

| 方案 | 描述 | 拒绝理由 |
|------|------|---------|
| Elasticsearch | 日志写入 ES，运行记录留 MySQL | 引入新组件，运维复杂度增加，V2 日志量尚不需要 ES 的检索能力 |
| MySQL + 对象存储归档 | 热数据 MySQL，冷数据自动迁移至对象存储 | 实现复杂度高，V2 阶段日志量可控，预留扩展点即可 |
| 仅内存不持久化 | 运行日志仅内存缓存，不落库 | 页面刷新后日志丢失，无法满足运维审计需求 |
