# ADR-001: 采用单体应用 + 模块化架构

## 状态

ACCEPTED

## 背景

能力开放平台（Capability Open Platform）是 XX 通讯系统的统一开放底座，需要支撑 API 管理、事件管理、回调管理、权限管理、审批管理等多个业务模块。

在架构选型时，面临以下选择：
1. **单体应用**：所有模块在同一个应用中，通过模块化设计解耦
2. **微服务架构**：每个模块独立服务，通过 RPC/消息队列通信
3. **BFF 模式**：为不同前端场景提供专属后端服务

## 决策

我们决定采用 **单体应用 + 模块化设计** 架构。

### 决策内容

1. **后端**：使用 Spring Boot 单体应用，通过 Package 划分业务边界
2. **前端**：使用 React 单体应用，通过路由和组件划分页面
3. **数据库**：使用 MySQL 数据库，通过表前缀区分业务
4. **部署**：单一部署单元，使用集群模式实现高可用

### 模块划分

```
Backend (Spring Boot)
├── modules/
│   ├── group/       # 分组管理
│   ├── api/         # API 管理
│   ├── event/       # 事件管理
│   ├── callback/    # 回调管理
│   ├── permission/  # 权限管理
│   ├── approval/    # 审批管理
│   ├── gateway/     # 消费网关
│   └── scope/       # Scope 授权
└── common/          # 公共模块
```

## 理由

### 为什么选择单体架构

| 维度 | 单体架构优势 |
|------|-------------|
| **MVP 适配** | 项目处于初期建设阶段，需要快速迭代验证 |
| **开发效率** | 统一代码仓库，减少跨服务协调成本 |
| **运维成本** | 单一部署单元，运维复杂度低 |
| **团队匹配** | 与现有 app-management 技术栈一致，学习成本低 |
| **事务简单** | 模块间调用在同一进程内，事务管理简单 |

### 为什么不选择微服务

| 维度 | 微服务劣势 |
|------|-----------|
| **运维复杂** | 需要服务发现、配置中心、链路追踪等基础设施 |
| **开发成本** | 多代码仓库、跨服务调试困难 |
| **分布式事务** | 跨服务调用需要处理分布式事务 |
| **过度设计** | 初期流量不大，微服务收益不明显 |

### 为什么不选择 BFF

| 维度 | BFF 劣势 |
|------|----------|
| **代码重复** | BFF 层可能存在重复逻辑 |
| **维护成本** | 多个服务需要独立维护 |
| **当前不需要** | 前端场景相对统一，无需针对性优化 |

## 后果

### 正面影响

1. **快速交付**：预计 80 人天完成 MVP，比微服务方案节省 65 人天
2. **低运维成本**：单一部署单元，无需复杂基础设施
3. **易于调试**：本地开发无需启动多个服务
4. **团队协作**：单代码仓库便于代码审查和知识共享

### 负面影响

1. **单点故障**：应用宕机影响全部功能
   - **缓解措施**：使用 PM2 集群模式 + 负载均衡 + 健康检查
2. **扩展受限**：无法按模块独立扩缩容
   - **缓解措施**：前期流量可预测，整体扩展即可满足需求
3. **代码耦合风险**：随着功能增加，模块边界可能模糊
   - **缓解措施**：严格模块化设计 + 代码审查 + 架构评审

### 演进路径

当业务规模增长到以下阈值时，可考虑微服务拆分：

| 阈值 | 说明 |
|------|------|
| 团队规模 > 20 人 | 需要独立团队负责不同模块 |
| 单模块 QPS > 10000 | 需要独立扩展特定模块 |
| 模块独立发布需求 | 需要频繁发布特定模块 |

拆分优先级建议：
1. **Gateway Service**：消费网关首先拆分，独立扩展
2. **Approval Service**：审批服务其次拆分，可复用
3. **Permission Service**：权限服务最后拆分，为核心服务
