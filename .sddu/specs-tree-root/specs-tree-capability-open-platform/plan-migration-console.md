# 迁移控制台设计文档

## 1. 迁移控制台概述

### 1.1 设计目标

迁移控制台是一个离线HTML页面工具，用于帮助开发人员和运维人员在系统迁移过程中可视化地管理数据迁移。核心设计理念：
- **离线可用**：单HTML文件，无需服务器，可直接在浏览器中打开
- **API驱动**：通过HTTP API与后端服务交互，支持不同环境
- **可视化操作**：提供图形界面，支持勾选、批量操作
- **实时反馈**：展示迁移进度和结果统计

### 1.2 核心功能

1. **配置管理**：配置API地址、上下文根、请求头等信息
2. **数据查询**：查询旧表数据、新表数据、迁移状态
3. **数据选择**：通过勾选框选择需要迁移的数据
4. **迁移操作**：执行单条/批量/全量迁移
5. **验证操作**：验证迁移数据的正确性
6. **结果展示**：展示迁移统计、失败详情、实时日志

### 1.3 使用场景

| 环境 | API地址示例 | 使用场景 |
|------|------------|----------|
| 开发环境 | `http://localhost:8080` | 开发人员本地测试迁移功能 |
| 测试环境 | `http://test-server:8080` | 测试人员进行迁移验证 |
| 生产环境 | `https://prod-server:8443` | 运维人员执行生产迁移 |

```javascript
    ]
}
```

## 3. 数据查询API设计

### 3.1 查询旧表数据API

**接口定义**
```
GET /api/v1/migration/{module}/source
```

**路径参数**
| 参数 | 类型 | 说明 |
|------|------|------|
| module | string | 模块名称（category/api/event/permission/approval/subscription） |

**查询参数**
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | integer | 否 | 页码，默认1 |
| size | integer | 否 | 每页大小，默认50 |

**响应示例**
```json
{
    "code": 0,
    "message": "success",
    "data": {
        "total": 150,

        "size": 50,
        "records": [
            {
```

## 4. 数据列表展示设计

### 4.1 数据表格结构

| 列名 | 说明 | 宽度 |
|------|------|------|
| 勾选框 | 复选框，用于选择数据 | 40px |
| 状态 | 已迁移/未迁移标识 | 80px |
| ID | 数据唯一标识 | 100px |
| 名称 | 数据名称 | 200px |
| 编码 | 数据编码 | 150px |
| 状态 | 数据状态 | 100px |
| 创建时间 | 数据创建时间 | 150px |

### 4.2 勾选框设计

- **单选**：点击行前的复选框，选中/取消选中单条数据
- **全选**：点击表头复选框，选中当前页所有数据
- **反选**：反转当前页的选中状态
- **只选未迁移**：仅选中状态为"未迁移"的数据

### 4.3 已迁移状态标识

通过对比旧表ID和新表的 `sourceId` 字段判断迁移状态：
- **已迁移**：旧表ID存在于新表的 `sourceId` 中，显示绿色标识
- **未迁移**：旧表ID不存在于新表的 `sourceId` 中，显示灰色标识

### 4.4 分页和筛选功能

**分页功能**
- 支持上一页/下一页/跳转指定页
- 显示总记录数和当前页码
- 支持切换每页显示数量（20/50/100）

**筛选功能**
- 按迁移状态筛选（全部/已迁移/未迁移）
- 按数据状态筛选（全部/启用/禁用）
- 按名称/编码模糊搜索

### 7.3 API调用封装

使用原生 `fetch` API 封装HTTP请求：

```javascript
async request(method, path, body = null) {
    const url = this.getFullUrl(path);
    const options = {
        method,
        headers: this.getHeaders()
    };
    
    if (body) {
        options.body = JSON.stringify(body);
    }
    
    try {
        const response = await fetch(url, options);
        const data = await response.json();
        
        if (data.code !== 0) {
            throw new Error(data.message);
        }
        
        return data.data;
    } catch (error) {
        console.error('API请求失败:', error);
        throw error;
    }
}
```

### 7.4 错误处理机制

**错误类型**
1. **网络错误**：API地址不可达、网络超时
2. **认证错误**：401未授权、403禁止访问
3. **业务错误**：后端返回错误码和错误信息
4. **数据错误**：数据格式不正确、数据缺失

**错误处理流程**
```
发生错误 → 捕获异常 → 记录日志 → 显示提示 → 用户处理
```

**错误提示方式**
- 使用 `alert()` 或自定义提示框显示错误信息
- 在结果展示区域显示错误详情
- 在实时日志中记录错误信息

## 9. 使用流程

### 9.1 准备工作
1. 启动后端服务
2. 使用浏览器打开 `migration-console.html`
3. 配置API地址和请求头

### 9.2 数据迁移流程
1. 选择要迁移的模块
2. 点击"查询旧数据"，查看待迁移数据
3. 勾选需要迁移的数据（或使用"只选未迁移"）
4. 点击"迁移勾选数据"执行迁移
5. 查看迁移结果和失败详情
6. 处理失败数据（重试或手动处理）

### 9.3 数据验证流程
1. 完成数据迁移后
2. 点击"验证勾选数据"或"全量验证"
3. 查看验证结果
4. 处理验证失败的数据

## 10. 注意事项

1. **安全性**：API地址和请求头信息保存在浏览器本地，请勿在公共电脑上保存敏感信息
2. **性能**：大批量数据迁移建议分批进行，避免一次性迁移过多数据
3. **幂等性**：迁移操作支持重复执行，已迁移的数据会被跳过
4. **日志保存**：实时日志仅在当前会话有效，刷新页面后清空
5. **跨域问题**：如果API服务和HTML文件不在同一域名下，需要后端配置CORS
