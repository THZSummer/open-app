# ADR-004: 事件/回调通道地址白名单控制

## 状态

ACCEPTED

## 背景

### 安全问题

能力开放平台 spec.md 定义了事件（FR-019/FR-029）和回调（FR-022/FR-030）的消费通道配置能力：消费方获批后可配置 `channelAddress`（通道地址），平台网关会主动将该地址作为目标发起 HTTP 请求，分发事件或路由回调。

这一设计存在 **SSRF（Server-Side Request Forgery）攻击面**：

| 维度 | 事实 |
|------|------|
| 写入入口 | FR-019（事件消费配置）、FR-022（回调消费配置）——消费方填写 `channelAddress` |
| 执行出口 | FR-029（事件网关分发）、FR-030（回调网关路由）——网关 POST 到 `channelAddress` |
| 现有校验 | spec EC-011 提到"连通性检查或格式校验"，但未定义任何地址级安全约束 |
| 数据库 | ADR-003 DDL 中 `channel_address VARCHAR(500)` 为纯字符串，无格式/安全约束 |

### 风险场景

消费方（或恶意攻击者接管消费方账户后）可将 `channelAddress` 配置为：

| 攻击类型 | 示例 | 危害 |
|---------|------|------|
| 内网服务探测 | `http://10.0.0.1:8080/admin` | 扫描内部服务 |
| 云元数据窃取 | `http://169.254.169.254/latest/meta-data/` | 窃取 AK/SK、IAM 凭证 |
| 本机回环 | `http://127.0.0.1:port/` | 攻击平台自身服务 |
| IPv6 绕过 | `http://[::ffff:127.0.0.1]/` | 绕过 IPv4 黑名单 |
| 协议滥用 | `file:///etc/passwd` | 读本地文件 |
| 内部 API 调用 | `http://internal-service/api/v1/delete` | 触发内部服务写操作 |

### 现有校验缺口

| 文档位置 | 校验内容 | 能否防御 SSRF |
|---------|---------|:---:|
| spec FR-019 / FR-022 | "填写通道地址"——无任何校验描述 | ❌ |
| spec EC-009 | 仅校验通道类型枚举合法性 | ❌ |
| spec EC-011 | "连通性检查或格式校验"——模糊描述，无具体规则 | ❌ |
| plan-api.md `channel_address` 字段 | 类型 `string`，条件必填——无格式/安全约束 | ❌ |
| ADR-003 DDL | `channel_address VARCHAR(500)`——纯字符串 | ❌ |

## 决策

我们决定采用 **平台级 URL 正则白名单** 机制，在消费方配置通道地址时进行前置校验。

### 决策内容

#### 1. 白名单存储

复用现有 `openplatform_property_t` 字典表（与数据字典 FR-DICTIONARY-001 同一张表），零 DDL 变更：

| 字段 | 值 | 说明 |
|------|-----|------|
| `path` | `channel_address_whitelist` | 分组路径 |
| `code` | `callback_url_regex_*` / `event_url_regex_*` | 前缀区分回调/事件 |
| `value` | 正则表达式 | 一条规则一行记录 |
| `status` | 0=失效 / 1=有效 | 可单独禁用某条规则 |

#### 2. 校验时机

本期**仅写入时校验**，运行时不校验：

| 调用点 | 触发场景 |
|--------|---------|
| `EventSubscriptionService.configureChannel()` | FR-019 保存事件接收参数 |
| `CallbackSubscriptionService.configureChannel()` | FR-022 保存回调接收参数 |

> 运行时（FR-029/FR-030 网关分发/路由）不在本期范围内，后续按需补充。

#### 3. 校验规则

- 地址必须命中白名单中**至少一条**正则（`Pattern.compile(regex).matcher(addr).matches()`）
- **空白名单 = 不限制**（兼容初期上线，全部放行）
- 全部不命中 → 拒绝配置，返回 400 + 提示「通道地址不在允许范围内」

#### 4. 管理入口

平台管理员通过现有**数据字典管理**（FR-DICTIONARY-001）维护白名单规则，复用其 CRUD + 导入导出 + 任务中心能力，无需新增管理页面。

#### 5. 对齐设计

| 维度 | 本方案 | 连接器 V3 FR-015（参考） |
|------|--------|-------------------------|
| 控制对象 | 消费方配置的通道地址 | 连接器出站 HTTP 调用 URL |
| 配置角色 | 平台管理员 | 连接器创建者 |
| 校验时机 | 仅写入时 | 仅运行时 |
| 空白名单语义 | 空 = 不限制 | 空 = 不限制 |
| 存储 | `openplatform_property_t` | 连接器版本快照 `urlWhitelist[]` |

## 理由

### 为什么选择正则白名单

| 维度 | 优势 |
|------|------|
| **灵活性** | 正则可精确匹配域名、路径前缀、端口等任意维度 |
| **复用性** | 直接复用数据字典管理能力，零新增管理页面 |
| **对称性** | 与连接器 V3 FR-015 形成统一的安全模型 |
| **零 DDL** | 复用现有 `openplatform_property_t`，无需新建表 |
| **渐进式** | 空白名单 = 不限制，可分阶段上线（先观察后拦截） |

### 为什么本期仅做写入时校验

| 维度 | 理由 |
|------|------|
| **范围控制** | 写入时校验已能阻止新增恶意地址，覆盖主要风险 |
| **存量兼容** | 运行时校验会导致存量配置突然失效，需要额外的迁移/通知机制 |
| **实现成本** | 写入时校验仅 2.5d，运行时校验需额外改造网关，成本高 |
| **后续可扩展** | 运行时校验可作为后续增强，不阻塞本期交付 |

### 为什么不选择其他方案

| 替代方案 | 不选原因 |
|---------|---------|
| IP 黑名单 | 维护成本高，无法覆盖域名级控制，DNS 解析可变 |
| 域名白名单（精确匹配） | 灵活性不足，无法支持通配符、路径前缀等场景 |
| 连通性检查（spec EC-011 原文） | 平台主动探测目标地址，反而可被利用为内网扫描工具 |
| 第三方 SSRF 防护库 | 引入外部依赖，且主要防护 IP 层，不解决业务层地址控制需求 |

## 后果

### 正面

- ✅ 堵住消费方配置通道地址的 SSRF 攻击面
- ✅ 平台管理员可集中管控允许的地址范围
- ✅ 与连接器 V3 FR-015 形成对称安全模型
- ✅ 零 DDL 变更，复用现有字典管理能力

### 负面

- ⚠️ 平台管理员需维护白名单规则，有一定运维成本
- ⚠️ 正则配置有一定门槛，错误正则可能导致合法地址被拒
- ⚠️ 运行时不校验，存量恶意地址无法自动清理

### 风险缓解

| 风险 | 缓解措施 |
|------|---------|
| 正则配置错误 | 字典写入时校验正则语法合法性（`Pattern.compile`） |
| 空白名单期被利用 | 上线后尽快录入已知合法地址规则，切换为拦截模式 |
| 存量恶意地址 | 后续版本补充运行时校验 + 存量地址巡检机制 |
| 白名单收紧影响存量 | 收紧前提前通知消费方重新配置，设置过渡期 |
