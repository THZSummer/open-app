## 分析过程总结

### 第一阶段：问题空间探索

**关键洞察**：
- 业务驱动：AI 时代战略，XX 通讯平台需快速开放能力给三方平台
- 核心痛点：已有能力较为薄弱，无法满足业务发展需要，缺失能力无法支撑业务演进
- 技术策略：相对独立的能力用 AI 完整构建，避免在历史债上继续加码
- 建设策略分层：
  - 沿用现有代码：应用管理、成员管理
  - 重新构建（有存量业务）：API 管理、事件/回调管理、权限管理
  - 完全新建：连接器等

### 第二阶段：能力分类模型

**关键决策**：
- 能力分为两类：
  1. 平台本身的能力：应用管理、成员管理、操作日志等
  2. 连接内外业务的能力：API、事件、回调、连接器等
- 连接业务能力进一步分为：
  - 公共连接能力：API、事件等（开放平台自己构建，不属于某个业务模块独有）
  - 特有连接能力：XX 业务能力（属于某个业务模块，如 IM 业务模块）
- 能力粒度：暂不考虑具体能力粒度，关注能力的归属设计
- 核心用户：能力消费方（设计以消费方体验为中心）

### 第三阶段：治理与架构确认

**核心设计原则**：
- 审批机制：属于谁的业务都由谁去审
- 平台定位：开放平台不做中间平台，避免多重依赖
- 与数据开放平台关系：能力开放平台是数据开放平台的"超集"，但能力开放不特意体现数据开放的东西
- 权限归属：能力开放是基础，权限管理属于能力的一部分
- MVP 范围：API 管理、事件/回调管理、权限管理、审批管理
- AI 构建范围：新的独立模块（API 管理、事件/回调管理、权限管理、审批管理）
- 部署模式：按需独立部署

## 与数据开放平台的关系

| 维度 | 关系说明 |
|------|---------|
| **定位** | 能力开放平台是数据开放平台的"超集" |
| **边界** | 能力开放不特意体现数据开放的东西 |
| **复用** | 权限管理属于能力的一部分，数据开放平台可复用能力开放平台的权限模型 |
| **部署** | 按需做独立部署 |

**分析**：
- 能力开放平台是基础平台，提供通用的能力管理框架
- 数据开放平台是能力开放平台在"数据"领域的特例应用
- 权限模型、审批流程等通用能力应由能力开放平台统一提供
- 数据开放平台可以复用能力开放平台的基础能力

### 与数据开放平台的协同（9.4 已确认）

**用户确认**（2026-04-13）：

| 事项 | 用户反馈 | 策略调整 |
|------|---------|---------|
| **边界和复用关系** | 数据开放平台依赖能力开放平台的能力（API/事件通道等），不要搞反了 | ✅ 能力开放平台是基础能力提供方 |
| **权限模型统一** | 能力开放平台构建统一的权限管理，数据开放平台等做复用 | ✅ 统一权限管理，各业务场景复用 |
| **迁移策略** | 能力开放平台先构建，数据开放平台后构建，先后依赖，不存在迁移 | ✅ 先后构建关系，非迁移关系 |

**关键洞察**：
1. **依赖关系**：数据开放平台是能力开放平台的"消费方"，而非"超集"
   - 数据开放平台借助能力开放平台提供的能力（API/事件通道等）
   - 能力开放平台是基础设置，数据开放平台是上层应用

2. **权限模型设计原则**：
   - 能力开放平台：构建统一的权限管理，适配不局限于 API、事件、数据等类型的权限管理
   - 数据开放平台：将数据对象关联到数据对象的权限之后，后续流程交给统一权限管理和审批管理
   - 各自设计：能力开放平台的权限设计无需体现数据开放，各业务场景使用统一权限设计时将各自场景的业务设计进去

3. **建设顺序**：
   ```
   阶段 1: 能力开放平台（基础能力）
   ├── 统一权限管理
   ├── 统一审批管理
   ├── API 管理
   └── 事件/回调管理
   
   阶段 2: 数据开放平台（上层应用）
   ├── 数据对象管理（业务层）
   ├── 数据注册/审批（复用能力开放平台）
   ├── 数据订阅/消费（复用能力开放平台）
   └── 数据治理（业务层）
   ```

**架构关系图**：
```
┌─────────────────────────────────────────────┐
│           数据开放平台（阶段 2）              │
│  - 数据对象管理、数据注册、数据订阅          │
│  - 数据治理                                  │
│  ↓ 复用/依赖                                 │
├─────────────────────────────────────────────┤
│         能力开放平台（阶段 1 - 基础）         │
│  - 统一权限管理 ←──────────────┐            │
│  - 统一审批管理 ←──┐           │            │
│  - API 管理        │           │            │
│  - 事件/回调管理   │           │            │
└────────────────────┼───────────┼────────────┘
                     │           │
        ┌────────────┘           │
        │                        │
┌───────┴────────┐    ┌─────────┴──────────┐
│ API 权限       │    │ 数据对象权限       │
│ (API 场景)     │    │ (数据开放场景)     │
└────────────────┘    └────────────────────┘
         ↓                      ↓
    统一权限管理 → 统一审批管理 → 审计日志
```

**分析**：
- 能力开放平台是底层基础设施，提供通用的能力管理框架
- 数据开放平台是上层业务应用，依赖能力开放平台的基础能力
- 权限模型统一设计，各业务场景（API、事件、数据等）复用统一权限管理，但各自设计业务层权限逻辑
- 建设顺序清晰：先能力开放平台，后数据开放平台，不存在迁移问题

### 需求挖掘本质洞察（基于第五轮对话）

**核心原则**：
用户指出"4.2 可粗略写，但是边界一定要清晰，API 管理、事件管理、回调管理、连接器管理，属于不同的能力请分开写"。
这表明需求挖掘不能停留在表面的“功能列表”，而要深入到**能力域的本质差异**。

**分析洞察**：
1.  **API 管理 (同步)**：核心是接口契约、版本控制、在线文档。处理的是“请求 - 响应”模型。
2.  **事件管理 (异步广播)**：核心是 Topic 定义、订阅关系、消息分发。处理的是“发布 - 订阅”模型。
3.  **回调管理 (Webhook)**：核心是回调地址配置、重试机制、签名验证。处理的是“状态变更通知”模型。
4.  **连接器管理 (集成流)**：核心是流编排、参数映射、调度执行。处理的是“系统集成”模型。

**平台底座复用**：
虽然这四种能力在业务层表现不同，但它们都依赖统一的基础设施：
*   **权限管理**（平台本身能力）：为所有能力类型提供统一的鉴权模型（Scope/RBAC）。
*   **审批管理**（平台本身能力）：提供统一的审批工作流引擎。
*   **应用身份集成**：复用现有应用管理系统（AKSK），作为所有能力的统一访问凭证。

这一洞察将作为后续 `@sdd-spec` 规范编写阶段的核心设计原则。
