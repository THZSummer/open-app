# ADR-007: 多版本模型下的引用稽核策略

## 状态
PROPOSED

## 背景

V2 多版本模型引入了一个关键问题：当连接器版本被连接流编排引用后，如何阻止该版本被标记失效/删除？如何高效查询"谁引用了这个版本"？

spec.md §1.6 已给出设计原则：
> 被引用方的校验通过查询引用方确定，不把引用关系固化到被引用方的状态或字段中。被引用方只关心自身配置，不关心"被谁用"。

需要具体实现这一原则的技术方案。

## 决策

### 决策 1：FlowVersion 部署引用——通过 `flow.deployed_version_id` 查询（1:1 关系）

连接流版本失效前校验：
```sql
-- 检查是否有 Flow 部署了此版本
SELECT COUNT(*) FROM openplatform_v2_cp_flow_t
WHERE deployed_version_id = #{versionId} AND lifecycle_status IN (2, 3); -- 运行中或已停止
```

关系特点：Flow ↔ deployed FlowVersion 是 1:1（一个 Flow 只能部署一个版本），O(1) 查询。

### 决策 2：ConnectorVersion 编排引用——通过 `connector_version_ref` 中间表管理（M:N 关系）

**新增中间表**：
```sql
CREATE TABLE openplatform_v2_cp_connector_version_ref_t (
    id BIGINT(20) NOT NULL,
    flow_version_id BIGINT(20) NOT NULL,       -- 连接流版本ID
    flow_id BIGINT(20) NOT NULL,               -- 连接流ID（冗余，免JOIN）
    connector_version_id BIGINT(20) NOT NULL,  -- 连接器版本ID
    connector_id BIGINT(20) NOT NULL,          -- 连接器ID（冗余，免JOIN）
    -- ...
);
```
> 💡 不存储 `node_id`——引用表不耦合编排配置的具体节点结构，仅记录"谁引用了谁"。`flow_id`/`connector_id` 冗余允许直接按实体查询引用关系，无需穿透版本表。

**写入时机**：连接流编排保存时，遍历所有 connector 节点，同步写入/更新引用关系。

**查询时机**：连接器版本标记失效前：
```sql
-- 检查是否有连接流版本引用此连接器版本
SELECT DISTINCT ref.flow_id, f.name_cn
FROM openplatform_v2_cp_connector_version_ref_t ref
JOIN openplatform_v2_cp_flow_t f ON ref.flow_id = f.id
JOIN openplatform_v2_cp_flow_version_t fv ON ref.flow_version_id = fv.id
WHERE ref.connector_version_id = #{connectorVersionId}
  AND fv.status IN (1, 2, 3, 4, 5); -- 草稿/待审批/已撤回/已驳回/已发布
```

**清理时机**：
- FlowVersion 删除时：删除该版本的所有引用关系
- FlowVersion 编排内容变更时：增量更新引用关系（删除不再引用的 + 新增新增的 + 保留不变的）

### 决策 3：不维护"被引用计数"字段

被引用方（Connector/ConnectorVersion/FlowVersion）**不维护** `ref_count` 字段。原因：
1. 与 spec §1.6 设计原则一致：被引用方不关心"被谁用"
2. 避免引用计数字段与实际引用关系不一致（分布式环境下的经典问题）
3. 引用关系查询代价可接受（均通过索引覆盖）

## 后果

### 正面影响

1. **数据一致性**：引用关系只有单一事实来源（中间表 / deployed_version_id），避免双写不一致
2. **删除安全**：失效/删除操作前实时查询引用关系，不会误删除被引用版本
3. **扩展性好**：新增引用类型时，只需新增中间表或查询条件，不影响现有表结构
4. **实现简单**：引用关系查询均为简单 SQL，无需额外缓存一致性维护

### 负面影响

1. **查询开销**：每次失效/删除操作需要额外查询引用关系（预期频率低，可接受）
2. **中间表维护**：编排保存时需要额外维护 `connector_version_ref` 表（批量操作，可接受）
3. **无反向导航**：从连接器版本无法直接看到所有引用方列表（需要额外查询，但 spec 未要求此功能）

### 新增索引

```sql
-- connector_version_ref 查询索引
CREATE INDEX idx_connector_version ON openplatform_v2_cp_connector_version_ref_t (connector_version_id);
CREATE INDEX idx_connector ON openplatform_v2_cp_connector_version_ref_t (connector_id);
CREATE INDEX idx_flow ON openplatform_v2_cp_connector_version_ref_t (flow_id);

-- flow.deployed_version_id 查询索引
CREATE INDEX idx_deployed_version ON openplatform_v2_cp_flow_t (deployed_version_id);
```

## 替代方案

| 方案 | 描述 | 拒绝理由 |
|------|------|---------|
| 引用计数 | 在被引用方维护 ref_count 字段 | 与 spec 设计原则冲突，存在计数不一致风险 |
| 直接查 FlowVersion JSON | 扫描所有 FlowVersion 的 orchestrationConfig JSON，用 LIKE/JSON_CONTAINS 查引用 | 无法利用索引，性能极差，1000 版本时不可行 |
| 外键约束 | MySQL 物理外键 + ON DELETE RESTRICT | 已有规范禁止物理外键（统一使用逻辑 FK），且 JSON 中的引用关系无法通过 FK 表达 |
