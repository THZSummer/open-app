# ADR-008: #54 调用连接流 — 透明穿透响应模式

## 状态

PROPOSED

## 背景

`#54 调用连接流`（`POST /api/v1/flows/{flowId}/invoke`）是 connector-api 运行时唯一的对外 HTTP 触发端点。外部系统通过此接口调用已部署的连接流，触发整个 DAG 执行。

V1 和 V2 早期设计中，`#54` 与其他管理面接口一样使用标准响应信封 `{ code, messageZh, messageEn, data, page }`，其中 `data` 内嵌 `{ executionId, status, resultData, durationMs }`。

此设计存在以下问题：

1. **响应体被平台包装污染**：外部调用方真正关心的是出口节点的出参数据（`resultData`），却被包在 `data.resultData` 两层嵌套中，增加了调用方的解析成本。
2. **入参/出参与编排配置脱节**：`#54` 的请求体和响应体结构在接口文档中硬编码，而实际上应由用户在连接流中通过触发器节点（`httpInputDef`）和出口节点（`httpOutputDef`）自由定义。平台应充当透明代理，而非强制约定结构。
3. **平台元数据与业务数据耦合**：`executionId`、`status`、`durationMs` 等平台级元数据与业务数据混在同一 JSON 信封中，调用方解析业务数据时需"跳过"平台字段。

## 决策

**#54 接口采用「透明穿透」（Transparent Passthrough）模式**：

1. **请求侧完全用户自定义**：调用方传入的 Header（除 `X-Sys-Token` 认证头）、Query 参数、Body 结构完全遵循连接流中触发器节点的 `nodeInput`（httpInputDef）定义——header/query/body 三段式，每段由用户在设计连接流时通过 Schema 编辑器配置。

2. **响应侧完全用户自定义**：调用方收到的响应 Header（用户自定义部分）和 Body 结构完全遵循连接流中出口节点的 `output`（httpOutputDef）定义——header/body 两段式，每段由用户在设计连接流时通过 Schema 编辑器配置。

3. **平台元数据统一放入 `X-` 响应头**：所有平台级参数——执行 ID（`X-Execution-Id`）、执行状态（`X-Status`）、耗时（`X-Duration-Ms`）、缓存状态（`X-Cache-Status`）、结果码（`X-Code`）、中英文消息（`X-Message-Zh`/`X-Message-En`）——均追加到 HTTP 响应头中，与用户自定义的响应 Header 和 Body 完全分离。

4. **不使用标准响应信封**：`#54` 不再使用 `{ code, messageZh, messageEn, data, page }` 格式。响应 Body 即为出口节点的出参数据本身，无任何平台包装。

5. **前置校验失败时的处理**：SYSTOKEN 不在白名单（401）、入站限流（429）、连接流未部署（503）等前置校验失败时，连接流未实际执行，仅返回 `X-Code`、`X-Message-Zh`、`X-Message-En` 三个响应头，Body 为空，不出现 `X-Execution-Id` 等执行期元数据头。

## 后果

### 正面影响

- **调用方解析零成本**：外部系统收到的响应 Body 直接就是出口节点的出参数据，无需做 `data.resultData.field` 这样的嵌套解包。
- **接口定义由用户掌控**：`#54` 的请求/响应结构不再由平台硬编码，而是由连接流设计者在编排画布中通过 Schema 编辑器自由定义。平台仅负责"按图施工"。
- **平台与业务彻底解耦**：平台元数据（执行追踪、监控告警）在响应头中，业务数据在响应体中，两者互不干扰。调用方可选择性读取平台元数据头。
- **HTTP 语义更自然**：标准 HTTP 语义中，Headers 承载元数据，Body 承载资源数据。此设计完全符合 HTTP 规范。
- **与 #53 调试接口一致**：`#53` 调试执行原本就由 open-server 代理、由前端处理响应格式，此次改造使两个运行时接口的设计哲学统一。

### 负面影响

- **与 open-server 管理面接口不一致**：open-server 全部 54 个管理面接口继续使用标准响应信封，只有 `#54` 采用透明穿透模式。开发者在实现时需要区分两套响应处理逻辑。
- **错误处理模式不同**：管理面接口通过 `code` 字段传递错误，`#54` 通过 HTTP Status Code + `X-Code` 头传递错误。调用方需要适配两套错误处理。
- **API 文档复杂度增加**：`#54` 的接口文档需要同时描述"平台固定部分"（URL、X-Sys-Token 认证）和"用户可变部分"（请求/响应 Schema 来源），文档结构与其他接口截然不同。
- **#54 的请求体不再能通过 Swagger/OpenAPI 静态生成固定 Schema**：因为请求/响应结构随连接流配置动态变化，只能标注为"由触发器节点 httpInputDef 定义"。

### 缓解措施

- connector-api 内部为 `#54` 编写独立的 `FlowInvokeController`，不依赖 open-server 的通用 `Result<T>` 响应包装器。
- API 文档中明确标注 `#54` 为「透明穿透模式」例外。
- 后续可考虑为 `#54` 提供 Schema 探测端点（如 `OPTIONS /api/v1/flows/{flowId}/invoke`），返回当前已部署版本的触发器入参 Schema 和出口出参 Schema。

### 相关接口

| 接口 | 影响 |
|------|------|
| #54 调用连接流 | **本次改造**：采用透明穿透模式 |
| #53 调试执行 | 无直接变更，但设计哲学一致（connector-api 内部 `#53` 也不使用标准信封） |
| #51 调试代理 | 无变更，open-server 代理层仍使用标准信封，由 open-server 自行处理 connector-api #53 的原始响应 |
