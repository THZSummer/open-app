# ADR-001: Admin 能力 CRUD 放在 market-server 扩展（替代原 open-server 方案）

## 状态
ACCEPTED（原 open-server 方案已废弃）

## 背景
- spec 指定"服务端：market-server"
- market-server 已有 `AbilityEntity` + `AbilityMapper`，直连 `openplatform_ability_t`
- market-server 已有 Flyway 基础设施（`V2__create_lookup_file_table.sql`）；但 `openplatform_ability_t` 表归属 open-server，V4 迁移脚本由表归属服务管理
- 原方案（open-server 扩展）与 spec 矛盾，已废弃

## 决策
在 market-server 新建独立 `ability` 模块，将 `approval` 模块中现有的 `AbilityEntity` / `AbilityMapper` 迁移至独立模块，新建 `AdminAbilityController` + `AdminAbilityService`。market-web 作为前端调用 market-server 的 admin 接口前缀 `/service/open/v2/ability/admin/`。

## 后果
### 正面
- 与 spec 一致，职责分离清晰，管理面与开放面代码隔离
- market-server 已有 AbilityEntity + AbilityMapper，直连同库，无需跨服务调用
- V4 迁移在表归属服务（open-server）管理，DB 一致性有保障

### 负面
- 需要在 market-server 侧新建 `AbilityPropertyMapper`（图标/示意图管理）
- 需要确认 market-server 的文件上传能力
- V4 迁移与 CRUD 代码分属两个服务，需协调部署顺序
