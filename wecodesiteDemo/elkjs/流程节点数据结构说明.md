# 流程结构节点数据结构说明

本文档基于当前实现整理，详细说明添加循环、错误处理、并行和条件分支节点时，会创建哪些节点（nodes）和连线（edges）数据。核心实现代码位于 `src/types/flow.ts` 和 `src/store/flowStore.ts`。

---

## 1. 基础数据结构

### 1.1 FlowNode 节点结构

每个节点本质上是 ReactFlow 的 `Node<FlowNodeData>`，核心结构如下：

```ts
{
  id: string;
  type: NodeType;
  position: {
    x: number;
    y: number;
  };
  data: {
    label: string;
    type: NodeType;
    description?: string;
    config?: Record<string, any>;
  };
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `string` | 节点唯一 ID，当前由 `nanoid(10)` 生成 |
| `type` | `NodeType` | ReactFlow 节点类型，决定使用哪个节点组件渲染 |
| `position` | `{x, y}` | 节点初始位置，插入后会被 ELK 自动布局重新计算 |
| `data.label` | `string` | 节点展示名称 |
| `data.type` | `NodeType` | 业务节点类型，通常和外层 `type` 保持一致 |
| `data.description` | `string?` | 基础节点下方说明文案，文本节点通常没有 |
| `data.config` | `Record<string, any>?` | 结构节点关系配置，用于布局、删除、分支增删、嵌套归属判断 |

---

### 1.2 FlowEdge 连线结构

```ts
{
  id: string;
  source: string;
  target: string;
  sourceHandle?: string;
  targetHandle?: string;
  type: 'insert-edge';
  animated: false;
  style: {
    stroke: '#6366f1';
    strokeWidth: 2;
  };
  data?: {
    hideInsertButton?: boolean;
    sourceEdgeMeta?: {
      total: number;
      index: number;
    };
    targetEdgeMeta?: {
      total: number;
      index: number;
    };
  };
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `string` | 连线唯一 ID |
| `source` | `string` | 起点节点 ID |
| `target` | `string` | 终点节点 ID |
| `sourceHandle` | `string?` | 起点连接点，当前结构节点通常使用 `'top'` 或继承旧连线 |
| `targetHandle` | `string?` | 终点连接点，当前结构节点通常使用 `'bottom'` 或继承旧连线 |
| `type` | `'insert-edge'` | 固定为 `'insert-edge'`，使用自定义插入连线组件 |
| `animated` | `boolean` | 当前固定为 `false` |
| `style` | `object` | 连线样式，固定颜色和粗细 |
| `data.hideInsertButton` | `boolean?` | 是否隐藏连线中间的加号按钮。`true` 表示结构辅助线，不允许用户插入节点 |
| `data.sourceEdgeMeta` | `object?` | 渲染时由 FlowCanvas 动态补充，用于多出边分叉绘制 |
| `data.targetEdgeMeta` | `object?` | 渲染时由 FlowCanvas 动态补充，用于多入边合并绘制 |

> **注意**：`sourceEdgeMeta` 和 `targetEdgeMeta` 不是插入结构时创建的原始数据，而是在画布渲染前由 `FlowCanvas.tsx` 动态补充，用于边的横折线路径计算。

---

## 2. 四类结构节点总览

| 结构类型 | 插入方法 | 主节点 type | 默认新增 nodes 数 | 默认新增 edges 数 | 分组字段 | 角色字段 |
|---|---|---|---:|---:|---|---|
| 循环节点 | `insertLoopV2` | `loop-v2` | 5 | 7 | `loopV2GroupId` | `loopV2Role` |
| 错误处理节点 | `insertLoopV2` | `error-handler` | 5 | 7 | `loopV2GroupId` | `loopV2Role` |
| 并行处理节点 | `insertParallel` | `parallel` | 6 | 8 | `parallelGroupId` | `parallelRole` |
| 条件分支节点 | `insertParallel` | `condition-branch` | 6 | 8 | `parallelGroupId` | `parallelRole` |

## 3. 添加循环节点时的数据结构

调用入口：

```ts
insertLoopV2({
  edgeId,      // 被拆分的旧连线 ID
  position,    // 循环主节点初始位置
  nodeType: NodeType.LOOP_V2
})
```

### 3.1 插入前的原始连线

假设原来有一条连线：

```ts
{
  id: 'edge-trigger-end',
  source: 'trigger-1',
  target: 'end-1',
  sourceHandle: 'top',
  targetHandle: 'bottom',
  type: 'insert-edge',
  animated: false,
  style: { stroke: '#6366f1', strokeWidth: 2 }
}
```

添加循环节点时，会：

1. 删除旧连线 `edge-trigger-end`。
2. 新增 **5 个节点**。
3. 新增 **7 条连线**。

---

### 3.2 新增的 nodes（5 个）

#### 3.2.1 循环主节点

```ts
{
  id: loopV2Id,           // nanoid(10) 生成
  type: NodeType.LOOP_V2,
  position,
  data: {
    label: '循环节点',
    type: NodeType.LOOP_V2,
    description: '带文本说明的循环结构',
    config: {
      loopV2GroupId: loopV2Id,         // 自身 ID 作为分组 ID
      parentLoopV2GroupId: undefined,   // 不在父循环右侧链路时为 undefined
      parentLoopV2Role: undefined,      // 不在父循环右侧链路时为 undefined
      ...parallelBranchConfig           // 如果插入在并行/条件分支内部，会携带父分支归属配置
    }
  }
}
```

`config` 字段说明：

| 字段 | 说明 |
|---|---|
| `loopV2GroupId` | 当前循环结构的分组 ID，等于循环主节点 ID |
| `parentLoopV2GroupId` | 如果循环节点插入在某个父级循环右侧链路中，这里记录父循环 ID |
| `parentLoopV2Role` | 如果属于父级循环右侧链路，值为 `'right-column-node'` |
| `...parallelBranchConfig` | 如果循环节点插入在并行/条件分支内部，会记录父分支归属 |

其中 `parallelBranchConfig` 可能包含：

```ts
{
  parentParallelGroupId: string;
  parentParallelBranchId: string;
  parentParallelBranchIndex: number;
  parentParallelRole: 'branch-node';
}
```

这表示当前循环结构整体属于某个并行或条件分支中的内容节点。

---

#### 3.2.2 循环区域文本节点

```ts
{
  id: loopRegionTextId,
  type: NodeType.TEXT,
  position: { x: loopRegionTextX, y: position.y + 140 },
  data: {
    label: '循环区域',
    type: NodeType.TEXT,
    config: {
      loopV2GroupId: loopV2Id,
      loopV2Role: 'region'   // 标识为循环区域说明节点

```

说明：

| 字段 | 说明 |
|---|---|
| `loopV2GroupId` | 归属的循环主节点 ID |
| `loopV2Role` | `'region'` 表示循环区域说明节点 |
| `label` | 固定为 `'循环区域'` |

这个节点在视觉上位于循环主节点左下方。

---

#### 3.2.3 循环开始文本节点


说明：

| 字段 | 说明 |
|---|---|
| `loopV2Role` | `'start'` 表示循环右侧链路入口 |
| `label` | 固定为 `'循环开始'` |

用户可以在“循环开始”到“循环结束”之间继续插入动作节点、循环节点、错误处理节点、并行节点、条件分支节点等。

#### 3.2.4 循环结束文本节点


#### 3.2.5 循环跳出文本节点

```ts
{

  type: NodeType.TEXT,

    label: '循环跳出',
    type: NodeType.TEXT,
```

| 字段 | 说明 |
|---|---|
| `loopV2Role` | `'break'` 表示循环结构的最终出口 |

循环跳出节点用于汇合两条路径：
1. 循环区域路径（`循环区域 -> 循环跳出`）。
2. 循环开始到循环结束这条右侧路径（`循环开始 -> ... -> 循环结束 -> 循环跳出`）。

## 4. 添加错误处理节点时的数据结构

错误处理节点完全复用循环节点结构，调用同一个方法。

调用入口：

```ts
insertLoopV2({
  edgeId,
  position,
  nodeType: NodeType.ERROR_HANDLER
})
```

### 4.1 与循环节点的相同点

- 新增 5 个 nodes。
- 新增 7 条 edges。
- 删除被拆分的旧连线。
- 使用 `loopV2GroupId` 作为结构分组 ID。
- 使用 `loopV2Role` 标识内部文本节点角色。
- 可以嵌套在循环右侧链路中（通过 `parentLoopV2GroupId`）。
- 可以嵌套在并行/条件分支内部（通过 `parallelBranchConfig`）。

### 4.2 与循环节点的差异

主要差异是节点类型和文案：

| 节点 | 循环节点 label | 错误处理节点 label |
|---|---|---|
| 主节点 | `'循环节点'` | `'错误处理节点'` |
| 主节点 description | `'带文本说明的循环结构'` | `'带文本说明的错误处理结构'` |
| 区域文本节点 | `'循环区域'` | `'错误处理区域'` |
| 开始文本节点 | `'循环开始'` | `'错误处理开始'` |
| 结束文本节点 | `'循环结束'` | `'错误处理结束'` |
| 跳出文本节点 | `'循环跳出'` | `'错误处理跳出'` |

### 4.3 错误处理主节点 config

```ts
{
  id: loopV2Id,
  type: NodeType.ERROR_HANDLER,
  position,
  data: {
    label: '错误处理节点',
    type: NodeType.ERROR_HANDLER,
    description: '带文本说明的错误处理结构',
    config: {
      loopV2GroupId: loopV2Id,
      parentLoopV2GroupId: rightColumnGroupId,
      parentLoopV2Role: rightColumnGroupId ? 'right-column-node' : undefined,
      ...parallelBranchConfig
    }
  }
}
```

## 5. 添加并行处理节点时的数据结构

调用入口：

```ts
insertParallel({
  edgeId,
  position,
  nodeType: NodeType.PARALLEL
})
```

### 5.1 插入前的原始连线

同循环节点假设有一条 `edge-trigger-end`。

添加并行节点时，会：

1. 删除旧连线。
2. 新增 **6 个节点**。
3. 新增 **8 条连线**。

---

### 5.2 新增的 nodes（6 个）

#### 5.2.1 并行处理主节点

```ts
{
  id: parallelId,
  type: NodeType.PARALLEL,
  position,
  data: {
    label: '并行处理节点',
    type: NodeType.PARALLEL,
    description: '多分支并行处理结构',
    config: {
      ...parallelBranchConfig,       // 如果插入在父并行/条件分支内部会携带
      parallelGroupId: parallelId,    // 自身 ID 作为分组 ID
      parallelRole: 'root'            // 标识为并行结构主节点
    }
  }
}
```

`config` 字段说明：

| 字段 | 说明 |
|---|---|
| `parallelGroupId` | 当前并行结构分组 ID，等于并行主节点 ID |
| `parallelRole` | `'root'` 表示并行结构主节点 |
| `...parallelBranchConfig` | 如果当前并行节点插入在另一个并行/条件分支内部，则记录父分支关系 |

---

#### 5.2.2 分支开始文本节点（2 个）

由 `createParallelBranchNodes` 函数创建（`flowStore.ts` 第 185-242 行）。

分支1开始：

```ts
{
  id: branchOneStartId,
  type: NodeType.TEXT,
  position,
  data: {
    label: '分支1开始',
    type: NodeType.TEXT,
    config: {
      parallelGroupId: parallelId,
      parallelRole: 'branch-start',     // 标识为分支开始节点
      parallelBranchId: branchOneId,     // 当前分支 ID
      parallelBranchIndex: 1            // 分支序号，用于显示和布局排序
    }
  }
}
```

分支2开始：

```ts
{
  id: branchTwoStartId,
  type: NodeType.TEXT,
  position,
  data: {
    label: '分支2开始',
    type: NodeType.TEXT,
    config: {
      parallelGroupId: parallelId,
      parallelRole: 'branch-start',
      parallelBranchId: branchTwoId,
      parallelBranchIndex: 2
    }
  }
}
```

**关键说明**：分支开始文本节点（`parallelRole: 'branch-start'`）上会显示删除按钮，删除逻辑在 `TextNode.tsx` 第 11-32 行。删除按钮只在该节点上展示，因为 `shouldShowDeleteButton` 判断条件为 `parallelRole === 'branch-start' && parallelGroupId && parallelBranchId`。

## 6. 添加条件分支节点时的数据结构

条件分支节点复用并行处理节点的插入逻辑，调用同一方法。

调用入口：

```ts
insertParallel({
  edgeId,
  position,
  nodeType: NodeType.CONDITION_BRANCH
})
```

### 6.1 与并行处理节点的相同点

- 新增 6 个 nodes。
- 新增 8 条 edges。
- 使用 `parallelGroupId` 表示结构分组。
- 使用 `parallelRole` 表示结构内角色（`root`、`branch-start`、`branch-end`、`merge`）。
- 使用 `parallelBranchId` 表示分支 ID。
- 使用 `parallelBranchIndex` 表示分支序号。
- 点击主节点可以继续添加分支。
- 分支开始节点可以删除分支。
- 可以嵌套在循环右侧链路中。
- 可以嵌套在并行/条件分支内部（通过 `parallelBranchConfig`）。

### 6.2 与并行处理节点的差异

条件分支主要差异是节点类型和文案：

| 节点 | 并行处理节点 label | 条件分支节点 label |
|---|---|---|
| 主节点 | `'并行处理节点'` | `'条件分支节点'` |
| 主节点 description | `'多分支并行处理结构'` | `'多条件分支处理结构'` |
| 分支1开始 | `'分支1开始'` | `'条件1开始'` |
| 分支1结束 | `'分支1结束'` | `'条件1结束'` |
| 分支2开始 | `'分支2开始'` | `'条件2开始'` |
| 分支2结束 | `'分支2结束'` | `'条件2结束'` |
| 合并节点 | `'并行合并'` | `'条件分支合并'` |

新增分支时，分支文案前缀从 `'分支'` 变为 `'条件'`（由 `branchLabelPrefix` 字段控制）。

### 6.3 条件分支节点 config

条件分支节点的 config 结构与并行处理节点完全一致：

```ts
{
  id: parallelId,
  type: NodeType.CONDITION_BRANCH,
  position,
  data: {
    label: '条件分支节点',
    type: NodeType.CONDITION_BRANCH,
    description: '多条件分支处理结构',
    config: {
      ...parallelBranchConfig,
      parallelGroupId: parallelId,
      parallelRole: 'root'
    }
  }
}
```

## 7. 嵌套结构时的额外 config

### 7.1 插入在循环右侧链路中

如果某个节点插入在循环结构的右侧链路中（通过 `getLoopV2RightColumnGroupId` 判断），会携带：

```ts
{
  parentLoopV2GroupId: string;
  parentLoopV2Role: 'right-column-node';
}
```

适用场景：

- 动作节点插入在 `循环开始 -> 循环结束` 链路中。
- 循环节点嵌套在另一个循环的右侧链路中。
- 并行节点嵌套在循环的右侧链路中。
- 条件分支节点嵌套在循环的右侧链路中。

作用：布局时会根据这些字段把节点排列在循环开始和循环结束之间。

### 7.2 插入在并行或条件分支内部

如果某个节点插入在并行/条件分支的某条分支内部（通过 `getParallelBranchConfig` 判断），会携带：

```ts
{
  parentParallelGroupId: string;
  parentParallelBranchId: string;
  parentParallelBranchIndex: number;
  parentParallelRole: 'branch-node';
}
```

适用场景：

- 动作节点插入在 `分支开始 -> 分支结束` 链路中。
- 循环节点嵌套在并行分支中。
- 条件分支节点嵌套在并行分支中。
- 并行节点嵌套在条件分支中。

作用：布局时会根据这些字段把节点放在对应分支的开始和结束节点之间。

## 10. 可插入节点的连线规则汇总

### 循环结构

| 连线 | `hideInsertButton` | 是否可插入节点 | 说明 |

| 循环主节点 -> 循环区域 | 是 | 否 | 结构辅助线 |
| 循环区域 -> 循环跳出 | 是 | 否 | 结构辅助线 |
| 循环主节点 -> 循环开始 | 是 | 否 | 结构辅助线 |
| 循环开始 -> 循环结束 | 否 | **是** | 循环内部主要可编辑链路 |
| 循环结束 -> 循环跳出 | 是 | 否 | 结构辅助线 |
| 循环跳出 -> 下一个节点 | 否 | 是 | 循环结构出口 |

### 并行 / 条件分支结构

| 连线 | `hideInsertButton` | 是否可插入节点 | 说明 |
|---|---|:---:|---|
| 上一个节点 -> 主节点 | 否 | 是 | 可以继续在结构前插入节点 |
| 主节点 -> 分支开始 | 是 | 否 | 结构辅助线 |
| 分支开始 -> 分支结束 | 否 | **是** | 分支内部主要可编辑链路 |
| 分支结束 -> 合并节点 | 是 | 否 | 汇合辅助线 |
| 合并节点 -> 下一个节点 | 否 | 是 | 结构出口 |
